## Producer→consumer landing order, per unified-lock graph

Producers land first, topologically (each before any producer that depends on
it): **`nota` → `schema` → `schema-rust`** (`schema` depends on `nota`;
`schema-rust` depends on `nota` + `schema`). Then the swept consumers land,
and each unified-lock graph regenerates its locks and build-verifies before the
next graph's consumers proceed:

- **nota graph — 81 consumers.** Every consumer that pins `nota` (via
  `nota-next.git`). Essentially the whole engine: `agent`, `cloud`, `lojix`,
  `mind`, `router`, `spirit`, all `signal-*` and `meta-signal-*`, `terminal`,
  `upgrade`, `skills`, … (full list in the plan).
- **schema graph — 10 consumers.** `meta-signal-orchestrate`,
  `meta-signal-upgrade`, `mind`, `orchestrate`, `persona`, `signal-orchestrate`,
  `signal-spirit`, `signal-upgrade`, `spirit`, `upgrade`.
- **schema-rust graph — 52 consumers.** `agent`, `cloud`, `lojix`, `mind`,
  `router`, `spirit`, most `signal-*`/`meta-signal-*`, `terminal`, `skills`, …
  (full list in the plan).

A consumer appears in several graphs (it pins several producers); its own edits
are independent text rewrites, so within a graph the consumer order is free —
the ordering that matters is producers-before-consumers and per-graph
lock-regen + verify.

Landing is the **gated** phase: commit producers-first to the `drop-next`
staging branch, never `main`; the GitHub renames, redirect checks, per-graph
lock regen, and verify are done there, not by this dry run.

## Flagged pins — 4, all in `mind/Cargo.toml` (need a manual landing call)

`mind` deliberately pins the families at **pre-rename revisions**. At those
commits the crate still publishes the old name, so a token rewrite would make
the dependency unresolvable (`no matching package named 'schema-rust'`). The
tool leaves all four byte-for-byte unchanged and flags them:

| line | entry | pinned rev |
|---|---|---|
| 59 | `schema-rust-next = { git = "ssh://…/schema-rust-next.git", rev = … }` (build-dep) | `bb4dfe29…` |
| 70 | `[patch."…/nota-next.git"]` → `nota-next = { …, rev = … }` | `7105c2be…` |
| 73 | `[patch."…/schema-rust-next.git"]` → `schema-rust-next = { …, rev = … }` | `bb4dfe29…` |
| 76 | `[patch."…/schema-next.git"]` → `schema-next = { …, rev = … }` | `b3be7d0f…` |

**Recommended resolution (per pin, a landing decision):**

- These are *deliberate* old-revision pins — the maintainer of `mind` chose
  those commits. The rename must not silently move them.
- Left unchanged, each still resolves after the GitHub rename: the old
  `…-next.git` URL redirects, and at the pinned commit the crate carries the old
  name, which the keyless entry (line 59) and the patch bodies expect. So the
  **safe default is to leave them and verify `mind` builds after the rename.**
- For the three `[patch]` blocks, confirm the patch still *applies*: the block
  patches a dependency identified by its `…-next.git` URL, but `mind`'s own
  `nota`/`schema`/`schema-rust` deps now point at the renamed URLs, so the patch
  target may no longer match. Per block, either (a) advance the pin to a
  post-rename revision and rewrite the key/URL to the new name, or (b) drop the
  block if the deliberate pin is obsolete. Recommend (a) if the pin is still
  wanted; verify with a build.

## Residue — 219 hits across 51 repos (surfaced, NOT rewritten)

The tool surfaces every old token no classified surface rewrote, so the plan is
honest over the whole corpus. By file kind:

| kind | hits | what it is | landing action |
|---|---|---|---|
| `.md` | 101 | `ARCHITECTURE.md` / `README.md` / `skills.md` prose naming the old repos | update doc prose if desired; cosmetic |
| `.rs` | 97 | **hyphenated** repo names as *string data* / comments — `dependency_boundary.rs` guards, `@generated by schema-rust-next` headers, URL constants | see below |
| `.schema` | 11 | `schema/lib.schema` data files naming the families | review per file |
| spirit `scripts/*` | 6 | `run-nix-integration-tests`, `check-local-schema-stack` pass `nota-next`/… as flake-override args | rename the repo-name args (and any coupled `*_REF`) |
| `.nota` | 3 | example NOTA data | review |
| `.py` | 1 | a helper script arg | review |

The `.rs` residue is the important class and splits into two:

- **`dependency_boundary.rs` guards** (in `meta-signal-lojix`,
  `meta-signal-upgrade`, `signal-lojix`, `signal-terminal`, `signal-upgrade`,
  `upgrade`, and `spirit/tests/nix_integration.rs`): these assert the crate has
  migrated *off* `"nota-next"` / `"schema-rust-next"`. Their string literals
  **must stay** — after the rename the dep tree genuinely won't contain
  `nota-next`, so the guard keeps passing. **Landing action: leave unchanged.**
  (An earlier tool version would have falsely rewritten these; fixed — see
  below.)
- **`@generated by schema-rust-next` headers** in generated `src/schema/*.rs`:
  the generated *code* (`pub use nota_next::…`) is rewritten to `pub use
  nota::…` (an edit, not residue); only the header comment is residue. **Landing
  action: regenerate the generated files** (F5) — the canonical fix — which also
  refreshes the header. Regenerate-and-diff to confirm no other drift.

## Correctness fix this dry-run surfaced (already landed in the tool)

The first real-engine run exposed a false-rewrite class: the Rust surface was
rewriting the *hyphenated* family name in `.rs` files, but a hyphen cannot be in
a Rust identifier — so such a token is always string data or a comment. That
would have flipped the `dependency_boundary.rs` guards into asserting the *real*
dependency is forbidden, breaking those tests on apply. Fixed: **RustSource now
rewrites only the underscore path-ident** (real imports/re-exports); hyphenated
`.rs` tokens are surfaced as residue. Effect on the plan: edits 2497→2400 (false
`.rs` rewrites gone), residue 122→219. Regression test:
`tests/surfaces.rs::rust_source_leaves_repository_name_string_data_alone`.

## Caveats / follow-ups

- The tool reads each repo's **working copy** (not its git tip), so the plan
  reflects current on-disk state; a working copy mid-edit by another agent would
  be reflected. For an authoritative landing plan, re-run against clean tips.
- `DryRunPlan.nota` is faithful, human-readable NOTA, but it does **not**
  round-trip through the strict NOTA decoder: the `nota` codec rejects
  re-reading a slash-bearing content token it itself encoded (a canonicalization
  quirk). The summarizer therefore re-derives the plan in memory rather than
  decoding the file. Follow-up: fix the `nota` codec canonicalization (or encode
  residue content in a decoder-accepted form) so plans round-trip.
- `mind`'s deliberate pins and the spirit script args are the only edits that
  are not pure mechanical renames; everything else (2400 edits across 89 repos)
  is format-preserving identity substitution.

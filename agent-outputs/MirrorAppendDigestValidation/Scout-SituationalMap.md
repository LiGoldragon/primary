## 2. Mirror Append handling (engine + decision)

Flow (engine.rs:5-10 doc + dispatch): `Input::Append(suffix)` ->
`ReadInput::CheckAppend(suffix)` (engine.rs:239-241) -> `Store::check_append`
loads ledger (store.rs:365-377) -> `CheckedAppend::into_decision`
(decision.rs:25-101) -> on `AcceptSuffix`, `WriteInput::PersistSuffix`
(engine.rs:261-264) -> `Store::persist_suffix` (engine.rs:422-426).

### Chain LINKAGE validation â€” YES. `mirror/src/decision.rs`

`CheckedAppend::into_decision` (decision.rs:27-101) runs, in order:
`suffix_inconsistency`, `expected_head_violation`, `known_divergence`
(decision.rs:49-57). The linkage checks, verbatim:

Internal consecutiveness + previous-digest chaining
(decision.rs:117-130):
```rust
fn suffix_inconsistency(&self, entries: &[EntryEnvelope]) -> Option<AppendRejectionReason> {
    for window in entries.windows(2) {
        let previous = &window[0];
        let next = &window[1];
        if next.sequence.clone().into_u64() != previous.sequence.clone().into_u64() + 1 {
            return Some(AppendRejectionReason::SequenceGap);
        }
        if next.previous_digest() != Some(&previous.digest) {
            return Some(AppendRejectionReason::HeadForked);
        }
    }
    None
}
```

Expected-head linkage to stored rows (decision.rs:135-170, key arms):
```rust
None => {
    if first_sequence != 1 || first.previous_digest().is_some() {
        return Some(AppendRejectionReason::SequenceGap);
    }
}
Some(mark) => {
    let mark_sequence = mark.sequence.clone().into_u64();
    if mark_sequence + 1 != first_sequence {
        return Some(AppendRejectionReason::SequenceGap);
    }
    if first.previous_digest() != Some(&mark.digest) {
        return Some(AppendRejectionReason::DigestMismatch);
    }
    match self.known_digest(mark_sequence) {
        Some(known) if known.digest == mark.digest => {}
        Some(_) => return Some(AppendRejectionReason::HeadForked),
        None => return Some(AppendRejectionReason::SequenceGap),
    }
}
```

Re-send divergence vs stored digests (decision.rs:177-191):
```rust
fn known_divergence(&self, entries: &[EntryEnvelope]) -> Option<AppendRejectionReason> {
    let head_sequence = self.head_sequence();
    for entry in entries {
        let sequence = entry.sequence.clone().into_u64();
        match self.known_digest(sequence) {
            Some(known) if known.digest == entry.digest => {}
            Some(_) => return Some(AppendRejectionReason::DigestMismatch),
            None if sequence <= head_sequence => {
                return Some(AppendRejectionReason::SequenceGap);
            }
            None => {}
        }
    }
    None
}
```

Interpretation: every `DigestMismatch` here compares one CARRIED digest against
ANOTHER CARRIED/STORED digest (entry.previous_digest vs mark.digest; entry.digest
vs the stored row's digest). The digest is treated as an opaque 32-byte token
that must CHAIN consistently. Sequence monotonicity and previous->digest linkage
are fully validated.

### Recompute digest from payload bytes â€” NO (confirmed).

There is no blake3/sha/hashing call anywhere in `decision.rs`, `store.rs`, or
`engine.rs`. The carried `EntryEnvelope.digest` is never compared to any
function of `EntryEnvelope.payload`. `Store::check_append` only loads the head
plus the known rows overlapping the suffix's sequence range (store.rs:365-377,
334-360); `KnownEntry` carries only `{ sequence, digest }` (schema/sema.rs:214-217),
not the payload. The mirror is payload-blind by construction (mirror Cargo.toml
description line 8: "Payload-blind append-ingest mirror daemon").

Exact insertion point where a `hash(body) == entry.digest` check COULD be added:
inside `CheckedAppend::into_decision` (decision.rs:27-101), as a new per-entry
guard alongside the existing `ledger.suffix_inconsistency(...)` /
`ledger.known_divergence(...)` calls (decision.rs:49-57), returning
`AppendRejectionReason::DigestMismatch`. A natural shape is a new method on
`RegisteredLedger` or on `EntryEnvelope` (verb-on-type) iterating `entries`
before the novelty filter. The `DigestMismatch` reason variant already exists
(signal-mirror schema/lib.rs:221) and currently has spare meaning available for
body-vs-digest.

### Persist + advance head â€” `mirror/src/store.rs`

```rust
pub fn persist_suffix(&mut self, suffix: &NovelSuffix) -> Result<AppendReceipt> {   // store.rs:403
    if !suffix.entries().is_empty() {
        self.commit_entry_rows(suffix)?;
    }
    self.advance_head(suffix)
}
pub fn commit_entry_rows(&mut self, suffix: &NovelSuffix) -> Result<()> {            // store.rs:415
    let mut commit = CommitRequest::new(self.entries);
    for envelope in suffix.entries() {
        commit = commit.assert(ReceivedEntry::from_envelope(&suffix.store, envelope));
    }
    self.engine.commit(commit)?;
    Ok(())
}
pub fn advance_head(&mut self, suffix: &NovelSuffix) -> Result<AppendReceipt> {      // store.rs:426
    self.engine.mutate(Mutation::new(
        self.heads,
        StoredHead::new(
            suffix.store.as_str().to_owned(),
            Some(HeadStamp::from_mark(&suffix.head)),
        ),
    ))?;
    Ok(AppendReceipt { store: suffix.store.clone(), head: suffix.head.clone() })
}
```
(`NovelSuffix` is the engine's decided suffix; `Store::engine` is a
`sema_engine::Engine`, store.rs:266-272.)

## 4. Does the mirror have access to `sema-engine`'s digest recompute?

### Dependency: YES, the mirror depends on sema-engine.
`mirror/Cargo.toml:43`:
```toml
sema-engine = { git = "https://github.com/LiGoldragon/sema-engine.git", branch = "main" }
```
`signal-mirror/Cargo.toml` does NOT depend on sema-engine (contract crate is
runtime-excluded; deps are only `nota`, `rkyv`, `signal-frame`). Confirmed by
grep: no `sema-engine`/`sema_engine` in signal-mirror.

### What digest type does the mirror use for the head?
The wire/decision/storage path uses `signal_mirror::EntryDigest = FixedBytes<32>`
(a raw 32-byte blob): `HeadMark.digest`, `EntryEnvelope.digest`, `KnownEntry.digest`,
and the stored `DigestBytes(FixedBytes<32>)`. It does NOT use
`sema_engine::EntryDigest` in the decision path. The only place the two cross is
the shipper's acknowledge callback, `mirror_head_from_mark` (shipper.rs:262-267),
which builds a `sema_engine::EntryDigest::new(*mark.digest.payload().payload())`
purely to call `component_engine.acknowledge_mirror(...)`.

### `EntryDigest::from_entry_fields` accessibility â€” NOT directly callable.
`sema-engine/src/versioning.rs:246`:
```rust
pub struct EntryDigest([u8; 32]);
```
exported `pub use versioning::{EntryDigest, ...}` (sema-engine lib.rs:74); it has
`pub const fn new([u8;32])` (versioning.rs:249) and `pub const fn bytes() ->
&[u8;32]` (versioning.rs:253). BUT the hashing constructor is crate-private:
```rust
pub(crate) fn from_entry_fields(                          // versioning.rs:257
    store_name: &VersionedStoreName,
    schema_hash: StoreSchemaHash,
    commit_sequence: CommitSequence,
    snapshot: SnapshotIdentifier,
    previous_entry_digest: Option<EntryDigest>,
    operations: &NonEmpty<VersionedLogOperation>,
) -> Self {
    let mut hasher = blake3::Hasher::new();
    Self::update_bytes(&mut hasher, b"sema-engine-versioned-commit-log-entry-v2");
    ... store_name, schema_hash, commit_sequence, snapshot, previous_entry_digest,
        operation count + each operation.update_digest(...) ...
    Self(*hasher.finalize().as_bytes())
}
```
So the digest is a domain-separated **blake3 over structured fields**
(store_name, schema_hash, commit_sequence, snapshot, previous_entry_digest, and
the operation set) â€” NOT a hash of an opaque "body" blob. `from_entry_fields`
and `update_bytes` are `pub(crate)`; the mirror cannot call them.

### What IS the carried digest / body? (from the shipper)
`ComponentShipper::envelope_for_entry` (shipper.rs:150-166) builds each envelope:
- `digest` = `entry.entry_digest()` raw bytes (the sema-engine entry's own stored
  blake3 digest), wrapped as `signal_mirror::EntryDigest`.
- `payload` = `rkyv::to_bytes::<_>(entry)` â€” the rkyv serialization of the WHOLE
  `sema_engine::VersionedCommitLogEntry`.

So `body == rkyv(VersionedCommitLogEntry)` and `carried_digest ==
entry.entry_digest()`. A literal `blake3(body) == carried_digest` is FALSE: the
digest is not a hash of the rkyv bytes (the rkyv body even CONTAINS the digest
field, versioning.rs:403-410).

### The reachable recompute path (public): `VersionedCommitLogEntry::new`.
`VersionedCommitLogEntry` is `pub` and rkyv-(de)serializable, exported from
sema-engine (lib.rs:75), def `versioning.rs:403-411`. Its public constructor
recomputes the digest internally:
```rust
pub fn new(                                               // versioning.rs:414
    store_name: VersionedStoreName,
    schema_hash: StoreSchemaHash,
    commit_sequence: CommitSequence,
    snapshot: SnapshotIdentifier,
    previous_entry_digest: Option<EntryDigest>,
    operations: NonEmpty<VersionedLogOperation>,
) -> Self {
    let entry_digest = EntryDigest::from_entry_fields(&store_name, schema_hash,
        commit_sequence, snapshot, previous_entry_digest, &operations);
    Self { ..., entry_digest, ... }
}
```
plus public accessors `store_name()/schema_hash()/commit_sequence()/snapshot()/
previous_entry_digest()/entry_digest()/operations()` (versioning.rs:441-469).
There is NO public `verify()`/`recompute()` on the entry (the only `verify` is
`checkpoint.rs:501`).

## 5. Reading back a stored entry's full body (for a re-derive test)

### Wire read ops on `Input` â€” only ObserveHeads and Restore (no ObserveEntries).
`Input` (signal-mirror schema/lib.rs:507-513) read-shaped variants:
- `ObserveHeads(HeadQuery)` -> `Output::HeadsObserved(HeadListing)`; a
  `HeadListing` carries `StoreHead { store, head: Option<HeadMark> }` â€” i.e.
  HEAD DIGEST ONLY, no bodies (store.rs:543-562).
- `Restore(RestoreQuery)` -> `Output::Restored(RestoreBundle)`; `RestoreBundle`
  carries the latest `CheckpointArtifact` plus a `suffix: Vec<EntryEnvelope>` of
  every entry PAST the checkpoint's covered_end (store.rs:510-540) â€” these are
  FULL `EntryEnvelope`s incl. `payload`. But `Restore` REQUIRES a checkpoint:
  with none it returns `RestoreRejectionReason::NoCheckpoint` (store.rs:521-526),
  and `UnknownStore` if the head row is absent.

There is NO `ObserveEntries` / read-one-entry op on the wire. Full bodies at or
below the checkpoint's covered_end are not reachable over the wire at all.

### In-process read-back (for a test linking the mirror crate).
`Store::load_restore(&self, query) -> Result<Result<RestoreBundle, RestoreRejection>>`
(store.rs:510-540) returns full `EntryEnvelope`s (via `ReceivedEntry::to_envelope`)
for the post-checkpoint suffix. The lower-level `Store::entry_rows` (store.rs:303-308)
returns `Vec<ReceivedEntry>` for any key range and is PRIVATE; a test would reach
bodies either through `load_restore` (needs a checkpoint) or by adding/using a
public entry-read surface. `ReceivedEntry::to_envelope` (store.rs:184-191) is the
landed-body -> `EntryEnvelope` reconstruction a re-derive test would use.

## CONCLUSION: can the mirror self-validate `hash(body) == digest`?

Literal `blake3(payload_bytes) == carried_digest`: **NO** â€” the digest is not a
hash of the body. It is a domain-separated blake3 over the sema-engine commit-log
entry's structured fields (`from_entry_fields`, versioning.rs:257-285), and the
body is the rkyv serialization of the whole `VersionedCommitLogEntry`.

Structured self-validation (re-derive the digest FROM the landed body): **YES,
with what the mirror already depends on â€” no new crate dependency needed.** The
mirror already depends on `sema-engine` (Cargo.toml:43), and the body is exactly
`rkyv(VersionedCommitLogEntry)` (shipper.rs:157-164), a `pub` rkyv type. A test
or a new check can:

1. `rkyv::from_bytes::<VersionedCommitLogEntry>(envelope.payload.as_slice())`.
2. Reconstruct via the public `VersionedCommitLogEntry::new(store_name,
   schema_hash, commit_sequence, snapshot, previous_entry_digest, operations)`
   using the decoded entry's public accessors â€” `new` recomputes the digest
   through `from_entry_fields` internally, so the crate-private function is NOT
   needed.
3. Assert `reconstructed.entry_digest().bytes() == decoded.entry_digest().bytes()`
   (body self-consistency, the real "hash(body)==digest"), AND
   `decoded.entry_digest().bytes() == envelope.digest.as_bytes()`
   (carried-vs-body).

Requirements / caveats (interpretation):

- It does NOT need a new sema-engine dependency (present) and does NOT need the
  `pub(crate)` `from_entry_fields` (reachable via `VersionedCommitLogEntry::new`).
- It DOES require the body to be the real serialized `VersionedCommitLogEntry`.
  Today it is (shipper.rs). A hostile/garbage body would fail rkyv decode, which
  is itself a usable rejection signal.
- This makes the mirror **payload-AWARE**: it would have to understand
  sema-engine's commit-log entry format, contradicting the stated
  "payload-blind" design (mirror Cargo.toml:8; signal-mirror
  description/INTENT). That is an architectural tradeoff for the decision-maker,
  not a code blocker.
- Over the WIRE the mirror cannot get bodies for entries at/below a checkpoint's
  covered_end (only `Restore`'s post-checkpoint suffix, and only when a
  checkpoint exists). A re-derive TEST is easiest in-process against
  `Store`/`ReceivedEntry::to_envelope`, or via `load_restore` after publishing a
  checkpoint.

### 6.2 · Boxes-for-unsized-fields absent (intent 404)

Intent 404 (Maximum, 2026-05-24):
> "NOTA mirrors the binary self-describing root object plus
> relative-pointer box layout. For complex schemas, the root
> object stays compact (sized fields inline); unsized or growing
> fields (String, Vec, Option-of-unsized, nested records with
> their own growth) become BOXES that appear AFTER the root in
> declaration order. No box-index naming is required because the
> order of boxes matches the order of declaration in the schema.
> A coordinate notation supports access into the box-vector:
> (vector-N element-M) reads as the M-th element of the N-th box
> vector."

This **substantially refines /164's schema grammar**. /164's §3.3
shows variant payloads as direct type references (`(Record Entry)`
means Record carries an Entry struct). Per intent 404, when a
struct has unsized fields (Strings, Vecs), the NOTA projection
becomes: root object inline (sized fields), then boxes
(String/Vec contents) after the root, in declaration order.
Access via `(vector-N element-M)` coordinates.

**/164 §3.4 ("Where structs go") and §3.5 ("Primitives") need a
substantial rewrite** to reflect:
- Sized fields inline in the root
- Unsized fields as ordered post-root boxes
- `(vector-N element-M)` coordinate notation
- Why this matches the rkyv binary form (it does — rkyv lays
  unsized data after the sized root with relative pointers)

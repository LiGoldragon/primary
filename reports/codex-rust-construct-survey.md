# Codex candidate-node survey: the standardized Rust subset

Scope: evidence-based inventory of Rust constructs this workspace's schema
generator (`schema-rust`) emits, plus what hand-written daemon code does
beyond that, a noise exhibit, and exclusion candidates — input for designing
"codex," a positional NOTA-style IR for Rust-as-data.

Method: direct reading of `schema-rust` source and its committed emission
goldens, plus a fork's `rg`-counted survey of hand-written code in
`sema-engine`, `spirit`, and `signal-spirit`. One research fork (the emitter
inventory) died mid-run without returning findings; its portion below was
regathered directly by this scout via `Read`/`grep` on the same sources, so
section 1 is scout-verified rather than fork-summarized. All claims are
tagged OBSERVED (direct file evidence) or INTERPRETATION (my synthesis).

Repos read: `/home/li/primary/repos/schema-rust` (emitter + goldens),
`/home/li/primary/repos/signal-spirit` (wire-contract consumer),
`/home/li/primary/repos/spirit` (daemon consumer, generated actor/runtime
surface), `/home/li/primary/repos/sema-engine` (storage-engine consumer).

## 1. Emitted-construct inventory

OBSERVED: `schema-rust/ARCHITECTURE.md` states the renderer is fully
token-based (`proc_macro2::TokenStream` + `quote`, one `prettyplease` pass at
`emit_item_tokens`); "No emitter code builds Rust source with
`format!`/`self.line`", except the literal `// @generated` header comment.
Per-noun `ToTokens` wrapper types (e.g. `RustScalarAliasTokens`,
`NewtypeInherentImplTokens`, `EnumVariantConstructorsTokens`, `RouteEnumTokens`,
`SignalFrameImplTokens`) each own one section of emission.

OBSERVED: schema-rust emits into **two distinct target shapes**, both driven
by the same `GenerationDriver`/`ModuleEmission` API (`build.rs` in each
consumer, e.g. `signal-spirit/build.rs` uses
`ModuleEmission::wire_contract_module("signal")`):

- **Wire-contract modules** (e.g. `signal-spirit/src/schema/signal.rs`,
  6150 lines from a 265-line `.schema` source) — pure data shapes, no doc
  comments, no logic bodies beyond trivial constructors/accessors.
- **Daemon-runtime modules** (`spirit/src/schema/{daemon,nexus,sema}.rs`,
  876/2616/1781 lines, all headed `// @generated by schema-rust`) — richer:
  traits with associated types, generic bounds, async fn, doc comments, and
  kameo `Actor`/`Message` impls with real handler bodies. This means the
  "emitted vs hand-written" boundary is not "data vs logic" — schema-rust
  already emits a nontrivial slice of the actor/runtime scaffolding.

Construct kinds observed, with frequency signal:

- **Tuple newtype struct** (dominant — hundreds per module): wraps one
  scalar/typed field.
  ```rust
  pub struct ConfigurationPath(String);
  ```
- **Named-field struct** (dominant, second most common shape): plain
  positional record.
  ```rust
  pub struct DatabaseMarker {
      pub commit_sequence: CommitSequence,
      pub state_digest: StateDigest,
  }
  ```
- **Unit-only enum** (common — used for closed atoms like `Kind`, `Magnitude`,
  `AuthorizationMode`):
  ```rust
  pub enum ObserverFilter { All, OperationsOnly, EffectsOnly }
  ```
- **Enum with tuple-variant payloads** (dominant for "root"/operation enums —
  e.g. `Input`, `Output` in `signal-spirit`, 20+ variants each):
  ```rust
  pub enum Input {
      State(State),
      Record(Record),
      ...
      Version,
      Marker,
      PublicIntent(PublicIntent),
  }
  ```
- **Struct-shaped error enum with named fields per variant**, seen in
  `schema-rust/tests/fixtures/collections_generated.rs:446-475` — a fully
  emitted `SignalFrameError` with `ArchiveEncode`, `FrameTooShort { found:
  usize }`, etc.
- **Manual-shaped `impl std::fmt::Display` with a full `match` over enum
  variants, plus `impl std::error::Error for X {}`** — emitted, not
  hand-written (same file, lines 454-475). Confirms `match` expressions and
  `Display`/`Error` trait impls are inside the emitted surface, at least for
  frame/codec error types. `match` appears in 6 of 9 sampled goldens
  (16-36 occurrences each in the larger ones) — common, not dominant.
- **`impl From<Payload> for Variant`** (dominant pattern for enum payload
  wrapping) — counted 7 to 138 times per golden file (`spirit_nexus_generated.rs`
  alone has 138).
- **Per-variant constructor factory methods** on the root enum, one method
  per operation (e.g. `Output::marker_reported(payload: DatabaseMarker) ->
  Self`) — dominant pattern, one per variant, so scales linearly with
  operation count (25+ in `signal-spirit`'s `Output`).
- **Generic struct/enum with type parameters**, no lifetime — used for the
  runtime "plane" family: `pub enum Plane<SignalRoot, NexusRoot, SemaRoot>`,
  `pub struct Signal<Root>`, `pub struct Nexus<Root>`, `pub struct
  Sema<Root>`, `pub struct MessageProcessed<Reply>` — appears identically
  across every "runner"/"plane" fixture (5 of 9 goldens), so this is a fixed,
  reused emission pattern, not ad hoc.
- **Generic struct with lifetime + type parameter + `where` clause on a trait
  impl**, `schema-rust/tests/fixtures/runner_generated.rs:1843-1894`:
  ```rust
  impl<'engine, Engine> triad_runtime::RunnerEngines
  for NexusRunnerAdapter<'engine, Engine>
  where
      Engine: NexusEngine,
  {
      type Reply = Output;
      ...
      async fn apply_sema_write(&mut self, write: Self::SemaWrite) -> Self::Work {
          let output: SemaWriteOutput = NexusEngine::apply_sema_write(...).await;
          NexusWork::sema_write_completed(output)
      }
  }
  ```
  This is emitted code with lifetimes, generic trait bounds, associated
  types, and multi-statement `async fn` bodies calling into trait methods —
  present in every "runner" golden (3 async fns each).
- **Trait definitions with methods**, emitted (not hand-written) — seen
  repeatedly: `MessageSentHook`, `MessageProcessedHook<Reply>`,
  `UpgradeFrom<Previous>: Sized`, `AcceptPrevious<Previous>:
  UpgradeFrom<Previous>` recur verbatim across 6 of 9 goldens. In
  `spirit/src/schema/daemon.rs` (also `@generated`): `pub trait
  ComponentDaemon: Sized + 'static`, `pub trait DaemonBinder: ComponentDaemon`,
  `pub trait StagedAdvance<Daemon: ComponentDaemon>: Send` — trait bounds
  combining `Sized`, `'static`, `Send`, and supertraits are emitted.
- **kameo `Actor`/`Message<T>` impls with async handler bodies**, emitted —
  `spirit/src/schema/daemon.rs:533-599`:
  ```rust
  pub struct EngineActor<Daemon: ComponentDaemon> { engine: Daemon::Engine }
  impl<Daemon: ComponentDaemon> Actor for EngineActor<Daemon> {
      type Args = Self;
      type Error = Daemon::Error;
      async fn on_start(actor: Self::Args, _actor_reference: ActorRef<Self>)
          -> Result<Self, Self::Error> { Daemon::start(&actor.engine)?; Ok(actor) }
      async fn on_stop(&mut self, ...) -> Result<(), Self::Error> { Daemon::stop(&self.engine) }
  }
  impl<Daemon: ComponentDaemon> Message<WorkingInput> for EngineActor<Daemon> { ... }
  ```
  This file also carries emitted doc comments (`/// The kameo actor that owns
  the component engine...`), the only doc comments observed anywhere in the
  survey — so doc-comment emission exists but is confined to the
  daemon-runtime target, not the wire-contract target (0 `///` lines across
  all 9 `schema-rust` test-fixture goldens, which are wire-contract-shaped).
- **`#[rustfmt::skip]` on essentially every emitted item** — the single most
  consistent attribute pattern; appears on every struct, enum, impl, trait,
  and type alias sampled, in every golden.
- **`pub use crate::path::X as X` re-export** for cross-schema type imports —
  43 consecutive lines in `signal-spirit/src/schema/signal.rs:1-70` alone
  (one per `signal-domain` type), dominant pattern for dependency-schema
  reuse.
- **`pub type Alias = std::primitive::Type`** scalar aliasing — small,
  fixed set (`String`, `Integer`, `Boolean`, `Path`) at the top of every
  wire-contract module.

**Derives** (OBSERVED, `grep -oh 'derive([^)]*)'` over all 9
`schema-rust` goldens): exactly two derive sets dominate —
`derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug,
PartialEq, Eq)` (236 occurrences) always paired with a `#[cfg_attr(feature =
"nota-text", derive(nota::NotaDecode, nota::NotaDecodeTraced,
nota::NotaEncode))]` (199 occurrences) immediately above it. Two smaller
variants exist: `derive(Clone, Debug, PartialEq, Eq)` (9×, no rkyv — seen on
the hand-composed `SignalFrameError`) and `derive(PartialEq, Eq, PartialOrd,
Ord)` (3×, ordering-sensitive types). The derive set is **constant** for
ordinary schema types, not varying per field type.

**Dependencies/traits**: `rkyv::{Archive, Serialize, Deserialize}` (every
data type), `nota::{NotaDecode, NotaDecodeTraced, NotaEncode}` (feature-gated,
every data type), `std::fmt::Display` + `std::error::Error` (frame/codec
error enum only), `triad_runtime::{RunnerEngines, NexusAction, ...}` and
`kameo::{Actor, Message, ActorRef, WeakActorRef, Context}` (daemon-runtime
target only).

**Module layout**: OBSERVED — one generated `.rs` file per schema module,
named to match (`signal.schema` → `src/schema/signal.rs`), collected under a
hand-written `src/schema/mod.rs` with `pub mod x;` lines (not itself
`@generated`-tagged, per `head -1` check). This matches
`schema-rust/ARCHITECTURE.md`'s stated `RustModulePath` convention: crate
namespace segment dropped, nested modules become files under `src/schema/`.

## 2. Hand-written-construct inventory (beyond schema)

Source: fork survey of `sema-engine/src` (19 files, 8942 LOC, no
`schema/` subdir), `spirit/src` minus the three `@generated` files under
`schema/` (30 files, 15946 LOC), `signal-spirit/src` (5 files, 7880 LOC — but
`schema/{domain,signal}.rs` there are also fully generated, so effectively
only `help.rs`/`lib.rs`, both small, are truly hand-written). rg counts are
the fork's own evidence, not estimates.

**Pervasive** (appears in nearly every hand-written file):
- `match` expressions with real multi-arm logic: 85 (sema-engine), 197
  (spirit hand-written).
- `Result<T, E>` + `?`-operator chains: 143/329/46 `Result` occurrences and
  371/494/26 `?` occurrences across sema-engine/spirit/signal-spirit.
- `impl Trait for Type` blocks with method bodies (not just derives): 142
  impl blocks in sema-engine alone.
- Iterator/closure chains (`.map`/`.filter`/`.and_then`, closures): 104-141
  closure sites and 41-130 iterator-combinator sites per repo.
- thiserror-style error enums with per-variant `#[error("...")]` messages —
  example `sema-engine/src/error.rs:7-40`,
  `Error::StorageLayoutMismatch { stored: u64, expected: u64 }` with an
  interpolated multi-line message. 1 (sema-engine), 16 (spirit), 2
  (signal-spirit) such error enums.

**Common** (recurring, not universal):
- Hand-written `trait` definitions used as extension traits, e.g.
  `spirit/src/store/mod.rs` defines 6 `*Ext` traits (`GuardianEntryExt`,
  `GuardianQueryExt`, etc.) — 20 trait defs total in spirit's hand-written
  code, 6 in sema-engine.
- `#[cfg(feature = "...")]` conditional-compilation guards throughout
  `spirit`.
- Explicit lifetimes in hand-written signatures: 30 (sema-engine), 70
  (spirit), 10 (signal-spirit).
- `where`-bounded generic functions: 45 (sema-engine), 39 (spirit), 0
  (signal-spirit).

**Rare/isolated**:
- `async fn`/`.await` in genuinely hand-written (non-`@generated`) code —
  spirit's async surface is overwhelmingly the generated actor/daemon
  scaffold (`schema/daemon.rs`), not hand code; direct `tokio::` imports
  outside generated files: 18 in spirit, 0 elsewhere. Direct `kameo::` use
  outside generated files: 0 — all kameo wiring is emitted.
- `unsafe` — **absent everywhere** (0 hits in all three repos);
  `spirit/src/lib.rs:27` has `#![forbid(unsafe_code)]` crate-wide.
- `macro_rules!` — **entirely absent** in all three repos (0 hits).
  Proc-macro derive is the only macro mechanism observed anywhere in this
  survey.

**Beyond a data-shape generator, definitively** (candidates for a "logic
tier" outside codex's initial node set, or a distinct later tier):
multi-arm `match` with real branching logic (example:
`spirit/src/guardian.rs:295-320`,
`RepresentativeReply::representative_reply` matching `SubReply` variants and
constructing different rejection payloads per arm), hand-written extension
traits, thiserror error enums with custom interpolated message text,
`?`-chained fallible pipelines, closures/iterator combinators, and the
`#[cfg(feature = ...)]` conditional-compilation layer.

**Unknowns flagged by the fork**: `sema-engine`'s 45 `where`-clause hits
were not individually read (boilerplate vs meaningful generic API,
undetermined); `signal-spirit/src/help.rs` (the one plausibly hand-written
file besides `lib.rs`) was not read in detail; `spirit/src/bin/*.rs` (6
binaries) were included in the 15946-LOC hand-written count but not broken
out separately, so binary-specific construct mix is unknown.

## 3. Noise exhibit

Source: `signal-spirit/schema/signal.schema` (265 lines, human-authored) →
`signal-spirit/src/schema/signal.rs` (6150 lines, `@generated by
schema-rust`) — a ~23x line-count expansion for the whole file. Two
compact exhibits within it:

**Exhibit A — a single 1-line record declaration:**

Schema (`signal.schema`, one line among 265):
```
DatabaseMarker { CommitSequence StateDigest }
```

Generated Rust (`signal.rs:724-733`, 8 lines for this one type, not counting
the two other types it references):
```rust
#[rustfmt::skip]
#[cfg_attr(
    feature = "nota-text",
    derive(nota::NotaDecode, nota::NotaDecodeTraced, nota::NotaEncode)
)]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug, PartialEq, Eq)]
pub struct DatabaseMarker {
    pub commit_sequence: CommitSequence,
    pub state_digest: StateDigest,
}
```
Every one of `CommitSequence`, `StateDigest` is itself a one-line schema
declaration (`CommitSequence Integer`) expanded to its own 8-line tuple
newtype elsewhere in the file — so the "true" expansion for this one record
is 24 lines across 3 declarations, none of it visible in the 1-line schema.

**Exhibit B — the top-level operation set:**

Schema (`signal.schema`, the bracketed operation-name lists near the top,
~4 lines total defining the `Input`/`Output` operation vocabularies) maps to
a 26-variant `pub enum Input { State(State), Record(Record), ...,
Version, Marker, PublicIntent(PublicIntent) }` (signal.rs:2109-2135) and a
matching `Output` enum, **plus** one hand-symmetric constructor factory
method per `Output` variant (signal.rs, ~5000-5060, e.g. `pub fn
marker_reported(payload: DatabaseMarker) -> Self { Self::MarkerReported(...)
}`) — roughly 30 extra emitted lines per operation, none of which appear as
separate schema text; they are a fixed multiplier applied uniformly across
every operation name in the schema's bracketed lists.

INTERPRETATION: the noise is structural, not incidental — it is the same
handful of emission patterns (derive pair, `#[rustfmt::skip]`, per-variant
`From`/constructor, re-export block) repeated once per schema declaration.
This is exactly the kind of repetition a positional codex IR could collapse
back toward 1:1 with the schema, since the multiplier is constant per
declaration kind rather than varying with content.

## 4. Exclusion candidates

OBSERVED via targeted `grep -c` across `schema-rust/src`, `sema-engine/src`,
`signal-spirit/src`, `spirit/src` (all directly counted, not inferred):

- **`unsafe`** — 0 occurrences in all four repos; actively `#![forbid]`-den
  in `spirit`. Strong exclusion candidate.
- **`macro_rules!`** (declarative macros) — 0 occurrences in all four repos.
  The only macro mechanism anywhere in this survey is proc-macro `derive`
  and `cfg_attr`. Strong exclusion candidate.
- **`union`** — 0 occurrences anywhere. Strong exclusion candidate.
- **Explicit `async_trait`-style macro usage** — 0 occurrences; the
  workspace instead relies on native `async fn` in traits (stable in current
  Rust), all through the generated kameo/daemon surface. So "async trait
  methods" as a raw language feature should stay in-scope, but the
  `async_trait` macro pattern specifically can be excluded.
  INTERPRETATION, weak signal: only sampled via one regex; not exhaustively
  verified against every repo in the workspace, only the four surveyed.

Not directly checked (named as unknowns, not claimed absent): raw pointers,
`impl Trait` in return position (`-> impl Trait`), trait objects (`dyn
Trait`) — a shallow grep found 0-3 `dyn` hits per repo (low but nonzero;
scout did not read those sites to confirm they're load-bearing or
incidental), so `dyn` is a weak-evidence exclusion candidate at most, not a
confirmed one. Const generics, GATs, and procedural-macro-authoring code
(as opposed to macro *usage*) were not searched at all in this pass.

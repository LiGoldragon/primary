A strawman so the psyche can SEE Logos and Nomos syntax. This is not a spec. Every
syntax choice is tagged either **[licensed by ruling]** (with the ruling cited) or

## 0. Open choices — flagged at top, decide these first

**OPEN CHOICE 1 (the big one): dotted-prefix `Head.{...}` vs headed-delimiter
`{ Head ... }`.** The landed invariant says "dotted-prefix application"
[licensed by ruling — prior-session invariants]. But the empirical parse (section 4)
shows the raw nota layer treats the dot as **ordinary atom text**: `Types.{...}`
parses as two adjacent root objects — an atom `"Types."` (dot included) and a
separate brace — never bound together. The headed-delimiter form `{ Head body }` /
`(Head args)` instead parses as **one clean tree with the head as first child**, and
that is exactly the shape nota's already-landed `StructuralMacroNode` example
consumes (`#[shape(pascal_head, arity = N)]`, `head = "Vec"`). So there is a real
tension between the stated dotted-prefix invariant and what the raw floor natively
structures. Three ways to resolve, psyche to pick:
- (1a) Keep dotted-prefix; the logos expectation layer pairs `Head.` with the
  following block. Costs: every level doubles its root-object count; the daemon
  re-pairs head+body.
- (1b) Teach the raw nota layer a dot-application rule so `Head.(...)` is one node.
  Costs: a change to nota's raw grammar (currently `.` is plain atom punctuation).
- (1c) Adopt headed-delimiter `(Head ...)` / `{ Head ... }` as the logos form.
  Costs: reads as "head is first positional child," which is close to (but not the
  same as) the forbidden bare space-pair; free ride on landed nota dispatch.

**OPEN CHOICE 2: per-kind section wrapper.** Is a types block written `Types.{ … }`
(a named kind head) or a bare `{ … }` (kind inferred from document position/filename)?
[proposal] — the schema uses a bare `{ … }` for its type section.

**OPEN CHOICE 3: field-name derivation.** TrueLogos carries no field names
(positionality, no named binding [licensed by ruling]). The mockup proposes the macro
derives Rust field names as snake_case of each field's type name
(`CommitSequence` -> `commit_sequence`). [proposal] — confirm the rule.

**OPEN CHOICE 4: tuple-newtype vs named-struct discriminator.** The oracle collapses
these by arity (verified: single-field `ObservedRecords { RecordSet }` emits a tuple
newtype `ObservedRecords(RecordSet)`; multi-field emits a named struct). The mockup
adopts arity as the discriminator so the special case dissolves. [proposal] — confirm.

**OPEN CHOICE 5: is `Vector` a named macro or a logos builtin?** The mockup models it
as a named macro (`Vector.(T)` -> `Vec<T>`) so collections live in the macro table
like everything else. [proposal].

**OPEN CHOICE 6: nota version.** The dispatch task said "nota 0.7.0 is on main," but
the workspace lockfile and nota's `origin/main` both resolve **nota 0.5.1**
(`repos/signal-spirit/Cargo.lock`, commit `ce7c564`). All parse evidence below is
against 0.5.1 — what the workspace actually builds. If a 0.7.0 exists elsewhere, the
verdict should be re-run against it. [evidence].

## 1. Worked example 1 — a plain wire-contract record (DatabaseMarker)

Three stages, left to right in the pipeline `schema -> macro expansion -> logos ->
Rust`. Stage 2 is the proposed TrueLogos; stage 3 is the actual generated Rust the
macros must lower to (the acceptance oracle [licensed by ruling — acceptance oracle]).

**Stage 1 — schema source** (`signal.schema`, real lines):

```
CommitSequence Integer
StateDigest Integer
DatabaseMarker { CommitSequence StateDigest }
```

**Stage 2 — proposed TrueLogos** (`samples/wire-types.logos`, dotted-prefix form):

```
CommitSequence.{ Integer }
StateDigest.{ Integer }
DatabaseMarker.{ CommitSequence StateDigest }
```

Syntax annotations:
- Brace-delimited per-kind block and positional bodies — [licensed by ruling:
  per-kind blocks; strictly typed positional].
- `CommitSequence`, `Integer`, `DatabaseMarker` capitalized-leading = objects —
  [licensed by ruling: capitalization semantics].
- No field names in the body; only field *types*, positionally — [licensed by ruling:
  positionality, no named binding]. Field names are derived downstream (OPEN CHOICE 3).
- No `pipe` delimiters, no bare `Name Type` space-pair (the schema's
  `CommitSequence Integer` becomes `CommitSequence.{ Integer }`) — [licensed by ruling:
  no pipe delimiters; no space-separated pair forms].
- The `.` dotted prefix — [licensed by ruling: dotted-prefix application], but see
  OPEN CHOICE 1: the raw layer does not bind it.
- Single-field body -> tuple newtype; multi-field body -> named struct —
  [proposal, OPEN CHOICE 4] (oracle-faithful).

**Stage 3 — actual generated Rust** (`signal.rs`, the oracle, verbatim):

```rust
#[rustfmt::skip]
#[cfg_attr(feature = "nota-text",
    derive(nota::NotaDecode, nota::NotaDecodeTraced, nota::NotaEncode))]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug, PartialEq, Eq)]
pub struct CommitSequence(Integer);

#[rustfmt::skip]
#[cfg_attr(feature = "nota-text",
    derive(nota::NotaDecode, nota::NotaDecodeTraced, nota::NotaEncode))]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug, PartialEq, Eq)]
pub struct StateDigest(Integer);

#[rustfmt::skip]
#[cfg_attr(feature = "nota-text",
    derive(nota::NotaDecode, nota::NotaDecodeTraced, nota::NotaEncode))]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug, PartialEq, Eq)]
pub struct DatabaseMarker {
    pub commit_sequence: CommitSequence,
    pub state_digest: StateDigest,
}
```

Everything in stage 3 that is not in stage 2 (the two derive attribute groups, the
`pub struct` keyword, the tuple-vs-named shape, the `commit_sequence`/`state_digest`
field names) is supplied by the Types-section structural macro in section 3. That is
the "noise multiplier" the survey measured, now named and localized in one macro.

## 2. Worked example 2 — a generic (Vector) through the same three stages

**Stage 1 — schema source** (`signal.schema`, real lines):

```
Domains (Vector Domain)
```

**Stage 2 — proposed TrueLogos** (`samples/wire-types.logos`):

```
Domains.{ Vector.(Domain) }
```

- `Vector.(Domain)` — named-macro application in paren form — [licensed by ruling:
  named macros dispatch as `Map.()`]. `Vector` is looked up by minted identity in the
  macro table [licensed by ruling: table keys on minted identity]; an unknown head in
  `X.()` position is an error [licensed by ruling].
- `Domains.{ … }` wraps the single field `Vector.(Domain)` -> tuple newtype
  [proposal, OPEN CHOICE 4].

**Stage 3 — actual generated Rust** (`signal.rs`, the oracle, verbatim):

```rust
#[rustfmt::skip]
#[cfg_attr(feature = "nota-text",
    derive(nota::NotaDecode, nota::NotaDecodeTraced, nota::NotaEncode))]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize, Clone, Debug, PartialEq, Eq)]
pub struct Domains(Vec<Domain>);
```

The macro `Vector` lowers `Vector.(Domain)` to `Vec<Domain>`; the Types-section
structural macro wraps it as the tuple newtype with the standard derives.

## 3. Nomos definition-file mockup (`samples/types-section.nomos`)

The first concrete Nomos syntax ever written. Nomos is presumed NOTA-shaped
([open] in design-v0; the file below does raw-parse, section 4). These definitions
are consumed in the logos daemon at the schema->logos conversion site
[licensed by ruling 2026-07-11]. All record shapes below are [proposal].

**Named macro — `Vector`** (`Vector.(T)` -> `Vec<T>`):

```
Macro.{
  Vector
  Match.( Head.(Vector) Arity.(1) )
  Lower.( RustGeneric.( Vec Element ) )
}
```

**Named macro — `Map`** (illustrative; `Map.(K V)` -> `BTreeMap<K, V>`):

```
Macro.{
  Map
  Match.( Head.(Map) Arity.(2) )
  Lower.( RustGeneric.( BTreeMap Key Value ) )
}
```

**Structural macro — the Types-section "particular-struct definition"** (the one
that adds the rkyv derive trio + optional NOTA-text derives, then chooses tuple vs
named by arity):

```
Structural.{
  ParticularStruct
  Applies.( Section.(Types) UnknownHead Brace )
  Derives.(
    RustfmtSkip
    CfgAttr.( nota-text Derive.( NotaDecode NotaDecodeTraced NotaEncode ) )
    Derive.( Archive Serialize Deserialize Clone Debug PartialEq Eq )
  )
  Lower.(
    Single.( TupleNewtype.( Head Field ) )
    Multiple.( NamedStruct.( Head Fields SnakeCaseFromType ) )
  )
}
```

Nomos annotations:
- `Macro.{ Name … }` vs `Structural.{ Name … }` names the two dispatch modes
  [licensed by ruling: named-then-structural dispatch]. The first positional slot is
  the macro's projected name; its identity in the table is minted, not the string
  [licensed by ruling: table keys on minted identity].
- `Applies.( Section.(Types) UnknownHead Brace )` encodes the structural trigger:
  in the Types section, a head with no macro-table entry, applied to a brace —
  [licensed by ruling: structural macros are per-section positional defaults for
  `X.{}` where X is not in the table].
- `Single`/`Multiple` under `Lower` is the arity discriminator — [proposal, OPEN
  CHOICE 4], oracle-faithful.
- `nota-text` is lowercase/kebab (a name, not an object) — [licensed by ruling:
  capitalization semantics]; it is the literal cargo feature gate from the oracle.
- Every record uses positional brace/paren bodies, no field labels, no pipes —
  [licensed by ruling: positional; no pipe delimiters].
- The dotted heads (`Match.`, `Lower.`, `Head.`) carry the same OPEN CHOICE 1 caveat.

## 4. Parsability verdict — real nota parse, honestly reported

Method: `nota::Document::parse` (nota 0.5.1, the workspace's resolved version) run
against all three sample files via a throwaway harness
(`scratchpad/notaparse`, path-dependency on `git-archive/nota`). Reproduce with a
one-file cargo bin calling `Document::parse(read_to_string(path))` and walking
`root_objects()`. Distinguishing **raw-parses today** from **meaningful parse needs
implementation** (a logos/nomos kind/expectation table):

**All three files RAW-PARSE today with zero errors.** The raw layer discovers
delimiter and atom structure exactly as designed; every atom classifies as a
PascalCase symbol (or, for `nota-text`, a kebab symbol; for `1`/`2`, string-only).

**What raw-parses vs what needs implementation:**

- **Delimiter/atom structure: raw-parses.** Braces, parens, nesting, comments (`;;`),
  and symbol classification all resolve now. `samples/wire-types-headed.logos` parses
  as a single clean tree: `{ Types { CommitSequence Integer } … { Domains (Vector
  Domain) } }` — head as first child, ready for nota's landed `pascal_head` /
  `head = "Vec"` structural dispatch.
- **Dotted-prefix binding: raw-parses, but NOT as application.** In
  `samples/wire-types.logos`, `Types.{…}` yields **two root objects** — atom
  `"Types."` (the dot absorbed into the atom) and a separate brace — and likewise at
  every nested level (`CommitSequence.` + `{ Integer }`, `Vector.` + `(Domain)`). The
  raw floor never pairs the dotted head with the following delimiter; that pairing is
  **meaning that needs the logos expectation layer** (or a raw-grammar dot rule — OPEN
  CHOICE 1). So dotted-prefix is parsable-as-text but not parsable-as-application
  today.
- **Type/kind meaning: needs implementation everywhere.** That `Integer` is a scalar,
  that `Vector` is a named macro, that a brace under `Types` is a struct definition,
  that arity picks tuple-vs-named — none of that is raw-layer knowledge. It requires
  the logos/Nomos expectation (kind) tables, which do not exist yet. The raw parse is
  necessary structural triage, not semantic acceptance.

One-line verdict: **the mockup files parse at the raw nota structural layer today
(both dotted and headed forms), but every bit of meaning — and, for the dotted form,
even the head-to-body application binding — awaits the logos/Nomos kind tables that
are not yet built.** The headed-delimiter form is the one already aligned with nota's
landed structural-node dispatch.

ID b0s4 | Clarification | certainty=Low importance=Minimum | domains=Technology.Software.Programming.DomainSpecificLanguages | referents=schema,macro,enum,nota,discriminator
DESC: The enum-versus-macro-invocation distinction CANNOT be resolved by delimiter-shape or position alone: a collection macro such as Vec applied to one element type, and a two-variant enum, are structurally identical paren objects and can appear at the same node position - the head symbol is an operator in one case and a variant name in the other, with nothing marking which. So an explicit DISCRIMINATOR on the paren head is required. This corrects the earlier framing that position alone resolves it. The struct bracket and the map brace do NOT have this problem - only the paren is doubly loaded as both enum and head-applied macro. Three candidate discriminators: a reserved operator-word set (implicit), a sigil marking operator heads (explicit, structural), or moving collections off the paren entirely onto suffix sigils and the brace - vector as a star suffix, option as a question suffix, map as a brace - so the paren stays pure enum. Open decision: which discriminator, and whether enum stays the bare paren default.

# Schema runtime actor upgrade vision — overview

Kind: Implementation synthesis
Topics: schema runtime nexus mail signal sema spirit

## What changed

This pass moves the current prototype from the older executor wording to the
new runtime triad:

- **Signal** is the communication plane: CLI/daemon wire roots, short headers,
  NOTA text at the edge, and rkyv signal frames across the process boundary.
- **Nexus** is the execution and mail-keeper plane: a decoded Signal payload
  becomes `NexusMail<Payload>` with a `MessageIdentifier`; while Nexus owns it,
  the mail is in processing state.
- **SEMA** is the durable state plane: Nexus lowers state-involving mail into a
  `SemaCommand`, SEMA returns `SemaResponse`, and Nexus turns that into a
  processed Signal reply.

The important implementation constraint stayed intact: generated schema nouns
are the method surface. The new runtime support is generated by
`schema-rust-next`; `spirit-next` implements behavior on those generated nouns
instead of inventing parallel free helpers.

## Implemented surfaces

### `schema-rust-next`

`RustEmitter` now emits a mail lifecycle and Nexus dispatch surface alongside
the existing NOTA/rkyv route code:

```rust
pub struct MessageSent {
    pub identifier: MessageIdentifier,
    pub root: MessageRoot,
    pub short_header: u64,
}

pub struct NexusMail<Payload> {
    pub identifier: MessageIdentifier,
    pub payload: Payload,
}

pub struct MessageProcessed<Reply> {
    pub identifier: MessageIdentifier,
    pub reply: Reply,
}

pub trait InputNexus {
    type Reply;
    type Error;
    fn record(&self, mail: NexusMail<Entry>) -> Result<Self::Reply, Self::Error>;
    fn observe(&self, mail: NexusMail<Query>) -> Result<Self::Reply, Self::Error>;
}
```

The generated `Input` root owns the crossing from Signal into Nexus:

```rust
impl Input {
    pub fn dispatch_mail_with_nexus<Nexus>(
        self,
        identifier: MessageIdentifier,
        nexus: &Nexus,
    ) -> Result<MessageProcessed<Nexus::Reply>, Nexus::Error>
    where
        Nexus: InputNexus,
    {
        let reply = match self {
            Self::Record(payload) => nexus.record(NexusMail::new(identifier, payload)),
            Self::Observe(payload) => nexus.observe(NexusMail::new(identifier, payload)),
        }?;
        Ok(MessageProcessed::new(identifier, reply))
    }
}
```

The test fixture uses a real `SpiritNexus` value implementing the generated
trait, so the dispatch proof is against generated objects, not a handwritten
mirror.

### `schema-next`

The macro/lowering side now has an explicit design example proving the three
schema planes can be declared by schema objects:

```nota
[
  (Record Entry)
  (Observe Query)
]
[
  (RecordAccepted RecordIdentifier)
  (RecordsObserved RecordSet)
]
{
  NexusAction {Record Entry Observe Query}
  NexusResult {Accepted RecordIdentifier Observed RecordSet}
  SemaCommand {Record Entry Observe Query}
  SemaResponse {Recorded RecordIdentifier Observed RecordSet}
}
```

That test keeps Nexus and SEMA in the schema namespace rather than hiding them
as Rust-only runtime enums.

### `spirit-next`

The Spirit engine now routes through generated Nexus mail:

```rust
impl Engine {
    pub fn handle(&self, input: Input) -> Output {
        let identifier = self.issue_message_identifier();
        let sent = input.message_sent(identifier);
        self.remember_message_sent(sent);

        let processed = input
            .dispatch_mail_with_nexus(identifier, self)
            .expect("engine nexus is infallible");
        self.remember_message_processed(&processed);
        processed.into_reply()
    }
}

impl InputNexus for Engine {
    type Reply = Output;
    type Error = Infallible;

    fn record(&self, mail: NexusMail<Entry>) -> Result<Self::Reply, Self::Error> {
        Ok(self.apply_sema_command(mail.into_sema_command()))
    }

    fn observe(&self, mail: NexusMail<Query>) -> Result<Self::Reply, Self::Error> {
        Ok(self.apply_sema_command(mail.into_sema_command()))
    }
}
```

The runtime test proves:

- `NexusMail<Entry>` lowers to generated `SemaCommand::Record`.
- `Store::apply` is still the single writer for SEMA mutation.
- `Engine::handle` increments sent and processed mail counts while preserving
  the existing record/observe behavior.

## Skill updates

The workspace skills now say the runtime triad is Signal / Nexus / SEMA, not
Signal / executor / SEMA. The important rule is:

- Signal decodes and frames.
- Nexus owns in-flight mail and execution decisions.
- SEMA owns durable mutation.
- Behavior belongs as methods or trait impls on generated schema nouns.

The affected skills are:

- `skills/component-triad.md`
- `skills/rust/methods.md`

## Still missing

- Nexus currently stores sent/processed mail counts in memory. It is not yet the
  durable mail manager with queued delivery, response correlation, database
  markers, or hash/counter state.
- `schema-rust-next` emits Nexus dispatch traits, but the full
  schema-diff-driven upgrade matrix is still only a partial proof.
- `signal-frame` has not yet been restacked as schema-derived generated source;
  Spirit still carries its own local signal-frame support in the generated
  schema source.
- The schema language can declare Nexus/SEMA nouns, but schema-authored macros
  for the entire macro system are not yet the only lowering mechanism.

## Recommendation

The next implementation slice should make the mail lifecycle durable:
`MessageSent`, `NexusMail`, `MessageProcessed`, and SEMA response markers should
be schema-declared objects with an explicit storage-backed mail state. That is
the point where Nexus stops being an in-memory proof and becomes the component's
real execution ledger.

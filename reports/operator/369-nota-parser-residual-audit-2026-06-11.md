# NOTA Parser Residual Audit

Operator report for the cleanup request: remove hand-rolled parsing and NOTA implementation traces from the Spirit/agent path and audit for leftovers.

## What landed

- `nota-next` now owns byte scalar NOTA codecs through `ByteSequence` and `FixedByteSequence`.
- `schema-rust-next` no longer emits generated `from_nota_block` / `to_nota` bridge methods, generated byte codec impls, or `ScopePathImplTokens` string-path helpers.
- `signal-agent`, `meta-signal-agent`, `agent`, and `spirit` were regenerated or adapted to the new derived surface.
- `agent` and `spirit` configuration writer binaries now parse their request envelopes through derived `NotaDecode` types instead of manual block walking.
- `spirit` no longer has the generated or handwritten DomainScope codec/path slop. Scope matching is structural over typed recursive enums.
- `CriomOS-home` pins the cleaned Spirit stack.

## Remaining global traces

A literal `/git/github.com/LiGoldragon` scan still finds old NOTA implementation traces outside the cleaned Spirit/agent path. The largest category is older checked-in generated schema files in other components that have not yet been regenerated against `schema-rust-next` `cedb2e06`.

The important active exception is `signal-spirit`: Spirit still depends on it for daemon configuration and production migration source types, and `signal-spirit` still contains handwritten `nota-text` projection code. Earlier handoff treated full `signal-spirit` hand-rolled to schema-driven migration as a separate larger question. Under the current standard, that migration is no longer optional if the requirement is literally zero active handwritten NOTA outside `nota-next`.

Other active or adjacent repos with residual hits include router/message/cloud/orchestrate contracts, repository-ledger, chronos, horizon/lojix surfaces, and generated signal/meta-signal crates. Those should be handled as a separate sweep: update their schema pins, regenerate checked-in schema artifacts, delete obsolete bridge methods, and either retire or schema-drive any remaining handwritten contract NOTA projection.

## Boundary

`nota-next` itself is the legitimate owner of raw NOTA parsing, `NotaBlock`, `Block::*`, scalar codecs, and the `NotaEncode` / `NotaDecode` trait implementations for core scalar/container types. Hits inside `nota-next` are not residual slop; they are the parser/codec implementation.

# 2 — Dimension: mentci daemon state & SEMA

*Scan dimension `daemon-state`. 4 confirmed / 5 raw findings (each adversarially verified against the actual code).*

## Assessment

The mentci daemon's runtime state is a hand-written in-memory `State` struct (state.rs) held inside a single kameo `StateOwner` actor, spawned fresh from `State::default()` on every `bind()`. NONE of the durable design in sema.schema is real: there is no build.rs, no schema codegen, no redb/StoreLocation, no rehydrate path — the five "persisted families" and the whole Nexus Work/Action reaction frame are aspirational schema text the code does not implement. So the summary's "in-memory, lost on restart" is correct and stronger than stated: pending questions, recorded verdicts, answer proposals, AND the subscription registry all evaporate on restart, and a restarted daemon cannot route a verdict because the in-memory question/evaluation linkage is gone. The pending-question lifecycle is also only half-built: present->answered works locally, but "verdict routed to criome" is NOT in the daemon at all — the daemon's `answer()` just records the verdict in a Vec; routing lives in a separate caller-driven `CriomeApprovalBridge` that the test, not the daemon, invokes. Critically the daemon is REQUEST/REPLY ONLY: `serve_forever` accepts one connection, handles it, returns; `observe`/subscriptions mint a token and store interest but there is NO socket-push, NO fan-out, NO held client connection — the daemon physically cannot PUSH an update to a watching CLI. This is the central blocker for a long-running daemon backing a live UI.

## Confirmed findings

### Daemon state is fully in-memory; no SEMA persistence exists despite the schema

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `mentci/src/daemon.rs:59-60 (bind: State::default() into StateOwner::spawn); mentci/src/state.rs:35-52 (Default); contrast mentci/schema/sema.schema:228-295 (five Family declarations)`
- **Evidence:** `bind()` does `StateOwner::spawn(StateOwner::new(State::default()))` — a fresh empty State per bind. State holds plain `Vec<ApprovalQuestion>`, `Vec<ApprovalVerdict>`, `Vec<AnswerProposalRecord>`, `BTreeMap<String,InterfaceInterest>` and `u64` counters, all in RAM. A repo-wide grep for redb/StoreLocation/persist/rehydrate/load/save in mentci/src returns nothing; there is no build.rs. sema.schema declares PendingQuestionsFamily/DecisionsFamily/AnswerProposalsFamily/SubscriptionsFamily/RevisionFamily as redb-style tables with the comment 'on restart the daemon self-resumes from them' — but no code reads or writes any table.
- **Impact:** A real long-running daemon backing the CLI UI loses every parked submission, recorded verdict, answer proposal, and the entire subscription registry on any restart/crash. Worse for the loop: the in-memory link between a minted QuestionIdentifier and the criome AuthorizationEvaluation needed to route the verdict is gone, so a restarted daemon literally cannot complete approval of a previously-parked submission. The CLI UI would show an empty pending list after restart while criome still holds parked work.
- **Direction:** Wire the daemon to a durable store. Either generate the SEMA engine from sema.schema (add the schema-next codegen + redb backing the schema already specifies) or, as a smaller first step, persist the State Vecs/map to a redb file in StateOwner and rehydrate in bind() before serving. The mentci INTENT already calls SEMA the canonical UI state, so this is the schema's intended home, not a bolt-on.
- **Verification:** Confirmed by reading the actual code. mentci/src/daemon.rs:59-60 spawns StateOwner from a fresh State::default() on every bind() — no rehydrate path. mentci/src/state.rs:13-52 holds all daemon state as plain in-RAM Vec<ApprovalQuestion>/Vec<ApprovalVerdict>/Vec<AnswerProposalRecord>/BTreeMap<String,InterfaceInterest> plus u64 counters; Default seeds them empty. mentci/Cargo.toml has NO dependency on sema/sema-engine/redb, there is no build.rs, and a grep over src/ for redb|persist|rehydrate|load|save|sema returns nothing. The sema.schema (lines ~228-295) declares PendingQuestionsFamily/DecisionsFamily/AnswerProposalsFamily/SubscriptionsFamily/RevisionFamily as redb-style Identified-keyed tables with the explicit comment that 'on restart the daemon self-resumes from them... it rehydrates the SEMA' — but nothing implements them; the schema is a pure spec. mentci/INTENT.md independently states 'The SEMA state is the canonical UI state' and names 'revision as durable daemon state', so persistence is the intended design, currently absent. So the core finding (fully in-memory, zero SEMA persistence despite the declared schema) is accurate and load-bearing: a long-running daemon backing the CLI UI loses every parked submission, verdict, answer proposal, and the subscription registry on any restart/crash — a real blocker for the goal's 'live long-running daemon' end state. ONE correction to the impact: the auditor's sentence about 'the in-memory link between a minted QuestionIdentifier and the criome AuthorizationEvaluation' being lost on restart overstates a mechanism that does not exist yet. State carries no AuthorizationEvaluation field, and CriomeApprovalBridge (src/criome_bridge.rs) is never invoked from src/daemon.rs or src/state.rs — it is exercised only by tests/criome_bridge.rs, where the evaluation is built inline. State.answer() records a verdict into the in-memory decisions Vec but never routes it to criome. So restart does not 'lose' a question→evaluation link; that verdict-routing wiring is independently unbuilt. The persistence gap is real on its own terms; that one causal sentence is inaccurate and should be dropped from the impact.
  - Refined: mentci/src/daemon.rs:59-60 (StateOwner::spawn(StateOwner::new(State::default())) per bind); mentci/src/state.rs:13-52 (in-RAM State + empty Default); mentci/Cargo.toml:10-23 (no sema/redb dep, no build.rs); mentci/schema/sema.schema:~239-296 (five *Family Identified tables with self-resume comment, unimplemented); mentci/INTENT.md:11-12,35-36 (SEMA = canonical/durable state). Correction evidence: src/criome_bridge.rs:31-43 submit_verdict exists but is referenced only by tests/criome_bridge.rs:184/370-371; no AuthorizationEvaluation field in State and no bridge call in daemon.rs/state.rs.

### Daemon is request/reply only — it cannot PUSH updates to a watching client

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `mentci/src/daemon.rs:81-91 (serve_forever / serve_next) and 98-114 (handle_connection); mentci/src/state.rs:122-130 (observe) and 208-227 (project)`
- **Evidence:** `serve_forever` is `loop { let (stream,_)=accept()?; handle_connection(&stream)?; }` and `handle_connection` reads exactly one frame, asks the actor once, writes exactly one reply, and the stream drops. `observe()` mints a SubscriptionToken and inserts `(token, interest)` into the `subscriptions` BTreeMap, returning a ONE-SHOT `InterfaceObservationOpened` snapshot. Nothing ever iterates `subscriptions` to deliver later updates — there is no held connection, no broadcast/notify, no InterfaceDelivery. The nexus.schema PublishInterfaceState/InterfaceFanOut/InterfaceDelivery effects (nexus.schema:417,458,472-475) and sema fan-out are entirely unimplemented; grep for publish/fan/Deliver/broadcast in src returns nothing.
- **Impact:** A long-running mentci client cannot watch for new parked submissions — it can only poll by reconnecting and re-issuing ObserveInterfaceState, getting a fresh snapshot each time. There is no way for criome->mentci->client to surface a newly-parked submission live. The end-state UI ('the human uses the CLI to see parked submissions') degrades to manual polling, and a TUI/status-bar that reacts to escalations is impossible without this.
- **Direction:** Make handle_connection able to hold a subscriber connection open and have StateOwner push InterfaceDelivery frames on every revision bump. Concretely: serve_forever must spawn/await connections concurrently (it is currently strictly serial — one slow subscriber blocks all other clients), and the actor needs a registry of live ActorRef/stream sinks keyed by SubscriptionToken so bump_revision fans projected state. This is the InterfaceFanOut the nexus schema already names.
- **Verification:** Confirmed by reading the cited code. daemon.rs:81-91 serve_forever is `loop { accept(); handle_connection(); }` — strictly serial. handle_connection (98-114) reads exactly one frame, blocks on a single actor ask, writes one reply, stream drops; it cannot hold a connection open. state.rs:122-130 observe() mints a SubscriptionToken, inserts (token, interest) into the subscriptions BTreeMap, and returns a ONE-SHOT InterfaceObservationOpened snapshot from project(). Nothing in the codebase ever iterates subscriptions to push later updates: grep for publish/fan/deliver/broadcast/notify across src/ returns nothing. The nexus.schema does name PublishInterfaceState / InterfaceFanOut / InterfaceDelivery (NexusEffectCommand and NexusEffectResult blocks, with the InterfaceFanOut record), and these are entirely unimplemented in src. client.rs:49-58 corroborates the impact: ClientCommand::run() connects, writes one frame, reads one reply, and exits — a pure one-shot client with no watch loop, so a watching client must reconnect and re-issue ObserveInterfaceState for a fresh snapshot each time. The finding's technical claims all hold. One correction on severity: for the LITERAL stated goal (a human runs the mentci CLI to SEE parked submissions and approve/reject/defer), snapshot-on-reconnect polling is sufficient — the end-to-end criome->mentci->verdict->criome loop can run without server push, because each CLI invocation already gets a fresh PendingQuestions projection. Push/fan-out is required only for a live reactive TUI/status-bar (which the meta-signal config even lists, NotificationClient::StatusBar), not for the minimum working CLI loop. Also, the 'one slow subscriber blocks all clients' point is latent, not active: with no held connections today every handle_connection returns immediately, so the serial loop is only a problem once held subscribers are added. So this is a real, schema-anticipated gap that blocks the reactive UI but not the minimum CLI-poll loop — high, not blocker.
  - Refined: mentci/src/daemon.rs:81-114 (serial serve_forever + one-shot handle_connection); mentci/src/state.rs:122-130 (observe inserts into subscriptions BTreeMap, returns one-shot snapshot, no consumer of the map); mentci/src/client.rs:49-58 (run() is one connect/write/read/exit — no watch loop); mentci/schema/nexus.schema PublishInterfaceState/InterfaceFanOut named but unimplemented (grep publish/fan/deliver/broadcast in src/ = empty)

### Verdict routing to criome is absent from the daemon; the present->routed lifecycle terminates at an in-memory Vec

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** unported
- **Location:** `mentci/src/state.rs:132-155 (answer); mentci/src/criome_bridge.rs:31-44 (submit_verdict); nexus.schema:421,427,457 (RetireQuestion/RouteVerdict/VerdictRouted)`
- **Evidence:** `State::answer()` removes the question from pending and does `self.decisions.push(verdict.clone())` then replies VerdictAccepted — it never contacts criome. Routing lives entirely in `CriomeApprovalBridge::submit_verdict`, which the daemon never calls; in tests/criome_bridge.rs the TEST calls `CriomeApprovalBridge::new(...).submit_verdict(evaluation, &verdict)` and the test holds the criome `AuthorizationEvaluation` that the daemon never sees. The nexus schema's RouteVerdict effect and the SignedVerdict/VerdictRouting envelope (nexus.schema:253-263) are unimplemented; the daemon stores no escalation->question mapping at all.
- **Impact:** The closed loop is not actually closed by the daemon. The 'verdict flows back to criome' step only works because a single-process test manually carries the AuthorizationEvaluation from criome's escalation reply over to the bridge. A real deployment where mentci-daemon owns parked submissions has no stored AuthorizationEvaluation to submit, so it cannot route the human's verdict to criome on its own.
- **Direction:** The daemon must persist the originating criome AuthorizationEvaluation alongside each parked ApprovalQuestion (extend ApprovalQuestion or add an escalation-origin family), and `answer()` must drive the bridge's submit_verdict using that stored evaluation as a Nexus RouteVerdict effect — not leave routing to an external caller. This depends on the escalation actually arriving at the daemon (next finding).
- **Verification:** Every cited claim holds up against the actual code. (1) state.rs:132-155 `answer()` does exactly `pending_questions.remove(index)` + `decisions.push(verdict.clone())` + replies `VerdictAccepted`; it never contacts criome. (2) criome_bridge.rs:31-44 `submit_verdict` exists and is the only routing code, but a grep across all of src/ shows the daemon (daemon.rs StateOwner holds only a `State`) never constructs or calls `CriomeApprovalBridge` — its sole occurrence is the test at tests/criome_bridge.rs:184/370. (3) The test confirms the manual carry: it builds `evaluation` locally (line 316), gets EscalateToPsyche from criome, separately and unlinked-ly calls PresentQuestion on mentci (line 340), then in PROOF (d) the TEST itself calls `CriomeApprovalBridge::new(...).submit_verdict(evaluation.clone(), &verdict)` (lines 370-371) holding the evaluation the daemon never sees. (4) The nexus schema (nexus.schema:421 RetireQuestion, 427 RouteVerdict, 457 VerdictRouted, 253-263 SignedVerdict/VerdictRouting, 235 CriomeEscalationRequest, 426 FrameEscalation) defines the intended escalation-frame and verdict-route effects, none of which are implemented — the daemon runs a flat State::apply with no escalation ingestion and no escalation->question mapping. ApprovalQuestion stores only {identifier, proposal}, no AuthorizationEvaluation. The auditor's impact is correct: a real two-daemon deployment where mentci-daemon owns parked submissions has no stored evaluation to route, so the loop is not closed by the daemon. Minor: the finding labels it kind=unported; it is more precisely unimplemented-against-its-own-schema, but the substance and blocker severity are right for the goal.
  - Refined: mentci/src/state.rs:147-153 (answer removes+pushes+replies, no criome call); mentci/src/daemon.rs:32-34,135-147 (StateOwner holds only State; ApplyRequest path has no bridge); grep of src/ finds criome only at daemon.rs:176 test socket string; mentci/tests/criome_bridge.rs:316,370-371 (test holds evaluation and calls submit_verdict itself); mentci/schema/nexus.schema:421,426-427,456-457,235,253-263 (RouteVerdict/VerdictRouted/FrameEscalation/SignedVerdict defined, none implemented)

### mentci never auto-subscribes to criome escalations — the escalation->question framing is faked by the test

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `mentci/src/criome_bridge.rs (whole file: only configure + submit_verdict, no subscribe); mentci/src/state.rs:83-95 (present_question takes a caller-supplied proposal); nexus.schema:415,426,443,456 (FrameEscalation/EscalationFramed)`
- **Evidence:** CriomeApprovalBridge has exactly two methods: `configure` and `submit_verdict`. There is no method that opens a criome subscription (criome has CriomeRequest::ObserveAuthorizedObjects and an AuthorizationPending reply at criome/src/actors/authorization.rs:120, but mentci never calls it). In the test, the criome EvaluateAuthorization reply (`EscalateToPsyche`) is read by the test, then the test independently fabricates a `question_proposal()` and sends `MentciRequest::PresentQuestion` to mentci — the proposal content is hand-written, not derived from the criome escalation. The nexus.schema FrameEscalation effect (CriomeEscalationRequest -> QuestionProposal) is unimplemented.
- **Impact:** There is no mechanism by which a parked criome submission becomes a mentci pending question automatically. The summary's 'caller-driven bridge / does not yet auto-subscribe' is exactly right and is a blocker: without mentci subscribing to criome escalations and framing them into QuestionProposals, the CLI can only ever show questions some external glue manually pushed, so the live two-daemon loop cannot run unattended.
- **Direction:** Add a CriomeApprovalBridge subscription path that opens criome's escalation/pending stream, and a Nexus FrameEscalation operation that converts each CriomeEscalationRequest into a QuestionProposal and AdmitQuestion. This requires the daemon to hold the criome connection open (same long-running-connection capability the fan-out finding needs) and to be running a real event loop rather than serial one-shot accept.
- **Verification:** Confirmed by reading the cited code. (1) mentci/src/criome_bridge.rs has exactly two methods, `configure` and `submit_verdict`; there is no subscribe/observe path and the bridge never opens a criome stream. (2) mentci/src/state.rs:83-95 `present_question` takes a caller-supplied `QuestionProposal` — it is never derived from a criome escalation. (3) In tests/criome_bridge.rs `mentci_closed_verdict_approves_criome_escalation_over_meta_socket`, the test sends EvaluateAuthorization, reads back EscalateToPsyche (PROOF a), then independently fabricates `question_proposal()` (hand-written literals "authorize-spirit-head"/"approve"/"criome-escalated-policy") and pushes it via MentciRequest::PresentQuestion. The `escalated` reply is NOT passed into or used to build the proposal — the escalation->question framing is entirely faked by the test. (4) FrameEscalation/EscalationFramed/CriomeEscalationRequest exist ONLY in nexus.schema (records + the NexusEffectCommand/Result enums at lines 235/415/426/443/456); grep over src/ for those symbols returns nothing — zero Rust implementation. (5) mentci/src/daemon.rs BoundDaemon holds no criome connection (only socket_path/listener/runtime/state/codec) and serves via serial one-shot accept (serve_next/serve_forever), so it cannot hold a long-running criome subscription. This genuinely blocks the goal: no mechanism turns a parked criome submission into a mentci pending question, so the live two-daemon loop cannot run unattended. One correction to the finding's location detail: AuthorizationPending (authorization.rs:120) is the reply to EvaluateAuthorization on the signing path, not an escalation/pending stream to subscribe to; the actual observation surface is ObserveAuthorizedObjects (root.rs:240) returning AuthorizedObjectUpdateSnapshot, and criome has no dedicated parked-escalation observation stream at all — which makes the gap slightly larger than stated, not smaller.
  - Refined: mentci/src/criome_bridge.rs (only configure+submit_verdict, no subscribe); mentci/src/state.rs:83-95 (present_question takes caller-supplied proposal); mentci/tests/criome_bridge.rs (question_proposal() fabricated at the `fn question_proposal()` helper and pushed via PresentQuestion at the `let question = thread::scope` block, independent of the `escalated` reply); mentci/src/daemon.rs:62-91 (BoundDaemon holds no criome conn; serial one-shot accept); nexus.schema:235,415,426,443,456 (FrameEscalation/EscalationFramed schema-only, no impl in src/)

### State machine ad hoc, not the schema-derived Nexus; bridge re-derives ProposalDigest with string formatting

- **Verdict:** DROPPED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** ugly-code
- **Location:** `mentci/src/state.rs:55-71 (apply hand-dispatch), 166-170 (ProposalDigest via format!), 133-139 (Defer special-case), 237-258 (string-formatted identifier minting)`
- **Evidence:** `State::apply` is a hand-written match over MentciRequest with no relation to the nexus.schema Work/Action reaction frame, NexusEffectCommand, or SEMA Write/Read rosters — the schemas describe an engine the code does not use. `propose_answer` builds the canonical digest with `ProposalDigest::new(format!("answer-proposal-{}-{}", ...))` — a stringly-typed digest, not a real content digest, and the comment in sema.schema:262 insists this digest is 'what criome later approves... never a loose answer string.' Identifier minting is `format!("question-{}", n)` etc. The `Defer` arm in answer() returns VerdictAccepted WITHOUT removing the question from pending or recording a decision — a special case that should collapse into the normal verdict path (a deferred question presumably stays pending by intent, but the divergence is silent and untyped).
- **Impact:** Does not block the loop but means the elaborate sema/nexus schemas are dead design while the live behavior is unvalidated hand code. The format!-based ProposalDigest will not match any real criome-side digest once cryptographic approval lands, so the answer-proposal path is a stub that will break at the criome boundary. The silent Defer divergence is a lifecycle-correctness smell.
- **Direction:** Either generate the State engine from sema.schema (making the schema load-bearing) or delete the unused schema sophistication and keep the schemas honest about the in-memory slice. Replace the format! digest with a typed content digest over the AnswerProposal body. Make Defer an explicit, tested lifecycle transition rather than an early-return that skips bookkeeping.

# 4 — Dimension: criome authorization & parking

*Scan dimension `criome-park`. 5 confirmed / 5 raw findings (each adversarially verified against the actual code).*

## Assessment

"Client approval mode" / parking is a genuine DESIGN GAP, not an existing concept. criome has exactly two closed AuthorizationMode variants — AutoApprove (short-circuits to Authorized) and Quorum (evaluates the contract synchronously). In Quorum mode, `EvaluateAuthorization` returns a verdict in the SAME request/reply turn and holds NO pending state: `record_evaluation_decision` (root.rs:331) persists/publishes ONLY when the decision is `Authorized`; for `EscalateToPsyche` or `Rejected` it constructs an `AuthorizationEvaluated` reply and drops the request entirely — nothing is parked, nothing is keyed, nothing is recoverable. The signal-criome `AuthorizationEvaluation` is never written to any store/table for the escalate path (the store only holds the slot-based `AuthorizationStateRecord` used by the separate BLS `AuthorizeSignalCall` signing flow, which is a different mechanism). `EscalateToPsyche` is reachable two ways — a contract whose Rule is literally `EscalateToPsyche` (language.rs:318), or the META approval path where the approver sends `Defer` (root.rs:375) — but in both cases criome merely emits a terminal reply and forgets. The meta approval handler `record_authorization_approval` (root.rs:351) is fully STATELESS: the approver (mentci) must re-supply the entire `AuthorizationEvaluation` inline in `AuthorizationApproval { evaluation, decision }` (meta-signal-criome lib.rs:77), and criome re-publishes it; criome cannot answer "what is parked?" because it never stored the original submission. There is therefore no way for an external submitter to discover its submission is parked nor to later learn it was authorized except by holding the evaluation itself and polling `ObserveAuthorizedObjects`. The goal — criome parks a submission and waits for an out-of-band mentci verdict referencing that parked item by id — has no backing state and is a blocker-level missing mechanism.

## Confirmed findings

### EscalateToPsyche drops the request with zero parked state retained

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** design-gap
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/root.rs:331-349 (record_evaluation_decision), called from evaluate_authorization root.rs:307-329`
- **Evidence:** record_evaluation_decision only persists/publishes when `decision == EvaluationDecision::Authorized` (root.rs:336 guards the publish). For EscalateToPsyche (and Rejected) it skips the publish and returns `CriomeReply::AuthorizationEvaluated { contract, decision }` (root.rs:345). The incoming `AuthorizationEvaluation` (object, evidence, contract) is moved into the reply and dropped at end of turn. No store::Store* call writes a pending/parked evaluation; the store snapshot only knows contracts and the unrelated slot-based AuthorizationStateRecord.
- **Impact:** The goal's parking semantics literally do not exist: when criome decides a submission needs psyche approval it forgets the submission in the same request turn. There is nothing for mentci to list, nothing for a later verdict to reference by id, and nothing for the original submitter to re-discover. The mentci->daemon->criome approval loop cannot park-and-resume because no parked state is held.
- **Direction:** Add a pending-authorization table (durable, in the StoreKernel) keyed by a new ParkedAuthorizationId. evaluate_authorization should, on EscalateToPsyche, persist the full AuthorizationEvaluation under that id and reply with the id + a Parked status instead of a terminal AuthorizationEvaluated. Add signal-criome roots to list parked items and to subscribe to parked->resolved transitions.
- **Verification:** Confirmed by reading the cited code and tracing the full loop across both repos. The finding's factual claims hold exactly: criome/src/actors/root.rs:331-349 (record_evaluation_decision) guards the publish on `decision == EvaluationDecision::Authorized` (line 336) and, for EscalateToPsyche/Rejected, performs no store write and returns CriomeReply::AuthorizationEvaluated{contract, decision} (line 345), dropping the AuthorizationEvaluation's `object` and `evidence`. EvaluateAuthorization is synchronous request/reply (root.rs:237-238); no store::Store* persists a parked evaluation, and no subscription publish fires on escalation (publish_authorized_object_update only runs in the Authorized branch). The slot-based AuthorizationStateRecord IS a real parking mechanism, but it belongs to a SEPARATE flow — AuthorizationCoordinator.authorize_signal_call (authorization.rs:104-126) for SignalCallAuthorization — and is not wired to the contract-evaluation/escalation path; the auditor correctly flags it as unrelated. Most decisively, I traced where the parked AuthorizationEvaluation actually lives in the running system: it lives NOWHERE. mentci's daemon State (mentci/src/state.rs:12-26) holds pending_questions: Vec<ApprovalQuestion> and decisions: Vec<ApprovalVerdict> — never an AuthorizationEvaluation. The end-to-end test mentci_closed_verdict_approves_criome_escalation_over_meta_socket (mentci/tests/criome_bridge.rs:255-380) only closes the loop because the TEST DRIVER keeps `evaluation` in a local variable across the escalation (line 325) and manually re-feeds evaluation.clone() to submit_verdict (line 371); the mentci question at line 336 is an independently hand-built question_proposal() with no link to the escalation. So in a real two-daemon deployment neither criome nor mentci retains the evaluation, and CriomeApprovalBridge.submit_verdict (mentci/src/criome_bridge.rs:31-44) would have no AuthorizationEvaluation to send back — the park-and-resume loop cannot run outside a test orchestrator. One correction to the finding: its prescribed fix location (a ParkedAuthorizationId table inside criome's StoreKernel) is presented as the only remedy, but nexus.schema:227-240 shows the intended architecture routes parking to mentci (it FRAMES the CriomeEscalationRequest into a QuestionProposal and mints the local id). The gap is real either way, but the fix could equally carry the evaluation into mentci's pending store; the auditor's criome-side prescription is an overstatement of one option, not a factual error about the gap's existence. Severity blocker is appropriate: the goal explicitly demands the loop RUN against two live daemons, and the parked state needed to resume does not exist in either daemon.
  - Refined: criome/src/actors/root.rs:336 (publish guarded on Authorized) and root.rs:345 (terminal AuthorizationEvaluated reply drops object+evidence); no store write on EscalateToPsyche. mentci/src/state.rs:12-26 State has no AuthorizationEvaluation field. mentci/tests/criome_bridge.rs:316-371 shows the test driver itself (not either daemon) holds `evaluation` and re-feeds it to submit_verdict — proving no live party retains parked state. Intended parking owner is mentci per mentci/schema/nexus.schema:227-240.

### Meta approval handler is stateless — approver must re-supply the whole evaluation; criome cannot look up a parked item by id

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** design-gap
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/root.rs:351-383 (record_authorization_approval); contract meta-signal-criome /git/github.com/LiGoldragon/meta-signal-criome/src/schema/lib.rs:77-80 (AuthorizationApproval)`
- **Evidence:** AuthorizationApproval is `{ evaluation: AuthorizationEvaluation, decision: AuthorizationApprovalDecision }` — the FULL evaluation travels inline from the approver. record_authorization_approval destructures `{ evaluation, decision }` and, on Approve with a matching digest, publishes an AuthorizedObjectUpdate built entirely from the supplied `evaluation` (root.rs:362-368). It never consults the store/registry for a previously parked item; there is no id-keyed lookup. Defer maps to EvaluationDecision::EscalateToPsyche (root.rs:375) and again stores nothing.
- **Impact:** The verdict path is not a verdict-on-a-parked-item; it is 'replay an evaluation the approver already holds'. This forces mentci to be the carrier of the full evaluation it never received from criome (criome never parked/sent it). It also means an approve cannot be authenticated against what was actually submitted, and a Defer cannot keep the item alive. The closed mentci-CLI->criome loop where mentci approves item-by-id is unbuildable on this contract.
- **Direction:** Change the meta approval contract to reference a parked item by ParkedAuthorizationId (the verdict carries id + decision, not the inlined evaluation). criome loads the parked AuthorizationEvaluation from its pending table, applies the verdict, transitions the parked row, and publishes. Defer should re-park (refresh), not emit a terminal EscalateToPsyche.
- **Verification:** Confirmed by reading the cited code. meta-signal-criome/src/schema/lib.rs:77-80 defines `AuthorizationApproval { evaluation: AuthorizationEvaluation, decision: AuthorizationApprovalDecision }` — the full evaluation travels inline; there is no id type anywhere in the meta contract. In criome/src/actors/root.rs:351-383 `record_authorization_approval` destructures `{ evaluation, decision }` and, on Approve with a matching digest (line 360), publishes an AuthorizedObjectUpdate built entirely from the supplied `evaluation` (lines 362-368). It performs NO store/registry lookup; nothing is persisted. Defer maps to terminal `EvaluationDecision::EscalateToPsyche` (line 375) and stores nothing; Reject maps to `Rejected(AgreementMissing)`. This is replay of an evaluation the approver already holds, not a verdict on a parked item. The auditor's framing of WHY this is fatal to THE GOAL is correct and reinforced by two facts it did not fully cite: (1) signal-criome's `AuthorizationMode` enum is only `[Quorum AutoApprove]` (schema/lib.schema:82) — there is no client-approval/park mode in the meta flow at all, and `evaluate_authorization` (root.rs:307-329) either auto-publishes (AutoApprove) or runs synchronous contract-store evaluation (Quorum); neither parks an evaluation keyed by id for later meta approval. (2) mentci's own criome_bridge.rs:31-44 `submit_verdict` is forced to carry the full `AuthorizationEvaluation` to criome, proving the consequence the finding predicts — mentci must be the carrier of an evaluation criome never parked or sent it. One correction to the auditor's DIRECTION: a store-backed pending table DOES exist, but only on the separate working-signal `AuthorizationCoordinator` path (request_slot-keyed `AuthorizationStateRecord`, authorization.rs:104-266); it is the Quorum/signing flow and is not wired to the meta approval handler. So the park infrastructure for the META flow that mentci drives genuinely does not exist, and the direction (add a ParkedAuthorizationId, park-load-transition-publish, re-park on Defer) is the correct shape. Blocker severity is right: the closed mentci-CLI->criome approve-by-id loop the psyche asked for is unbuildable on this contract.
  - Refined: meta-signal-criome/src/schema/lib.rs:77-80 (AuthorizationApproval carries inline `evaluation`, no id); criome/src/actors/root.rs:351-383 (record_authorization_approval: no store lookup, publishes from supplied evaluation, Defer->EscalateToPsyche terminal, nothing persisted); signal-criome/schema/lib.schema:82 (AuthorizationMode [Quorum AutoApprove] — no park mode); criome/src/actors/root.rs:307-329 (evaluate_authorization never parks for meta approval); mentci/src/criome_bridge.rs:31-44 (submit_verdict forced to supply full AuthorizationEvaluation).

### No discovery path: an external submitter cannot learn its submission is parked or later authorized

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/root.rs:307-349 + subscription.rs:104-119 (OpenAuthorizedObjectSubscription / publish_authorized_object_update)`
- **Evidence:** The only post-evaluation notification surface is AuthorizedObjectUpdate, published ONLY on Authorized (root.rs:336-343) into SubscriptionRegistry.authorized_object_updates (subscription.rs:144). There is no 'pending/parked' update kind and no parked-list query — ObserveAuthorizedObjects filters only over AuthorizedObjectUpdate (subscription.rs:104-119). EvaluateAuthorization is request/reply: the submitter gets one synchronous AuthorizationEvaluated and that is the whole interaction.
- **Impact:** Even with parking added to state, there is no signal-criome root by which the submitter (or mentci) enumerates parked submissions or is notified when one flips to authorized. Both the human-facing 'show parked submissions' list and the submitter's 'am I authorized yet' wait have no contract behind them.
- **Direction:** Add signal-criome roots: ObserveParkedAuthorizations (list + subscribe) and a ParkedAuthorizationUpdate event emitted on park and on resolution. The submitter's EvaluateAuthorization reply should return Parked{id} so it can subsequently observe that id.
- **Verification:** Confirmed by reading the code. The EvaluateAuthorization flow is purely request/reply (signal-criome/schema/lib.schema:23; root.rs:307-329 evaluate_authorization) with branches only for AutoApprove (authorize) and otherwise a synchronous contract store.evaluate() — there is NO park branch, and AuthorizationMode is literally just [Quorum AutoApprove] (lib.schema:82), so no client-approval/park mode exists. The ONLY post-evaluation push surface is AuthorizedObjectUpdate, published exclusively when the decision is Authorized (root.rs:336-343 and 362-368); ObserveAuthorizedObjects filters only over AuthorizedObjectUpdate. So there is no 'pending/parked' update kind, no parked-list query, and no Parked reply variant. Critically, record_authorization_approval (root.rs:351-383) requires the approver to SUPPLY the entire AuthorizationEvaluation, and the mentci bridge (criome_bridge.rs:31-44) does exactly that — meaning criome persists nothing mentci can enumerate; mentci must already hold the evaluation out-of-band. The auditor's claim of 'no discovery path' is therefore accurate and load-bearing for the goal's 'show parked submissions' list and the submitter's 'am I authorized yet' wait. Corrections to the finding: (1) line-number imprecision — subscription.rs:104-119 is open_authorized_object_subscription, and the ObserveAuthorizedObjects dispatch is actually root.rs:240-248; substance unaffected. (2) The finding omits that a SEPARATE coordinator flow exists (AuthorizeSignalCall -> AuthorizationPending + ObserveAuthorization by request_slot, authorization.rs:104-158) which does park-and-observe by slot — but it is the BLS quorum-signing root, not wired to EvaluateAuthorization or the mentci bridge, and even it offers only by-slot lookup, not an enumeration, so it does not close the gap. (3) The gap is somewhat downstream of an even more fundamental missing piece: criome has no client-approval/park mode at all, so 'add a parked-list root' presupposes parking first.
  - Refined: criome/src/actors/root.rs:307-349 (evaluate_authorization has no park branch; AuthorizedObjectUpdate published only on Authorized) + root.rs:351-383 (record_authorization_approval requires caller-supplied evaluation) + root.rs:240-248 (ObserveAuthorizedObjects dispatch); signal-criome/schema/lib.schema:23 (EvaluateAuthorization request/reply), :82 (AuthorizationMode [Quorum AutoApprove] — no park mode), :24 (ObserveAuthorizedObjects over AuthorizedObjectUpdateStream only); mentci/src/criome_bridge.rs:31-44 (submit_verdict supplies full AuthorizationEvaluation). Separate non-wired park path: criome/src/actors/authorization.rs:104-158.

### Two unrelated authorization mechanisms (slot-based AuthorizeSignalCall vs contract EvaluateAuthorization) — only the wrong one holds durable-ish pending state

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `high`)
- **Kind:** design-gap
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/authorization.rs:104-126 (authorize_signal_call -> AuthorizationPending) vs src/actors/root.rs:307 (evaluate_authorization)`
- **Evidence:** authorize_signal_call DOES create persisted pending state: it calls create_authorization_state(store::CreateAuthorizationState::signing(..)) and replies AuthorizationPending with a request_slot + AuthorizationObservationToken (authorization.rs:111-125), observable later via ObserveAuthorization (authorization.rs:146-158). But this is the BLS multi-signer signature-collection flow (RouteSignatureRequest/SubmitSignature/RejectAuthorization), entirely separate from EvaluateAuthorization, which is the path the mentci/criome approval goal actually flows through (root.rs:237, evaluate_authorization). The contract-evaluation path has no equivalent pending slot.
- **Impact:** There is already a working 'park + observe by token' pattern in the codebase, but it is wired to the signing flow, not the contract-approval flow the goal needs. Risk of two parallel half-built approval models; the one the goal targets is the one without parking.
- **Direction:** Either fold contract EvaluateAuthorization escalations into the existing AuthorizationStateRecord/request_slot/observation-token machinery (reuse AuthorizationPending + ObserveAuthorization) or deliberately unify the two into one 'pending authorization' concept. Reusing the slot model is the cheaper path to a real parked item with an observation token mentci can poll.
- **Verification:** Confirmed by reading the cited files. Two distinct authorization mechanisms exist and only the wrong one parks durable state. (1) authorize_signal_call (criome/src/actors/authorization.rs:104-126) genuinely persists pending state via create_authorization_state(CreateAuthorizationState::signing(..)) and replies AuthorizationPending with a request_slot + AuthorizationObservationToken, later observable through ObserveAuthorization (authorization.rs:146-158) — but this is the BLS multi-signer signing flow (RouteSignatureRequest/SubmitSignature/RejectAuthorization). (2) The contract path the goal actually flows through, EvaluateAuthorization (root.rs:237 -> evaluate_authorization root.rs:307), parks NOTHING: in non-AutoApprove mode it evaluates the contract and returns AuthorizationEvaluated synchronously; record_evaluation_decision (root.rs:331-349) only touches the store to publish an AuthorizedObjectUpdate when the decision is already Authorized, and does nothing for EscalateToPsyche. I verified the live loop in tests/daemon_skeleton.rs:1254-1283: EvaluateAuthorization returns EscalateToPsyche with no stored item, then mentci must re-supply the ENTIRE evaluation in SubmitAuthorizationApproval to approve. The mentci side corroborates the gap: mentci/src/criome_bridge.rs submit_verdict requires the full AuthorizationEvaluation to be passed in (criome never stored it), and mentci/src/state.rs keeps its own pending_questions list rather than fetching parked items from criome. So there is no criome-side parked queue for mentci to list/observe — exactly the capability the goal ("criome PARKS it... the human uses mentci to SEE parked submissions") requires. One correction/extension to the finding: AuthorizationMode only has variants [Quorum AutoApprove] (signal-criome schema/lib.schema:82) — there is no distinct ClientApproval mode; escalation rides on a contract Rule::EscalateToPsyche, and the missing-parking gap is broader than just reusing the slot model (there is also no observe path from mentci into criome escalations), but the finding's stated direction (fold contract escalations into the AuthorizationStateRecord/request_slot/observation-token machinery) is the correct and cheapest fix. Severity high is appropriate: this is the central mechanism the goal needs and it is absent on the relevant path.
  - Refined: criome/src/actors/authorization.rs:111-125 (signing path persists pending + observation token); criome/src/actors/root.rs:307-349 (evaluate_authorization/record_evaluation_decision store nothing on EscalateToPsyche); criome/tests/daemon_skeleton.rs:1254-1283 (live loop re-supplies full evaluation on approval, no parked item); mentci/src/criome_bridge.rs:30-43 (submit_verdict requires caller-held evaluation); mentci/src/state.rs:14,85,109 (mentci-local pending_questions, not sourced from criome); signal-criome/schema/lib.schema:82 (AuthorizationMode [Quorum AutoApprove] — no ClientApproval park mode)

### AuthorizationMode is a closed two-variant enum with no 'client approval' / hold mode

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `high`)
- **Kind:** missing-feature
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/root.rs:312 + daemon.rs:44; enum defined in signal-criome`
- **Evidence:** evaluate_authorization branches only on `AuthorizationMode::AutoApprove` (root.rs:312) vs the default Quorum contract-evaluation path. AutoApprove short-circuits to Authorized; Quorum runs the contract and returns whatever decision the rule yields. There is no mode whose semantics are 'park every submission for an external client to approve'. The goal's 'client approval mode' has no enum variant and no code path.
- **Impact:** Even though a contract can be authored with Rule::EscalateToPsyche to force escalation, that is per-contract policy, not a daemon mode that uniformly parks submissions for mentci. The goal's framing ('criome runs in client approval mode and parks submissions') maps to nothing in the mode enum.
- **Direction:** If 'client approval mode' is meant to be a daemon-wide hold (park everything), add the variant and route evaluate_authorization through the new pending-park machinery for it. If it is meant to be expressed per-contract via EscalateToPsyche, keep the two modes but make EscalateToPsyche park (per finding 1) — then 'client approval mode' is just 'contracts that escalate'. The psyche should confirm which framing; today neither parks.
- **Verification:** Verified by reading the actual code. The finding's technical claims all hold. AuthorizationMode is a closed two-variant enum `[Quorum AutoApprove]` (signal-criome/src/schema/lib.rs:247-250; schema/lib.schema:82). `evaluate_authorization` (criome/src/actors/root.rs:307-329) branches only on `AutoApprove` (line 312, short-circuit to Authorized) vs the `Quorum` contract-evaluation path; there is no mode whose semantics are "park every submission for an external client." The only escalation path is per-contract via `Rule::EscalateToPsyche` (signal-criome lib.rs:432; criome/src/language.rs:318), which yields `EvaluationDecision::EscalateToPsyche`, not a daemon mode. So the finding correctly maps the goal's "client approval mode" to nothing in the enum, and correctly frames the open question (daemon-wide hold vs per-contract escalate).

One important correction that strengthens (not weakens) the finding: the per-contract `EscalateToPsyche` path also does NOT park. `record_evaluation_decision` (root.rs:331-349) only persists/publishes when the decision is `Authorized`; for `EscalateToPsyche` it merely returns `CriomeReply::AuthorizationEvaluated` to the caller and stores nothing pending. The mentci bridge (mentci/src/criome_bridge.rs) is a pure client that requires the caller to already hold both the `AuthorizationEvaluation` and the `ApprovalVerdict`. The existing end-to-end test `mentci_closed_verdict_approves_criome_escalation_over_meta_socket` (mentci/tests/criome_bridge.rs:255-380) demonstrates this: it hand-carries the `evaluation` object across stages and hand-injects an unrelated mentci `PresentQuestion`; criome never parks the escalated submission nor surfaces it to mentci. So the finding's conclusion "today neither parks" is accurate and is in fact the crux of why the goal (criome parks submissions and waits for mentci/psyche) is unmet. This is a genuine missing-feature gap central to the goal; high severity is right.

# 3 ŌĆö Dimension: the criomeŌåömentci bridge loop

*Scan dimension `bridge-loop`. 5 confirmed / 6 raw findings (each adversarially verified against the actual code).*

## Assessment

The live park->present->approve->verdict loop does NOT exist; the bridge is entirely caller-driven and there is no process that connects the two daemons continuously. CriomeApprovalBridge (mentci/src/criome_bridge.rs) is a stateless 53-line wrapper with two synchronous one-shot methods (configure, submit_verdict) ŌĆö the caller must already hold the AuthorizationEvaluation and the mentci ApprovalVerdict and hand both in; nothing subscribes to criome. The "proof" test (mentci/tests/criome_bridge.rs) is a single-process puppet: it manually drives criome's EvaluateAuthorization to get EscalateToPsyche, then manually constructs an unrelated mentci PresentQuestion, manually answers it, then manually calls submit_verdict with the evaluation it kept in a local variable ŌĆö there is zero data flow from the escalation into the mentci question. Worse, the open loop is structural on the criome side: an EscalateToPsyche result from EvaluateAuthorization is returned in the request reply and then dropped ŌĆö criome never records the parked submission, never publishes an event (record_evaluation_decision only publishes when decision==Authorized), and there is no escalation/pending event variant in the CriomeEvent enum nor any request to observe parked submissions. So a submission to criome does not "park and wait" anywhere queryable; it just returns a verdict to whoever called EvaluateAuthorization. To make the loop real, four things must be built: (1) criome must persist escalations as parked records and expose them via observe/event; (2) a long-running mentci bridge client/task must subscribe to criome and turn escalations into mentci PresentQuestion calls carrying the AuthorizationEvaluation; (3) that client must watch mentci for answered verdicts and route them back through submit_verdict; (4) the AuthorizationEvaluation must travel inside the mentci question (the QuestionProposal has no field for it today) so the verdict can be correlated to the parked submission.

### Verdict cannot be correlated to a parked submission: the mentci question carries no AuthorizationEvaluation

- **Location:** `mentci/src/criome_bridge.rs:31-44 (submit_verdict) vs the QuestionProposal carried in mentci state (mentci/src/state.rs:83-95, present_question) and signal_mentci::QuestionProposal fields used in tests/criome_bridge.rs:158-169`
- **Evidence:** submit_verdict takes the AuthorizationEvaluation as a SEPARATE argument from the ApprovalVerdict, because the verdict (signal_mentci::ApprovalVerdict { question, decision, answered_by }) carries only a QuestionIdentifier, not the criome evaluation. - **Verification:** Verified against the actual source. submit_verdict (mentci/src/criome_bridge.rs:31-44) takes the AuthorizationEvaluation as a SEPARATE argument from the &ApprovalVerdict, exactly as described. ### criome subscription push primitive is explicitly unimplemented, so even existing subscriptions never receive events

- **Verdict:** DROPPED ŌĆö verified severity `drop`, confidence `high` (auditor rated `high`)
- **Kind:** unported
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/subscription.rs:13-27 (SubscriptionRegistry doc + state)`
- **Evidence:** The SubscriptionRegistry doc comment states verbatim: 'The push primitive that emits IdentityUpdate events to active subscribers is not implemented in this daemon yet; this registry tracks subscription presence so retraction is meaningful even before the push side lands.' open_authorized_object_subscription (lines 104-119) returns only a one-shot snapshot of already-accumulated authorized_object_updates; it never streams later updates. publish_authorized_object_update (lines 140-146) merely appends to a Vec ŌĆö no fan-out to open subscriber sockets.
- **Impact:** Subscriptions in criome are snapshot-on-open only. A bridge cannot get a live feed of escalations (or even of authorized-object pulses) pushed to it; it would have to poll. Combined with the missing escalation event, the daemon has no mechanism to wake mentci when a new submission parks. The continuous live loop depends on this push primitive existing.
- **Direction:** Implement the push side: hold each open subscriber's writable connection (or an actor mailbox bridged to it) in the registry and, on publish_authorized_object_update / a new publish_escalation, fan out the CriomeEvent to matching open subscribers over the streaming frame channel.
- **Verification:** The finding's cited evidence is literally accurate: criome/src/actors/subscription.rs carries the verbatim doc comment that the push primitive is unimplemented (lines 13-27), open_authorized_object_subscription (104-119) returns only a one-shot snapshot, and publish_authorized_object_update (140-146) only appends to a Vec. (1) The criome daemon is architecturally pure request-reply: handle_connection in src/daemon.rs:139-154 reads exactly one request, writes one reply, and drops the connection; there is no persistent streaming frame channel and the registry cannot hold open subscriber sockets, so the finding's proposed direction (fan out over a 'streaming frame channel') describes a capability the daemon was not built to have.

# 7 — Dimension: unported / stub catalog

*Scan dimension `stubs`. 5 confirmed / 6 raw findings (each adversarially verified against the actual code).*

## Assessment

The six repos are remarkably free of literal stub macros: zero todo!/unimplemented!/unreachable! anywhere, and RequestUnimplemented is only a schema variant — never actually produced by either daemon. The master_key "placeholder" the summary flags is RETIRED: criome/src/master_key.rs is real BLS12-381 signing with on-disk 0600 key custody, so criome-owned signing of verdicts IS built (the summary's claim that it is "NOT built" is wrong). The real unported work is structural, not marker-tagged: (1) criome never PARKS an escalation — EscalateToPsyche is returned synchronously in the reply and stored nowhere; (2) there is no push/subscribe primitive (subscription.rs self-documents "the push primitive ... is not implemented in this daemon yet") and meta-signal-criome's Input enum offers no way to OBSERVE parked escalations, only Configure + SubmitAuthorizationApproval; (3) nothing maps a criome AuthorizationEvaluation into a mentci QuestionProposal nor correlates a mentci verdict back to it — the bridge test glues these by hand with a hardcoded fixture; (4) mentci SEMA is non-durable, booting fresh from State::default() every time. These four are the blockers/high gaps for a live two-daemon loop. Incidental: one free-function discipline violation in criome.

## Confirmed findings

### criome never parks an escalation — EscalateToPsyche is a fire-and-forget reply, not a held pending question

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/root.rs:331-349 (record_evaluation_decision) and :375`
- **Evidence:** record_evaluation_decision only calls publish_authorized_object_update when decision == EvaluationDecision::Authorized (root.rs:336). For EscalateToPsyche it stores NOTHING and just returns CriomeReply::AuthorizationEvaluated{decision: EscalateToPsyche}. A grep for pending/park/escalat across criome/src finds no escalation queue or pending store — the only 'pending' is the contract-time-check partition in subscription.rs:155. The submission is not held awaiting approval; the evaluation evidence is dropped after the reply.
- **Impact:** The goal's core mechanic — criome PARKS a submission and the submission WAITS for mentci/psyche approval — does not exist. criome answers 'escalate' and forgets; there is nothing for mentci to later approve against, and no way for the eventual SubmitAuthorizationApproval to be matched to a still-waiting submission. The end-to-end park-then-authorize loop literally cannot run.
- **Direction:** Add a pending-escalation table in the store actor keyed by request/object digest; evaluate_authorization writes the full AuthorizationEvaluation into it on EscalateToPsyche, and record_authorization_approval looks it up by digest, applies the verdict, then removes it. The held evaluation becomes the source of truth instead of the caller re-supplying it.
- **Verification:** The finding's factual claims about criome are accurate. In criome/src/actors/root.rs, record_evaluation_decision (331-349) publishes an AuthorizedObjectUpdate ONLY when decision == Authorized (336-343); for EvaluationDecision::EscalateToPsyche it stores nothing and returns a fire-and-forget CriomeReply::AuthorizationEvaluated. grep over criome/src for park/pending/escalat finds no escalation queue — the AuthorizationPending in authorization.rs:120 is a different signing-solicitation flow, and subscription.rs:155 'pending' is the contract-time-check partition. So criome does NOT park the escalation and drops the evaluation evidence. record_authorization_approval (351-383) requires the CALLER to re-supply the full AuthorizationEvaluation on SubmitAuthorizationApproval. I must correct the auditor's framing, though: the intended architecture parks on the MENTCI side, not criome. mentci's daemon holds pending_questions (state.rs) and is meant to re-supply the stored evaluation via CriomeApprovalBridge::submit_verdict. The auditor's proposed 'direction' (a criome-side pending table keyed by digest) is therefore not the only fix and not clearly the intended one. BUT the loop is still broken, even worse than stated: (1) mentci's CriomeApprovalBridge::submit_verdict has ZERO callers — the only reference outside criome_bridge.rs is the `pub mod criome_bridge` in lib.rs:10; mentci's answer() (state.rs:132-155) just removes the pending question and records the decision, never invoking the bridge, so the verdict never flows back to criome at all; (2) mentci's persisted ApprovalQuestion/QuestionProposal schema (sema.schema:159-172) carries source/prompt/suggested-answer/explanation/context but NOT the AuthorizationEvaluation, so even if the bridge were wired, the evaluation criome demands is never persisted across the wait. Neither daemon holds the evaluation across the park: criome drops it, mentci never captures it. The end-to-end park-then-authorize loop literally cannot run.
  - Refined: criome/src/actors/root.rs:336-344 (publish only on Authorized; EscalateToPsyche stores nothing) and :351-383 (record_authorization_approval requires caller-resupplied AuthorizationEvaluation); no park/pending/escalate store in criome/src (grep). Compensating mentci path is absent: mentci/src/criome_bridge.rs:31-44 submit_verdict has no callers (only mentci/src/lib.rs:10 `pub mod criome_bridge`); mentci/src/state.rs:132-155 answer() drops the question without re-supplying criome; mentci/schema/sema.schema:159-172 QuestionProposal/ApprovalQuestion does not persist AuthorizationEvaluation.

### No subscribe/observe path for mentci to learn about parked escalations — meta Input has only Configure + SubmitAuthorizationApproval

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `meta-signal-criome /git/github.com/LiGoldragon/meta-signal-criome/src/schema/lib.rs:132-135 (enum Input); criome /git/github.com/LiGoldragon/criome/src/actors/subscription.rs:17-22 (doc comment)`
- **Evidence:** meta-signal-criome Input = {Configure, SubmitAuthorizationApproval} — no ObservePendingEscalations / poll / stream variant. SubscriptionRegistry's own doc comment states: 'The push primitive that emits IdentityUpdate events to active subscribers is not implemented in this daemon yet'. publish_authorized_object_update (subscription.rs:140-146) merely pushes into an in-memory Vec authorized_object_updates; no subscriber socket is ever written. signal-criome has no PendingApproval/AwaitingApproval input either.
- **Impact:** Even if criome parked escalations, mentci has no wire request to discover them. The mentci bridge (criome_bridge.rs) is purely caller-driven: submit_verdict requires the caller to ALREADY hold the AuthorizationEvaluation. mentci can never auto-subscribe to criome escalations, which the summary lists as the missing long-running-client piece. Without an observe/push contract the loop can only be driven by an external test harness, not by the daemons.
- **Direction:** Add a meta-signal-criome Input::ObservePendingEscalations (request/reply snapshot is enough for a one-shot CLI) returning the parked AuthorizationEvaluations with their digests; later a push channel for a long-running mentci client. mentci's CriomeApprovalBridge gains a poll method that lists parked escalations rather than requiring the caller to supply the evaluation.
- **Verification:** The gap is real but the finding mislocates it. The finding's headline — mentci has no wire path to DISCOVER parked escalations it does not already know about — holds. But several cited specifics are wrong: (1) The finding says the missing observe path belongs in meta-signal-criome (Input = {Configure, SubmitAuthorizationApproval}); that enum is correctly minimal — observe is a WORKING-signal operation. (2) The finding claims signal-criome has no PendingApproval/observe input; FALSE — signal-criome has Input::ObserveAuthorization(AuthorizationObservation), Output AuthorizationObservationSnapshot/AuthorizationPending, and an AuthorizationUpdate event, all wired through criome root.rs:215 -> AuthorizationCoordinator (authorization.rs:146 observe_authorization) -> store. (3) The subscription.rs push-not-implemented doc comment and publish_authorized_object_update-into-a-Vec evidence is accurate but mostly irrelevant: that is the authorized-object-update stream, a different surface from authorization-request observation. The genuinely material gap: observe_authorization looks up exactly ONE state by the AuthorizationRequestSlot the caller already supplies (lookup -> into_iter().collect() => 0 or 1 state), so it cannot enumerate unknown escalations; and while StoreKernel has ReadAuthorizationSnapshot / authorization_snapshot() that lists ALL stored states (store.rs:431,618), grep confirms NO CriomeRequest variant routes to it (root.rs request match has no list-all arm) — it is unreachable over the socket. mentci's CriomeApprovalBridge (criome_bridge.rs) has only configure + submit_verdict, and submit_verdict requires the caller to already hold the full AuthorizationEvaluation. So mentci cannot discover parked submissions over the wire; the evaluation/slot must be handed to it out-of-band, which blocks the daemon-driven loop the goal demands. Correct fix direction: route a list-all-pending wire request (e.g. expose authorization_snapshot via a new signal-criome Input) and give the bridge a poll/list method — NOT a new meta-signal-criome Input as the finding proposes.
  - Refined: criome/src/actors/authorization.rs:146-158 observe_authorization (single-slot lookup, 0-or-1 result); criome/src/actors/store.rs:431,618-627 authorization_snapshot/ReadAuthorizationSnapshot exists but grep shows no CriomeRequest routes to it; criome/src/actors/root.rs:172-271 request match has ObserveAuthorization but no list-all arm; signal-criome/src/schema/lib.rs:1216-1241 Input includes ObserveAuthorization(AuthorizationObservation) (contradicts finding's no-observe claim); mentci/src/criome_bridge.rs only configure+submit_verdict, submit_verdict requires caller-supplied AuthorizationEvaluation

### Nothing maps a criome escalation into a mentci QuestionProposal, nor correlates the mentci verdict back — the bridge test glues both halves by hand with a hardcoded fixture

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** unported
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/criome_bridge.rs (whole file, 53 lines) and mentci/tests/criome_bridge.rs:336-371`
- **Evidence:** criome_bridge.rs exposes only configure() and submit_verdict(evaluation, verdict); it never constructs a signal_mentci::QuestionProposal from a signal_criome::AuthorizationEvaluation. In the test the mentci question is a hardcoded fixture: MentciRequest::PresentQuestion(question_proposal()) (criome_bridge.rs test:340), NOT derived from the escalated evaluation captured at test:316-333. The test then manually carries the same `evaluation` variable into submit_verdict(evaluation.clone(), &verdict) (test:371). State::present_question (state.rs:83-95) takes whatever QuestionProposal it is handed; nothing in src builds one from criome.
- **Impact:** The translation layer at the heart of the UI — escalation -> human-readable approval question -> verdict -> back to criome keyed to the right submission — is absent. The 'proven happy path' is a single-process test that hand-wires the correlation; in two live daemons there is no code that knows a given mentci verdict answers a given criome escalation. The approval UI cannot present real parked submissions or route real verdicts.
- **Direction:** Add an impl From<AuthorizationEvaluation> (or a bridge method) that builds a QuestionProposal carrying the criome request/object digest, and a verdict path that submits the verdict tagged with that digest so criome's pending table (finding 1) can match it. This is the operator's primary build target.
- **Verification:** Confirmed by reading the actual files. criome_bridge.rs (53 lines) exposes exactly configure() and submit_verdict(evaluation, verdict) plus a private map_decision(); it never constructs a signal_mentci::QuestionProposal from a signal_criome::AuthorizationEvaluation, and there is no `impl From<AuthorizationEvaluation>` anywhere in mentci/src (grep: zero From impls, the only QuestionProposal reference in src/ is state.rs:83 receiving a pre-built proposal). In tests/criome_bridge.rs the mentci question IS a hardcoded fixture — question_proposal() at test:158-169 (ApprovalSource::CriomeEscalation, prompt 'authorize-spirit-head', etc.), passed at test:340 — derived from nothing in the escalated evaluation captured at test:316-333. The same `evaluation` variable is then hand-carried into submit_verdict(evaluation.clone(), &verdict) at test:371; the test's shared local variable IS the correlation. Crucially, the bridge is never wired into the daemon at all: daemon.rs has zero references to criome_bridge/CriomeApprovalBridge, and configure()/submit_verdict() have no callers anywhere in src/ — only the test invokes them. On the criome side this is even worse for the goal than the finding states: root.rs SubmitAuthorizationApproval (root.rs:353) consumes the FULL AuthorizationEvaluation back inside the AuthorizationApproval payload, and record_evaluation_decision for an escalation stores nothing digest-keyed for later matching — so to approve, the submitter must already hold the entire original evaluation, which today only the single-process test holds. The translation/correlation layer at the heart of the approval UI (escalation -> human-readable question -> verdict -> back to criome keyed to the right submission) is genuinely absent from production code. Severity blocker stands: in two live daemons no code knows which mentci verdict answers which criome escalation, so the UI cannot present real parked submissions or route real verdicts. Minor correction to the auditor's direction: the fix isn't only a digest-tagged verdict against a criome pending table — criome currently demands the whole evaluation round-tripped, so mentci must also capture and carry the escalated AuthorizationEvaluation (or criome must gain a digest-keyed pending lookup); either way the bridge build is the operator's primary target.
  - Refined: mentci/src/criome_bridge.rs (whole 53-line file: only configure(), submit_verdict(), map_decision() — no QuestionProposal construction, no From<AuthorizationEvaluation>); mentci/src/daemon.rs (no criome_bridge reference — bridge unwired); grep across mentci/src shows the sole QuestionProposal use is state.rs:83 present_question() taking a pre-built proposal; tests/criome_bridge.rs:158-169 hardcoded question_proposal(), used at :340, with evaluation hand-carried from :316-333 into submit_verdict at :371; criome/src/actors/root.rs:353-380 SubmitAuthorizationApproval consumes the full AuthorizationEvaluation (no digest-keyed pending match).

### mentci SEMA is non-durable — daemon boots fresh from State::default() with no load/persist

- **Verdict:** CONFIRMED — verified severity `medium`, confidence `high` (auditor rated `high`)
- **Kind:** missing-feature
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/daemon.rs:60 (StateOwner::spawn(StateOwner::new(State::default()))) and state.rs:35-52 (Default)`
- **Evidence:** bind() spawns the state actor as StateOwner::new(State::default()). A grep for persist/sema/durable/load_state/restore/snapshot across mentci/src returns zero hits. State holds pending_questions/decisions/answer_proposals in plain Vec/BTreeMap (state.rs:13-26) with monotonic counters reset to 1 each boot.
- **Impact:** Every mentci-daemon restart discards all parked questions and answer proposals and resets identifier counters. For an approval UI whose whole job is to hold submissions until a human acts, a restart loses the queue and can re-mint colliding QuestionIdentifiers. Not a hard blocker for a first demo run, but the loop is not trustworthy across restarts — and the workspace contract requires daemons to self-resume from persisted SEMA.
- **Direction:** Persist State to a SEMA file (rkyv) on mutation and load it in on_start; keep identifier counters in the persisted state. Aligns mentci with the 'daemon self-resumes from persisted SEMA state' override.
- **Verification:** Confirmed accurate by reading the actual files. daemon.rs:59-60 spawns the state actor as StateOwner::spawn(StateOwner::new(State::default())); on_start (daemon.rs:127-133) is a trivial passthrough that returns the actor unchanged with no load/restore. state.rs holds pending_questions/decisions/answer_proposals in plain Vec and subscriptions in BTreeMap (state.rs:14-26), with Default setting next_question/next_proposal/next_subscription to 1 (state.rs:44-46). A grep across src/ for persist/sema/durable/load_state/restore/snapshot returns zero implementation hits — the only matches are the trivial on_start and schema comments. So every mentci-daemon restart discards all parked questions, decisions, and proposals and resets the monotonic identifier counters, which can re-mint colliding QuestionIdentifiers. This contradicts the daemon's own schema/sema.schema, which declares SEMA as 'the persistence layer — what the daemon STORES' and (Spirit 7x5z) the canonical UI state the daemon resumes from, and it violates the workspace hard-override 'On restart a daemon self-resumes from persisted SEMA state.' Correction to the finding's framing: it is a real durability/contract gap but not a blocker for the immediate goal — a single continuous run of both daemons over their sockets (no restart) satisfies the demanded end-to-end loop. The auditor itself concedes it is 'not a hard blocker for a first demo run,' so its own 'high' severity is slightly overstated relative to THE GOAL; medium fits better. The location, evidence, and direction are otherwise correct.
  - Refined: mentci/src/daemon.rs:59-60 (StateOwner::spawn(StateOwner::new(State::default()))) and :127-133 (on_start passthrough, no load); mentci/src/state.rs:35-52 (Default with in-memory Vec/BTreeMap and counters reset to 1); schema/sema.schema:4-7,16-22 declares the SEMA as the persistence layer the daemon stores/resumes; grep persist|sema|durable|load_state|restore|snapshot over src/ yields no implementation.

### Authorized-object updates accumulate in an in-memory Vec that is never drained or pushed

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** design-gap
- **Location:** `criome /git/github.com/LiGoldragon/criome/src/actors/subscription.rs:140-146 (publish_authorized_object_update) and field authorized_object_updates`
- **Evidence:** publish_authorized_object_update does self.authorized_object_updates.push(update) and returns a unit AuthorizedObjectPublication marker. The Vec is read only by run_due_contract_checks (subscription.rs:163-166) for absence checks; subscribers registered via OpenAuthorizedObjectSubscription are never sent anything. The actor's own doc comment (lines 17-22) admits the push side is unbuilt.
- **Impact:** Once a verdict authorizes a submission, no subscriber (including a future long-running mentci client) is notified — the authorization is only observable by a subsequent polling Observe. For the approval UI's 'verdict flows back to criome which then authorizes' to surface to the human, this fan-out is needed. Medium because request/reply observation can stand in for a first demo.
- **Direction:** Implement the push channel the comment describes: on publish, write AuthorizedObjectUpdate frames to each open subscription socket; bound/evict the in-memory Vec or move it to the store actor.
- **Verification:** The technical description is accurate. In criome/src/actors/subscription.rs the authorized_object_updates Vec is append-only: publish_authorized_object_update (lines 140-146) only pushes; open_authorized_object_subscription (lines 104-119) returns a one-time AuthorizedObjectUpdateSnapshot of the current Vec; and the Vec is read again only by run_due_contract_checks (lines 153-183) for absence checks. Registered subscribers in authorized_object_subscriptions (line 24) are never sent any live frame. The actor's own doc comment (lines 17-20) confirms the push side is unbuilt. So the auditor's "in-memory Vec never drained or pushed to subscribers" claim holds (note: run_due_contract_checks DOES drain contract_time_checks, but not authorized_object_updates, which is what the finding is about).

However, the finding's relevance to THE GOAL is overstated. I traced the actual mentci approval loop. mentci NEVER calls ObserveAuthorizedObjects on criome — grep across all of mentci src/ and tests/ shows zero references to AuthorizedObjectUpdate/ObserveAuthorizedObjects/subscriptions against criome. mentci's CriomeApprovalBridge (mentci/src/criome_bridge.rs) only does Configure and submit_verdict (SubmitAuthorizationApproval). mentci tracks its own pending_questions/decisions in mentci/src/state.rs. The verdict-flows-back-and-authorizes path is criome root.rs record_authorization_approval (lines 351-379), which on Approve calls publish_authorized_object_update and returns authorization_approval_recorded(AuthorizationEvaluated) — that meta-signal reply is how the human's verdict surfaces, and it works regardless of subscription fan-out. The AuthorizedObjectUpdate subscription is a separate downstream observation channel for other criome clients/peers (exercised only in criome's own daemon_skeleton tests as a snapshot-on-open poll), not part of the mentci approval UI. The finding itself concedes "request/reply observation can stand in for a first demo." So this is a genuine but pre-existing design gap in criome's subscription actor, NOT on the critical path of the mentci approval loop the psyche asked for. Dropping severity to low for the goal.
  - Refined: criome/src/actors/subscription.rs:140-146 (push-only), :104-119 (snapshot-on-open, no live send), :17-20 (doc comment admits push side unbuilt); criome/src/actors/root.rs:351-379 (record_authorization_approval returns authorization_approval_recorded directly — the verdict surfaces via meta reply, not subscription); mentci/src/criome_bridge.rs (only Configure + submit_verdict, never ObserveAuthorizedObjects); grep of mentci src/ + tests/ shows zero AuthorizedObjectUpdate/subscription usage against criome.

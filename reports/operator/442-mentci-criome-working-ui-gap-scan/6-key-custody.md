# 6 — Dimension: key custody & verdict signing

*Scan dimension `key-custody`. 1 confirmed / 4 raw findings (each adversarially verified against the actual code).*

## Assessment

The master key is REAL: criome/src/master_key.rs is genuine BLS12-381 (min-pk via blst) with a domain-separation tag, length-delimited preimage, sign/verify round-trip tests, and a real-but-transitional custody flow (generate-on-first-run, persist secret to a 0600 file, fsync, symlink/permission guards, startup reconciliation against the registered criome identity). The summary's claim that "real criome-owned BLS signing is NOT built" is therefore wrong as stated — the signing primitive and its custody bootstrap exist and are exercised. HOWEVER, the production gap is real and severe at a DIFFERENT seam: an approval verdict does NOT produce a signature. The mentci verdict path (meta-signal SubmitAuthorizationApproval -> root::record_authorization_approval) and the synchronous evaluate path (root::record_evaluation_decision) both terminate by publishing an UNSIGNED AuthorizedObjectUpdate pulse and returning an AuthorizationEvaluated{contract,decision} record that the signal-criome schema gives NO attestation, signature, or public-key field. The fully-built AttestationSigner (signer.rs, holding the master key) is reachable only from the separate Sign/AttestArchive/AttestChannelGrant/AttestAuthorization request paths — root's approval/evaluation handlers never call ask_signer. So a human approval becomes a decision-plus-pulse, never a cryptographic attestation. Custody at rest is plaintext (the 0600 file is not encrypted and there is no unlock flow), and mentci has ZERO key/sign/custody/unlock surface anywhere in its src or schema. The signing engine exists; the wire from "approval verdict" to "BLS attestation" does not.

### The approval/evaluation reply and AuthorizedObjectUpdate schema have no field to carry an attestation

- **Verdict:** DROPPED — verified severity `drop`, confidence `high` (auditor rated `high`)
- **Kind:** design-gap
- **Location:** `signal-criome/schema/lib.schema:260-262 (AuthorizationEvaluated) and :273-278 (AuthorizedObjectUpdate); meta-signal-criome/src/schema/lib.rs:85 (AuthorizationApprovalRecorded wraps AuthorizationEvaluated)`
- **Evidence:** AuthorizationEvaluated is {contract.ContractDigest, decision.EvaluationDecision} — no attestation/signature/public-key field. AuthorizedObjectUpdate is {object, contract, decision.EvaluationDecision, stamp.AttestedMoment} — also unsigned (EvaluationDecision::Authorized is a bare enum variant, lib.schema:204-208). meta-signal-criome's AuthorizationApprovalRecorded simply re-exports/wraps AuthorizationEvaluated, so the meta reply mentci receives structurally cannot return a signature.
- **Impact:** Even if root called the signer, there is nowhere on the wire to put the resulting Attestation, so mentci/the psyche could never receive or verify the cryptographic proof. The schema, not just the handler, blocks the production boundary.
- **Direction:** Add an Attestation (or Option<Attestation>) field to the Authorized outcome — either widen AuthorizationEvaluated, or introduce an AuthorizationAttested reply carrying signal-criome::Attestation, and thread it through meta-signal-criome::Output::authorization_approval_recorded. Per workspace 'no backward compatibility pre-production', widen the type directly.
- **Verification:** The finding's structural claims are factually accurate but its conclusion is wrong because it contradicts explicit, load-bearing criome intent. Verified facts: signal-criome/schema/lib.schema:260-263 AuthorizationEvaluated is {contract.ContractDigest, decision.EvaluationDecision} with no attestation field; :275-280 AuthorizedObjectUpdate is {object, contract, decision, stamp.AttestedMoment}, also unsigned; EvaluationDecision (:204-208) has a bare Authorized variant; meta-signal-criome/src/schema/lib.rs:85 AuthorizationApprovalRecorded(AuthorizationEvaluated) and :238 authorization_approval_recorded wraps it. So the reply mentci receives carries only the decision, no signature. However, criome/INTENT.md states this as a DELIBERATE constraint: 'Attestations are out-of-band only; content records carry no embedded proof... Content records do not carry embedded proof fields — the origin context, not proof material discipline stays inviolate.' AuthorizationEvaluated/AuthorizedObjectUpdate are precisely such content records, so the absence of an in-band attestation is intended, not a gap. The finding's prescribed fix (widen the type to embed an Attestation) would directly violate intent. The out-of-band attestation path already exists: signal-criome exposes AttestAuthorization -> AuthorizationAttestationRequest -> AttestationReceipt(Attestation) (schema :14, :35, :527-531, :561-563), wired through root.rs:198-210 to the signer actor, producing a separate Attestation record that references the authorization content. Against THE GOAL — a working approve/reject/defer loop where the verdict flows back and criome authorizes — record_authorization_approval (root.rs:351-383) already returns the EvaluationDecision and publishes the AuthorizedObjectUpdate on Approve; mentci learns the outcome from the decision on the wire. The loop does not require a cryptographic attestation returned to mentci. The finding's own premise ('even if root called the signer') is moot: the approval path intentionally never calls the signer. severityAdjusted=drop: the cited 'gap' is an intentional design decision, the recommended remediation contradicts intent, and the capability it claims is missing exists out-of-band.
  - Refined: criome/INTENT.md (constraint 'Attestations are out-of-band only; content records carry no embedded proof'); signal-criome/schema/lib.schema:14,527-531,561-563 (AttestAuthorization -> AuthorizationAttestationRequest -> AttestationReceipt out-of-band path); criome/src/actors/root.rs:198-210 (AttestAuthorization routed to signer) and :351-383 (record_authorization_approval returns EvaluationDecision + publishes AuthorizedObjectUpdate, by design never invoking the signer). Schema facts confirmed at signal-criome/schema/lib.schema:260-263 and :275-280 and meta-signal-criome/src/schema/lib.rs:85,238.

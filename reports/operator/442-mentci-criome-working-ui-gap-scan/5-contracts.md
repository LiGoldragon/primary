# 5 — Dimension: signal contract coherence (4 repos)

*Scan dimension `contracts`. 6 confirmed / 6 raw findings (each adversarially verified against the actual code).*

## Assessment

The four signal contracts are individually well-formed, freshly generated (all src/schema/lib.rs are newer than their .schema as of 2026-06-20), positional-NOTA and closed-enum disciplined, and the meta/working layers are consistent: meta-signal-criome correctly cross-imports CriomeDaemonConfiguration/AuthorizationEvaluation/AuthorizationEvaluated from signal-criome so startup and meta-reconfigure share one type identity, and meta-signal-mentci mirrors that shape. Two summary claims are now STALE and should be retired: signal-criome is NOT on "stale dot-field notation" (it lowers cleanly under positional grammar today), and "client approval mode" IS representable today — AuthorizationMode::Quorum already parks via EscalateToPsyche and is consumed by the criome root actor. The real coherence gap is the parked-then-answer LOOP: when criome's EvaluateAuthorization yields EscalateToPsyche it returns AuthorizationEvaluated synchronously and stores/streams NOTHING, so there is no wire type for an "awaiting-approval" evaluation, no observation stream that carries parked evaluations, and meta SubmitAuthorizationApproval forces the answerer to re-supply the entire AuthorizationEvaluation criome already saw. On the mentci side ApprovalSource::CriomeEscalation is a bare nullary variant carrying no criome request reference, so a presented question cannot be tied back to the criome object it must authorize. These are the contract shapes that block a real two-daemon loop; the existing happy-path test only works because one in-process test variable is handed to both daemons.

## Confirmed findings

### No wire type for a parked / awaiting-approval evaluation; EscalateToPsyche is returned and forgotten

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `criome/src/actors/root.rs:307-349 (evaluate_authorization, record_evaluation_decision); signal-criome/schema/lib.schema:48 (AuthorizationEvaluated), :198-202 (AuthorizationEvaluation)`
- **Evidence:** record_evaluation_decision only calls publish_authorized_object_update when decision == Authorized (root.rs:336). For EscalateToPsyche it just returns CriomeReply::AuthorizationEvaluated{contract,decision} (root.rs:345) and persists nothing — no StoreEvaluation message, no parked table (grep for parked/StoreEvaluation/escalat in store.rs/tables.rs/subscription.rs returns nothing). AuthorizedObjectUpdate is published only for Authorized, so the AuthorizedObjectUpdateStream never carries a parked item.
- **Impact:** A live criome cannot tell mentci 'something is parked awaiting your approval'. The submission's EscalateToPsyche verdict dies at the caller of EvaluateAuthorization; nothing in the contract lets mentci/psyche discover parked submissions independently. The end-to-end loop cannot start because the 'park' event is invisible over the wire.
- **Direction:** Add a signal-criome reply/event for a parked evaluation (e.g. AuthorizationParked carrying the AuthorizationEvaluation + a stable park slot id) and have criome persist escalated evaluations and publish them on an observation stream (extend AuthorizedObjectUpdate.decision usage to carry EscalateToPsyche, or add a dedicated ParkedAuthorizationStream). Make EscalateToPsyche a stored, observable state, not a synchronous throwaway.
- **Verification:** Confirmed by reading the actual code. record_evaluation_decision (criome/src/actors/root.rs:331-349) publishes AuthorizedObjectUpdate ONLY when decision == Authorized (line 336); for EscalateToPsyche it persists nothing and returns CriomeReply::AuthorizationEvaluated, which per schema (signal-criome/schema/lib.schema:260-263) carries only {contract, decision} — the AuthorizationEvaluation body (object + evidence) is dropped. grep for parked/StoreEvaluation/escalat in criome src returns no storage path — escalated evaluations are never persisted, never re-emitted, never observable. The AuthorizedObjectUpdateStream therefore can never carry a parked item.

The auditor's core claim is not only accurate, it is decisively proven by the integration test criome/tests/daemon_skeleton.rs:1248-1283: the test holds evaluation.clone() in a LOCAL VARIABLE across the EvaluateAuthorization->EscalateToPsyche reply (1257-1265) and replays that same in-memory value into the meta SubmitAuthorizationApproval call (1271-1272). This is exactly the single-process shortcut the goal forbids. In a real two-daemon deployment the verdict submitter (mentci) cannot do this: mentci's only inbound-criome capability is CriomeApprovalBridge::submit_verdict(evaluation, verdict) (mentci/src/criome_bridge.rs:31-43), which REQUIRES a full AuthorizationEvaluation as input but has no wire path to obtain one — mentci's daemon/state (state.rs, daemon.rs) only manage mentci's own QuestionProposal/ApprovalVerdict concepts, never criome AuthorizationEvaluations. So a parked submission is genuinely invisible over the wire; the end-to-end approval loop cannot start. One correction to the finding's wording: it understates that the EscalateToPsyche path returns AuthorizationEvaluated which strips object+evidence, so even the synchronous caller loses the data needed to later submit a verdict — the throwaway is even more total than 'returned and forgotten' implies. The direction (add a parked reply/event carrying the full AuthorizationEvaluation + stable slot id, persist escalated evaluations, publish on an observation stream) is the right fix.
  - Refined: criome/src/actors/root.rs:336 (publish gated on Authorized only) and :345-348 (EscalateToPsyche returns AuthorizationEvaluated, persists nothing); signal-criome/schema/lib.schema:260-263 (AuthorizationEvaluated = {contract, decision} only — evaluation body lost) and :580/585 (AuthorizedObjectUpdateStream only carries AuthorizedObjectUpdate, published Authorized-only); mentci/src/criome_bridge.rs:31-43 (submit_verdict requires an AuthorizationEvaluation with no wire path to obtain it); criome/tests/daemon_skeleton.rs:1257-1272 (test smuggles evaluation.clone() across the escalation reply in-process — the forbidden single-process shortcut).

### SubmitAuthorizationApproval requires the answerer to re-supply the entire AuthorizationEvaluation criome already evaluated

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** design-gap
- **Location:** `meta-signal-criome/schema/lib.schema:46-49 (AuthorizationApproval{evaluation,decision}); criome/src/actors/root.rs:351-383 (record_authorization_approval)`
- **Evidence:** AuthorizationApproval carries the full AuthorizationEvaluation by value, and record_authorization_approval re-checks evaluation.object.digest against evaluation.evidence.operation locally (root.rs:359-360) rather than looking up a parked slot. In tests/criome_bridge.rs the same in-process `evaluation` variable is passed to EvaluateAuthorization (line 325) AND to submit_verdict (line 371). The loop only closes because one process holds the object twice.
- **Impact:** Over two real sockets the mentci answerer does NOT possess the evaluation criome parked — it would have to reconstruct the full Evidence/AttestedMoment/AuthorizedObjectReference, which it has no source for. Approval is therefore not addressable by a parked-id; it is a content re-submission. This makes a genuine cross-process approve/reject impossible.
- **Direction:** Change the approval payload to reference a criome-minted parked slot id (request_slot / park token) plus the closed decision, and have criome resolve the stored evaluation by that id. The answerer should answer BY ID, never by re-supplying the evaluation.
- **Verification:** Confirmed by reading the cited code. meta-signal-criome/schema/lib.schema:46-49 defines AuthorizationApproval{evaluation.AuthorizationEvaluation, decision.AuthorizationApprovalDecision} — the approval carries the ENTIRE evaluation by value, with no slot/park-id field. criome/src/actors/root.rs:351-383 (record_authorization_approval) destructures that evaluation and, on Approve, re-checks evaluation.object.digest == evaluation.evidence.operation.object_digest() locally (root.rs:360, mirroring the same check at root.rs:308) then republishes from the supplied object — it never looks up a stored parked slot. mentci/src/criome_bridge.rs:31-44 (submit_verdict) requires the caller to hand in the full AuthorizationEvaluation; it has no lookup path and even discards the verdict's identifying content (only verdict.decision is mapped via map_decision; the question/answered_by are dropped). The test mentci/tests/criome_bridge.rs builds one `evaluation` (lines 316-320) and passes it to BOTH EvaluateAuthorization (line 325) and submit_verdict(evaluation.clone(), &verdict) (line 371) — the loop closes only because one process holds the object twice, exactly as the finding states. I verified the negative claim too: the EscalateToPsyche branch of record_evaluation_decision (root.rs:331-349) neither persists the evaluation under any id nor returns a handle — AuthorizationEvaluated carries only {contract, decision}. So over two real sockets mentci has no source for Evidence/AttestedMoment/AuthorizedObjectReference and cannot reconstruct them; mentci's own daemon only ever sees a QuestionProposal (prompt/answer/explanation/context text), not the criome evaluation. The gap is genuine and defeats the goal's required cross-process approve-by-reference. Notably the signal-criome SignalCall path already implements the correct pattern — AuthorizationPending returns a request_slot, state is stored, and observe/reject/verify resolve by request_slot (authorization.rs:104-224, store.rs LookupAuthorizationState) — so answer-by-id is idiomatic here and conspicuously missing from the meta approval path, confirming this is an unfinished design rather than an intentional choice. The finding's line references and direction (reference a criome-minted parked slot id and resolve the stored evaluation by it) are accurate.
  - Refined: meta-signal-criome/schema/lib.schema:46-49 (AuthorizationApproval{evaluation,decision}, no slot id); criome/src/actors/root.rs:355-383 (record_authorization_approval re-checks digests at :360, no slot lookup) vs the persisted EscalateToPsyche path root.rs:331-349 storing nothing keyed; mentci/src/criome_bridge.rs:31-44 (submit_verdict demands full evaluation by value); mentci/tests/criome_bridge.rs:316-320 + 325 + 371 (one in-process `evaluation` reused for both EvaluateAuthorization and submit_verdict); contrast criome/src/actors/authorization.rs:104-224 + store.rs LookupAuthorizationState (request_slot park/lookup that the meta path lacks).

### mentci ApprovalSource::CriomeEscalation carries no reference back to the criome evaluation it must authorize

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** design-gap
- **Location:** `signal-mentci/schema/lib.schema:113-117 (ApprovalSource), :137-143 (QuestionProposal), :179-183 (ApprovalVerdict); mentci/schema/sema.schema:146-150`
- **Evidence:** ApprovalSource is the nullary closed enum [CriomeEscalation AgentQuestion LocalSystemPrompt] — CriomeEscalation carries no payload. QuestionProposal/ApprovalQuestion store source + prompt + suggested answer + context only. ApprovalVerdict carries question.QuestionIdentifier + decision + answered_by — a mentci-local id, with no criome request_slot/ContractDigest/park-id field anywhere.
- **Impact:** When a parked criome submission becomes a mentci question, the question has no typed handle to the criome object. The mentci daemon, on receiving an AnswerQuestion, cannot construct the criome-facing approval addressing the right parked evaluation. The two id-spaces (mentci QuestionIdentifier vs criome request_slot) are never bridged in the contract.
- **Direction:** Give CriomeEscalation a payload carrying the criome park reference (request_slot or a CriomeAuthorizationReference newtype), and thread that reference through ApprovalQuestion so the daemon can map an AnswerQuestion verdict onto the correct criome approval submission. This is the missing join between the two contracts.
- **Verification:** VERIFIED REAL and material to the goal. I confirmed by reading the actual files (line numbers in the finding are slightly off but the cited symbols are exactly as described). signal-mentci/schema/lib.schema: ApprovalSource (lines 113-117) is the nullary closed enum [CriomeEscalation AgentQuestion LocalSystemPrompt] with no payload on CriomeEscalation; QuestionProposal (137-143) carries only source/prompt/SuggestedAnswer/explanation/Context; ApprovalQuestion (151-154) carries identifier+proposal; ApprovalVerdict (179-183) carries question(QuestionIdentifier)+decision+answered_by. Nowhere is a criome reference (AuthorizationEvaluation, ContractDigest, request_slot, or park id) threaded through the mentci working contract.

The gap is NOT implemented elsewhere — it is confirmed in the implementation too. mentci/src/state.rs State stores pending_questions: Vec<ApprovalQuestion> and decisions: Vec<ApprovalVerdict> but holds NO map from QuestionIdentifier to a criome AuthorizationEvaluation; State::answer() is purely mentci-local and never reaches criome. The CriomeApprovalBridge (src/criome_bridge.rs) takes the criome AuthorizationEvaluation as a SEPARATE argument to submit_verdict(evaluation, verdict) — the evaluation is not derived from the verdict or any stored question state. grep shows CriomeApprovalBridge/submit_verdict are used ONLY in tests/criome_bridge.rs, never wired into the daemon: the StateOwner actor does not even own a bridge. In the end-to-end test (tests/criome_bridge.rs:370-371) the join between the two id-spaces exists only because the TEST holds `evaluation` as a local variable and hands it to submit_verdict manually; the daemon has no path to reconstruct which parked criome evaluation an incoming AnswerQuestion authorizes. This is a genuine blocker for the goal's two-live-daemon loop.

ONE CORRECTION to the finding's direction: criome's meta-approval addressing is NOT request_slot. meta-signal-criome AuthorizationApproval carries the FULL AuthorizationEvaluation (evaluation + decision), and criome matches by that content, not by an AuthorizationRequestSlot (request_slot belongs to a separate signature-solicitation/quorum subsystem, signal-criome lib.schema ~370-478, not the meta approval path the bridge uses). So the correct join is to carry/store the criome AuthorizationEvaluation (or enough to reconstruct it) against the question — e.g. a CriomeEscalation(AuthorizationEvaluation) payload threaded through QuestionProposal/ApprovalQuestion and persisted in State so the daemon can call submit_verdict with the right evaluation on AnswerQuestion. The auditor's request_slot suggestion is mis-specified, but the core join gap is accurate.
  - Refined: signal-mentci/schema/lib.schema: ApprovalSource nullary enum (CriomeEscalation no payload, ~L113-117), QuestionProposal (~L137-143), ApprovalQuestion (~L151-154), ApprovalVerdict (~L179-183) — no criome handle. mentci/src/state.rs:13-26 (State fields, no question→evaluation map) and 132-155 (answer() is criome-blind). mentci/src/criome_bridge.rs:31-44 (submit_verdict takes AuthorizationEvaluation as a separate arg). mentci/tests/criome_bridge.rs:316-320, 352-371 (test holds `evaluation` locally and passes it to submit_verdict; daemon never does). meta-signal-criome/schema/lib.schema:46-49 (AuthorizationApproval addresses by full AuthorizationEvaluation, not request_slot).

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `high`)
- **Kind:** missing-feature
- **Location:** `mentci/src/criome_bridge.rs:1-53 (CriomeApprovalBridge — only configure + submit_verdict); signal-criome/schema/lib.schema:16,24 (ObserveAuthorization/ObserveAuthorizedObjects streams exist but unused by mentci)`
- **Evidence:** CriomeApprovalBridge exposes exactly configure() and submit_verdict(); there is no subscribe/observe method and no consumption of AuthorizationObservationStream or AuthorizedObjectUpdateStream. grep for subscribe/Subscribe/ObserveAuthorization/escalat in mentci/src/{criome_bridge,daemon,state}.rs finds nothing. The bridge is purely caller-driven: the caller already holds the AuthorizationEvaluation.
- **Impact:** Even once criome parks observably, mentci has no client that subscribes to learn of new parked items and inject them as PresentQuestion. The 'criome PARKS -> mentci sees it' half of the loop has no contract consumer on the mentci side; it depends on an out-of-band caller, which the live two-daemon topology lacks.
- **Direction:** Once criome exposes a parked-evaluation stream, add a bridge method that opens that observation, maps each parked evaluation into a signal-mentci PresentQuestion (CriomeEscalation with the park reference), and owns the pending-question lifecycle. This is the long-running subscriber the reports flag as deferred — but the CONTRACT must first expose the stream to subscribe to.
- **Verification:** Verified by reading the actual code. CriomeApprovalBridge (mentci/src/criome_bridge.rs:1-53) exposes exactly configure() and submit_verdict() — no subscribe/observe method, and it never consumes AuthorizationObservationStream or AuthorizedObjectUpdateStream. grep across mentci/src confirms the only "observe" is ObserveInterfaceState (mentci's own UI state, state.rs:66), nothing toward criome escalations. The mentci daemon (daemon.rs) is strictly one-frame request/reply with no long-running outbound subscriber. The integration test tests/criome_bridge.rs:254-406 is itself the out-of-band bridge: after criome returns EscalateToPsyche synchronously (root.rs:307-329), the TEST hand-builds question_proposal() and pushes PresentQuestion into mentci, then feeds the same evaluation (which it already held in scope) back via submit_verdict — so submit_verdict is caller-driven and requires the full AuthorizationEvaluation up front, exactly as the finding states. I would strengthen the finding: the gap is deeper than "mentci doesn't subscribe." On the EvaluateAuthorization->EscalateToPsyche path, criome PERSISTS NOTHING and publishes to no stream (record_evaluation_decision at root.rs:331-349 only publishes AuthorizedObjectUpdate when decision==Authorized; escalations are fire-and-forget). There is no parked-escalation store and zero "mentci"/"PresentQuestion"/"park" reference anywhere in criome/src. The ObserveAuthorization/AuthorizationObservationStream that does exist serves the separate signature-quorum flow (AuthorizeSignalCall -> AuthorizationPending with request_slot), not the escalation flow the approval UI uses — so even the named streams are the wrong ones. The auditor's "the CONTRACT must first expose the stream to subscribe to" caveat is correct and the heart of the gap. This directly blocks the goal's "criome PARKS -> mentci sees it" half over two live daemons.
  - Refined: mentci/src/criome_bridge.rs:22-44 (only configure + submit_verdict, caller supplies AuthorizationEvaluation); mentci/src/daemon.rs:98-114 (one-shot request/reply, no outbound subscriber); criome/src/actors/root.rs:307-349 (EscalateToPsyche returned synchronously, nothing persisted/published — no parked store); criome/src grep: no mentci/park/PresentQuestion; tests/criome_bridge.rs:336-375 (test, not a daemon client, manually bridges escalation->PresentQuestion->submit_verdict)

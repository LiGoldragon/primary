# 8 — Dimension: ugly code / discipline sweep

*Scan dimension `ugly`. 6 confirmed / 6 raw findings (each adversarially verified against the actual code).*

## Assessment

Both daemons are real, compile-shaped, actor-structured code over signal-frame Unix sockets — the criome side genuinely speaks two sockets (working + meta) and persists through a sema engine, and mentci genuinely owns UI state behind a kameo actor. So the discipline problems here are aesthetic/structural, not blockers to the loop: none of them prevent the mentci-CLI->daemon->live-criome path from running. The most-cited workspace override — "every fn is a method/associated-fn; free functions forbidden" — is violated in at least five places (synthetic_exchange, rejection, actor_reply, authorization_store_rejection, active_status, the last of which is also dead code). The sharper structural smells are: synchronous sema-engine disk IO performed directly inside async actor message handlers (no-blocking-IO-in-actor-handlers), a stringly-typed actor-error boundary (ActorCall(String) plus format!("{other:?}") frame errors) that throws away typed replies, the length-prefixed frame read duplicated verbatim three times, and record_authorization_approval re-deriving its EvaluationDecision in a way that duplicates evaluate_authorization rather than collapsing into the normal evaluation path. These are quality debts the psyche's beauty criterion would flag, not gaps in the working surface.

## Confirmed findings

### Free functions in both crates violate the method-only override

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** ugly-code
- **Location:** `criome/src/transport.rs:14 (synthetic_exchange); criome/src/actors/mod.rs:26,30 (rejection, actor_reply); criome/src/actors/authorization.rs:293 (authorization_store_rejection); criome/src/actors/store.rs:695 (active_status)`
- **Evidence:** Five bare `fn`/`pub fn` at module scope, none inside an impl: `fn synthetic_exchange() -> ExchangeIdentifier`, `pub fn rejection(reason: RejectionReason) -> CriomeReply`, `pub fn actor_reply(reply: CriomeReply) -> CriomeActorReply`, `fn authorization_store_rejection(error: crate::Error) -> CriomeReply`, and `pub fn active_status(identity: &StoredIdentity) -> bool`. The grep `^pub fn |^fn ` returns exactly these plus the legitimate `fn main`s.
- **Impact:** Does not block the loop, but each is a placement decision the workspace forbids: `rejection`/`actor_reply` should be `CriomeReply::reject(...)` / `CriomeActorReply::from(reply)`; `synthetic_exchange` belongs on the codec or an ExchangeIdentifier constructor; `authorization_store_rejection` is a `From<Error> for CriomeReply`; `active_status` should be `StoredIdentity::is_active(&self)` and is in fact dead (no caller in the crate).
- **Direction:** Move each onto its owning noun: `impl CriomeReply { fn reject }`, `impl From<CriomeReply> for CriomeActorReply`, `impl ExchangeIdentifier { fn synthetic }`, `impl From<crate::Error> for CriomeReply`, `impl StoredIdentity { fn is_active }`; delete `active_status` outright since nothing calls it.
- **Verification:** Confirmed by reading the files. `grep -E '^(pub )?fn '` over criome/src returns exactly the five cited functions plus the three `fn main`s — no others. Each description holds: `rejection` (mod.rs:26) wraps `CriomeReply::Rejection(Rejection::new(...))` and is called ~28 times across root/authorization/subscription/signer/registry; `actor_reply` (mod.rs:30) is a one-line wrapper over `CriomeActorReply::new`, called ~24 times — both are constructor-shaped free functions that should live on their owning nouns (`impl CriomeReply`, `impl From<CriomeReply> for CriomeActorReply`). `synthetic_exchange` (transport.rs:14) builds an `ExchangeIdentifier` and is called 4x — belongs on a constructor. `authorization_store_rejection` (authorization.rs:293) maps `crate::Error -> CriomeReply` (used 2x) — the canonical `From<Error>` case. `active_status` (store.rs:695) is `pub fn active_status(identity: &StoredIdentity) -> bool` returning `identity.status() == PrincipalStatus::Active`; grep for both `active_status` and `is_active` across the whole repo (src + tests) shows ZERO callers — it is genuinely dead and should be deleted (or be `StoredIdentity::is_active`). All five genuinely violate the method-only hard override in AGENTS.md. The one correction to the finding's framing: relative to THE GOAL (a live two-daemon approval loop over Unix sockets), this is pure code-discipline, not a functional gap — the auditor itself says it does not block the loop. For a goal-scoped severity that makes it low, not medium; it remains a valid finding within the ugly-code-discipline dimension.
  - Refined: criome/src/actors/mod.rs:26 (rejection), :30 (actor_reply); criome/src/transport.rs:14 (synthetic_exchange, called at lines 62,89,141,172); criome/src/actors/authorization.rs:293 (authorization_store_rejection, called at 116,138); criome/src/actors/store.rs:695 (active_status — grep confirms zero callers in src or tests, dead code)

### Synchronous sema-engine disk IO runs inside async actor message handlers

- **Verdict:** CONFIRMED — verified severity `medium`, confidence `high` (auditor rated `high`)
- **Kind:** ugly-code
- **Location:** `criome/src/actors/store.rs:515-693 (every StoreKernel Message handler) -> src/tables.rs:520-527 (engine.mutate_keyed / assert_keyed) -> sema-engine `pub fn commit``
- **Evidence:** StoreKernel `async fn handle` bodies call `self.store_identity(...)`, `self.create_authorization_state(...)`, etc., which call `self.tables.put_*` / `self.engine.mutate_keyed(...)` / `assert_keyed(...)`. The engine API is synchronous (`pub fn commit<RecordValue>(...)` in sema-engine engine.rs, no `async`), so every persisting message handler performs blocking file IO on the actor's executor thread with no `spawn_blocking`/`block_in_place`.
- **Impact:** Directly violates no-blocking-IO-inside-actor-handlers. With the daemon's single shared tokio runtime, a slow fsync in StoreKernel stalls the whole actor system, including the authorization/evaluation handlers that the approval loop drives. It works today only because traffic is single-shot and serial.
- **Direction:** Either run the store engine off-runtime (a dedicated blocking thread the actor messages over a channel, or `tokio::task::spawn_blocking` around the commit), or make StoreKernel a non-async/blocking actor explicitly isolated from the request-serving runtime so the discipline is satisfied by construction.
- **Verification:** Every technical claim holds up against the actual code. In criome/src/actors/store.rs every persisting Message handler (StoreIdentity 515-526, StoreRevocation, StoreAttestation, StoreAuthorizationState, CreateAuthorizationState 593-603, StoreContract, StoreSignatureSolicitation, StoreSignatureSubmission 682-693) is `async fn handle` that synchronously calls store methods (e.g. store_signature_submission -> tables.put_*). Those land in tables.rs upsert (502-529) which calls engine.mutate_keyed (523) / assert_keyed (526). In sema-engine/src/engine.rs commit (669, no async), mutate_keyed (499), assert_keyed (399) are all synchronous `pub fn`, and commit drives `self.storage.write(|transaction| ...)` against a redb-backed store (engine.rs:78 comment, :132/:150/:201 writes) — i.e. real blocking disk IO. The daemon runs one shared tokio runtime (daemon.rs:106 `Runtime::new()`) on which all actors including StoreKernel are spawned (root.rs spawn calls), and there is no spawn_blocking/block_in_place anywhere in store.rs or tables.rs (grep returned nothing). So the finding correctly identifies blocking sema-engine disk IO inside async kameo handlers on the request-serving runtime — a genuine violation of the no-blocking-in-actor-handlers discipline (skills/actor-systems.md / rust-discipline). Correction to weight, not facts: this is a discipline/latent-scalability defect, not a functional blocker. The auditor itself concedes 'It works today only because traffic is single-shot and serial.' The psyche's goal is a WORKING end-to-end approval loop over two live daemons; the approval loop runs correctly with this code — a slow fsync would only add latency/serialization under concurrent load, which the single-shot park->approve->authorize flow does not exercise. So it is real and worth fixing, but high overrates it relative to the goal; medium is the honest severity for an ugly-code-dimension finding that does not impede the working end state.
  - Refined: criome/src/actors/store.rs:515-693 (async handlers -> store_* methods); criome/src/tables.rs:502-529 (upsert -> engine.mutate_keyed/assert_keyed); sema-engine/src/engine.rs:399,499,669 (synchronous assert_keyed/mutate_keyed/commit) with self.storage.write transactions at engine.rs:132/150/201 over redb; criome/src/daemon.rs:106 single shared tokio Runtime; no spawn_blocking/block_in_place in store.rs or tables.rs (grep empty).

### Actor-error boundary is stringly-typed: ActorCall(String) plus format!("{other:?}") frame errors

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** ugly-code
- **Location:** `criome/src/error.rs Error::ActorCall(String) (used root.rs:150,165,240,252,264,276,288; authorization.rs:240 etc.); criome/src/transport.rs:54,73,78,82,100,129,156,160 (UnexpectedSignalFrame { got: format!("{other:?}") }); mentci/src/error.rs:59 Error::ActorCall(String); mentci/src/daemon.rs:107`
- **Evidence:** Every actor `.ask(...)` mapping is `.map_err(|error| crate::Error::ActorCall(error.to_string()))`, collapsing a typed kameo SendError into a String. The frame codec turns any unexpected `FrameBody`/`SubReply` into `Error::UnexpectedSignalFrame { got: format!("{other:?}") }` — the typed variant is Debug-formatted into a string the moment it is rejected.
- **Impact:** Contradicts typed-per-crate-Error / no-stringly-typed-dispatch: a caller can never match on why an actor call failed (mailbox closed vs. panic) nor on which unexpected frame arrived. Information is destroyed at exactly the boundary where the approval loop most needs to distinguish 'criome rejected' from 'transport broke'.
- **Direction:** Give `ActorCall` a typed payload (wrap the kameo error type, or a small closed enum of MailboxClosed/Panicked/Timeout). For `UnexpectedSignalFrame`, carry the unexpected typed value (or a closed FrameKind enum) instead of its Debug string.
- **Verification:** Confirmed by reading the actual files. criome/src/error.rs:8 defines `ActorCall(String)` and :41 `UnexpectedSignalFrame { got: String }`, exactly as cited. Every actor `.ask(...)` site collapses the typed kameo `SendError` via `.map_err(|error| Error::ActorCall(error.to_string()))` (criome daemon.rs:150,167; authorization.rs:240,252,264,276,288; registry.rs:172,181; root.rs:165; signer.rs:187; store.rs:294,330; mentci daemon.rs:107). kameo's SendError (kameo-0.20.0/src/error.rs:89) is a real structured enum — ActorNotRunning / ActorStopped / MailboxFull / HandlerError / Timeout — so the cause distinction genuinely exists and is being destroyed. transport.rs read paths (54,73,77,81,129,152,156,160) Debug-format the rejected FrameBody/SubReply/Reply into the `got` string. So the discipline violation (stringly-typed actor-error and frame boundary, contra typed-per-crate-Error) is accurate. Two corrections to the finding: (1) line-number attribution is imprecise — the auditor lists `root.rs:150,165,240,252,264,276,288` but 150/240/252/264/276/288 are actually in daemon.rs (150) and authorization.rs (240-288); the pattern itself is pervasive and the claim stands. (2) The goal-relevance is real but narrower than 'medium' implies: there is a concrete consequence — authorization.rs:293-298 `authorization_store_rejection` matches only AuthorizationReplayAttempted and funnels every other Error, including a store-mailbox-closed/panicked ActorCall, into `_error => MalformedRequest`, so an actor/transport failure is reported to the client as a malformed request, conflating exactly the 'criome rejected vs. transport broke' distinction the approval loop wants. But this is a diagnosis/quality defect; it does not break the end-to-end two-daemon socket loop, which still runs. For an ugly-code sweep against a working-UI goal it is a legitimate but low-impact finding.
  - Refined: criome/src/error.rs:8 `ActorCall(String)`, :41 `UnexpectedSignalFrame { got: String }`; collapse sites authorization.rs:240/252/264/276/288, registry.rs:172/181, signer.rs:187, root.rs:165, daemon.rs:150/167, mentci daemon.rs:107; transport.rs:54/73/77/81/129/152/156/160 `format!("{other:?}")`; structured source type kameo-0.20.0/src/error.rs:89 SendError enum; goal-relevant collapse visible at criome/src/actors/authorization.rs:293-298 (ActorCall falls into `_error => MalformedRequest`).

### Length-prefixed frame read duplicated verbatim three times

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** ugly-code
- **Location:** `criome/src/transport.rs:95-109 (CriomeFrameCodec::read_frame) and 178-192 (CriomeMetaFrameCodec::read_frame); mentci/src/frame_codec.rs:25-45 (FrameCodec::read_mentci_frame)`
- **Evidence:** Three near-identical bodies: read 4-byte big-endian prefix, bounds-check against maximum_frame_bytes, allocate `4 + length`, copy prefix, resize, `read_exact` the rest, then `decode_length_prefixed`. CriomeFrameCodec and CriomeMetaFrameCodec differ only in the concrete Frame type; mentci's copy differs only in the error type it returns on overflow.
- **Impact:** Repetition that should resolve into one named pattern. The two criome codecs are identical wrappers around different `*::decode_length_prefixed` functions — the maximum-bytes guard and prefix framing want to be one generic LengthPrefixedReader parameterised over the frame type, with the per-codec part being only the typed decode.
- **Direction:** Extract a single `LengthPrefixedFrameReader` (or a trait method on a shared codec type) generic over `Frame: DecodeLengthPrefixed`; the two criome codecs and the mentci codec become thin newtypes selecting the frame type and max size.
- **Verification:** The code description is accurate. I read all three bodies: CriomeFrameCodec::read_frame (transport.rs:95-109), CriomeMetaFrameCodec::read_frame (transport.rs:178-192), and FrameCodec::read_mentci_frame (frame_codec.rs:25-45). All three perform the identical sequence: read 4-byte big-endian prefix, bounds-check length against maximum_frame_bytes, allocate Vec with_capacity(4+length), copy prefix, resize, read_exact the remainder, then call <FrameType>::decode_length_prefixed. The two criome codecs differ only in the concrete Frame type; the mentci copy differs in the overflow error it constructs (signal_frame::FrameError::LengthMismatch vs Error::UnexpectedSignalFrame). One nuance the finding understates: signal-frame already exposes decode_length_prefixed (frame.rs:271/302) which itself calls strip_length_prefix, so the manual re-prefix-then-decode dance is doubly wasteful — the codecs hand-read the prefix only to prepend it back and let decode_length_prefixed strip it again. A LengthPrefixedFrameReader generic over the frame type is a sound direction, though it requires introducing a DecodeLengthPrefixed trait (currently decode_length_prefixed is an inherent macro/impl method, not a trait, so no trait bound exists to dispatch on yet). The finding is a genuine ugly-code/duplication observation under the workspace anti-duplication discipline. However it is NON-functional: all three paths work correctly, so it has no bearing on whether the end-to-end approval loop runs against two live daemons. For THE GOAL (a working loop) medium overrates it; it is a low-severity cleanup.
  - Refined: criome/src/transport.rs:95-109 and :178-192 are byte-identical apart from CriomeFrame vs CriomeMetaFrame and the decode_length_prefixed target; mentci/src/frame_codec.rs:29-44 is the same algorithm with FrameError::LengthMismatch on overflow. signal-frame/src/frame.rs:271 shows decode_length_prefixed already strips the prefix it is fed, confirming the manual prefix reassembly is redundant work.

### record_authorization_approval re-derives the verdict, duplicating evaluate_authorization's normal path

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** ugly-code
- **Location:** `criome/src/actors/root.rs:351-383 (record_authorization_approval) vs 307-349 (evaluate_authorization / record_evaluation_decision)`
- **Evidence:** record_authorization_approval re-checks `evaluation.object.digest == evaluation.evidence.operation.object_digest()` (the same digest guard evaluate_authorization does at 308), and on Approve manually rebuilds the `AuthorizedObjectUpdate { object, contract, decision: Authorized, stamp }` literal that record_evaluation_decision already builds at 337-342. The decision is then derived through a separate match producing `EvaluationDecision::Rejected(AgreementMissing)` for Approve+digest-mismatch and Reject, and `EscalateToPsyche` for Defer.
- **Impact:** Two parallel code paths compute and publish an authorized-object update from an AuthorizationEvaluation, with the digest-guard and the update-literal duplicated. A future change to what 'authorized' publishes (or to the digest check) must be made in both, and the approval verdict's mapping (Reject-as-AgreementMissing) is buried in an inline match rather than a named ApprovalDecision -> EvaluationDecision conversion.
- **Direction:** Route the approval through the same `record_evaluation_decision` after mapping the verdict to an EvaluationDecision via a typed conversion (`impl From<AuthorizationApprovalDecision> for EvaluationDecision`, or a method that also takes the digest-match result), so the publish-on-authorized step exists once.
- **Verification:** Verified by reading criome/src/actors/root.rs:307-383. The finding's claims hold exactly. record_authorization_approval (351-383) is the handler for SubmitAuthorizationApproval (dispatched at 285-286) — the verdict-from-mentci/psyche flowing back to criome, squarely on the goal's critical path. It re-checks the same digest guard evaluate_authorization uses (line 360 vs 308), and on Approve manually rebuilds the identical `AuthorizedObjectUpdate { object, contract, decision: Authorized, stamp }` literal plus publish_authorized_object_update call that record_evaluation_decision already performs (362-368 vs 337-343). The verdict mapping (Approve/Reject -> Rejected(AgreementMissing), Defer -> EscalateToPsyche) lives in an inline match rather than a named AuthorizationApprovalDecision -> EvaluationDecision conversion. Two real differences exist that the finding under-states: the return wrappers differ (CriomeReply::AuthorizationEvaluated vs meta_signal_criome::Output::authorization_approval_recorded), and the approval path produces EscalateToPsyche for Defer, which the eval path never emits — so a clean refactor would map the verdict to an EvaluationDecision (via a typed From) and route through a shared publish step, but cannot trivially reuse record_evaluation_decision wholesale because of the differing reply type. The duplication and the buried mapping are genuine. This is a maintainability/discipline issue, not a correctness bug; the loop works. Medium over-rates an ugly-code duplicated-literal finding; low is appropriate.
  - Refined: criome/src/actors/root.rs:308 & :360 (duplicated digest guard); :337-343 & :362-368 (duplicated AuthorizedObjectUpdate literal + publish_authorized_object_update call); :371-376 (inline Approve/Reject->AgreementMissing, Defer->EscalateToPsyche mapping that should be a typed From<AuthorizationApprovalDecision> for EvaluationDecision). Handler dispatched at :285-286 for meta_signal_criome::Input::SubmitAuthorizationApproval.

### key_registry rebuilds the registry with an N+1 await loop over identities

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `low`)
- **Kind:** ugly-code
- **Location:** `criome/src/actors/root.rs:498-525 (CriomeRoot::key_registry)`
- **Evidence:** After `registry.ask(ReadIdentitySnapshot)` returns a full IdentitySnapshot, the code loops `for identity in snapshot.into_identities()` and issues one further `registry.ask(ResolveIdentity::new(identity.identity))` per identity to fetch each public key, building KeyRegistry one admit at a time.
- **Impact:** This runs on the Quorum evaluation hot path (evaluate_authorization at 318 calls key_registry on every non-AutoApprove submission), so each approval evaluation does 1 + N round-trips to the registry actor. The snapshot it already holds should carry enough to build the registry directly; the per-identity resolve is repetition the snapshot was meant to eliminate.
- **Direction:** Make ReadIdentitySnapshot (or a dedicated ReadKeyRegistry message) return identity+public_key pairs so KeyRegistry is built from one reply, removing the per-identity ask loop.
- **Verification:** Confirmed by reading the actual code. CriomeRoot::key_registry (root.rs:498-525) does exactly what the finding describes: it asks registry::ReadIdentitySnapshot, then loops over snapshot.into_identities() issuing one registry.ask(ResolveIdentity::new(identity.identity)) per identity, calling key_registry.admit(...) one pair at a time — a 1+N round-trip pattern. The reason it must re-resolve is real: IdentityRegistry::snapshot() (registry.rs:152-165) builds IdentitySnapshot from IdentityReceipt records carrying only identity+status, NOT the public key, so the snapshot alone cannot build KeyRegistry. The auditor's direction is sound and there is an even cleaner in-crate fix available: registry already exposes snapshot_records() returning Vec<StoredIdentity> (which carry public_key() via stored.public_key()), so a dedicated message returning StoredIdentity records (or identity+public_key pairs) would build KeyRegistry from one reply. One scope correction to the impact claim: key_registry() runs on the non-AutoApprove evaluate_authorization path (root.rs:318), which is indeed the client-approval-mode evaluation path the goal cares about — so the auditor is right that it is on that path. But these are all in-process kameo actor asks against a small host-identity registry, the loop is functionally correct, and it does not block the end-to-end two-daemon approval loop from running. It is a genuine but low-severity code-quality/efficiency wart, exactly as the auditor rated it.
  - Refined: criome/src/actors/root.rs:498-525 (key_registry N+1 ResolveIdentity loop) confirmed; root cause at criome/src/actors/registry.rs:152-165 where snapshot() maps StoredIdentity into IdentityReceipt{identity,status} dropping public_key; cleaner fix path: snapshot_records() (registry.rs:176-183) already returns Vec<StoredIdentity> with public_key().

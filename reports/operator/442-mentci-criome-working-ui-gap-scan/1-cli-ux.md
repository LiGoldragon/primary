# 1 — Dimension: mentci CLI as UI

*Scan dimension `cli-ux`. 6 confirmed / 7 raw findings (each adversarially verified against the actual code).*

## Assessment

The `mentci` CLI (mentci/src/client.rs) is a pure one-shot NOTA-frame pipe, not a usable human-approval UI, and the gap to the psyche's goal is the dominant gap on this dimension. It has NO subcommands or verbs: it takes exactly one positional argument that must be a complete `MentciRequest` (the full signal-mentci `Input` enum) authored as inline NOTA, a `.nota` file, or a binary frame — there is no `list`, `watch`, `approve`, `reject`, or `defer` surface. To answer a question the human must hand-author a NOTA `AnswerQuestion(ApprovalVerdict(...))` record carrying the exact minted `QuestionIdentifier`, which they can only learn by separately hand-authoring an `ObserveInterfaceState` request and then decoding the raw binary reply the CLI dumps to stdout (`reply.encode_length_prefixed()` -> `stdout`, client.rs:55-56) — the output is not human-readable at all. There is NO interactive mode and NO watch/subscribe: each invocation opens one socket, sends one frame, reads one reply, exits (client.rs:49-58); the daemon is plain request/reply with no event push despite a `MentciEvent` type existing in the frame definition. Critically, the whole criome side of the loop is absent from the runtime: `CriomeApprovalBridge` (criome_bridge.rs) is referenced only in lib.rs and the in-process test, never by the daemon or CLI; the daemon never connects to criome, never auto-subscribes to escalations, and the configured `home_criome_socket_path()` (configuration.rs:56) is dead. The end-to-end loop the psyche wants exists only as the single-process test mentci/tests/criome_bridge.rs, which manually wires criome-evaluate -> mentci-present -> mentci-answer -> bridge-submit in test code; nothing in the shipped binaries performs that gluing. Reports' claim that "criome meta Configure was implemented" is true at the bridge/library level, but it is not reachable from the CLI.

## Confirmed findings

### CLI has no approval verbs — only a single raw NOTA-frame argument

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/client.rs:39-65 (request_frame/input_argument)`
- **Evidence:** `input_argument` matches `[argument] => Ok(...)` and errors otherwise (client.rs:60-65); `request_frame` treats that single argument as either a file path or an inline NOTA string parsed straight into a full `MentciRequest` via `NotaSource::new(source).parse::<MentciRequest>()` (client.rs:80-86). There is no argv parsing for subcommands — no `list`, `watch`, `approve <id>`, `reject <id>`, `defer <id>`. main.rs:4 just calls `ClientCommand::from_environment().run()`.
- **Impact:** A human cannot drive approve/reject/defer from the CLI without hand-authoring a complete `AnswerQuestion(ApprovalVerdict(question ... decision ... answered_by ...))` NOTA record by hand, including the exact daemon-minted QuestionIdentifier. This is the single largest barrier between the current code and the 'working mentci approval UI' the psyche asked for.
- **Direction:** Add a real verb layer to ClientCommand (e.g. `mentci list`, `mentci watch`, `mentci approve <question-id>`, `mentci reject <id>`, `mentci defer <id>`) that constructs the corresponding signal-mentci request internally, so the human supplies an id + decision, never NOTA.
- **Verification:** Confirmed by reading the actual code. mentci/src/client.rs:60-65 (input_argument) matches exactly one argument and errors otherwise (Error::ClientArgumentCount); request_frame (39-47) treats that single argument as a file path or inline NOTA, and request_frame_from_nota (80-86) parses it straight into a full MentciRequest via NotaSource::new(source).parse::<MentciRequest>(). main.rs:3-4 just calls ClientCommand::from_environment().run() with no argv dispatch. A grep across src/ for approve/reject/defer/list/watch/subcommand finds only daemon/state/bridge internals — no client-side verb layer exists. The signal-mentci schema confirms the human-facing verdict is AnswerQuestion(ApprovalVerdict{question: QuestionIdentifier, decision: ApprovalDecision, answered_by: SubscriberName}), and the schema explicitly states the DAEMON mints the QuestionIdentifier (never the client). So a human cannot approve/reject/defer without hand-authoring that complete NOTA record including the daemon-minted id. The cited line numbers and the impact statement are accurate. This is a real, central blocker to the goal's 'working mentci approval UI' — without a verb layer the CLI is unusable as the human approval surface.
  - Refined: mentci/src/client.rs:60-65 (input_argument matches [argument] only), client.rs:39-47 + 80-86 (request_frame parses the single arg into a full MentciRequest), main.rs:3-4 (no argv dispatch); signal-mentci/schema/nexus.schema ApprovalVerdict + the schema note that the daemon mints QuestionIdentifier; grep of src/ shows no client verb layer.

### CLI output is raw binary, not human-readable — cannot observe pending questions

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** design-gap
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/client.rs:54-56 (run)`
- **Evidence:** `run` does `let bytes = reply.encode_length_prefixed()?; io::stdout().lock().write_all(&bytes)?;` — it writes the length-prefixed binary reply frame straight to stdout. There is no rendering of `InterfaceState` / `PendingQuestionsView` / `ApprovalQuestion` into readable text. The only way to 'see' pending questions is to send a hand-authored `ObserveInterfaceState` request and then decode the binary blob with a separate tool.
- **Impact:** Even though the daemon state (state.rs:208-227 project / PendingQuestionsView) can return the pending-question list, the CLI gives the human an opaque byte stream. A human literally cannot read what is parked awaiting approval, so they cannot know which id to approve. The observe-then-answer cycle the UI needs is unusable.
- **Direction:** Decode the MentciReply and pretty-print it: render PendingQuestionsView as a numbered list (id, source, prompt, suggested answer, context), render VerdictAccepted/Rejection as a one-line status. Keep a `--raw`/binary mode only for machine pipelines.
- **Verification:** Confirmed by reading the actual files. mentci/src/client.rs `run()` (lines 49-58) connects, writes the request frame, reads the reply, then does `let bytes = reply.encode_length_prefixed()?; io::stdout().lock().write_all(&bytes)?;` — it writes the raw length-prefixed binary reply frame straight to stdout. There is no Display impl, no println, no NOTA-text emission, and no rendering of PendingQuestionsView/ApprovalQuestion anywhere in the client path (src/main.rs only adds eprintln for errors; there is no other bin or render module). The daemon side genuinely can return the data: state.rs:208-227 `project()` builds InterfaceProjection::PendingQuestionsProjection(PendingQuestionsView::from_questions(...)), and signal-mentci defines PendingQuestionsView (schema/lib.rs:452) over ApprovalQuestion. So the human's only way to read parked questions via the CLI is to capture an opaque binary blob and decode it with a separate tool — unusable for the observe-then-approve loop the goal requires. One correction to the auditor's 'direction': these signal types already implement NotaEncode/to_nota() (signal-mentci/src/schema/lib.rs:434), so the idiomatic fix per workspace NOTA conventions is to decode the reply and emit its to_nota() text (optionally a numbered list), keeping the binary frame behind a machine/--raw mode — not necessarily a hand-rolled pretty-printer. The gap itself (no human-readable output) is real and directly blocks the goal.
  - Refined: mentci/src/client.rs:54-56 (run): reply.encode_length_prefixed() + stdout.write_all — raw binary, no rendering. No Display/println/to_nota call in client path (src/main.rs, src/bin/ confirm). Data is available: state.rs:219-221 PendingQuestionsProjection; signal-mentci/src/schema/lib.rs:452 PendingQuestionsView and :434 to_nota() exist but are never invoked by the CLI.

### No interactive or watch/subscribe mode — every action is a fresh exit-after-one-reply process

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `blocker`)
- **Kind:** missing-feature
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/client.rs:49-58 (run); daemon.rs:98-114 (handle_connection, single request/reply)`
- **Evidence:** `run` connects, writes one frame, reads exactly one reply, returns (client.rs:52-57). The daemon's `handle_connection` reads one frame and writes one reply per connection (daemon.rs:99-113); `serve_forever` loops over connections but never pushes unsolicited frames. A `MentciEvent` type exists in the frame definition (signal-mentci/src/lib.rs:16 `MentciFrame = StreamingFrame<Input, Output, MentciEvent>`) but the daemon never emits one and the client never reads a stream of them. The `ObserveInterfaceState` path mints a subscription token and stores interest (state.rs:122-130) but nothing ever delivers updates to that subscriber.
- **Impact:** The human cannot 'watch' for new parked submissions/criome escalations; they must repeatedly re-run a one-shot observe command and re-decode binary. There is no live approval queue. Subscriptions are recorded but inert — fan-out is unbuilt, so the 'new question arrives -> human is notified' half of the loop does not exist.
- **Direction:** Add a long-running `mentci watch` client that holds the socket open and renders pushed InterfaceState revisions, backed by daemon-side subscription fan-out (deliver fresh InterfaceState to each registered InterfaceInterest on every bump_revision).
- **Verification:** Every cited fact holds up against the actual code. client.rs run() (49-58) connects, writes one frame, reads exactly one reply, returns — a pure one-shot client; main.rs confirms the only client entrypoint is this exit-after-one-reply run(). daemon.rs handle_connection (98-114) reads one request and writes one reply per connection; serve_forever (81-86) accepts connections in a loop but never emits an unsolicited frame. signal-mentci/src/lib.rs:16 declares MentciFrame = StreamingFrame<Input, Output, MentciEvent>, and a workspace grep confirms MentciEvent is never constructed or written anywhere in src/. state.rs observe() (122-130) mints a token and stores InterfaceInterest in self.subscriptions, but the map is only ever read by retract() (185-193); bump_revision() (237-240) just increments counters and never fans out to subscribers. So subscriptions are genuinely inert and the daemon->client push path does not exist. This matters to the goal: the 'new parked submission -> human is notified' half of the loop has no live delivery; the human can only re-run one-shot observe and re-decode binary. I downgrade severity from blocker to high: the goal's hard requirement is that the loop RUN over two live sockets, which a repeated-polling observe client can technically satisfy without watch mode, so the missing push degrades the UX to manual polling rather than outright preventing the loop from running. The auditor's title/evidence/direction are otherwise correct.
  - Refined: mentci/src/client.rs:49-58 (one-shot run); mentci/src/main.rs (sole client entry is run()); mentci/src/daemon.rs:81-114 (serve_forever loop + single request/reply handle_connection, no unsolicited emit); signal-mentci/src/lib.rs:16 (MentciEvent declared, never emitted — grep-confirmed); mentci/src/state.rs:122-130 (observe stores InterfaceInterest) and 185-193/237-240 (subscriptions only read by retract; bump_revision never fans out)

### Daemon never connects to criome — the approval loop is not wired into any running binary

- **Verdict:** CONFIRMED — verified severity `blocker`, confidence `high` (auditor rated `blocker`)
- **Kind:** unported
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/criome_bridge.rs (whole) vs daemon.rs; configuration.rs:56-58 (home_criome_socket_path)`
- **Evidence:** `grep -rn CriomeApprovalBridge src/` finds it only in lib.rs:10 (module decl) and criome_bridge.rs itself — never in daemon.rs, client.rs, or command.rs. The daemon (daemon.rs:46-114) only owns a `State` actor and a mentci socket; it never opens criome's meta or ordinary socket and never auto-subscribes to criome escalations. `DaemonConfiguration::home_criome_socket_path()` (configuration.rs:56) is dead — no caller. The only place the bridge runs is mentci/tests/criome_bridge.rs, where the test itself calls `CriomeApprovalBridge::new(&criome_meta_socket).submit_verdict(...)` after manually evaluating on criome and presenting on mentci.
- **Impact:** There is no running-binary path where a criome escalation becomes a parked mentci question, nor where an accepted mentci verdict is forwarded to criome's meta socket. The entire cross-daemon loop the psyche wants exists only as in-process test orchestration. Approving a question via the CLI updates mentci's in-memory state and removes it from the queue (state.rs:140-154) but sends NOTHING to criome.
- **Direction:** Wire CriomeApprovalBridge into the daemon: on configured home_criome_socket, subscribe to criome escalations and turn each into a PresentQuestion; on AnswerQuestion acceptance, look up the originating AuthorizationEvaluation and call bridge.submit_verdict so the verdict reaches criome's meta socket. The CLI then only needs the approve verb; the daemon owns the criome glue.
- **Verification:** Confirmed by reading the cited files. CriomeApprovalBridge (src/criome_bridge.rs) is wired into NO running binary: grep shows it only in lib.rs:10 (mod decl), criome_bridge.rs, and the two test files. daemon.rs (BoundDaemon/StateOwner/serve_forever) owns only a State actor and the mentci UnixListener; it never opens criome's socket, never subscribes to escalations, never constructs the bridge. command.rs::run just calls Daemon::run. configuration.rs:56 home_criome_socket_path() has no production caller (only tests/configuration.rs:49). State::answer (state.rs:132-155) on an accepted verdict removes the pending question and pushes to `decisions`, returns VerdictAccepted, and sends NOTHING to criome — and State holds no AuthorizationEvaluation, so forwarding a verdict to criome is structurally impossible in the binary. The only place submit_verdict runs is tests/criome_bridge.rs:370, which orchestrates both daemons in-process via thread::scope and manually calls serve_next_meta()/submit_verdict. So the entire cross-daemon loop exists only as in-process test orchestration — a real blocker against the goal of an end-to-end loop over two live sockets. Minor correction: the test configures criome with AuthorizationMode::AutoApprove, not park/client-approval mode, which if anything strengthens the finding (the escalation→PresentQuestion direction is wholly absent). The auditor's direction is sound but understates scope: the daemon currently has no async criome client/subscription plumbing at all, and State retains no evaluation to forward, so this is more than a small wiring change.
  - Refined: src/daemon.rs:81-114 (serve_forever/handle_connection only handle mentci frames, no criome socket); src/criome_bridge.rs:31-44 (submit_verdict, only test-called); src/state.rs:132-155 (answer drops verdict locally, no bridge, no retained AuthorizationEvaluation); src/configuration.rs:56-58 (home_criome_socket_path, no prod caller); tests/criome_bridge.rs:184,370 (sole runtime invocation, in-process thread::scope orchestration)

### Verdict carries no link back to the originating criome evaluation

- **Verdict:** CONFIRMED — verified severity `high`, confidence `high` (auditor rated `high`)
- **Kind:** design-gap
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/state.rs:132-155 (answer); criome_bridge.rs:31-44 (submit_verdict signature)`
- **Evidence:** `State::answer` removes the question and pushes an `ApprovalVerdict` (state.rs:147-148) but the verdict only holds `question`/`decision`/`answered_by` — no AuthorizationEvaluation. `CriomeApprovalBridge::submit_verdict` REQUIRES `evaluation: AuthorizationEvaluation` as a separate argument (criome_bridge.rs:32-35); in the test it is the caller that still holds `evaluation` from the earlier EvaluateAuthorization call. mentci's State never stores the criome evaluation alongside the question, so even if the daemon were wired to criome it could not reconstruct which evaluation a verdict approves.
- **Impact:** Forwarding a CLI-entered verdict to criome is impossible from daemon state alone — the mapping question-id -> AuthorizationEvaluation does not exist anywhere in mentci. This is the concrete data-model gap that blocks closing the loop even after the daemon is taught to talk to criome.
- **Direction:** When a criome escalation is turned into a question, store the originating AuthorizationEvaluation (or its digest/contract+object) in the ApprovalQuestion/State so that on AnswerQuestion the daemon can rebuild the exact evaluation to hand to bridge.submit_verdict.
- **Verification:** Confirmed accurate on every cited point. State::answer (state.rs:132-155) removes the question and pushes verdict.clone() into self.decisions; ApprovalVerdict (signal-mentci src/schema/lib.rs:182-187) holds only question/decision/answered_by. QuestionProposal (lib.rs:145-151) and ApprovalQuestion (lib.rs:156-159) carry source/prompt/suggested_answer/explanation/context — NO AuthorizationEvaluation, contract, or object digest. CriomeApprovalBridge::submit_verdict (criome_bridge.rs:31-44) genuinely requires the full evaluation: AuthorizationEvaluation as a separate argument and wraps it into AuthorizationApproval { evaluation, decision }. The mentci State struct (state.rs:13-26) stores nothing criome-related. The integration test tests/criome_bridge.rs proves the gap concretely: at line 370-372 the bridge is called with evaluation.clone() that the TEST caller still holds from the earlier EvaluateAuthorization (line 316-320, 325) — it is never retrieved from mentci daemon state; question_proposal() (line 158-169) builds the proposal with no evaluation link. I also confirmed the daemon is not even wired to the bridge: grep over src/ shows CriomeApprovalBridge/submit_verdict referenced ONLY in criome_bridge.rs and the test, never in daemon.rs (which only has a hardcoded /tmp/criome-test.socket string). So forwarding a CLI-entered verdict to criome from daemon state alone is impossible: the question-id -> AuthorizationEvaluation mapping does not exist anywhere in mentci. This is a genuine data-model gap that blocks the goal's end state (verdict flowing back to criome) independent of, and in addition to, the missing daemon->criome wiring. Direction (store the originating evaluation, or its digest/contract+object, in ApprovalQuestion/State) is the correct fix shape. Severity high is right: it is a core blocker for closing the loop, though it sits behind the also-missing daemon-to-bridge call.
  - Refined: mentci/src/state.rs:13-26 (State has no evaluation store), :132-155 (answer pushes verdict-only); signal-mentci/src/schema/lib.rs:145-159 (QuestionProposal/ApprovalQuestion carry no AuthorizationEvaluation), :182-187 (ApprovalVerdict = question/decision/answered_by); mentci/src/criome_bridge.rs:31-44 (submit_verdict requires evaluation arg); mentci/tests/criome_bridge.rs:316-320,370-372 (bridge fed caller-held evaluation, not daemon state); daemon.rs has zero bridge references.

### Mismatched verb naming between CLI-needed actions and signal-mentci payloads forces NOTA verbosity

- **Verdict:** CONFIRMED — verified severity `low`, confidence `high` (auditor rated `low`)
- **Kind:** ugly-code
- **Location:** `signal-mentci /git/github.com/LiGoldragon/signal-mentci/src/lib.rs:14 (MentciRequest = Input) + state.rs:55-71 (apply dispatch)`
- **Evidence:** The human-facing actions map to verbose typed records: approve = `AnswerQuestion(ApprovalVerdict{ question, decision: ApproveSuggestedAnswer, answered_by })`, observe = `ObserveInterfaceState(InterfaceStateObservation{ interest })`. With no CLI verb layer, the human must author these full nested positional NOTA records by hand (per the bare-atom NOTA rules) for every single action, including spelling the `answered_by` SubscriberName each time.
- **Impact:** Even a NOTA-fluent operator faces high-friction, error-prone input for routine approvals; a non-technical psyche cannot use it at all. This is the ergonomic tax that a verb layer (finding 1) is meant to erase.
- **Direction:** Let the CLI verbs supply the constant fields (decision variant, default answered_by from the configured PersonaIdentity, FullInterfaceState/PendingQuestions interest) so the human types only the id.
- **Verification:** Confirmed by reading the code. The mentci CLI surface (src/main.rs -> src/client.rs ClientCommand) takes exactly one argument and routes it as a NOTA file path, an rkyv file path, or an inline NOTA string, parsing it DIRECTLY into a full MentciRequest via NotaSource::new(source).parse::<MentciRequest>() (client.rs:80-86). There is no verb layer (no approve/reject/defer/observe subcommands) anywhere in the client path. signal-mentci/src/lib.rs:14 indeed aliases MentciRequest = Input, and state.rs:55-71 dispatches over the verbose typed variants. The schema (signal-mentci/schema/lib.schema:179-182, 266-268) confirms the human must hand-author nested positional NOTA: approve = (AnswerQuestion (ApprovalVerdict question-1 ApproveSuggestedAnswer psyche)) with the answered_by SubscriberName typed every time; observe = (ObserveInterfaceState (InterfaceStateObservation psyche FullInterfaceState)). One correction to the finding's evidence: the ApprovalVerdict record is positional (question, decision, answered_by), not the labeled/struct form the finding's prose implies with `{...}` braces — NOTA records are positional, so the friction is slightly different in form but the substance (verbose hand-authored nested record per action) holds. criome_bridge.rs is a server-side daemon-to-criome verdict bridge, not a CLI verb layer, so it does not close this gap. This matters to the goal of a usable approval UI, but it is UX friction rather than a correctness blocker for the live two-daemon loop, and it explicitly depends on / duplicates the verb-layer finding (finding 1). Low severity is right.
  - Refined: mentci/src/client.rs:39-47,80-86 (single-arg, parses straight to MentciRequest, no verb layer); mentci/src/main.rs:1-11 (CLI entrypoint is ClientCommand); signal-mentci/src/lib.rs:14 (MentciRequest = Input); mentci/src/state.rs:55-71 (verbose variant dispatch); signal-mentci/schema/lib.schema:179-182 (ApprovalVerdict question/decision/answered_by) + :266-268 (InterfaceStateObservation subscriber/interest)

## Findings dropped on verification

### Defer verdict accepted but never persisted or actioned, and unknown-id approval is the only failure surfaced

- **Verdict:** DROPPED — verified severity `low`, confidence `high` (auditor rated `medium`)
- **Kind:** bug
- **Location:** `mentci /git/github.com/LiGoldragon/mentci/src/state.rs:132-155 (answer)`
- **Evidence:** For `ApprovalDecision::Defer`, `answer` returns `VerdictAccepted` immediately WITHOUT checking the question exists, WITHOUT removing it from pending_questions, and WITHOUT recording the verdict (state.rs:133-139). So a Defer on a nonexistent id reports success, and a Defer on a real id leaves it pending with no record that the human deferred it. Approve/Reject correctly reject unknown ids (state.rs:145).
- **Impact:** From the UX loop, 'defer' is a silent no-op that lies about success — the deferred question is indistinguishable from an untouched one, and there is no deferred-state the human can later revisit. For a human approval queue, defer needs to be a first-class, observable state.
- **Direction:** Validate the question id for Defer too, and model deferral as an explicit question state (e.g. move it to a deferred list / mark it) so the watch/list views can show deferred items and the human can return to them.
- **Verification:** The finding's central premise is contradicted by the contract. signal-mentci/schema/lib.schema:163 specifies Defer as "non-binding deferral; leaves the question pending" — so State::answer keeping the question in pending_questions and NOT recording a decision is the specified behavior, not a bug. The accompanying test mentci/tests/state.rs:76-105 (defer_keeps_question_open_for_later_answer_proposal) confirms the intent: Defer deliberately keeps the question open so a later ProposeEditedAnswer still applies, whereas Approve closes it. A deferred question therefore remains fully visible in the pending-questions projections (full_state / PendingQuestionsProjection), so it is NOT "indistinguishable from untouched" in the sense of being invisible — it is still in the queue. The finding's recommended direction (move to a separate deferred list / mark as a distinct state) actually contradicts the schema, which says deferral leaves it pending. What IS real is one narrow asymmetry: for Defer, answer returns VerdictAccepted at state.rs:133-139 BEFORE the existence check (state.rs:140-146), so deferring a nonexistent question id reports success while Approve/Reject return Rejection(UnknownQuestion). That validation laxity is a genuine minor inconsistency, but it does not break the goal's end-to-end approve/reject loop: Defer maps to AuthorizationApprovalDecision::Defer (criome_bridge.rs:50) meaning "stay parked," and the criome object simply remains un-authorized — the parked end state. Since the finding mostly mischaracterizes spec'd behavior as a bug and the one real sub-issue is a non-loop-affecting nit, the finding as written does not hold up.
  - Refined: signal-mentci/schema/lib.schema:163 ("Defer — non-binding deferral; leaves the question pending") and mentci/tests/state.rs:76-105 establish that leaving the question pending without a decision record is intended. Real (but narrow) defect: mentci/src/state.rs:133-139 returns VerdictAccepted for Defer before the unknown-id check at state.rs:140-146, unlike Approve/Reject which reject unknown ids — a minor validation asymmetry, not a loop blocker.

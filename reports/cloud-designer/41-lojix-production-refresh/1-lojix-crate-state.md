## Startup argument shape

`lojix-daemon` takes exactly one argument and it MUST be a signal-encoded
(rkyv) file — `command.signal_file_argument()` then `into_signal_file()`
(`src/bin/lojix-daemon.rs:17-22`). Inline NOTA and `.nota` paths are
rejected at the argument boundary. The file decodes to
`DaemonConfiguration` via `rkyv::from_bytes` (`src/lib.rs:128-132`). The
config carries five fields: `ordinary_socket_path`,
`ordinary_socket_mode`, `owner_socket_path`, `owner_socket_mode`,
`state_directory_path` (`src/lib.rs:118-125`). On any error the daemon
prints `(DaemonRejected [..])` and exits 2. There is NO virgin-daemon /
meta-signal-config-message path yet — the binary always requires the rkyv
file. On-restart SEMA self-resume is not implemented (state is in-memory).

## Sockets bound today

Two authority-tiered Unix sockets via `triad-runtime`'s
`AsyncMultiListenerDaemon` on one tokio multi-thread runtime
(`src/daemon.rs:75-96`):

- **Ordinary** (`ListenerRole::Ordinary`) — peer-callable `signal-lojix`
  reads/watch/unwatch. CLI default `/run/lojix/ordinary.sock`, env
  override `LOJIX_ORDINARY_SOCKET` (`src/client.rs:18-21`).
- **Owner/meta** (`ListenerRole::Owner`) — `meta-signal-lojix`
  Deploy/Pin/Unpin/Retire. Default `/run/lojix/owner.sock`, env
  `LOJIX_OWNER_SOCKET`.

Owner-socket hardening is real: `validate_owner_socket_mode` refuses any
mode granting `0o007` other-access at startup (`src/daemon.rs:112-117`),
and each owner connection must present kernel-vouched peer uid/gid
matching the daemon's effective uid/gid (`OwnerPeerAuthority::authorize`,
`src/daemon.rs:176-187`; checked in `serve_owner`, `src/daemon.rs:254`).

Request handling: each connection is one request/one reply, served on its
own tokio task with a per-listener concurrency cap of 64
(`MAXIMUM_CONCURRENT_REQUESTS`, `src/daemon.rs:124`), an 8 MiB frame cap
(`src/daemon.rs:25`), and a 10s read timeout (`src/daemon.rs:30`).
Length-prefixed rkyv frames decode to the contract `Input`, drive
`SchemaRuntime::execute` (generated Nexus runner), and the reply encodes
back on the same tier. The engine is async; child `nix` processes are
awaited directly (`tokio::process`), no blocking-pool bridge.

## Schema files + syntax flavor

Two checked-in `.schema` files, both daemon-owned runtime (Nexus/SEMA)
schemas — NOT wire contracts (those live in `signal-lojix` /
`meta-signal-lojix`):

- `schema/nexus.schema` (85 lines) — engine feature catalog: imports
  block, `[NexusWork]`/`[NexusAction]` operation roots, then a record
  body declaring `SignalInput`/`SignalOutput`, the `EffectCommand`
  catalog (ResolveFlakeAuth, MaterializeHorizon, NixEval, NixBuild,
  CopyClosure, ActivateGeneration, PathInfoGc), `EffectResult`, and the
  NexusWork/NexusAction model.
- `schema/sema.schema` (83 lines) — durable-state plane: read/write
  operation variants, reply unions, `StateMarker`, and the four table
  records (`LiveSetTable`, `GcRootsTable`, `EventLogTable`,
  `ContainerLifecycleTable`).

Syntax flavor: this is **positional NOTA-record schema** in the
`schema-rust-next` flavor (the cloud `nexus.schema`/`sema.schema` shape).
Top-of-file imports use `local-name dependency:module:Type` lines inside
`{ }`; operation roots are bare `[Root]` lines; the body is a `{ }` block
of `Name { field Type ... }` records, `*`-shorthand promoted fields
(e.g. `GenerationIdentifier *`), and `[Variant ...]` / `[(Tag Type) ...]`
unions. `;;` line comments. No `(key value)` Lisp/JSON pairs. Generated
into `src/schema/{nexus,sema}.rs` by `build.rs`, freshness-checked via
`LOJIX_UPDATE_SCHEMA_ARTIFACTS` (`build.rs:36-37`). build.rs pulls the
two contract schemas from cargo metadata for `signal-lojix` /
`meta-signal-lojix`; missing metadata only warns and skips generation
(`build.rs:81-110`).

## Deploy pipeline — what runs vs. what is stubbed

The routing brain is `SchemaRuntime::decide*` in `src/schema_runtime.rs`.
A meta `Deploy` opens a continuation chain tracked by `DeployPipeline`
(`schema_runtime.rs:101-124`) with stages
Submitted→BuildingRecorded→CopyingRecorded→ActivatedRecorded
(`schema_runtime.rs:131-142`).

Pipeline as implemented (effects do real IO via `NixCommand` +
`tokio::process`, `schema_runtime.rs:1727-1920`):

1. `ResolveFlakeAuth` → `nix flake metadata --json` (real).
2. `MaterializeHorizon` (only when `build_attribute` is None — production
   shape) → loads the cluster proposal NOTA file, projects via
   `horizon-rs`, writes `horizon`/`system`/`deployment` tiny flake inputs
   under `<state>/generated-inputs/<cluster>/<node>/<shape>/`, hashes each
   with `nix hash path --type sha256 --sri`, emits `--override-input`
   values (`schema_runtime.rs:1411-1725`). Real.
3. `NixEval` → `nix eval --refresh --raw <flake>#<attr>.drvPath` with the
   override inputs (real). Production attr is
   `nixosConfigurations.target.config.system.build.toplevel` (System) or
   `homeConfigurations.<user>.activationPackage` (Home),
   `schema_runtime.rs:188-197`.
4. `NixBuild` → `nix build --no-link --print-out-paths` (+ `--builders
   ssh-ng://` for remote, + substituter options) (real,
   `schema_runtime.rs:1782-1808`).
5. `CopyClosure` → `nix copy --to ssh-ng://<node>` (real).
6. `ActivateGeneration` → `ssh <node> nix-env -p .../system --set
   "$CLOSURE"` — NOTE: the closure path is referenced as a literal
   `"$CLOSURE"` shell var that is never set, so activation is NOT
   target-safe (`schema_runtime.rs:1863-1873`).

**Reject-guard gates this hard** (`unsupported_deploy_reason`,
`schema_runtime.rs:534-550`): only System `Eval`/`Build` and Home `Build`
are accepted today. Every activating action (System Boot/Switch/Test/
BootOnce, Home Profile/Activate) is rejected with
`UnsupportedDeployAction` because copy/activate is not target-safe —
confirmed by test `activating_deploy_is_rejected_until_activate_lands`
(`tests/engine_routing.rs:168`). So end-to-end today: production
**eval and build** work through Horizon materialization; **copy +
activate + live-set mutation never execute via a real deploy.** Deferred
work tagged M2/M3 in comments (`schema_runtime.rs:186,532,829`).

## Live-set / durable / SEMA-engine state

State is fully **in-memory**, NOT sema-engine/redb. `Store` wraps
`Mutex<StoreState>` (`src/lib.rs:232-251`); `StoreState` holds the four
generated table nouns (`LiveSetTable`, `GcRootsTable`, `EventLogTable`,
`ContainerLifecycleTable`) plus four monotonic `u64` sequence counters
(`src/lib.rs:145-155`). Commit "sequence" doubles as the state digest.
Writes (`record_deploy_submitted`, `record_phase_transition`,
`record_generation_activated`, pin/unpin/retire,
`record_container_transition`) and reads (`query_generations`,
`read_event_log`, `check_key_material`) are all hand-rolled against this
locked map (`schema_runtime.rs:910-1279`). The lib doc and ARCHITECTURE
both flag sema-engine/redb durability as "the next storage cutover"
(`src/lib.rs:9-12`, `ARCHITECTURE.md:122-123,148-149`). No persistence,
no crash-restart resume.

Pin/unpin/retire are real in-memory gc-root slot mutations and DO work
end-to-end (`schema_runtime.rs:1031-1133`); they are owner-socket calls.

## Push / subscription — declared but NOT delivered

ARCHITECTURE.md (§"Deploy event log", lines 64-69, 150-158) and
INTENT.md (lines 46-49) promise pushed `DeploymentObservation` /
`CacheRetentionObservation` events bridged downstream of commit via a
sema-engine `SubscriptionSink`. **This is not implemented.**
`WatchDeployments`/`WatchCacheRetention` only mint a subscription token
and reply once (`open_subscription`, `schema_runtime.rs:463-478`;
`Unwatch` just acks). There is no `SubscriptionSink`, no
`StreamingFrameBody::SubscriptionEvent` emission, no `LaneSequence`
minting, no outbound delivery lane in the code — grep for those terms in
`src/` returns nothing. So the "push, never poll" invariant is currently
contract-shaped only.

## Container lifecycle — write path only, no source

The `RecordContainerTransition` write path and `ContainerLifecycleTable`
exist (`schema_runtime.rs:1135-1162`), but there is NO systemd/dbus
subscription producing those transitions — grep for `dbus`/`systemd`
in `src/` returns nothing. ARCHITECTURE.md:70-72 describes the dbus mirror
as owned surface; it is unimplemented.

## CLI client

`src/client.rs`: takes one `ComponentArgument`, resolves to an ordinary or
owner contract `Input`, connects to the matching socket, exchanges one
length-prefixed frame, prints the reply (NOTA when `nota-text`, else
`{:?}`). KNOWN BUG flagged in-code (audit R7, `src/client.rs:98-108`):
the two contracts' short-header ordinals collide (meta `Deploy` ==
ordinary `Query` == 0x0), so signal-file tier disambiguation relies on
rkyv layout divergence, not a structural tier discriminator — "the proper
fix is a tier bit in the short header (upstream of lojix)".

## Biggest gaps to production

1. **No durable storage** — all state in-memory; sema-engine/redb cutover
   + crash-restart resume is the headline gap (INTENT/ARCH both note it).
2. **Activating deploys rejected** — copy/activate not target-safe;
   `activate_system` references an unset `$CLOSURE` shell var. No real
   deploy ever mutates the live set / gc-roots end-to-end.
3. **Subscription push entirely missing** — token handshake only; the
   core "push, never poll" delivery path is unbuilt.
4. **Container/dbus mirror missing** — write path exists, no event source.
5. **No virgin-daemon meta-signal config path / no SEMA self-resume** —
   daemon hard-requires an rkyv startup file every launch.
6. **Wire tier-ambiguity** (client.rs R7) — needs an upstream short-header
   tier bit in `signal-frame`/contracts.
7. **No Nix flake check surface** — only the cargo suite; live nix tests
   are all `#[ignore]`.
8. **Node identity injection deferred (M3)** — System builds a fixed
   `nixosConfigurations.target...` attr; per-node identity via the horizon
   override is still TODO (`schema_runtime.rs:186`).

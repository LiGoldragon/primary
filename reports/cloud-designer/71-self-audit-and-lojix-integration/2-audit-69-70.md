## Report 69 — provider tokens + Cloudflare DNS

### Stands

- **Wrong gopass path bug.** `cloud/flake.nix:46` reads
  `gopass show -o cloudflare/api-token`, while the DO path at
  `flake.nix:71` uses the `.com` domain form
  `gopass show -o digitalocean.com/api-token`. The report cites line 46
  exactly, and the asymmetry vs DO is real. The chain that makes this a
  real daemon blocker is intact: the daemon wires
  `cloudflare::ProviderClient::production()` (`cloud/src/lib.rs:561`) →
  `FlarectlApi` (`cloud/src/cloudflare.rs:292`) → token via `CF_API_TOKEN`
  env (`cloud/src/cloudflare_cli.rs:18,50`), and the only thing that fills
  `CF_API_TOKEN` is the flarectl wrapper at `flake.nix:46`. The daemon's
  own `postInstall` wrapper (`flake.nix:96-98`) injects `HCLOUD_TOKEN` and
  `DIGITALOCEAN_ACCESS_TOKEN` but **not** `CF_API_TOKEN`, so the CF path
  depends entirely on that one wrapper line — a wrong path breaks it.

- **Cloudflare mutation capability exists.** `cloud/src/cloudflare.rs`
  carries `create_record` (`cloudflare.rs:62`), `update_record`
  (`:68`), `delete_record` (`:75`) on the `Api` trait, with real HTTP
  POST/PATCH/DELETE impls (`:240-280`). `ProviderClient::apply_plan`
  (`:360-378`) does delete-then-upsert reconciliation exactly as
  described: `delete_named_records` then `upsert_record` over
  create+update (`:368-371`). The "15 record kinds" count is exact —
  `RecordKindName::from_record_kind` enumerates A, AAAA, CNAME, TXT, MX,
  NS, PTR, SRV, CAA, SSHFP, TLSA, URI, SVCB, HTTPS, LOC = 15
  (`cloudflare.rs:555-573`).

- **"Built path, no live witness yet."** Fair: `apply_plan` is real code
  but the report never claims a live CF mutation occurred — it explicitly
  says it has no live witness. Correctly hedged.

### Overstated / to flag

- **"`apply_plan` ... against a live zone" framing.** `apply_plan` is
  API-impl-agnostic; in production it runs through `FlarectlApi` (CLI
  shell-out), not the `HttpApi` whose POST/PATCH/DELETE I cited. The
  reconciliation logic is identical either way, so the capability claim
  stands, but the report leans on the HTTP impl shape while production
  uses the flarectl impl. Minor — does not change the conclusion.

- **"0 zones means under-scoped OR empty account" — adequately hedged.**
  The report does present both branches ("lacks `Zone:Read` ... **or**
  that account has no domains") and lands on the correct shared
  conclusion ("either way the current token cannot drive a DNS change").
  This is fairly hedged, not overstated. The open question to the psyche
  (does that account hold the domains) is the right disambiguator. No
  correction needed.

### Unwitnessed (live-probe assertions — not code-verifiable here)

- The DO capability table (§2: which scopes 200 vs 403), the CF token
  being `active`/40-char/sees `Anna.bird@protonmail.com's Account`, and
  the `9109 Unauthorized` on `GET /user/tokens/{id}` are all live-API
  probe results captured at report time. They cannot be re-verified from
  source and are not claimed to be. Treated as report-asserted facts; the
  recommendations built on them (§2 priority list, §3 threat model) are
  internally consistent with them.

- **DO-capabilities recommendations are sane.** The priority order
  (`account` read to close the `/v2/account` 403 gap, then `reserved_ip`,
  `firewall`, `domain`, deferring `database`/`kubernetes`/`cdn`/Spaces)
  matches what cloud actually does — the daemon's provider surface is
  droplets + ssh keys + images + snapshots + vpc + projects, and there is
  no database/k8s/cdn code path. Reasonable and well-scoped.

- **Full-caps threat model is sound.** The (a) agent-capability-surface
  vs (b) token-leak-blast-radius split is the correct decomposition:
  scope is not an agent-containment boundary when the agent can mint via
  browser, but is the whole game for a secret-at-rest leak. The
  job-scoped + expiry recommendation follows. The reasoning is sound and
  the Spirit references (`ravc`, `cjrl`, `iprx`, `h03z`) are used as
  pointers, not load-bearing claims — I did not re-verify their content
  in this lane (that is lane 3's job).

## Report 70 — Tier-2 spine + socket protocol

### Stands

- **`Observation` host variant is `Servers`, not `ObserveServers`.**
  `signal_cloud::Observation::Servers(HostQuery)` at
  `signal-cloud/src/lib.rs:398` (and the `ObservationResult::Servers`
  mirror at `:445`). The CLI parses the **hand-written**
  `signal_cloud::Operation` (`cloud/src/client.rs:154`,
  `parse::<CloudOperation>()`) and `meta_signal_cloud::Operation`
  (`client.rs:159`), so the report's instruction to "drive the
  hand-written names" is exactly right. Citation is correct.

- **Plan ids are the derived deterministic strings claimed.** Create:
  `format!("{}-{:?}-host-plan", host_name, provider)`
  (`cloud/src/lib.rs:1274-1278`) → `<host>-<provider>-host-plan`. Destroy:
  `format!("{}-{:?}-host-destruction-plan", ...)` (`lib.rs:1318-1322`) →
  `<host>-<provider>-host-destruction-plan`. The `{:?}` debug-formats
  `Provider::DigitalOcean` to `DigitalOcean`, so the witness id
  `criome-spine-test-DigitalOcean-host-plan` (report line 20) is exactly
  what this code mints. A caller can construct ids without the prepare
  round-trip, as claimed.

- **Variant payloads nest; the nesting rule is accurate.** Decode is
  `(VariantHead Payload)`. `Registration` (3 fields,
  `meta-signal-cloud/src/lib.rs:45-49`) and `HostDestruction` (2 fields,
  `:223-226`) are multi-field structs → flat record payloads.
  `HostPlanPreparation` (`:216-218`) is a **single-field struct** wrapping
  `DesiredHostState` (`:205-211`), so its payload nests one extra level
  (double parens) — structurally exactly as the report says. The parser
  arity error the report quotes is real:
  `nota-next/src/codec.rs:65` emits
  `"expected {type_name} to hold {expected} root objects, found {found}"`.

- **Credential env var fact.** `DIGITALOCEAN_ACCESS_TOKEN` is the handle
  name (`cloud/src/digitalocean.rs:79`,
  `TOKEN_ENVIRONMENT_VARIABLE`), and the daemon reads the token by env-var
  name from the registered `CredentialHandle` (same
  `EnvironmentCredentialSource` pattern, `cloudflare.rs:52-56`). The boot
  snippet using `RegisterAccount (... DIGITALOCEAN_ACCESS_TOKEN)` as the
  handle is therefore correct.

- **write_config arg order + socket modes.** Example takes
  `<out.rkyv> <ordinary.sock> <meta.sock>`
  (`cloud/examples/write_config.rs:30` usage), and socket modes default to
  `0o600` (`write_config.rs:42,44`). Matches the report's boot block and
  its "modes default 0o600" note.

- **Daemon binds both sockets.** `Daemon::bind`
  (`cloud/src/schema/daemon.rs:163-198`) builds a working listener socket
  and pushes a meta listener socket (`:181-190`) into an
  `AsyncMultiListenerDaemon`. Two sockets bound at startup, as the report
  shows.

### Overstated / loose wording (true core)

- **"`HostPlanPreparation` is an alias newtype over `DesiredHostState`."**
  Not literally a Rust newtype or type alias — it is a named struct with
  one field `desired_host_state: DesiredHostState`
  (`meta-signal-cloud/src/lib.rs:216-218`). The **structural**
  consequence the report draws (one inner object → one extra nesting →
  double parens) is correct; only the "alias newtype" label is imprecise.

- **Quoted parser error is lightly paraphrased.** Report writes
  `"HostPlanPreparation to hold 1 root object, found 5"`; the codec emits
  `"... to hold 1 root objects, found 5"` (plural "objects", prefixed
  "expected"). Substance identical; the quote is not verbatim.

### Unwitnessed (asserted runtime facts — not code-verifiable, no stored artifact)

- **The live witness itself** (droplet `578873541` created → observed →
  destroyed over the sockets) appears only in report 70's own narrative;
  no witness log / transcript is checked into the session dir
  (`reports/cloud-designer/71-.../` holds only `0-frame-and-method.md`).
  The code path supporting the flow is real and consistent
  (`apply_digitalocean_host_plan` at `cloud/src/lib.rs:1637`, host-plan
  store at `:1286`, observe path), so the claim is **plausible and
  code-consistent**, but the specific run is report-asserted, not
  independently re-verifiable.

- **"binds both sockets in ~1s."** A runtime timing observation. The bind
  path is real (above) but the ~1s figure is not derivable from source.
  Treat as report-asserted.

- **Post-destroy stale `Initializing` read.** The honest caveat (DO
  eventual-consistency; the independent sweep showed `droplets: []`) is a
  runtime observation, consistent with the eventually-consistent provider
  API but not verifiable here. The follow-up note is fair.

### Nothing in the code contradicts report 70

No code claim in 70 is contradicted by source. The remaining-`68`-risks
paragraph (single-`EngineActor` blocking shape, two-tree drift,
Hetzner-unshipped, `ad53` half) is carried forward as still-open, not
re-asserted as fixed — appropriate. The blocking-actor claim ("held one
socket for the full duration of each blocking provider call") is
consistent with the synchronous `Store` + per-call provider shell-outs,
though the precise concurrency behavior is lane 1's territory.

## Bottom line

Both reports are **substantially accurate and well-anchored**. Every
code-anchored claim verified (the flake line 46 bug, the CF mutation
surface, the `Servers` variant, the derived plan ids, the nesting rule,
the env-var handle, the bind). The only corrections are wording-level:
"alias newtype" for a single-field struct, a non-verbatim parser-error
quote, and the HTTP-vs-flarectl impl framing in 69. The live-run facts in
70 (droplet id, ~1s bind, stale read) are plausible and code-consistent
but have no checked-in witness artifact — flagged Unwitnessed, not Wrong.
69's threat model and DO-capability recommendations are sound and the
"0 zones" question is fairly hedged.

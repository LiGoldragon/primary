# consumer migration path — Consumer-invocation model + migration path

cloud-designer, 2026-06-11. Grounded run for report 40. Area: Consumer-invocation model + migration path: how lojix-cli is invoked in production today, how the new lojix-daemon model differs, and the concrete steps + smallest-viable first migration to repoint a consumer.

## Findings

### ✅ confirmed — Production lojix-cli is invoked BY HAND by an operator from a shell — it is NOT a flake app, justfile/Makefile target, or scripted CI step. It ships into the home profile as two installed binaries (raw lojix-cli + a Python operator wrapper lojix-run), and an operator types one NOTA request string as the single argument.

- **Evidence:** CriomOS-home/modules/home/profiles/min/default.nix:191-197 `deploymentPackages = with pkgs; [ inputs.lojix-cli.packages.${...}.default (pkgs.callPackage ../../../../packages/lojix-run { inherit inputs; }) age sops ssh-to-age ];`. No `apps.`/`mkApp`/`writeShellApplication` for lojix in any flake (grep CriomOS/flake.nix empty). lojix-run.py:205-213 `subprocess.run([self.lojix_cli, self.request.text], ...)` — wrapper shells lojix-cli with one NOTA arg. CriomOS-home/skills.md:214-220 documents lojix-run as the operator-facing wrapper; raw lojix-cli when the stdout store path is the interface.
- **Cutover implication:** There is no central pinned invocation point to repoint — migrating the invocation means swapping the installed package set in the home profile (replace the lojix-cli binary + lojix-run wrapper with the new `lojix` thin client) AND standing up a daemon process somewhere. The operator habit (type a NOTA request) can be preserved only if the new thin client accepts inline NOTA, which it currently does NOT (see InlineNotaUnsupported finding).

### ✅ confirmed — CriomOS does NOT pin lojix-cli as a direct flake input at all. Its four deploy axes (system/horizon/deployment/secrets) are local stub path inputs (`path:./stubs/no-*`) that lojix overrides at deploy time via `--override-input`. CriomOS-cli appears in CriomOS/flake.lock ONLY transitively through criomos-home.

- **Evidence:** CriomOS/flake.nix:48,61,65,69 `system.url=path:./stubs/no-system`, `horizon.url=path:./stubs/no-horizon`, `deployment.url=path:./stubs/default-deployment`, `secrets.url=path:./stubs/no-secrets`; `grep -c lojix-cli.url CriomOS/flake.nix` = 0. CriomOS/flake.lock:526 `"lojix-cli": "lojix-cli"` is an INPUT OF the criomos-home node (line 728 lists it under criomos-home); the lojix-cli node at :1134-1160 pins rev fc2ff0287f53 with nixpkgs following `["criomos-home","nixpkgs"]` — i.e. it is criomos-home's input, not CriomOS's.
- **Cutover implication:** CriomOS itself is already daemon-agnostic: it consumes WHATEVER tool writes its four override inputs. Migrating CriomOS's SYSTEM deploy needs no flake.nix edit — only a different orchestrator producing the same four content-addressed override flakes. The lojix-cli pin in CriomOS/flake.lock is pure transitive noise inherited from criomos-home and is removed automatically once criomos-home drops lojix-cli.

### ✅ confirmed — Nothing consumes lojix-cli at flake EVAL time. CriomOS-home pins `lojix-cli.url` as a direct input but uses it in exactly two NON-eval-load-bearing places: it builds the lojix-cli BINARY into the home profile, and it builds the lojix-run wrapper (which substitutes the lojix-cli binary path). The input pin is purely for the binary, never for eval-time data.

- **Evidence:** CriomOS-home/flake.nix:120-121 declares `lojix-cli.url`; grep for `lojix` in CriomOS-home outputs returns only :120,:121 (input), :329 (the lojix-run CHECK), and modules/home/profiles/min/default.nix:192-193 (the two installed packages). packages/lojix-run/default.nix:4 `lojixCli = inputs.lojix-cli.packages.${system}.default` — consumes the package, not a nix value. mkHomeConfiguration (flake.nix:335-353) uses only horizon/user, no lojix. No CriomOS module imports any lojix nix attr.
- **Cutover implication:** The flake input pin is load-bearing ONLY for the binary — there is no hidden eval-time coupling to break. A consumer migration is a package swap, not an eval-graph surgery. This de-risks the cutover substantially: you can drop the lojix-cli input the moment the home profile installs the new `lojix` client instead.

### ✅ confirmed — The new model is structurally incompatible with the current 'run a CLI by hand' habit at the argument layer: the thin `lojix` client takes a PRE-ENCODED signal frame (or NOTA file), and inline NOTA is explicitly UNSUPPORTED — so an operator cannot type a `(System (...))` deploy request the way they type lojix-cli's `(FullOs ...)` today.

- **Evidence:** lojix/triad-port/src/client.rs:71-77 `resolve_request`: SignalFile -> decode; NotaFile -> decode; `InlineNota(_) => Err(Error::InlineNotaUnsupported)`. README.md:10-12 'lojix — thin CLI client. Reads one Nota request, sends it to the daemon'. ARCHITECTURE.md:72-75. lojix-cli by contrast (report 39, main.rs:8-15 / request.rs:185-219) accepts inline NOTA directly as argv.
- **Cutover implication:** Migration must ALSO supply a request-builder/encoder step: someone must turn an operator's intent into a meta-signal-lojix `DeployRequest` signal frame on disk before `lojix` can send it. Until inline NOTA lands (or a thin encoder wrapper is written, the new lojix-run equivalent), the daemon model is not hand-drivable. This is a concrete, currently-missing migration artifact, not just daemon plumbing.

### ✅ confirmed — The wire contract carries substituters PRE-RESOLVED (url + public_key), but neither the daemon nor the thin client performs node-name -> URL/key resolution. In lojix-cli that resolution is done against the horizon (ExtraSubstituters::from_horizon_nodes). The resolution logic has nowhere to live in Stack B today — confirming report 38 open question 3.

- **Evidence:** meta-signal-lojix/triad-port/schema/lib.schema: `ExtraSubstituter { url String public_key String }` and `substituters (Vec ExtraSubstituter)` in both SystemDeployment and HomeDeployment. Daemon only copies them through: lojix/triad-port/src/schema_runtime.rs:260-268 `convert_substituters` maps meta::ExtraSubstituter{url,public_key} -> nexus::ExtraSubstituter{url,public_key}. lojix-cli/src/request.rs:24,35,47 `substituters: Option<Vec<NodeName>>` (BARE NODE NAMES); lojix-cli/src/deploy.rs:99-100 `ExtraSubstituters::from_horizon_nodes(&horizon, &request.substituters)` and :210-220 resolves each NodeName to node.nix_url + node.nix_pub_key_line. The thin client (client.rs) does no horizon read at all.
- **Cutover implication:** Migration must decide WHERE node-name->cache-URL+key resolution moves. Two options: (a) the daemon gains horizon read for substituters (it already projects horizon for the override inputs, so the data is in reach) and the wire reverts to node names; or (b) the request-builder/operator hand-resolves before encoding the frame. The current wire bakes in choice (b) — every deploy frame must arrive with url+key already filled, which the operator/encoder must produce. This is a required, undecided design step before the first cutover.

### ✅ confirmed — Concrete migration steps to repoint a consumer (the 'migrate a consumer' definition): (1) package + deploy the daemon — no flake.nix/default.nix/systemd unit exists for the new lojix anywhere; (2) run lojix-daemon as a long-lived cluster-operator-owned process binding two sockets, started from a pre-generated rkyv config (today it parses NOTA from argv — override violation); (3) provide a request-builder that resolves substituters and encodes a DeployRequest frame; (4) install the thin `lojix` client into the home profile in place of lojix-cli+lojix-run; (5) run both stacks in parallel and switch one consumer/one node at a time.

- **Evidence:** No flake.nix/default.nix in lojix repo (`find ... -name flake.nix -o -name default.nix` empty); no systemd/module/socket wiring in any consumer (grep `lojix-daemon|*.sock|LOJIX_ORDINARY|LOJIX_OWNER` across CriomOS/CriomOS-home/goldragon -> only false positive repository-receive.nix:26 repository-ledger-owner.sock). bin/lojix-daemon.rs:17 `nota_config::ConfigurationSource::from_argv()?.decode()?` parses NOTA config from argv (report 38 gate 6). ARCHITECTURE.md:162-164 'cluster-operator-owned, one instance per operator workstation/shared deploy host'. client.rs:16-20 sockets from env LOJIX_ORDINARY_SOCKET/LOJIX_OWNER_SOCKET (default /run/lojix/{ordinary,owner}.sock). active-repositories.md §Cutover discipline (report 38 file 8): cover every consumer, run both in parallel, switch one at a time.
- **Cutover implication:** 'Migrate a consumer' = build a Nix package for the daemon+client, write a NixOS module that runs the daemon (with rkyv config, not NOTA argv) and a home-profile swap that installs the thin client + a request-builder, relocate substituter resolution, then repoint exactly ONE consumer's deploy path. The largest untbuilt block is packaging+systemd (entirely absent), not the Rust spine. Daemon-arg fix (rkyv-only) is a hard precondition because the override forbids NOTA-parsing daemons.

### 🟡 partial — The SMALLEST viable first migration is the HOME deploy path on a secret-free node, NOT a system deploy: it sidesteps the secrets-override break and the broken activate bodies are still in scope but Home Build needs no activation. The cleanest first cut is repointing CriomOS-home's `HomeOnly ... Build` on a node that needs neither the secrets override nor horizon-resolved substituters (tiger or zeus — EdgeTesting/Edge, no router/WPA3, no LLM).

- **Evidence:** Report 38 file 10: daemon admits ONLY System Eval/Build and Home Build; rejects all activating actions; secrets override unmaterialized breaks EVAL only for router(WPA3)/LLM nodes (file 9 §D: router/default.nix:28,36 fires only when wpa3SaePassword set; llm.nix:163 only when LLM active). datom.nota grounding (task brief): tiger & zeus = EdgeTesting/Edge, no router, no LLM = secret-free candidates; prometheus needs secrets. lojix-run check itself models a zeus FullOs Eval (checks/lojix-run/default.nix). UNTESTED: I did not run a real build of the new daemon against a zeus Home Build closure (non-mutating scope + no daemon packaging to invoke).
- **Cutover implication:** First cutover target: a secret-free node (zey/tiger) doing Home Build or System Eval/Build — the only capabilities the daemon actually has. This avoids gates 1 (activate), 2 (secrets) for that node. But even this minimal cut still requires: daemon packaging, the request-builder/encoder, substituter resolution (or a node needing no extra substituters), and the OsOnly firmware-field reconciliation (file 9 §B) if a System Build is in scope. A Home Build on zeus is the genuinely smallest piece; a full activating deploy on any node remains blocked by gate 1.

### ✅ confirmed — lojix-cli is STATELESS and run per-invocation, so the consumer-invocation model has no shared-state or lifecycle to migrate — every lojix-cli call is independent. The new daemon introduces a long-lived process lifecycle (start, persist, self-resume) that is net-new operational surface the consumer side must now own.

- **Evidence:** Report 39:13-15 'lojix-cli keeps no state... each invocation independent'. lojix-run.py creates a fresh RunDirectory per call (:150-167) and shells lojix-cli once (:205-213). New stack ARCHITECTURE.md:5-9 daemon 'long-lived... owns the live generation set, GC roots tree, deploy event log'; :162-164 one instance per operator host. Report 38 file 8: state in-memory (Mutex<StoreState>), no redb/Engine::open, restart loses all deploy state.
- **Cutover implication:** Migration adds a daemon-lifecycle responsibility absent today: who starts/supervises lojix-daemon, where its state lives, and what happens on restart. Per the override the daemon must self-resume from persisted SEMA state — but state is in-memory today (gate 4). For a FIRST cutover this is the psyche's reserved open-question 2 (in-memory-on-a-small-node-set acceptable, or durable-state-first). The invocation migration cannot complete without choosing a supervision model (systemd unit) that does not yet exist.

## Notes

- INVOCATION MODEL TODAY, precisely: operator types a NOTA request -> `lojix-run '(FullOs goldragon zeus [..] [github:LiGoldragon/CriomOS/main] Eval None None)'` OR raw `lojix-cli '(...)'`. lojix-run is a Python wrapper installed in the home profile that: rewrites known branch refs to exact jj revs (ExactReferenceRewriter, REPOSITORIES map at lojix-run.py:101-104), captures stdout/stderr to a run dir, redacts store hashes, runs SSH postchecks, and has a root-home-fallback that SSHes root@<node>.<cluster>.criome to set+activate a home generation. None of this wrapper logic exists in the new thin client.
- The grammar maps cleanly: lojix-cli `(FullOs cluster node source criomos action builder? substituters?)` -> wire `(Deploy (System (SystemDeployment cluster node FullOs source flake action builder? substituters)))`. The DeploymentKind discriminant (FullOs/OsOnly/HomeOnly) carries the split, so one Deploy verb covers all three (meta-signal-lojix schema header comment confirms this is a deliberate reconstruction of lojix-cli/src/request.rs:13-45).
- KEY ASYMMETRY for the cutover plan: CriomOS (system blueprint) is ALREADY orchestrator-agnostic — it consumes override inputs, not a lojix input. CriomOS-home is the ONLY repo with a direct lojix-cli dependency, and only for the BINARY (home profile install). So 'migrate the consumers' is really 'migrate CriomOS-home's home-profile package set + stand up a daemon', plus producing the four override inputs from the daemon's projection instead of lojix-cli's. CriomOS needs no edit.
- goldragon (the cluster proposal repo) has ZERO lojix references of any kind (grep empty) — it is pure data read per-request by horizon-rs; nothing to migrate there.
- Build sanity: `cargo build --offline --bins` in lojix/triad-port returned EXIT=0 (Finished in 0.06s, warm cache) — consistent with report 38's green-offline finding. I did NOT run cargo test or the --ignored real-nix tests (tasks 19-21 left pending; non-mutating scope and the migration-path question did not require them).
- Two missing migration ARTIFACTS that no report enumerated as a single list: (1) a Nix package + NixOS systemd module for lojix-daemon (none exists), (2) a request-builder/encoder that resolves substituters and writes a DeployRequest signal frame to disk for the thin client to send (the client refuses inline NOTA). Both are prerequisites even for the smallest Home-Build cutover.

## Findings

### ✅ confirmed — Router node prometheus is HARD-BLOCKED without the secrets override: nix eval of toplevel.drvPath fails on the missing WPA3 secret, and prometheus has no other eval blocker (empty swapDevices), so the secrets gate is the sole cause.

- **Evidence:** `nix eval ...horizon-prometheus...` ended: `error: Failed assertions:\n- router Wi-Fi secret routerWifiSaePasswords is missing from inputs.secrets.sopsFiles\n- router backup Wi-Fi secret routerBackupWifiPassword is missing from inputs.secrets.sopsFiles`. Stack: `while evaluating the option 'system.build.toplevel'` -> top-level.nix. prometheus io.swapDevices=[] (no swap bug to confound). Projected node: behavesAs.router=true, largeAi=true; routerInterfaces.wpa3SaePassword.name=routerWifiSaePasswords.
- **Cutover implication:** prometheus (LargeAiRouter) cannot be a first-cutover target. Stack B must materialize the secrets override (the missing 4th input — report 38 gate #2) before it can even BUILD prometheus. Defer prometheus to a later cutover wave once Secrets materialization lands.

### ✅ confirmed — The secrets gate fires via NixOS ASSERTIONS (router/default.nix:89-97), not only the let-binding throw at :36 that report 38 cited. The assertion path is stricter/more robust: it fires for any active router module regardless of which let bindings get forced, and surfaces BOTH the primary and backup Wi-Fi secrets as missing.

- **Evidence:** Worktree router/default.nix:91-92 `assertion = routerWifiSopsFileExists; message = [|router Wi-Fi secret ${...} is missing from inputs.secrets.sopsFiles|]` and :96-97 backup; the eval error reproduced both messages verbatim. The throw at :36 also exists but the assertion is what production eval surfaces.
- **Cutover implication:** Report 38's gate-#2 framing (throw at router/default.nix:36) is directionally right but should be updated: the live gate is a NixOS assertion at :89-97. A cutover gate-check / parity test should assert against the assertion message text, since that is what production eval emits.

### ✅ confirmed — Secret-free node zeus (Edge, no router/LLM) evaluates PAST the secrets reference to a complete drvPath — it is deployable today with the no-secrets stub.

- **Evidence:** `nix eval ...horizon-zeus...` returned `"/nix/store/azyabb1qrxyijd6nidln0pzms6rlf2fs-nixos-system-zeus-26.05.20260422.0726a0e.drv"` (full toplevel drv, exit 0, no secrets error). Projected zeus: behavesAs.router=false, largeAi=false, routerInterfaces=null, io.swapDevices=[].
- **Cutover implication:** zeus is the cleanest first-cutover candidate: full system closure evaluates with zero secrets, so Stack B's existing eval/build path (its one working capability) can target it now. It exercises the cutover end-to-end without needing the unbuilt Secrets materialization. (Activation gaps from report 38 gate #1 still apply — out of this area's scope.)

### 🟡 partial — Secret-free nodes tiger (EdgeTesting) and ouranos (EdgeTesting) also evaluate PAST the secrets reference — they never throw on inputs.secrets — but currently fail eval on an UNRELATED option-name bug in the swap module, not on secrets.

- **Evidence:** Both runs failed with `error: The option swapDevices."[...]".sizeMebibytes does not exist ... In .../modules/nixos/disks/preinstalled.nix ... Did you mean ...".size"...`, under `while evaluating the option assertions` -> encrypted-devices.nix — NO secrets error. Projected io.swapDevices: tiger=[{device:/dev/disk/by-uuid/4dcf...,sizeMebibytes:null}], ouranos=[{device:/swapfile,sizeMebibytes:32768}]; zeus=[] (why zeus passes). Source: disks/preinstalled.nix:3,40 `inherit (horizon.node.io) ... swapDevices; ... inherit swapDevices;` feeds a `sizeMebibytes` field the nixpkgs swapDevices option doesn't have.
- **Cutover implication:** tiger/ouranos are secret-free (so the secrets gate is NOT what blocks them), but a separate CriomOS swap-module bug (horizon emits swapDevices.sizeMebibytes; nixpkgs expects .size) blocks any node with a swap entry. This is a distinct, smaller fix independent of Stack B. For a first cutover, prefer zeus (no swap); to add tiger/ouranos, fix the sizeMebibytes->size mapping in disks/preinstalled.nix first.

### ✅ confirmed — Quantified: of the 5 goldragon nodes, exactly 1 (prometheus) is secret-bearing; 3 (zeus, tiger, ouranos) are secret-free; 1 (balboa) is projected away.

- **Evidence:** Species (datom.nota) -> behavesAs (horizon-rs node.rs:185 router=hybrid||router||large_ai_router; :184 large_ai=large_ai||large_ai_router): prometheus=LargeAiRouter -> router+largeAi (BOTH secret gates: router/default.nix + llm.nix:163 localLlmApiToken). zeus=Edge, tiger=EdgeTesting, ouranos=EdgeTesting -> router=false,largeAi=false,routerInterfaces=null (no secret reference). balboa=Center with trust=Zero -> dropped by projection (report 39; not a deploy target). All four projections ran exit 0.
- **Cutover implication:** The first cutover CAN dodge the secrets gate: 3 of 5 nodes are secret-free. Sequence the cutover wave: wave 1 = zeus (evaluates clean today); wave 1b after the trivial swap fix = tiger + ouranos; final wave = prometheus, which requires the Secrets materialization shape (report 38 gate #2) AND the LLM secret (llm.nix:163). Stack A (lojix-cli) keeps owning prometheus until then, per cutover discipline (run both in parallel, switch one consumer/node at a time).

### ✅ confirmed — The no-secrets stub does NOT silently mask the router gate — the gate is real and active, confirming report 39's 'asymmetric stub' description.

- **Evidence:** CriomOS/stubs/no-secrets/flake.nix returns `{ sopsFiles = {}; }` (empty), and with that stub prometheus still throws the assertion while zeus succeeds. So the stub lets non-router nodes evaluate but does not paper over a router node's missing secret.
- **Cutover implication:** No risk that a secret-free first cutover accidentally ships a mis-configured router: a router node simply will not evaluate without real secrets, so the failure mode is loud (eval fails), not silent (deploy a broken router).

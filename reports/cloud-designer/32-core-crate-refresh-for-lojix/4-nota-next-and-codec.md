# nota-next + nota-codec refresh for lojix

Survey for the cloud-designer lane: characterize the LATEST development of the
NOTA codec crates and contextualize for the new lojix triad stack
(`/git/github.com/LiGoldragon/lojix/triad-port`).

## Headline

For lojix, this is a near-zero-impact refresh. Both crates moved, but the moves
either (a) target the `StructuralMacroNode` derive surface that lojix does NOT
use, or (b) are already AT lojix's pin. The one must-do is a one-line lockfile
bump of `nota-next` to pick up two safe commits; everything else is
nice-to-have or non-applicable. The quoted-string rejection that the brief
flags as a possible breakage is already in lojix's pin AND lojix's only live
NOTA-text input is bracket-form, so there is no breakage.

## nota-next: the two missing commits

### `d996a30` — remove the abandoned at-binder syntax

What changed: deletes the entire `@`-binding parser surface from
`nota-next/src/parser.rs` (−155 lines): `parse_atom_or_at_binding`,
`parse_named_declaration_binding`, `parse_named_member_binding`,
`parse_at_delimited`, the `opening_starts_declaration` free fn, and the
`AtBindingOpening` trait. After this, `@` is an ordinary atom character; the
fallthrough is `Some(_) => Ok(self.parse_atom())`. Per psyche 2026-06-06
(Spirit `own9`, superseding `n2z3`). Also drops the at-binding design-example
test and an ARCHITECTURE section. The retired-syntax paragraph in nota-next
INTENT.md (lines 29-37) records this: schema declaration meaning is carried by
position + delimiter shape, not by an `@` interface.

Bearing on lojix — NONE (confirmed by reading lojix's actual schema files):

- `triad-port/schema/sema.schema` and `triad-port/schema/nexus.schema` use
  ONLY positional bracket/brace/parenthesis declaration forms. The whole
  schema body is `{ ... }` with operation roots `[NexusWork]` / `[NexusAction]`
  and union variants like `EffectCommand [(NixEval NixEvalCommand) ...]`. There
  is not a single `@` in either file (`grep '@'` on `schema/` and
  `src/schema/` returns only the `// @generated by schema-rust-next` header
  banner in the emitted Rust — not NOTA content).
- lojix's schemas are read by schema-next at lojix build time, and the
  at-binder commit notes "schema-next already rejects @" — so even at the build
  toolchain boundary lojix was never on the at-binder path.
- Removing dead parser code cannot regress a consumer that never used it. This
  is a strict simplification of the crate lojix depends on.

Verdict: safe to adopt, irrelevant to lojix's behavior. It only matters as
part of bumping the pin (you can't take `f0e435a` without it; it's the parent).

### `f0e435a` — add `#[shape(keyword = "...")]` structural-macro-node shape

What changed (`derive/src/lib.rs`, +21): adds a `Keyword { keyword: String }`
variant to the `StructuralVariantShape` enum in the `StructuralMacroNode`
derive. A `#[shape(keyword = "opens")]` variant:
- carries ZERO fields (`check_field_count` expects 0),
- lowers to `BlockShape::literal(keyword).into_structural_variant(...)`,
- decodes via `block.demote_to_string() == Some(keyword)`,
- encodes the literal back (`#keyword.to_owned()`).

The test (`tests/macro_nodes.rs`) shows the motivating use: a streaming /
relation signature `(Watch WatchRequest opens RecordStream)` decoded by a
4-arity `pascal_head` variant whose third field is itself a
`DerivedRelationKeyword` enum with `#[shape(keyword = "opens")]` /
`#[shape(keyword = "belongs")]` variants — i.e. an inner marker atom is its
own recursively-decoded structural macro node, discriminating sibling forms
with NO variant-name string comparison in hand-written code. The commit message
states the purpose explicitly: "Unblocks schema-next StructuralMacroNode derive
adoption: opens/belongs inner markers become recursively-decoded keyword
nodes."

Bearing on lojix — NONE today, and here is the adversarial reasoning, because
this is the one commit that could plausibly matter:

1. `StructuralMacroNode` is a SCHEMA-AUTHORING / schema-language consumer
   feature (schema-next reads `.schema` files via this typed-node layer). lojix
   does not embed schema-next; lojix's `.schema` files are lowered by
   schema-next at BUILD time (a build-dependency, `schema-rust-next`), and the
   product is checked-in Rust under `src/schema/`. lojix's *runtime* never
   parses schema text.
2. lojix's generated code uses ONLY `NotaDecode` / `NotaEncode` (the
   value-codec derives), never `StructuralMacroNode`. Confirmed by grep across
   `src/schema/nexus.rs` and `src/schema/sema.rs`: every type carries
   `#[cfg_attr(feature = "nota-text", derive(nota_next::NotaDecode,
   nota_next::NotaEncode))]` and the hand-emitted `from_nota_block` / `to_nota`
   wrappers delegate to those two traits. There is no `#[shape(...)]`, no
   `StructuralMacroNode`, no `keyword` anywhere in lojix.
3. lojix's schemas express their unions with the ordinary tagged-variant form
   (`LoggedEvent [(Deployment ...) (CacheRetention ...) (Container ...)]`),
   NOT the `(X Y opens Z)` streaming-relation keyword form. So even the schema
   *source* does not exercise the new shape.

Where it COULD matter later: if a future lojix schema revision adopts the
streaming-relation signature form (the `opens` / `belongs` stream-relation
vocabulary that the cloud nexus schema is exploring — lojix's `nexus.schema`
header says "Shape follows cloud/schema/nexus.schema exactly"), then schema-next
emitting lojix's generated code would need a `nota-next` recent enough to
provide `#[shape(keyword=...)]`. That is a transitive requirement that lands
through `schema-rust-next`'s own `nota-next` dependency, not through lojix's
direct `nota-next` pin. It is a watch-item, not an action.

Verdict: adopt the bump for hygiene + to keep lojix's direct `nota-next`
aligned with what schema-rust-next will expect, but it changes nothing in
lojix's current generated code or decode behavior.

## nota-codec: the quoted-string + block-string properties

lojix uses nota-codec on exactly one LIVE text-decode path and one type-derive
path:

- `triad-port/src/lib.rs:18` `use nota_codec::NotaRecord;` — derived on
  `DaemonConfiguration` (lib.rs:94). The daemon's single NOTA argument is
  decoded through `nota_config::ConfigurationSource::from_argv()?.decode()?`
  (`src/bin/lojix-daemon.rs:17`), and `nota_config` internally calls
  `nota_codec::Decoder::new(text)` (`nota-config/src/source.rs:116`). THIS is
  the path that the quoted-string rejection governs.
- `triad-port/src/lib.rs:34` `Nota(#[from] nota_codec::Error)` — lojix's `Error`
  enum absorbs nota-codec errors (including the new `QuoteStringDelimiter`).

### `f761421` — reject quoted string delimiters (already lojix's pin)

What changed: the lexer's `b'"'` arm now returns
`Err(Error::QuoteStringDelimiter { offset })` instead of reading a legacy quote
string; `read_legacy_quote_string` / `read_inline_string` /
`read_multiline_string` are deleted; `Token::Str` is removed from the token
enum, the decoder's `read_string` / `read_path` / `read_map_key` drop their
`Token::Str` arms; the `NotaStringKind::Legacy` value-tree variant is gone.
Net: the decoder is now STRICT — quotation marks are not NOTA string delimiters
at all, not even as legacy migration input.

Note the asymmetry vs nota-codec's own INTENT.md (`/git/.../nota-codec/INTENT.md`
§"Decoder accepts legacy quoted strings for migration"): the INTENT.md prose
still describes a `read_legacy_quote_string` lenient-decoder path that `f761421`
actually DELETED. The INTENT.md is stale relative to the code at lojix's pin —
intent record 703 authorized legacy-acceptance removal "once all emitter sites
migrate", and `f761421` executed that removal. This is a nota-codec-internal
doc-drift, not a lojix problem, but worth flagging to whoever owns nota-codec.

Does this break any lojix input path? NO. Decisive evidence:

- lojix's only hand-authored NOTA-text fixture is the daemon config in
  `tests/build_smoke.rs:104-108`:
  `format!("([{}] 432 [{}] 432)", ordinary_socket.display(), owner_socket.display())`.
  The socket-path `String` fields are emitted in BRACKET form `[{}]`, never
  quoted. The accompanying comment (lines 100-103) explicitly documents the
  bracket-string, head-less positional record shape.
- lojix's CLI does NOT decode inline NOTA text at all yet:
  `src/client.rs:73` is `ComponentArgument::InlineNota(_) =>
  Err(Error::InlineNotaUnsupported)`. SignalFile and NotaFile arguments are
  decoded as rkyv signal frames (`decode_signal_file` →
  `Input::decode_signal_frame`), not parsed as NOTA text. So there is no
  request-text path through nota-codec that could receive a quoted string.
- A repo-wide scan for `"` in NOTA position across `schema/`, `src/schema/`,
  and `tests/` finds quotation marks ONLY inside Rust string literals (error
  messages, `panic!`, `.to_string()` test data) — never inside NOTA content.

Is lojix relying on quoted strings anywhere (it must not)? NO — confirmed by
the scan above. lojix is already canonical-compliant: bracket strings for all
`String`-position config, and no live text request path.

Verdict: the property is correct and already in lojix's pin. No action; lojix
satisfies the "must not rely on quoted strings" rule.

### `9e855d4` — emit block strings for bracket-safe text (ancestor of pin)

What changed: `should_write_block` in the encoder now prefers the `[|...|]`
block form for content containing `]` (when safe). This is an ENCODER
preference; it only affects what nota-codec WRITES.

Bearing on lojix: lojix barely uses the nota-codec encoder on a live path —
the daemon config is hand-`format!`ed in the test, and replies go out as rkyv
signal frames, not encoded NOTA text. If lojix ever calls
`DaemonConfiguration`'s encoder (e.g. a config-emit helper), it would now get
block-form for bracket-bearing paths, which is strictly more canonical. No
behavioral risk. Already in lojix's pin regardless.

### `a7aa75b` — expand structural shape helpers for schema macros (ancestor)

What changed: +181 lines in `nota-codec/src/value.rs` adding structural shape
predicates for the schema macro-pattern layer (the `NotaValue` tree helpers).
This is consumed by schema-next / the brilliant-macro library, NOT by typed
value codec users.

Bearing on lojix: NONE. lojix consumes nota-codec only via `NotaRecord`/the
typed `Decoder` for config; it does not use the `NotaValue` structural-shape
tree. Already in lojix's pin. Non-applicable.

## What this means for the rkyv-vs-text split in lojix

A structural observation worth recording: lojix's NOTA-text surface is
deliberately thin and entirely gated behind the `nota-text` feature (off by
default — `Cargo.toml` `default = []`, `nota-text = ["dep:nota-next"]`). The
generated `from_nota_block` / `to_nota` methods on every schema type are
`#[cfg(feature = "nota-text")]`. The live daemon and CLI run on rkyv signal
frames (`decode_signal_frame` / `encode_signal_frame`); NOTA text is a
developer/debug and config-authoring affordance, not the request hot path. This
is why the nota-codec strictness changes have such low blast radius for lojix:
the request plane simply isn't NOTA-text.

## Recommendations

1. MUST-PORT (trivial): bump lojix's `nota-next` lockfile pin from `fb600e3`
   to `f0e435a` (takes both `d996a30` + `f0e435a`). Both are safe — a dead-code
   parser deletion lojix never exercised, and a derive-shape addition lojix
   doesn't use. Effort small (lockfile `cargo update -p nota-next` on the
   designer feature branch, no source change). Risk near-zero; the only
   behavioral surface that moved (the `StructuralMacroNode` derive) is not
   referenced by lojix. Rationale: keeps lojix's direct `nota-next` aligned
   with what `schema-rust-next` will emit against, and clears the
   already-flagged drift.

2. NICE-TO-HAVE (docs hygiene, not lojix-owned): flag to the nota-codec owner
   that `nota-codec/INTENT.md` §"Decoder accepts legacy quoted strings for
   migration" is stale — `f761421` deleted the legacy quote-decode path it
   describes. lojix should not act, but a cloud-designer sweep over the triad
   crates may want to capture this. Effort small; risk none (doc-only).

3. WATCH-ITEM (no action now): if a future lojix schema revision adopts the
   streaming-relation signature form (`(X Y opens Z)` / `belongs`) — which the
   `nexus.schema` "follows cloud nexus exactly" note hints at — the
   `#[shape(keyword=...)]` capability becomes a transitive build requirement
   via `schema-rust-next`'s own `nota-next` dependency. Track schema-rust-next's
   `nota-next` pin at that point, not lojix's direct pin. Effort n/a until
   triggered; risk is a build break if schema-rust-next emits keyword-shaped
   code against an old `nota-next` — but that is schema-rust-next's pin to
   manage, surfaced in the sibling schema-rust-next survey.

4. NON-ACTION (explicitly rejected as not applicable): do NOT migrate any lojix
   generated type to `StructuralMacroNode`. The value-codec `NotaDecode` /
   `NotaEncode` derives are the correct layer for lojix's data-bearing schema
   nouns; `StructuralMacroNode` is the schema-LANGUAGE consumer layer
   (schema-text authoring), which lojix is downstream of, not a user of.

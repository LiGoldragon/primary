### Substituters arrive pre-resolved on the wire — daemon does not resolve node-name to cache URL/key from horizon
*finder: 🟡 minor → verifier: **confirmed**, 🟡 minor; production gate: no*

- **State:** The meta-signal-lojix `ExtraSubstituter` wire type carries `url` + `public_key` already resolved, and the daemon passes them straight through. lojix-cli instead takes substituter NODE NAMES in the request and resolves each to its `nixUrl` + `nixPubKeyLine` from the projected horizon (preferring the Yggdrasil address so bootstraps don't depend on DNS). In the new split, that resolution work has no home: the thin CLI has no horizon access, and the daemon never resolves names. Either the operator must hand-resolve cache URLs/keys (a regression in ergonomics and correctness vs lojix-cli's name-based selection), or this resolution must be added to the daemon. Not a hard activate-blocker but a real parity gap in builder/substituter handling.
- **Evidence:** meta-signal-lojix/schema/lib.schema ExtraSubstituter { url String public_key String }; schema_runtime.rs:261-271 convert_substituters (pass-through). Parity bar: lojix-cli/src/deploy.rs:210-246 (ExtraSubstituters::from_horizon_nodes resolves name -> nixUrl/nixPubKeyLine, ygg-address preference), README.md:275-281.
- **Fresh re-read:** meta-signal-lojix/triad-port/schema/lib.schema:76 ExtraSubstituter { url String public_key String } — resolved on the wire. schema_runtime.rs:261-271 convert_substituters is pure pass-through (url/public_key copied).

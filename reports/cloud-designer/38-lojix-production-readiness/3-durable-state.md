# durable state — Durable state — is the live generation set persisted or lost on restart

cloud-designer, 2026-06-10. lojix main HEAD `7d66d2e`. Audit dimension `durable-state`.

**Finder status:** `ready` → **Verified status (after adversarial re-check):** `blocked`

## Net assessment (after re-reading current source)

CONFIRMED on main at HEAD 7d66d2e (2026-06-10). The entire lojix state plane — live generation set, GC-roots tree, append-only event log, container mirror, and all four monotonic sequence counters — lives only in an in-memory Mutex<StoreState> of Vecs, recreated empty on every process start. lib.rs:132-145 (StoreState::default all-empty), lib.rs:185-187 (Store::new -> default), daemon.rs:142 (one Store::new per process), daemon.rs:153-155 (actor start() hook is a literal Ok(()) no-op). Cargo.toml [dependencies] carries NO redb, sema-engine, sled, or rusqlite — there is no durable backing of any kind. The ARCHITECTURE.md:50-51 ('BTreeMap persisted via sema-engine') and sema.schema:13-14 ('registers each family with sema-engine at startup') claims are target/aspirational, contradicted by the actual Vec-backed in-memory tables; INTENT.md:32-36 and ARCHITECTURE.md:122-123,145-146 are honest ('next storage cutover'). The often-cited GC-roots filesystem mitigation does NOT hold: zero symlink-creation code exists (only std::os::unix use in all of src is UnixStream in client.rs:11), and the live_set/gc_roots tables are written ONLY by RecordGenerationActivated (schema_runtime.rs:976-993), reachable only after a deploy passes the reject-guard — which rejects every activating action (schema_runtime.rs:525-541, guard fires at line 486-490 before submission). So today the tables stay essentially empty and the amnesia has little live state to lose, but the durability gate is real and acute the instant activation lands. The misleadingly-named DaemonConfiguration.state_directory_path is used ONLY to derive generated-inputs for horizon flake materialization (schema_runtime.rs:67-68), never for state persistence. No restart/persist/reload test exists. Net: this is a self-stated, accurately-documented (in INTENT.md) production blocker for replacing lojix-cli as the durable deploy authority — but it is the SECOND blocker behind the activation gate, not the first to bite at cutover.

## Findings

### Entire state plane is in-memory only — recreated empty on every restart
*finder: 🔴 blocker → verifier: **confirmed**, 🔴 blocker; production gate: **yes***

- **State:** The four durable tables (live generation set, GC-roots, event log, container mirror) plus all four monotonic sequence counters live in `StoreState`, held behind a single `Mutex` inside `Store`. `Store::new()` is `Self::default()`, and `StoreState::default()` initializes every table to `Vec::new()` and every counter to 0. The daemon constructs exactly one `Store::new()` per process at `LojixRuntime::new` and never reads any prior state. On any restart or crash the daemon comes up with an empty live set, empty GC-roots tree, empty event log, and commit_sequence reset to 0 — it forgets which generation runs on every node, the entire point of its existence as 'source of truth.'
- **Evidence:** /git/github.com/LiGoldragon/lojix/triad-port/src/lib.rs:120-198 (StoreState fields + Default impl all-empty; Store = Mutex<StoreState>; Store::new -> default); /git/github.com/LiGoldragon/lojix/triad-port/src/daemon.rs:142 (`store: Arc::new(Store::new())`); lib.rs:11-12 doc admits in-memory Mutex-backed state, redb persistence a 'noted follow-on'
- **Fresh re-read:** lib.rs:121-145 (StoreState fields: live_set/gc_roots/event_log/containers + 4 u64 counters; Default impl sets every table to Vec::new() and every counter to 0); lib.rs:179-187 (Store = Mutex<StoreState>, Store::new()->Self::default()); daemon.rs:140-146 (LojixRuntime::new constructs Arc::new(Store::new()) once per process); no read of any prior state anywhere. HEAD 7d66d2e confirmed via git log -1.

### No persistence backing exists at all — no redb, no sema-engine, no DB crate in the dependency tree
*finder: 🔴 blocker → verifier: **confirmed**, 🔴 blocker; production gate: **yes***

- **State:** Despite ARCHITECTURE.md:50-52 describing the live set as 'persisted via sema-engine' and sema.schema:13-14 claiming 'the daemon registers each family with sema-engine at startup,' neither is true in code. Cargo.toml lists no redb, no sema-engine, no sled, no rusqlite — no database dependency whatsoever. A crate-wide grep for redb/persist/load_state/fsync/sync_all/File:: in src/ returns only the in-memory doc comments and the unrelated `generated_inputs_directory` (horizon flake materialization). `SchemaRuntime` implements the `sema::SemaEngine` trait (schema_runtime.rs:1937) but every read/write goes straight to the in-memory `Mutex<StoreState>`; there is no registration, no on-disk table family, no commit to durable storage.
- **Evidence:** /git/github.com/LiGoldragon/lojix/triad-port/Cargo.toml [dependencies] (no redb/sema-engine/sled/rusqlite); grep across src/ for redb|persist|fsync|sync_all|File:: -> only lib.rs:11-12,177 doc comments + schema_runtime.rs:67,1392 (generated-inputs/proposal, not state); schema_runtime.rs:1937 (SemaEngine impl over in-memory store); sema.schema:13-14 (aspirational 'registers with sema-engine' claim, unimplemented)
- **Fresh re-read:** Cargo.toml [dependencies] read in full: horizon-lib, nota-codec/config/next, meta-signal-lojix, signal-lojix, rkyv, serde_json, thiserror, tokio, triad-runtime — NO redb/sema-engine/sled/rusqlite. grep across src/ for redb|sema_engine|persist|load_state|fsync|sync_all|File:: returns only doc comments (lib.rs:10-12,119,175-177) and state_directory_path->generated-inputs (schema_runtime.rs:67). SemaEngine impl at schema_runtime.rs:1937-1954 routes apply_inner/observe_inner straight to apply_sema/observe_sema over the in-memory Mutex<StoreState>; no registration, no on-disk family, no durable commit.

### Daemon does NOT self-resume persisted SEMA state on startup — violates the one-argument-daemon rule
*finder: 🔴 blocker → verifier: **confirmed**, 🔴 blocker; production gate: **yes***

- **State:** The workspace hard override states 'On restart a daemon self-resumes from persisted SEMA state.' The lojix daemon does the opposite: its actor-runtime `start()` hook is `async fn start(&self) -> Result<()> { Ok(()) }` — a pure no-op with no load, restore, recover, or bootstrap of any prior state (a grep for those verbs in src/ finds nothing). The bin entry (`lojix-daemon.rs`) decodes the NOTA config and calls `Daemon::run`, which binds sockets and serves; no persisted state is ever consulted. A virgin start and a restart-after-a-week-of-deploys are byte-identical empty states. This is a direct, verifiable violation of the stated restart-resume contract, not merely a missing optimization.
- **Evidence:** /git/github.com/LiGoldragon/lojix/triad-port/src/daemon.rs:153-155 (start() => Ok(())); /git/github.com/LiGoldragon/lojix/triad-port/src/bin/lojix-daemon.rs:16-19 (run: decode config then Daemon::run, no state load); grep for load|restore|recover|bootstrap in src/ -> no state-load path
- **Fresh re-read:** daemon.rs:153-155 (async fn start(&self) -> Result<()> { Ok(()) } — pure no-op); bin/lojix-daemon.rs:16-19 (run() decodes ConfigurationSource then Daemon::new(configuration).run(); no state load); grep for load|restore|recover|bootstrap in src/ finds zero state-load path. Directly violates the workspace hard override 'On restart a daemon self-resumes from persisted SEMA state.' A virgin start and a restart-after-deploys are byte-identical empty states.

### GC-roots filesystem mitigation does NOT apply — no on-disk symlinks are written, and the index can't reconstruct from disk
*finder: 🟠 major → verifier: **confirmed**, 🟠 major; production gate: no*

- **State:** The task premise that the GC-roots tree is on the filesystem (/nix/var/nix/gcroots/criomos/...) and could reconstruct the in-memory index is FALSE on this build. There is zero symlink-creation code: a grep for symlink|gcroots|add-root|/nix/var/nix/gcroots|std::os::unix in schema_runtime.rs returns nothing; all `fs::` calls (lines 1476-1578) write generated flake inputs for horizon materialization, not GC-root symlinks. The `GcRootsTable` is mutated only as an in-memory Vec (push/find/remove at schema_runtime.rs:986,1019,1046,1074-1083). Moreover, GC-root rows are only ever inserted by `RecordGenerationActivated`, which is reachable only by a deploy that passes the reject-guard — and all activating deploys are rejected (next finding). So nothing is on disk to reconstruct from, and even if it were, there is no reconstruction code path. ARCHITECTURE.md:53-55 describes the on-disk symlink tree as the target design; it is unimplemented.
- **Evidence:** grep symlink|gcroots|add-root|/nix/var/nix/gcroots|std::os::unix in /git/github.com/LiGoldragon/lojix/triad-port/src/schema_runtime.rs -> no matches; schema_runtime.rs:986,1019,1046,1074-1083 (GcRootsTable mutated as in-memory Vec only); schema_runtime.rs:1476-1578 (fs:: writes are generated flake inputs); ARCHITECTURE.md:53-55 (on-disk gcroots tree is target, not built)
- **Fresh re-read:** grep symlink|gcroots|add-root|/nix/var/nix/gcroots|std::os::unix across ALL of src/ returns only client.rs:11 (use std::os::unix::net::UnixStream — socket connect, not a GC root). schema_runtime.rs:986-993 mutates gc_roots only as an in-memory Vec push; pin/unpin at 1004-1068 mutate it in place. ### State loss is largely moot TODAY because no activating deploy ever populates the tables
*finder: ℹ️ info → verifier: **confirmed**, ℹ️ info; production gate: no*

- **State:** The live-set and GC-roots tables are written only via `RecordGenerationActivated`, reached only at the end of the deploy pipeline. The reject-guard `unsupported_deploy_reason` accepts ONLY `SystemAction::Eval`/`Build` and `HomeMode::Build`; every activating action (Switch/Boot/Test/BootOnce, Home Profile/Activate) is rejected with `UnsupportedDeployAction` because 'copy/activate is not yet target-safe.' Eval/Build do not write the live set. Consequently, in the current build the in-memory tables stay essentially empty (only pin/unpin/retire could touch GC-roots, but those require a prior activation that can't happen), so the restart amnesia has little live state to lose right now. This does NOT downgrade the storage gate: the moment activation lands, in-memory-only state becomes an acute hazard, and durable backing must precede or accompany activate. It is the activation gate, not durability, that blocks cutover first; durability is the immediately-following blocker.
- **Evidence:** /git/github.com/LiGoldragon/lojix/triad-port/src/schema_runtime.rs:521-541 (reject-guard: only Eval/Build/Home-Build accepted); schema_runtime.rs:976-998 (live_set/gc_roots written only in RecordGenerationActivated path); ARCHITECTURE.md:16 ('reject until copy/activate is target-safe')
- **Fresh re-read:** schema_runtime.rs:525-541 (unsupported_deploy_reason accepts ONLY SystemAction::Eval|Build and HomeMode::Build; all else -> UnsupportedDeployAction); schema_runtime.rs:486-490 (guard fires at the top of decide_meta_input Deploy arm, replying DeployRejected before any RecordDeploySubmitted). live_set/gc_roots written only via RecordGenerationActivated (schema_runtime.rs:891-892->956-998, reached at line 638 only after a full pipeline that an activating deploy never enters). Eval/Build do not write the live set. Pin/Unpin/Retire require a prior activation that cannot happen. So tables stay empty today; durability becomes acute the instant activation lands. Activation is the first cutover blocker, durability the immediately-following one.

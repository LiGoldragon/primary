### No timeout or kill on the nix/ssh child process — the one residual gap shared with cloud, but far less severe
*finder: 🟡 minor → verifier: **confirmed**, 🟡 minor; production gate: no*

- **State:** NixCommand::run() (schema_runtime.rs:1838-1855) does Command::new(...).output().await with NO tokio::time::timeout and NO kill_on_drop. A hung `nix build`/`nix eval`/`ssh activate`/`ssh nix-store --gc` (network stall, hung TLS, auth prompt on the remote) ties up its tokio task and one of the 64 concurrency permits on that listener indefinitely. The only timeout in the daemon is the 10s REQUEST_READ_TIMEOUT on the inbound frame read (daemon.rs:30,228), which bounds connect-and-never-send, NOT the subprocess. This is the same 'timeout-free subprocess' class cloud flagged. BUT the blast radius is fundamentally smaller than cloud's: because there is no shared lock held across the await and connections are per-task, one hung build does NOT stall the daemon — it consumes one permit; the daemon keeps accepting and serving other requests until enough hung builds (up to 64 on one listener) exhaust that listener's pool. The gap is undocumented: neither ARCHITECTURE.md nor INTENT.md mentions a subprocess-timeout follow-on.
- **Evidence:** /git/github.com/LiGoldragon/lojix/triad-port/src/schema_runtime.rs:1838-1855 (no timeout); daemon.rs:30,228 (only the frame-read timeout); grep of ARCHITECTURE.md/INTENT.md for timeout/hung/kill returns nothing
- **Fresh re-read:** NixCommand::run (schema_runtime.rs:1838-1854) is Command::new(&self.program).args(...).output().await with NO tokio::time::timeout and NO kill_on_drop. Whole-src grep for tokio::time::timeout/kill_on_drop/.kill()/timeout( returns only daemon.rs:228 — the 10s REQUEST_READ_TIMEOUT (daemon.rs:30) which wraps ONLY codec.read_body_async (daemon.rs:228-229), bounding connect-and-never-send, not the subprocess. grep of ARCHITECTURE.md/INTENT.md for timeout|hung|kill|subprocess returns NONE — the gap is undocumented. Blast radius correctly scoped: a hung nix/ssh consumes one task + one of 64 permits on one listener; the daemon keeps accepting/serving (no shared lock held, per-task isolation) until that one listener's 64-permit pool is exhausted by hung builds. Minor for THIS dimension; would matter more for an availability/resilience dimension.

### Per-listener accept loop does head-of-line block on permit exhaustion (bounded, single-listener only)
*finder: 🟡 minor → verifier: **confirmed**, 🟡 minor; production gate: no*

- **State:** Within one listener, serve_next_connection awaits acquire_permit BEFORE spawning the handler (actor_runtime.rs:897-901, gate ask at :914). So if all 64 permits on a listener are held by in-flight (possibly hung, per the timeout finding) deploys, that listener stops accepting new connections until a permit frees. This is bounded backpressure by design (caps live work so a flood cannot exhaust resources, daemon.rs:120-124) and is isolated to the saturated listener — the OTHER socket keeps serving (separate gate). The gate actor's mailbox itself stays responsive (acquire is delegated via context.spawn at actor_runtime.rs:472, proven by the request_gate_mailbox_stays_available_while_permit_waits test). This is acceptable backpressure behavior, not a stall, but combined with the missing subprocess timeout it means a burst of hung deploys could wedge one listener's admission until restart.
- **Evidence:** triad-runtime@069ba8ed:src/actor_runtime.rs:897-901,914,472; lojix daemon.rs:120-124; gate tests at actor_runtime.rs:1152-1196
- **Fresh re-read:** Per-listener ActorListenerTask::serve_next_connection (actor_runtime.rs:897-901): accept_connection().await, then accepted_connection() which calls acquire_permit().await (:909-910) BEFORE spawn_connection (:901). So if all 64 permits on one listener are held, that listener stops accepting until a permit frees — bounded backpressure, isolated to the saturated listener (the other socket's separate gate keeps serving). The gate actor's mailbox stays responsive: AcquireRequestPermit handler delegates via context.spawn(async move { pool.acquire().await }) returning DelegatedReply (:462-473), so the gate does not block on a pending acquire. Acceptable backpressure, not a stall; combined with the missing subprocess timeout, a burst of hung deploys could wedge one listener's admission until restart — minor, single-listener-scoped, non-gating for this dimension.

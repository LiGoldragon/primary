## 1. signal-cloud (Ordinary Contract) — Operation & Reply Enumeration

**File:** `/git/github.com/LiGoldragon/signal-cloud/src/lib.rs`

### Operations
The `Operation` enum (generated by `signal_channel!` macro at line 325-329) defines two operations:

- **`Observe(Observation)`** (line 327): Read-only queries to retrieve provider/capability/zone/record/redirect/plan state.
  - Observation payload carries discriminated union of query types (lines 231-237): Capabilities, Zones, Records, Redirects, Plan
  - Each query type carries provider, account, zone, or plan identifier context

- **`Validate(Validation)`** (line 328): Validation of desired state (records and redirects) against provider rules.
  - Payload: `DesiredState` (provider, zone, records, redirects) at line 200-205

### Replies
The `Reply` enum (lines 330-335) defines five variants:

- **`Observed(ObservationResult)`**: Success response carrying observation results (line 331)
  - Result discriminated union (lines 263-269): Capabilities, Zones, Records, Redirects, Plan

- **`Validated(ValidationReport)`**: Success response with validation findings (line 332)
  - Findings vec of severity + message (lines 287-289)

- **`RequestUnsupported(RequestUnsupported)`**: Stub reply (line 333)
  - Reason enum (lines 294-301): ProviderNotBuilt, ProviderNotCompiled, ProviderNotConfigured, CapabilityNotCompiled, CapabilityNotConfigured, CapabilityUnauthorized

- **`RequestRejected(RequestRejected)`**: Rejection reply (line 334)
  - Reason enum (lines 313-318): InvalidDesiredState, PlanExpired, ProviderRateLimited, ProviderUnavailable

No `Reserved` variants. All observation payloads are concrete records; no TODO markers in the contract layer.

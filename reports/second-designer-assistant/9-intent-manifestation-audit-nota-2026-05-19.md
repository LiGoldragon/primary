## What's not manifested

Three psyche decisions about the intent-log surface itself are recorded as Maximum-certainty but haven't yet landed in the intent files or their governing skills. They affect every intent record written from now on, including ones I've been writing this week.

The first is the **two-field timestamp shape** (intent/workspace.nota lines 203–208, Maximum, 2026-05-19 18:30): *"why? that looks better. two fields, date then time."* The decision settles a still-open complaint from earlier the same evening that the `T13:07:19Z` form was *"a little bit excessive ... what the fuck is that about?"* (workspace.nota lines 182–187). The settled shape is a bare date field followed by a bare time field with the NOTA positional separator (a space) between them, no `T`, no `:00Z`. **Nothing on disk uses this yet.** Every recorded record — including the three NOTA entries I authored this week — still uses the old single-field ISO-8601 form. The decision authorises a script that rewrites existing timestamps; that script hasn't been written, the rewrite hasn't happened, and `skills/intent-log.md` still specifies the single-field form in §"Record shape". This is a workspace-wide sweep that touches every `intent/*.nota` file.

The second is the **restatement reaffirmation mechanism** (intent/workspace.nota lines 189–215). The problem statement landed Maximum: *"if an intent comes through one of my prompts that already has been worded in close enough words before ... a properly behaving agent would not just add more, just add another entry"* — the gap is that recording a fresh entry every time loses the recency signal on the *original* intent. The proposed solution (workspace.nota lines 210–215) is Medium: turn the `verbatim` field from a single string into a vector of per-statement structs `(date time quote)`. **Neither the rule (don't duplicate restatements) nor the shape (verbatim becomes a vector) is encoded in `skills/intent-log.md` yet.** The skill still describes a single verbatim string per record and says "Query prior entries on the topic" without naming reaffirmation as a distinct outcome from supersession. The vector-of-statements shape would also fight slightly with the homogeneous-list rule the psyche established this session (intent/nota.nota line 38: *"vectors are of the same type"*) — an inner record `(date time quote)` per statement is fine, but the brainstorm is still open.

The third is the **verbatim-italics convention for INTENT.md and ESSENCE.md** (intent/workspace.nota lines 287–304, Maximum). Skill `skills/intent-manifestation.md` already documents both the single-line (`*…*`) and multi-paragraph (markdown blockquote with italics inside) forms, and explains *why* blockquote is needed for multi-paragraph (CommonMark italics close at any blank line). **But the nota repo's `INTENT.md` that I created this week contains zero italicised verbatim quotes** — it's pure agent synthesis. That's not strictly wrong (the convention says italics flag *load-bearing* original wording, not every paragraph), but the three psyche statements driving the nota intent (positional records, the wrapping-type rule, the typed-Path direction) each have a quotable verbatim that could load-bear if italicised. Worth one revisit pass.

## What needs psyche clarification

Four questions surfaced from this audit. None are critical-blocking, but each is the kind of judgment-call where inferring would create more pollution than asking.

The first is the **restatement-vector shape**. The Maximum-certainty decision is that agents must detect restatements and reaffirm rather than duplicate. The Medium-certainty brainstorm is *"the verbatim field becomes a vector of (date, time, quote) structs"*. Should I treat the vector shape as decided and start authoring new records in that form, or hold off until the psyche commits the brainstorm? The downstream cost differs significantly: writing in the vector shape today means every reader of `intent/*.nota` has to handle both the legacy single-string and the new vector form during a transitional period; holding off means the gap stays open. The shape itself is also load-bearing because it interacts with the homogeneous-list rule — every inner element being `(date time quote)` is homogeneous (the same record type), which works, but the vector field then has a different inner type from the record's other fields, and it'd be the first place in the intent log where a record's fields are themselves a vector of records rather than scalar values.

The second is the **timestamp-rewrite scope and trigger**. The two-field shape is settled, but no record has been migrated. Options: (one) sweep all existing timestamps now via a small script, (two) leave existing records on the old form and write only new records in the two-field shape, (three) wait until persona-spirit ships and have it migrate automatically. The intent-log skill's "no transitional shape" discipline argues for option one, but it touches every intent file across the workspace — a coordinated edit pass.

The third is whether **the existing nota repo `INTENT.md` I created this week should be revisited to add italicised verbatim quotes**. The current synthesis is faithful but voice-less; adding a few load-bearing italicised psyche quotes (the wrapping-type *"this is really stupid. You're wrapping every single entry"* line, the typed-Path *"in a object that expects a pass"* phrasing, the three-case rule's *"PascalCase at the start of `(…)` is ALWAYS an enum variant"* constraint) would let the psyche's voice come through. Worth doing or premature aesthetic polish?

The fourth, narrower: **does the no-labeled-fields rule extend to data tables in markdown reports?** A two-column "Commit | Description" table carries genuinely tabular data; a "Status: …", "Pros: …" labeled card is the noise pattern the psyche flagged. I've been treating the two as distinct, but the rule in workspace.nota line 197 says *"no rendered tables with field-name chips"* — that phrase reads ambiguously. If everything table-shaped is out, this report needs another pass.

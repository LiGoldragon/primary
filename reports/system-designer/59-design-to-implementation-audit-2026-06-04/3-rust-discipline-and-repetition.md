# Slice A3 — Rust discipline and repetition (the missing-abstraction hunt)

Audit of `persona-spirit`, `signal-persona-spirit`, and `spirit-next`
against `skills/rust-discipline.md` (+ sub-files), `skills/abstractions.md`,
and `skills/actor-systems.md`. Read-only. Scope: the variant-ladder /

## Intent Anchors

[Spirit operations should support a simpler-to-more-complex variant ladder: short forms with summary defaults for normal operations, complex forms with full metadata for custom operations.] (Spirit 1474 High)

[Common Spirit operations should have short default forms that lower to the full data types, so agents do not compose every rarely-configured field for routine calls.] (Spirit 1545 High)

[Operations that extract or emit content accept a customizable output-target enum as the final field. Variants: Stdout, Stderr, File(path). Not an error channel. Keeps the wire interface uniform across extraction operations.] (Spirit 1548 High)

[Spirit defines a small-record data type carrying core load-bearing fields — identifier, topics, kind, description summary, magnitude, daemon-stamped date and time. It is what variant-ladder short-form reads and CollectRemovalCandidates emit; what archiving and downstream tools consume.] (Spirit 1549 High)

[Spirit gains a RecordDefault short-form recording operation taking only fields agents commonly customize — topics, kind, description, magnitude — with defaults injected for the rest (privacy Zero/open, daemon-stamped date/time). Record remains the canonical full-fidelity operation.] (Spirit 1550 High)

[ESSENCE: beauty = repetition resolves into a single named pattern.]

## Orientation — what is hand-written vs generated

The repetition findings concentrate where the code is **hand-written**:
`persona-spirit` (the production daemon — INTENT.md §"current production
target") and `signal-persona-spirit` (the hand-written working contract,
whose `spirit.schema` is fed to `signal_frame::signal_channel!` only for
the operation roots / reply / event tree). `spirit-next/src/schema/lib.rs`
is `// @generated by schema-rust-next`; its parallel `Input`/`Output`/
`*Route` enums are macro emission, not hand-rolled mirrors, so they are
out of scope for the missing-abstraction hunt. Where the two daemons share
a shape, only the hand-written one is flagged.

## V2 — Identical duplicated free function (V1 ∩ Part 2)

`daemon_configuration_argument_text` (daemon.rs:722) and
`migration_configuration_argument_text` (migration.rs:247) have
**byte-identical bodies** (verified by `diff` — zero body lines differ):

```rust
fn daemon_configuration_argument_text(argument: signal_frame::SingleArgument) -> Result<String> {
    let value = argument.as_str();
    if value.starts_with('(') {
        Ok(value.to_string())
    } else {
        fs::read_to_string(value).map_err(Error::input_output)
    }

```

Two failures stack: each is a free function (V1), AND the same logic
exists twice (Part 2). The noun is the single-argument the CLI accepts —
this is `(NOTA-inline-or-file-path)` resolution, the same logic the
single-argument rule (`AGENTS.md` §"NOTA is the only argument language")
describes for every component binary. It belongs as one method, e.g.
`impl SingleArgument { fn into_nota_text(self) -> Result<String> }` or a
`SpiritArgument` newtype, attached once and called from both daemon and
migration entry points.

## V3 — ZST namespace method holders (free functions in disguise)

`skills/rust/methods.md` §"No ZST method holders" + the erase-the-name
test. Two unit structs in `store.rs` exist only to park functions:

`store.rs:427` —

```rust
struct HandoverClock;
...
impl HandoverClock {
    fn read() -> HandoverClockReading { ... }
}
```

`HandoverClock` carries no data; `read()` takes no `self`. Erase the name
and `read()` is a free function. The verb is "read the wall clock into a
civil date/time" — it belongs on the `ClockSource` noun that already owns
clock reads in `clock.rs` (see A1 below), or on `HandoverClockReading`
as a `from_now()` constructor.

`store.rs:473` `HandoverCivilDate` and `clock.rs:84` `CivilDate` are the
twin Hinnant-algorithm holders — they DO carry `year/month/day` fields, so
they are data-bearing, not ZSTs; their problem is duplication (A1), not the
ZST anti-pattern. (Also `daemon.rs:1782` `struct SocketBinding;` exists;
not read in depth here — flag for A1/A2 follow-up.)

## V4 — Stringly-typed errors leaking the typed boundary

`persona-spirit/src/error.rs` is a proper per-crate `thiserror` enum
(good — `skills/rust/errors.md` satisfied at the enum level). But the
variants have collapsed the typed domain into `String`:

```rust
#[error("persona-spirit store error: {reason}")]
SpiritStore { reason: String },
...
#[error("persona-spirit request rejected before execution: {reason}")]
RequestRejected { reason: String },
```

`SpiritStore { reason: String }` is built via `Error::spirit_store(error:
sema_engine::Error)` which immediately calls `error.to_string()`
(error.rs:119-123). The typed `sema_engine::Error` is erased into a
`String` at the boundary — the caller can no longer pattern-match what the
store actually failed at. `skills/rust/errors.md`: "Foreign error types
convert via `#[from]`" and "carry the data needed to render a useful
message." The disciplined shape is `SpiritStore(#[from] sema_engine::Error)`.
Same pattern recurs in `SignalFrame`, `ActorRuntime`, `Migration`,
`InputOutput`, `ArchiveWrite` — all stringify a typed source. `RequestRejected
{ reason: String }` is the most-load-bearing offender: it is matched
elsewhere by `reason.contains("exact Zero")` (store.rs test line 1080),
which is exactly the "dispatching on string prefix" smell
(`skills/rust/methods.md` §"Don't hide typification in strings"). The
rejection reason should be a typed `RejectionReason` enum.

## V5 — Single-argument / flag-soup rule: PASS

`spirit.rs` is one line: `signal_frame::signal_cli!(spirit,
signal_persona_spirit);`. The single-argument contract (`AGENTS.md` §"NOTA
is the only argument language") is enforced in the shared `signal-frame`
kernel; `Error::FlagArgument` (error.rs:11) is the typed rejection. No
flag soup. Good.

## V6 — Blocking in actor handlers: PASS (within scope read)

The `SpiritCommandExecutor` handlers (`dispatch.rs`) reach downstream
planes via `ActorRef::ask().await` — non-blocking message sends, correct
per `skills/actor-systems.md` §"Blocking is a design bug". The store/file
IO (`RemovalCandidateArchive::write_to_file`, store.rs:408) runs inside the
`RecordStore` actor handler synchronously; that is a `std::fs` blocking
call on the mailbox. Flagged as a watch-item (B3) rather than a hard
violation because `RecordStore` is the durable-resource owner actor and
the write is short; the discipline's destination is a `FileWriter` plane.

# Part 2 — Repetition → the missing named abstraction

This is the priority output. Each finding quotes ≥2 sites and names the

## A1 — The civil-date algorithm exists THREE times (the headline)

The Howard Hinnant days-from-civil algorithm is implemented
character-for-character in two type holders plus a third `from_unix_seconds`
seconds-splitter wrapper.

Site 1 — `persona-spirit/src/actors/clock.rs:90`:

```rust
impl CivilDate {
    fn from_unix_days(days: i64) -> Self {
        let zero_based_days = days + 719_468;
        let era = if zero_based_days >= 0 {
            zero_based_days
        } else {
            zero_based_days - 146_096
        } / 146_097;
        let day_of_era = zero_based_days - era * 146_097;
        let year_of_era =
            (day_of_era - day_of_era / 1_460 + day_of_era / 36_524 - day_of_era / 146_096) / 365;
        ...
    }
}
```

Site 2 — `persona-spirit/src/store.rs:480`:

```rust
impl HandoverCivilDate {
    fn from_unix_days(days: i64) -> Self {
        let zero_based_days = days + 719_468;
        let era = if zero_based_days >= 0 {
            zero_based_days
        } else {
            zero_based_days - 146_096
        } / 146_097;
        let day_of_era = zero_based_days - era * 146_097;
        let year_of_era =
            (day_of_era - day_of_era / 1_460 + day_of_era / 36_524 - day_of_era / 146_096) / 365;
        ...
    }
}
```

The bodies are identical. The two differ ONLY in (a) the type name and
(b) which `Date`/`Time` the surrounding `*Reading` constructs —
`clock.rs::ClockReading` builds `signal_persona_spirit::Date`,
`store.rs::HandoverClockReading` builds `signal_version_handover::Date`.
A third copy of the seconds-of-day splitting (`/ 86_400`, `/ 3_600`,
`% 60`) lives in both `ClockReading::from_unix_seconds` (clock.rs:69)
and `HandoverClockReading::from_unix_seconds` (store.rs:458).

The noun is missing. This is the canonical ESSENCE case: "repetition
resolves into a single named pattern." The verb "Unix instant →
(year, month, day, hour, minute, second)" belongs on ONE data-bearing
type — a `CivilInstant { date: CivilDate, ... }` or just a shared
`CivilDate` with `from_unix_days` and a sibling `CivilTime::from_seconds_of_day`.
The two `Date`/`Time` projections then become `impl From<CivilDate> for
signal_persona_spirit::Date` and `impl From<CivilDate> for
signal_version_handover::Date` (verb-on-noun via `From`, per
`skills/rust/methods.md` §"Constructors"). The algorithm is written once.

**This abstraction wants its own home.** Per `skills/micro-components.md`,
civil-date conversion is a textbook micro-component (a named calendar
crate) consumed by both `clock.rs` and `store.rs`; both files import a
typed `CivilDate`. At minimum, within `persona-spirit`, the algorithm
must collapse to one `impl CivilDate` in one module, with the two
component-`Date` projections as `From` impls.

## A2 — The dispatch trace-shuttle: `snapshot → ask → into_parts → replace` × 14

`persona-spirit/src/actors/dispatch.rs` repeats one shape **14 times**
(verified: 14 `into_parts()`, 14 `self.trace.replace(trace)`). Two
representative sites:

`dispatch.rs:299` (`remove_entry`):

```rust
async fn remove_entry(&self, identifier: ...) -> Result<WorkingReply> {
    let trace = self.trace.snapshot();
    let pipeline = self
        .store
        .ask(store::RemoveEntry { identifier, trace })
        .await
        .map_err(Self::store_send_error)?;
    let (reply, trace) = pipeline.into_parts();
    self.trace.replace(trace);
    Ok(reply)
}
```

`dispatch.rs:314` (`change_certainty`):

```rust
async fn change_certainty(&self, change: ...) -> Result<WorkingReply> {
    let trace = self.trace.snapshot();
    let pipeline = self
        .store
        .ask(store::ChangeCertainty { change, trace })
        .await
        .map_err(Self::store_send_error)?;
    let (reply, trace) = pipeline.into_parts();
    self.trace.replace(trace);
    Ok(reply)
}
```

`collect_removal_candidates`, `capture_stamped_entry`, `observe_records`,
`observe_record_identifiers`, `observe_topics`, `observe_state`,
`observe_questions`, `retract_state_subscription`,
`retract_record_subscription`, `shape_unimplemented` — all the same five
lines around a single `ask`. The verb is "ask a plane a trace-carrying
message, splice the returned trace back into shared state, return the
reply." That verb belongs on `SharedTrace` (the noun that owns the trace
mutation). Sketch:

```rust
impl SharedTrace {
    async fn ask_pipeline<Actor, Message>(
        &self,
        actor: &ActorRef<Actor>,
        build: impl FnOnce(ActorTrace) -> Message,
        map_error: impl FnOnce(SendError<Message, Actor::Error>) -> Error,
    ) -> Result<WorkingReply>
    where Actor: Message<Message, Reply = PipelineReply>, ...
    {
        let pipeline = actor.ask(build(self.snapshot())).await.map_err(map_error)?;
        let (reply, trace) = pipeline.into_parts();
        self.replace(trace);
        Ok(reply)
    }
}
```

Each handler collapses to one call. Generic bounds need care (the actors
have different `Error` types — see A3), but the splice-the-trace verb is
indisputably `SharedTrace`'s, and writing it once is the named pattern.
This is **partial-portable**: the refactor has no behavior change, but the
generic bound across `Infallible`-reply vs `Error`-reply actors is a real
design choice (see openDecision).

## A3 — Six near-identical `*_send_error` helpers

`dispatch.rs:512-537` — six methods, five of them identical:

```rust
fn state_send_error<Message>(error: SendError<Message, Infallible>) -> Error {
    Error::actor_runtime(error.to_string())
}
fn classifier_send_error<Message>(error: SendError<Message, Infallible>) -> Error {
    Error::actor_runtime(error.to_string())
}
fn clock_send_error<Message>(error: SendError<Message, Infallible>) -> Error {
    Error::actor_runtime(error.to_string())
}
fn subscription_send_error<Message>(error: SendError<Message, Infallible>) -> Error {
    Error::actor_runtime(error.to_string())
}
fn reply_send_error<Message>(error: SendError<Message, Infallible>) -> Error {
    Error::actor_runtime(error.to_string())
}
```

Five `SendError<_, Infallible>` → `Error::actor_runtime` bodies are
identical; only `store_send_error` (Error-reply) differs. The repetition
is the per-plane naming — the verb is "an infallible-reply actor's send
failed → actor-runtime error", which does not vary by plane. One method
`SharedTrace::infallible_send_error<M>(SendError<M, Infallible>) -> Error`
(or a `From<SendError<M, Infallible>> for Error` impl) replaces the five.
Portable: pure dedup, no behavior change. (Folds naturally into A2's
`map_error` argument.)

## A4 — `Effect` ↔ `WorkingReply`: two parallel 13-variant enums with inverse 1:1 maps

`persona-spirit/src/observation.rs` defines `enum Effect` (13 variants,
line 38) whose every variant wraps the same payload type as a
`WorkingReply` variant. `Effect::from_reply` (line 91) and
`Effect::into_reply` (line 119) are inverse identity maps:

`from_reply` (observation.rs:92):

```rust
WorkingReply::RecordAccepted(payload) => Self::RecordAccepted(payload),
WorkingReply::RecordRemoved(payload) => Self::RecordRemoved(payload),
WorkingReply::CertaintyChanged(payload) => Self::CertaintyChanged(payload),
...
```

`into_reply` (observation.rs:120):

```rust
Self::RecordAccepted(payload) => WorkingReply::RecordAccepted(payload),
Self::RecordRemoved(payload) => WorkingReply::RecordRemoved(payload),
Self::CertaintyChanged(payload) => WorkingReply::CertaintyChanged(payload),
...
```

26 match arms expressing one fact: `Effect` IS `WorkingReply` minus the
two reply variants `Effect` never carries (`SubscriptionRetracted` aside,
the sets differ only by `ObserverSubscriptionOpened`). The discipline
question (`skills/rust/methods.md` §"One type per concept"): does `Effect`
earn its separate existence, or is it `WorkingReply` wearing a
component-local label? `Effect` adds `ToSemaOutcome` (line 165) and the
`sema_observation_for` projection — that is real behavior. So the fix is
not "delete `Effect`"; it is to express the conversion as `impl From<WorkingReply>
for Effect` / `impl From<Effect> for WorkingReply` (the projection IS a
`From`, per `skills/abstractions.md` §"reach for `impl From`"), AND to ask
whether the two enums should share a derive so the arm list is generated
rather than hand-mirrored. Mirror-by-hand of a 13-variant enum is exactly
the "parallel hand-rolled mirror of schema-generated types" the brief
calls out: `WorkingReply` is macro-emitted from `spirit.schema`'s `(Reply
...)` block; `Effect` is its hand-written shadow. **Blocked**: collapsing
toward a shared emission touches the `signal_channel!` schema surface
(undecided whether the component-local executor enum should be
schema-derived).

`RecordQuery` has six fields. Constructing the "everything-but-X" query is

let query = RecordQuery {

## A6 — `RecordFilter` is a hand-copied projection of `RecordQuery`

`store.rs:434` `RecordFilter` holds five fields that are a field-for-field
copy of five of `RecordQuery`'s six fields:

```rust
struct RecordFilter<'query> {
    topic_selection: &'query TopicSelection,
    kind: Option<Kind>,
    certainty_selection: CertaintySelection,
    recorded_time_selection: RecordedTimeSelection,
    privacy_selection: PrivacySelection,
}

impl<'query> RecordFilter<'query> {
    fn new(query: &'query RecordQuery) -> Self {
        Self {
            topic_selection: &query.topic_selection,
            kind: query.kind,
            certainty_selection: query.certainty_selection,
            recorded_time_selection: query.recorded_time_selection,
            privacy_selection: query.privacy_selection,
        }
    }
    fn matches(&self, record: &StoredRecord) -> bool { ... }
}
```

`RecordFilter` re-shelves `RecordQuery`'s fields (dropping only `mode`,
which is a reply-shaping concern not a filter concern) purely to host the
`matches` verb. Per `skills/abstractions.md` §"verb belongs to noun": the
`matches`-against-a-record verb belongs on `RecordQuery` itself —
`query.matches(record)` — not on a parallel struct that copies its fields.
The five `matches_topic`/`matches_kind`/… sub-methods move to `RecordQuery`
as private methods. `RecordFilter` then vanishes. Note the selection types
(`TopicSelection`, `CertaintySelection`, …) ALREADY carry their own
`matches` methods in the contract (lib.rs:484, 561, 586, 617) — so
`RecordQuery::matches` is just five delegations, no logic of its own. The
mirror struct earns nothing. Portable: behavior-preserving move-method.

## A7 — Topic validation written twice (contract + daemon)

`Topics::validate` (signal-persona-spirit/lib.rs:79) and
`SpiritStore::validate_topics` (persona-spirit/store.rs:324) implement the
same two checks — non-empty + no-duplicate-topic — with the same
BTreeSet-seen loop. Contract side (lib.rs:87):

```rust
let mut seen = std::collections::BTreeSet::<&str>::new();
for topic in value {
    if !seen.insert(topic.as_str()) {
        return Err(nota_codec::Error::Validation { ... });
    }
}
```

Daemon side (store.rs:330):

```rust
let mut seen = std::collections::BTreeSet::<&str>::new();
for topic in topics.as_slice() {
    if !seen.insert(topic.as_str()) {
        return Err(Error::RequestRejected { ... });
    }
}
```

Same algorithm, different error type. The validity rule is `Topics`'
invariant — it belongs once on `Topics` (it already does, via the
NotaDecode path). The daemon re-validates because it constructs `Topics`
internally (classifier, migration) without going through decode. The
abstraction: `Topics::new` should be the validating constructor (or a
`Topics::try_new -> Result<Self, _>`) so the invariant is enforced at
construction and the daemon never re-checks. Currently `Topics::new`
(lib.rs:51) is non-validating — that is the gap that forces the
duplicate. Portable within the contract crate.

# Connection to the proposed auditor role (records 234/235)

The findings above are exactly the auditor's beat: A1 (triple algorithm),
A2 (×14 boilerplate), A3 (×5 identical helpers), A4/A6 (hand-mirrored
parallel enums/structs) are mechanical pattern-detections a smaller
pattern-checking model finds with grep + structural diff. Concrete
seeds for an automated auditor:

- **Identical-body detector** — A2, A3, V2 are all "≥2 functions whose
  token stream after the signature is identical." `diff`-of-bodies caught
  V2 in this audit; that is automatable.
- **Parallel-enum detector** — A4: two enums where variant-N wraps the
  same payload type and a function is a 1:1 arm-for-arm map between them.
- **Field-copy struct detector** — A6: struct B's fields are a subset of
  struct A's fields, and B's only constructor copies them from an A.
- **Free-function detector** — V1: `fn` at module scope outside
  `#[cfg(test)]` / `fn main()`. Trivially greppable; the `.claude`
  PreToolUse hook is the per-keystroke version, an auditor pass is the
  sweep version.

# Gaps against the design intent (1548, 1549, 1550)

## G1 — `OutputTarget { Stdout, Stderr, File }` is unimplemented; current `ArchiveTarget` is the wrong shape

Spirit 1548 decided the final field of extraction operations is an
`OutputTarget` enum with variants **Stdout, Stderr, File(path)**. Current
source has `ArchiveTarget { Inline, File(ArchivePath) }`
(signal-persona-spirit/lib.rs:802, spirit.schema:64):

```rust
pub enum ArchiveTarget {
    Inline,
    File(ArchivePath),
}
```

`grep` confirms **zero** occurrences of `OutputTarget`, `Stdout`, or
`Stderr` anywhere in the triad. The decided three-variant enum does not
exist; `Inline` is a different concept (return-in-reply) than `Stdout`
(write to process stdout stream). Spirit 1544 ("output target — file path
OR process stream") and 1548 (Stdout/Stderr/File) together name a
process-stream target the current `Inline` does not provide.

**Portability: blocked.** Although the field-level variant list IS given
(Stdout, Stderr, File), this is NOT a clean rename of `ArchiveTarget`: the
psyche must decide whether `Inline` survives as a fourth variant (the
current daemon and tests depend on `Inline` returning summaries in the
reply, store.rs:403, 1002) or whether `Stdout` replaces it. The naming
collision (`ArchiveTarget` vs decided `OutputTarget`; `ArchivePath` field)
and the Inline-vs-Stdout semantic question are open. Surfacing > inferring.

## G2 — `RecordDefault` short-form operation is unimplemented

Spirit 1550 (High) decided a `RecordDefault` operation root taking only
{topics, kind, description, magnitude}, defaulting privacy + daemon date/
time, with `Record` remaining canonical. `grep` confirms **zero**
occurrences of `RecordDefault` in the triad. The `spirit.schema` operation
list (lines 6-15) and `OperationKind` (line 97) have `Record` but no
`RecordDefault`. The variant-ladder primitive `Entry::open` (lib.rs:388)
exists (defaults privacy to Zero) — that is the lowering target — but the
short-form OPERATION ROOT that an agent calls is absent.

**Portability: blocked.** The decision is field-level clear ({topics,
kind, description, magnitude}), but landing it requires a new operation
root in the `signal_channel!` schema, a new `OperationKind` variant, a new
`Command`/`Effect` lowering arm, and a new dispatch route — and the
ordering of the four fields and whether `magnitude` here means `certainty`
(the contract aliases `Certainty = Magnitude`, lib.rs:375) needs the
psyche's confirmation against 1549's "magnitude" wording. The wire shape
of a NEW operation is a schema decision the psyche owns.

## G3 — The small-record type (1549) lacks daemon-stamped date+time

Spirit 1549 decided a small-record carrying **identifier, topics, kind,
description summary, magnitude, AND daemon-stamped date and time**. The
existing `RecordSummary` (signal-persona-spirit/lib.rs:856) carries
identifier, topics, kind, description, certainty, privacy — but **no date
and no time**:

```rust
pub struct RecordSummary {
    pub identifier: RecordIdentifier,
    pub topics: Topics,
    pub kind: Kind,
    pub description: Description,
    pub certainty: Certainty,
    pub privacy: Privacy,
}
```

The date/time live one level up in `RecordProvenance` (lib.rs:866, wraps
a `RecordSummary` + `Date` + `Time`). So today there are TWO record-view
types (`RecordSummary` and `RecordProvenance`) where 1549 describes ONE
small-record-with-stamp. This is the `-Details`/`-Info` companion
anti-pattern (`skills/rust/methods.md` §"One type per concept"):
`RecordProvenance` is "the real version of `RecordSummary` next door."
1549 resolves it — the small-record IS the stamped one.

**Portability: blocked.** 1549 names the field SET but adding date/time to
`RecordSummary` (and likely retiring `RecordProvenance` + the
`SummaryOnly`/`WithProvenance` `ObservationMode` split that exists to
toggle the two, lib.rs:332) is a contract reshape touching the reply tree,
the `ObservationMode` enum, and every reply-shaping arm. Whether
`RecordProvenance` is absorbed or kept is a design decision the psyche
should confirm — 1549 strongly implies absorption ("It is what
variant-ladder short-form reads… what archiving and downstream tools
consume") but does not say "retire RecordProvenance" in words.

## What HAS landed (honesty about state)

The design is substantially implemented, not greenfield:

- `CollectRemovalCandidates` operation root exists (spirit.schema:14,
  OperationKind:97), with `RemovalCandidateCollection`,
  `RemovalCandidatesCollected`, `SkippedRemovalCandidate`,
  `RemovalCandidateSkipReason` types (lib.rs:814-922) and a full
  store implementation (store.rs:127-160) that archives-then-retracts and
  preserves candidates on archive failure (Spirit 1543, 1542). Tested
  (store.rs:944, 1042, 1089).
- The variant-ladder DECODE for `RecordQuery` is built — the
  progressively-shorter NOTA forms are accepted by the hand-written
  `NotaDecode for RecordQuery` (lib.rs:663-737), which peeks for a trailing
  `Mode` token at each position and injects defaults for the omitted
  middle fields (Spirit 1474/1545 read-side). This is the variant ladder
  working for `Observe` — it is `RecordDefault` (write-side, G2) that is
  missing.
- The garbage-collection path is distinct from deletion (Spirit 1541):
  `CollectRemovalCandidates` (archive+retract) is a separate root from
  `Remove` (RecordIdentifier).

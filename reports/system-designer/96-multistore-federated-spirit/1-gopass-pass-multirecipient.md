# GoPass / pass and the multi-recipient envelope: mapping key-possession-gated stores onto Sema

## Scope

The psyche's direction (Spirit `dun9`) proposes rethinking Spirit as a *federation of key-scoped stores* on the GoPass model: each store encrypted to a set of recipient keys, where access equals key possession and the per-store key is simultaneously the access boundary and the crypto-shred erasure unit. This report validates the **GoPass/pass mental model and the multi-recipient envelope pattern** (age, SOPS, git-crypt) against the strongest prior art, maps it onto a Sema store = versioned encrypted log, and isolates the load-bearing distinction the psyche already flagged: **revocation is not erasure**.

## GoPass / pass: the multi-store mental model

`pass` (the original "standard unix password manager") is the cleanest possible substrate: a password store is a directory tree of `.gpg` files, and a `.gpg-id` file lists the GPG recipient key IDs that every secret in that directory (and below) is encrypted to. Multiple key IDs may be listed, one per line; a `.gpg-id` in a subdirectory overrides the parent for that subtree, so the encryption recipient set is **scoped by path**. ([password-store](https://git.zx2c4.com/password-store/about/), [ManKier pass(1)](https://www.mankier.com/1/pass))

GoPass generalizes this into the architecture the psyche is reaching for:

- **Stores are first-class, not just subdirectories.** GoPass lets you *mount* independent substores into one virtual tree, and explicitly recommends multiple stores — each encrypted to a *uniform recipient set* — as "the preferred approach to encrypting different sets of secrets for different recipients," in preference to per-subdirectory recipient lists. It targets up to ~100 mounted substores without noticeable performance impact. ([gopass multi-store](https://github.com/gopasspw/gopass/blob/master/docs/usecases/multi-store.md), [gopass features](https://github.com/gopasspw/gopass/blob/master/docs/features.md))
- **Each store carries its own recipient set.** Every store/mount has its own `.gpg-id` plus a `.public-keys/` folder holding the recipient public keys by fingerprint, so a store is self-describing about who can read it. `gopass recipients add/remove --store <mount>` operates per store. ([gopass mounts](https://github.com/gopasspw/gopass/blob/master/docs/commands/mounts.md))
- **Lookup routes across stores transparently.** A secret name is matched against the mounted substores and the action executes on the owning store — the operator sees one tree, but each leaf is gated by its store's recipients. This is exactly the psyche's "view = union of decryptable stores, queries fan out over accessible stores."

The decryption rule is the whole point: **which private keys you hold determines what you can read.** A daemon (or person) with key K decrypts exactly the stores whose recipient set includes K; everything else is opaque ciphertext it may still hold and replicate. The view is the union of the decryptable stores — by construction the *same store-set returns different results to different key-holders*, which is precisely the federation semantics dun9 calls for.

## The multi-recipient envelope pattern (age, SOPS, git-crypt)

Under the hood, none of these re-encrypt the *content* per recipient. They use **envelope encryption**: one symmetric data key encrypts the payload; that data key is then wrapped once per recipient public key.

- **age**: encrypts the file body under a per-file data key with ChaCha20-Poly1305; for each recipient it puts a stanza in the header wrapping the file key (X25519: an ephemeral public key plus the file key encrypted under an HKDF-derived wrap key). Any one recipient's private key unwraps the file key and decrypts; overhead is ~200 bytes per recipient, *not* a re-encryption of the body. ([age X25519](https://kula.tproa.net/lnt/2020/06/x25519-encryption-in-age/), [age(1)](https://www.mankier.com/1/age))
- **SOPS**: explicit envelope — a single data key is encrypted to N masters simultaneously (AWS/GCP/Azure KMS, age, PGP). Any one master decrypts the data key, which decrypts the values. ([getsops/sops](https://github.com/getsops/sops), [getsops docs](https://getsops.io/docs/))
- **git-crypt**: AES under one symmetric repo key; the symmetric key is wrapped to each collaborator's GPG public key in `.git-crypt/`. ([git-crypt](https://github.com/AGWA/git-crypt))

**Recipient-change = re-wrap the data key, not re-encrypt content.** SOPS makes this explicit: `--add-pgp`/`--rm-pgp`/`--add-kms`/… edit the *envelope* (who the data key is wrapped to), and only `rotate` generates a fresh data key and re-encrypts values. ([SOPS rotate/add/rm](https://github.com/getsops/sops)) The same separation exists in age (re-encrypt header stanzas) and git-crypt (re-wrap the shared key to remaining collaborators).

This is the single most important structural fact for Sema: **adding/removing a reader is a cheap envelope edit; making old data unreadable to a removed reader requires rotating the data key AND re-encrypting forward — and even then it does nothing about ciphertext they already hold.**

## Revocation vs erasure — the distinction the psyche named, confirmed by prior art

The psyche's instinct that "removing a recipient does NOT erase (they may have cached)" is the central, repeatedly-documented limitation of every one of these systems:

- **pass / gopass**: re-initializing a store to a new `.gpg-id` re-encrypts current files to the new recipient set, but a removed recipient who pulled the repo (or ran gpg-agent) already has the plaintext and the old ciphertext; the password itself should be considered compromised and rotated at the source. (Implied throughout [pass(1)](https://www.mankier.com/1/pass); standard team-pass guidance.)
- **git-crypt** states it most bluntly: there is no `del-gpg-user`; removing a collaborator only removes their wrapped key and re-keys *future* updates — "the ex-collaborator will still be able to retrieve past versions of encrypted data" from history. Revocation is forward-only; the past is not erased. ([git-crypt #47](https://github.com/AGWA/git-crypt/issues/47))
- **Matrix Megolm**: when someone leaves a room you "create a new key" — but every recipient who held the prior Megolm session can still decrypt everything sent before rotation. Forward-only again. ([Megolm docs](https://gitlab.matrix.org/matrix-org/olm/blob/master/docs/megolm.md))
- **p2panda** formalizes the inherent decentralized delay: until *all* members observe a removal and rotate, some peers keep encrypting to the removed member — an unavoidable "post-compromise window," and for its symmetric *data-encryption* scheme a removed member who synced the ciphertext keeps the key for all data it covered. ([p2panda group encryption](https://p2panda.org/2025/02/24/group-encryption.html))

**Erasure is a different operation with a different unit.** True deletion-by-key is **crypto-shredding**: render data unrecoverable by destroying its key, the basis for GDPR Article 17 right-to-erasure-without-deleting-rows. ([Crypto-shredding (Wikipedia)](https://en.wikipedia.org/wiki/Crypto-shredding), [crypto-shredding for GDPR](https://granit-fx.dev/blog/crypto-shredding-gdpr-erasure-without-deleting-rows/)) Crypto-shred works only when the key was never disclosed to the party you want to erase from — it erases *the data for everyone who only ever held ciphertext* (e.g. the zero-knowledge mirror), but it provably cannot erase data from a holder who already decrypted and cached the plaintext or who still holds an old data key. SecuPi's caution is the right adversarial note: crypto-shredding is "not nirvana" for erasure where copies and cached keys exist. ([SecuPi](https://secupi.com/crypto-shredding-is-not-nirvana-for-right-of-erasure-or-rtbf-compliance/))

So the two operations factor cleanly for Sema:

| Operation | Mechanism | Unit | Effect |
|---|---|---|---|
| **Revocation** (remove a reader going forward) | rotate the store's data key; re-wrap envelope to remaining recipients; re-encrypt new payloads under the new key | the *envelope* + a key epoch | forward-only; old ciphertext/plaintext the removed reader cached is unaffected |
| **Erasure** (make data unrecoverable) | crypto-shred: destroy the data key custodied in criome | the *per-store data key* | erases for every ciphertext-only holder (the ouranos mirror); cannot reach already-decrypted caches |

## Mapping onto Sema (validation of the dun9 model)

The GoPass model maps onto our design with almost no impedance, because our store is *already* the right shape — a typed, blake3-hash-linked, payload-bearing event log with a redb materialized view, committing over **ciphertext**, mirrored to ouranos:

- **A Sema store = a `.gpg-id`-style recipient-scoped versioned log.** The store's data key is wrapped (via criome's BLS-identity-held keys) to a set of recipient keys — the direct analog of `.gpg-id` + `.public-keys/`. A Spirit *instance* holding key K materializes exactly the stores whose envelope wraps to K. Because the log commits over ciphertext (per our design), the *log/hash-chain replicates to instances that cannot read it* — the GoPass "you hold the ciphertext but not the key" property is native.
- **View = union of decryptable stores; queries fan out** over the stores the instance can decrypt — gopass's transparent multi-store routing is the exact operational model, and it directly delivers dun9's "same query, different results per instance, by design."
- **The Privacy magnitude becomes a store-routing function, not a per-record threshold.** Today Privacy is a per-entry magnitude classified by the guardian (IntakePolicy). In the federation model, Privacy *selects the destination store* (its recipient set) — the magnitude is realized as *which envelope the entry's data key is wrapped to*, which is strictly stronger than a magnitude compared at read time, because a non-holder of the key can't read the entry regardless of how it evaluates a threshold.
- **Erasure is already the per-store key in criome.** Our "erasure = crypto-shred the key" is exactly the per-store data key being the erasure unit; criome custodying keys is the right home, and the per-store key being *both* access boundary and shred unit is the GoPass insight stated precisely.

## The rigorous version: capability-based encrypted storage

GoPass is the mental model; the prior art that makes it rigorous is **capability-as-key**, and our blake3+BLS basis already lives in that world:

- **Tahoe-LAFS** is the canonical "the key *is* the capability" system: a write-cap, read-cap, and verify-cap form a one-way *diminishing* chain — the read-cap is derived by hashing the write-key, the storage-index by hashing the read-key, so a write-cap holder can hand out a read-cap (but not vice-versa), and the storage index lets a server locate/validate a share *without being able to read it*. ([Tahoe Capabilities](https://tahoe-lafs.org/trac/tahoe-lafs/wiki/Capabilities), [Tahoe outline](https://tahoe-lafs.readthedocs.io/en/latest/specifications/outline.html)) This is the model for **tiering Sema access by capability strength**: a verify-cap-equivalent lets ouranos (or another instance) validate the hash-chain and BLS checkpoints of a store it cannot read — exactly the zero-knowledge mirror posture — while a read-cap decrypts. That collapses "should the mirror know a store exists?" into a verify-cap question.
- **Cryptree** (the Wuala folder-tree scheme) shows the hierarchical-key version and, crucially, **lazy revocation**: defer re-encryption until the next write, accepting that the window between revoke and next-write is forward-insecure — the disciplined name for git-crypt's behavior, and a legitimate option for low-churn Sema stores. ([Cryptree paper](https://tik-db.ee.ethz.ch/file/146566189b90f952b8ab1dcf98010781/))
- **Keybase/KBFS** shows production multi-device key rotation: a Per-Team Key wraps data keys; revoking a device or removing a member *rotates the PTK*, re-encrypts it to remaining members, and records the rollover in a signed sigchain (note the parallel to our BLS-signed checkpoints). New devices auto-gain access on provisioning. ([Keybase team crypto](https://book.keybase.io/docs/teams/crypto), [Keybase PUK](https://book.keybase.io/docs/teams/puk)) This is the reference design for "a new Spirit instance is provisioned a key and immediately materializes its stores."
- **Matrix Megolm** is the per-room-key analog of per-store-key, including the rotation-on-membership-change and forward-secrecy/late-joiner tradeoffs. ([Megolm](https://gitlab.matrix.org/matrix-org/olm/blob/master/docs/megolm.md))

## Federated / multi-instance query over encrypted stores

For "multiple Spirit instances share stores as ciphertext and query the intersection of what their keys unlock," the directly-relevant 2024-2026 prior art is **Willow + Meadowcap** and **p2panda**:

- **Meadowcap** gives unforgeable, delegable, *restrictable* read/write capabilities scoped to a 3D area (subspace × path × time), checked at the network layer *before* any data is transmitted — possession of the cap literally determines what a peer will send you. Two namespace models (communal: each subspace owned by an author; owned: one owner mints all caps) map onto "instance-owned stores" vs "psyche-owned federation." Revocation via *time-bounded caps that simply aren't renewed* is a clean, no-rekey revocation primitive worth adopting for Sema delegations. ([Meadowcap spec](https://willowprotocol.org/specs/meadowcap/index.html))
- **Willow's prefix-based deletion** (deleting a path prefix deletes its whole subtree, hierarchically) is the multi-writer-safe deletion model for a log-structured store — relevant to per-store migration/compaction. ([Willow Protocol notes](https://jzhao.xyz/thoughts/Willow-Protocol))
- **p2panda** (2024-2025) is the closest whole-system analog: append-only logs, versioned/migratable schemas, a convergent offline-first access-control CRDT, and two encryption schemes — *message encryption* (Double Ratchet, per-message forward secrecy, but **no late-joiner access to history**) vs *data encryption* (symmetric, late-joiners get all prior secrets, but **no forward secrecy** and a removed-but-synced member keeps the key). ([p2panda group encryption](https://p2panda.org/2025/02/24/group-encryption.html), [p2panda access-control CRDT](https://p2panda.org/2025/08/27/notes-convergent-access-control-crdt.html)) That fork is the decision Sema must make per store: a Spirit log is wiki-shaped (late instances must read history) → it wants p2panda's *data-encryption* posture with explicit data-key rotation on revoke, **not** per-message forward secrecy.

## The real new problems (and best-known resolutions)

1. **Store-existence as a metadata leak.** Is knowing a store *exists* a privacy leak? Willow's answer is the strongest available: **Private Area Intersection / Confidential Sync** — peers exchange salted hashes (each salts with its own salt and compares received hashes under the *other peer's* salt) so neither can confirm a namespace/subspace/path the other doesn't already know, *provided IDs carry enough entropy to be unguessable*; what can be guessed (timestamps) is not hidden. ([Willow PIO spec](https://willowprotocol.org/specs/pio/index.html)) **Resolution for Sema: tier the store registry by capability** — a verify-cap-equivalent (Tahoe-style) lets the mirror validate a store's chain without learning its identity; give private stores high-entropy unguessable IDs; use PIO-style salted-hash intersection for instance-to-instance discovery rather than a global enumerable registry.
2. **Atomicity across stores.** A single intent capture or a branch/merge may touch entries that route to different stores. Envelope systems are per-file/per-store atomic but have no cross-store transaction. **Resolution:** keep the authoritative per-store log atomic; model cross-store consistency as *referential* (event-sourced), not transactional — a saga/causal-DAG (p2panda's approach) with the blake3 hash-link providing causal ordering across stores, accepting eventual consistency rather than synchronous multi-store commit.
3. **Cross-store references / referential integrity across an encryption boundary.** An entry in a readable store may reference an entry in a store the instance can't decrypt. **Resolution:** references are by blake3 content-hash (which the instance can hold and validate even as a verify-cap-only ciphertext holder); a dangling reference resolves to "exists-but-not-readable," never a hard error — Tahoe's storage-index-without-read-key property generalized.
4. **Per-record vs per-store granularity.** Per-store is the right *access + erasure* unit (it is the GoPass insight and the crypto-shred unit), but it is coarse: every privacy reclassification of a single record means moving it between stores (re-encrypt to a different envelope). **Resolution:** make the store the access/erasure boundary, and where finer erasure is needed, give each record its own data key wrapped *into* the store key (Tahoe/Cryptree-style key tree) so a single record can be crypto-shredded without rekeying the store — at the cost of a per-record key in criome. Reserve this for stores where per-record erasure is a stated requirement; default to per-store.
5. **Per-store guardian / schema / branch / migration.** Each store now needs its own IntakePolicy guardian, schema/migration lineage, and branch/fork/merge policy — the gopass per-store `.gpg-id` generalizes to per-store *governance*. p2panda's "versioned, migratable schemas distributed with the data" and Meadowcap's per-area policy are the precedents; this is additive design, not a blocker.
6. **Revocation ≠ erasure, made operational.** The system must expose *two distinct verbs* and never conflate them in the UI or the guardian: `revoke(store, recipient)` = rotate data key + re-wrap + re-encrypt forward (forward-only, cheap, leaves caches); `erase(store|record)` = crypto-shred the data key (erases ciphertext-only holders incl. ouranos; cannot reach decrypted caches). The guardian should refuse to present revocation as erasure, mirroring git-crypt's and SecuPi's explicit warnings.

## Bottom line

The GoPass-to-Sema mapping is sound and well-supported: a store as a recipient-scoped versioned encrypted log, view as the union of decryptable stores, queries fanned out over accessible stores, and the per-store key as the joint access-boundary/erasure-unit are all directly attested by pass/gopass, the age/SOPS/git-crypt envelope pattern, and the capability-storage canon (Tahoe, Cryptree, Keybase, Matrix, Willow/Meadowcap, p2panda). The one non-negotiable correction to keep front-and-center — already in the psyche's framing — is that **revocation (re-wrap, forward-only) and erasure (crypto-shred, ciphertext-only holders) are different operations with different reach, and the federation must implement both as distinct verbs.** The cleverest realizations to adopt are **Tahoe's diminishing capability chain** (verify-cap = let the mirror validate without reading; read-cap = decrypt) for tiering the store registry and resolving the store-existence-leak, and **Willow's PIO/Confidential Sync** for discovering shared stores without leaking the existence of stores a peer can't access.

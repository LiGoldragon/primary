# 5 — Research synthesis: the federated key-gated model

## Spirit as a Federation of Key-Scoped Stores — Design Synthesis

This synthesizes four research findings (GoPass/pass envelope model; capability-encrypted storage canon: Tahoe-LAFS / Cryptree / KBFS / Megolm; federated encrypted sync: Willow/Meadowcap / Keyhive-Beelay / p2panda / automerge; and the new-problems survey) into a concrete design for the direction in Spirit `dun9` (the psyche's proposal to rethink Spirit as a federation of GoPass-style key-gated stores). The verdict is unanimous across all four: the model is **not speculative** — it is the convergent design of the 2023-2026 local-first encrypted-sync field, and every property `dun9` asks for has a shipped or near-shipped reference. The single non-negotiable correction, which the psyche already flagged and every system confirms, is that **revocation is not erasure**.

The strongest prior art for our exact stack (typed hash-linked event log + materialized view + BLS-signed checkpoints + zero-knowledge mirror + crypto-shred erasure + capability-gated multi-instance query) is **Willow** (store/area/reconciliation + prefix-pruning deletion) plus **Ink & Switch's Keyhive/Beelay** (zero-knowledge relay syncing ciphertext it cannot decrypt — the direct ouranos precedent), with **Meadowcap** for delegated 3D-area capabilities, **Tahoe-LAFS** for the diminishing capability chain, **p2panda** for re-wrap-not-re-encrypt key rotation and access tiers, and **Cryptree** for subtree key derivation and lazy revocation.

### (a) The store mapping: a recipient-scoped versioned encrypted log

A Sema store is a `.gpg-id`-style recipient-scoped versioned log, and it is *already the right shape* — our existing design (typed, blake3-hash-linked, payload-bearing event log committing over ciphertext, redb materialized view, BLS-signed checkpoints, mirrored to ouranos) maps onto Willow's `Store` ("a set of `AuthorisedEntries`" forming a join semi-lattice / CRDT) with almost no impedance.

- **Store identity.** A store maps to a (namespace, subspace) pair. The store ID is **blake3 of a key-derived secret, never a human name**, so ouranos and any keyless instance see only opaque unlinkable handles — existence is itself behind the key. This mirrors Willow's "sufficiently long and random-looking" NamespaceIds.
- **Entry vs payload separation (load-bearing).** Willow's Entry is metadata only (`namespace_id`, `subspace_id`, `path`, `timestamp`, `payload_digest`, `payload_length`); the payload is a separate bytestring referenced by digest. **The log commits over the digest**, so the payload can be randomized-AEAD-encrypted and crypto-shredded independently while the hash commitment and BLS checkpoints survive. This is exactly our "log commits over ciphertext" requirement, and it gives crypto-shred a genuine advantage over GoPass-over-git: erasure is integrity-preserving (the chain stays valid, the ciphertext just becomes permanently opaque) rather than a history rewrite.
- **View = union of decryptable stores.** This is literally Willow's join: a peer's view is the join of every store it can authorise; the federation query fanning out over key-held stores *is* the `AreaOfInterest` intersection. This delivers `dun9`'s explicit goal — the same query returns different results to different instances, by design — for free.
- **Conflict resolution.** Adopt Willow's deterministic "newer entry" total order (greater timestamp, then greater `payload_digest`, then greater `payload_length`) for the materialized-view last-writer-wins rule, replacing ad-hoc merge logic.

### (b) Capability / key model

A store key is **three capabilities, not one** (Tahoe-LAFS diminishing chain), mapped onto our blake3+BLS basis:

| Capability | Power | Holder | Sema realization |
|---|---|---|---|
| **write-cap** | append authority | authoring instances | Meadowcap-style signed delegation |
| **read-cap** | the AEAD decryption key | recipient set | per-store KEK unwraps per-record DEK |
| **verify-cap** | validate every hash/signature without decrypting | ouranos, checkpoint auditors | the blake3 hash chain + BLS checkpoints |

The chain is **one-way diminishing**: derive weaker from stronger, never amplify. The **verify-cap is the single most important realization** — it is precisely and only what a zero-knowledge mirror and a BLS-checkpoint auditor need: validate the log without decrypting any payload. Our log-over-ciphertext already produces a natural verify-capability distinct from the read-capability. Name these three powers explicitly.

**Envelope encryption (KEK → DEK → EDEK) decouples access from erasure**, giving both granularities at once:
- The **per-store KEK** is the access/recipient boundary. Recipient-change = **re-wrap the KEK to the new recipient set, NOT re-encrypt content** (age/SOPS/git-crypt/p2panda all share this shape; SOPS makes it explicit — `--add-pgp`/`--rm-pgp` edit the envelope, only `rotate` mints a new data key).
- The **per-record DEK** is the crypto-shred erasure unit. Shredding one DEK erases one entry without touching the store; the hash-link chain stays intact because the commit is over ciphertext.

**Generational, signed key history (Keybase/KBFS):** each recipient-set change mints a new **store generation** whose public attestation lands on the BLS checkpoint chain; criome custodies and distributes the per-generation seed (encrypted to each instance's per-instance key). Derive a store's read-key, verify-key, and schema-key from one generation seed via labeled derivation (`HMAC(seed, label)`), not independent keys. Consider Keybase's server-held **mask** (`app_key = HMAC(seed,label) XOR criome_mask`) as a second factor so mirrored ciphertext on ouranos is insufficient to read without criome — cleanly separating the zero-knowledge mirror from the key custodian.

**Two distinct verbs the guardian must never conflate:**
- `revoke(store, recipient)` = rotate KEK + re-wrap envelope to remaining recipients + re-encrypt forward. **Forward-only, cheap, leaves caches.** The removed recipient's old key still reads all prior ciphertext forever.
- `erase(store | record)` = crypto-shred the DEK (or store KEK). Erases every ciphertext-only holder (the ouranos mirror); provably **cannot reach already-decrypted caches**.

**Revocation primitives, ranked:**
- **Meadowcap revocation-by-non-renewal** (time-bounded caps simply not reissued) — the cleanest primitive for *delegated readers*; no data-key rekey needed.
- **Lazy revocation** (Cryptree/Fu/Backes-Cachin) — on removal mark the KEK dirty, re-wrap on the next write; fits our append-only log perfectly ("next write" = "next entry"). Name the forward-insecure window explicitly; high-sensitivity stores cannot afford it.
- **Eager rotation** — for stores where the post-removal window is unacceptable.

### (c) Multi-instance federated query + cross-store references

**Federated query** is Willow Confidential Sync: **3d range-based set reconciliation** (exchange fingerprints over (namespace, subspace, time) ranges, recursively subdivide on mismatch, transmit only differing entries), scoped to the intersection of both peers' AreasOfInterest — true partial/sparse replication, the federation scoping primitive. For the ouranos relay specifically, adopt **Keyhive/Beelay**: sync the membership graph first (to decide what a peer may see), then **RIBLT set-reconciliation** of state hashes whose symbols encode `(document_id, hash(heads, cgka_ops))` — the relay reconciles encrypted documents *without ever comparing ciphertext*, and a single decoded symbol identifies the differing doc with no extra round trip.

**Cross-store references** resolve by **blake3 content-hash**, which a verify-cap-only holder can validate even without read access. A reference into an undecryptable store resolves to **"exists-but-not-readable," never a hard error** (Tahoe's storage-index-without-read-key generalized). Two disciplines:
- **Referential integrity is per-instance and partial by design.** The same reference resolves for a key-holding instance and dangles for one without the key — `dun9`'s explicit goal — which means there is **no global view**; integrity claims are always qualified by which keys the instance holds. No prior system offers global cross-encryption-boundary integrity; **do not promise it.**
- **Dangling is correct, not corruption.** Crypto-shredding a referenced record's DEK leaves the referrer's hash-link intact (commit is over ciphertext) but semantically dangling — a shredded target *should* dangle. Render it as "redacted/erased"; repair lazily. The database-world frame is relaxed/eventual referential integrity under causal consistency (Shapiro et al., PaPoC 2018), which is *provably sufficient* — valid cross-store references without any global transaction.

### (d) The new problems and best-known resolutions

**Store-existence as a metadata leak.** Knowing a store exists can leak. Resolution: **Willow Private Area Intersection / Private Interest Overlap** — read-access enforcement lives at the *replication layer*, not the data model; peers run PSI (Diffie-Hellman + salted hashes, per-session salt to block mirror/echo attacks) over fragments of their read capabilities, learning only the namespaces/paths they *both* already hold a cap for. A store you have no key for is **non-enumerable**. The honest limit, which we inherit: PSI **cannot prevent confirmation of a correct guess** — so existence-privacy reduces to **store-id unguessability** (long, random, per-subspace-encrypted ids), and **timestamps are conceded as guessable/out of scope**. The store identifier *is* the discovery capability (Earthstar's share-address pattern): no separate public registry to tier — discoverability is a property of who holds the high-entropy id. Tier the registry by capability: verify-cap-equivalent lets the mirror validate a chain without learning identity.

**Atomicity across stores.** No envelope or CRDT system offers cross-store transactions. Resolution: per-store-atomic appends + **saga / causal-DAG with compensating actions** (p2panda-style), blake3 hash-links providing causal ordering, accepting eventual consistency. A half-applied cross-store op is treated as a dangling ref until both arrive. **Do not build two-phase commit across stores** — it contradicts offline-first partial replication. The honest escape: where a synchronous cross-store invariant truly must hold, **co-locate that data in one store** — atomicity becomes a reason to choose store boundaries, feeding routing (e).

**Per-store guardian / schema / branch / migration.** Per-store governance is the prior-art norm (Willow per-namespace write policy; p2panda Pull/Read/Write/Manage tiers; Tahoe dirnode caps carry schema). Resolution: the IntakePolicy/guardian is **shared as meta-signal code** (one classifier definition) but **instantiated and versioned per store**. Per-store membership/recipient state is itself a **replicated log** (p2panda membership-CRDT-over-DAG with "strong removal": a removed member can't add a member even at the same logical time) — generalizing our per-component-policy-over-log. **Global migration is impossible** (can't decrypt all stores at once, and daemons never parse config) and must run **client-side, per-store, lazily, on the store's own branch**; cross-store migrations are themselves sagas.

**Granularity — three distinct boundaries, deliberately not collapsed:**
- **crypto-shred granularity = per-store key** (the recipient-set / erasure boundary)
- **access granularity = sub-store** (Meadowcap 3D area: path + time inside one key)
- **existence-erasure granularity = per-path-prefix** (Willow prefix-pruning, down to one record)

Forcing one-key-per-record explodes key management and adds KMS latency on every read/write (the cost Cryptree/BeeKEM key-trees exist to fight); forcing per-store-only loses path-scoped delegation and record-level pruning. **Use all three layers.** Default to per-store; opt into per-record DEKs only where fine-grained erasure is a stated requirement.

**Two-tier erasure.** Crypto-shred (destroy the key) erases the payload plaintext but leaves the entry skeleton; Willow **prefix-pruning** erases the existence skeleton tracelessly — and it achieves this *specifically by refusing any cryptographic proof-of-completeness* (giving up provable-full-history is what buys deletability; IPFS/hypercore structurally cannot delete). They compose: crypto-shred for "content gone, slot remains," prefix-pruning for "slot's existence gone." Neither can unsend data a peer already pulled.

### (e) Privacy magnitude → store routing

The transform of the old per-record threshold is the cleanest single shift in the design. Today Privacy is a per-entry magnitude the guardian compares against a read-time gate. In the federation, **Privacy becomes a routing function**: the IntakePolicy classifier computes *which key-scoped store (recipient set) an entry's DEK is wrapped to* — Willow's point-placement, not a gate.

- **Routing-by-store is strictly stronger than a read-time threshold**: a non-key-holder cannot read the entry regardless of how it evaluates a threshold.
- **It expresses what a scalar threshold cannot**: "private to a specific recipient set," because the store *is* the recipient set. `Zero`-privacy public intent routes to a broadly-shared (possibly existence-public) store; high-privacy records route to narrow-recipient, existence-private stores discoverable only via PSI.
- **Changing a record's privacy = re-routing = re-wrap under a different store key** (write to the new store + prefix-prune the old slot), never an in-place flag flip — because migration can't run on the zero-knowledge relay and the commit is over ciphertext.

**Routing safety rules:** the guardian must be **deterministic and single-destination** (a record routed to two stores of different privacy leaks the higher-privacy fact via the lower store), **fail-closed to the narrowest store on ambiguity**, and **append-only-auditable**.

### Late-joiner vs forward-secrecy — the forced trade, decided

A Spirit log is wiki-shaped: new instances must read history. That rules out per-message forward secrecy (p2panda Message-Encryption / Matrix Megolm style) for the log payloads. So the log adopts p2panda's **Data-Encryption posture**: stable symmetric key, late-joiners get prior secrets, **weak forward secrecy by design**, with explicit data-key rotation on revoke. The accepted consequence: a compromised store key exposes all data ever under that key until rotation — mitigate with periodic rotation and small erasure-unit stores. Genuinely ephemeral data needs a **separate Double-Ratchet / Message-Encryption store class** — two store classes, not one. (Megolm is the cautionary anti-pattern here, with formally-confirmed weak post-compromise security — IACR 2023/1300, Nebuchadnezzar — not a model.)

### Bottom line

Adopt: Willow's store-as-join-semilattice + 3dRBSR + prefix-pruning (log/view/existence-erasure); Keyhive/Beelay's membership-gated, RIBLT + sedimentree, zero-knowledge-relay sync (ouranos); Meadowcap's 3D-area delegated read/write caps with revocation-by-non-renewal (criome already has BLS signing); Tahoe's diminishing read/verify/write capability chain (verify-cap = the mirror's exact and only power); p2panda's re-wrap-not-re-encrypt rotation and access tiers; envelope KEK/DEK for the access-vs-erasure split; PIO/PAI for discovery with the honest "existence is hidden only as well as the store-id is unguessable" threat model. Keep **revocation ≠ erasure** as a first-class, two-verb distinction throughout. Treat cross-store referential integrity as per-instance, partial, dangling-tolerant — the frontier the field has no better answer for; do not promise global cross-store atomicity or integrity.


## Clever ideas

- Verify-cap as the mirror's exact and only power (Tahoe-LAFS): the weakest capability that validates every hash and BLS signature without decrypting any payload. It is precisely and only what a zero-knowledge mirror (ouranos) and a checkpoint auditor need; our log-over-ciphertext already produces a natural verify-cap distinct from the read-cap, so name read-cap/verify-cap/write-cap as the three one-way-diminishing powers of a store key.
- Envelope encryption decouples access from erasure (KEK -> DEK -> EDEK): the per-store KEK is the access/recipient boundary (re-wrap to change recipients, never re-encrypt content) while the per-record DEK is the crypto-shred unit — giving per-store access AND per-record erasure simultaneously, avoiding both 'erasure nukes the whole store' and 'key-per-row performance collapse'.
- Willow prefix-pruning buys deletability by REFUSING a completeness proof: traceless existence-erasure is possible precisely because Willow gives up any cryptographic proof-of-full-history (the same trade our crypto-shred makes at the payload layer); append-only-with-completeness systems like IPFS/hypercore structurally cannot delete.
- Two-tier / three-boundary erasure-and-access that are deliberately NOT collapsed: crypto-shred = per-store key, access = sub-store (Meadowcap 3D path+time area inside one key), existence-erasure = per-path-prefix (Willow pruning down to one record) — use all three layers rather than forcing one-key-per-record (key explosion) or per-store-only (loses path-scoped delegation).
- Store identifier IS the discovery capability (Willow PIO/PAI + Earthstar share-address): store id = blake3 of a key-derived secret, discovered only via PSI against a held cap, so existence-private stores are non-enumerable and there is no separate public registry to tier — discoverability collapses into 'who you hand the high-entropy id to', with the honest concession that PSI cannot stop confirmation of a correct guess.
- Keyhive/Beelay reconciles encrypted documents WITHOUT comparing ciphertext: sync the membership graph first to decide visibility, then RIBLT-reconcile state hashes whose symbols already encode (document_id, hash(heads, cgka_ops)) — one decoded symbol names the differing doc with no extra round trip, and the relay learns nothing. Sedimentree picks compaction chunk boundaries by trailing-zero-bits of the commit hash, so divergent encrypted replicas independently choose identical boundaries and the zero-knowledge server compacts ciphertext it cannot read.
- Privacy magnitude becomes a routing/point-placement function, not a read-time gate: the guardian computes WHICH recipient-set store an entry's DEK is wrapped to (strictly stronger than a threshold, since a non-key-holder can't read regardless), and this expresses 'private to a specific recipient set' which a scalar threshold cannot — changing privacy = re-route = re-wrap under a different store key + prefix-prune the old slot.
- Coordination-free revocation via causal dominance (Keyhive blanking / p2panda strong removal): a removed instance's concurrent writes lose the merge to the blank and a removed member can't add a member even at the same logical time — correct revocation with zero quorum/consensus, exactly the property a federation of independently-online Spirit instances needs; Meadowcap's revocation-by-non-renewal (time-bounded caps simply not reissued) gives the same for delegated readers with no data-key rekey.

## New problems

- Revocation is not erasure, structurally and unavoidably: re-wrapping the store KEK to a new recipient set stops FUTURE access but the removed recipient's old key reads every historical ciphertext forever and may have cached plaintext. Only crypto-shredding the DEK erases, and even that cannot reach already-decrypted caches — it only erases ciphertext-only holders (the ouranos mirror). The guardian must expose revoke() and erase() as separate verbs and refuse to present revocation as erasure (confirmed by pass/gopass, git-crypt, KBFS, Megolm, p2panda).
- Store-existence as a metadata leak: knowing a store EXISTS can leak. Willow PIO/PAI (DH-PSI + salted hashes) hides existence against blind probing but CANNOT prevent confirmation of a correct guess, so existence-privacy reduces entirely to store-id unguessability (long, random, per-subspace-encrypted ids), and timestamps are conceded as guessable/out of scope. GoPass leaks this badly (mount names, .public-keys/<ID>, git tree all plaintext) — a counterexample to avoid.
- No cross-store ACID atomicity is possible (different recipient sets, different instances, the relay can't read ciphertext to coordinate): a multi-store write is a saga of per-store-atomic appends + compensating actions under causal consistency, with a window where the operation is partially visible. Synchronous cross-store invariants force co-location in one store; two-phase commit across stores is forbidden as it contradicts offline-first partial replication.
- Cross-store references are dangling-cap hazards and referential integrity is per-instance/partial BY DESIGN: the same reference resolves for a key-holder and dangles for a non-holder (the explicit goal), so there is no global view and integrity must be qualified by held keys. Crypto-shredding a referenced record leaves the referrer's hash-link intact but semantically dangling — which is correct (a shredded target should dangle, render as redacted), not corruption. No prior system offers global cross-encryption-boundary integrity; do not promise it.
- Per-record envelope DEKs explode key count and add KMS/criome latency on every read/write ('a key per row -> unacceptable performance degradation'): needs batching and a tiered scheme (per-store KEK default, per-record DEK only where fine-grained erasure is a stated requirement), not per-record everywhere.
- Forward secrecy is in irreducible tension with late-joiner history: a wiki-shaped Spirit log needs new instances to read history, which rules out per-message forward secrecy for log payloads (p2panda Message-Encryption / Megolm), so a compromised store data key exposes all data ever under that key until rotation. Genuinely ephemeral data needs a SEPARATE Double-Ratchet store class — two store classes, not one. Mitigate the log class with periodic rotation and small erasure-unit stores.
- Routing determinism / dual-write leak: if the guardian routes the same record to two stores of different privacy, the higher-privacy fact leaks via the lower store. Routing must be deterministic, single-destination, fail-closed to the narrowest store on ambiguity, and append-only-auditable.
- Removal-propagation / post-compromise window in a decentralized federation: until all instances observe a recipient removal and rotate, some keep encrypting to the removed recipient — revocation latency is not instantaneous across instances, and a removed-but-synced member retains the key for all data it covered until rotation.
- Global schema migration is impossible under federation (can't decrypt all stores at once, and daemons never parse config): migrations run client-side, per-store, lazily, on the store's own branch; lazy revocation/rotation likewise leaves a forward-insecure window where not-yet-rewritten data is still readable under the old key, and a store can drift into an inconsistent recipient state until the next write or an fsck pass — needs an explicit eager-vs-lazy policy per privacy tier.

## Open questions

- Eager vs lazy revocation per privacy tier: which Privacy magnitudes mandate eager re-encryption on recipient removal (no forward-insecure window) vs accept lazy rotation-on-next-write? High-sensitivity stores likely cannot afford lazy, but eager bulk re-encryption on an append-only log over ciphertext has real cost — what is the cutoff and who decides it?
- Should criome adopt the Keybase server-held-mask second factor (app_key = HMAC(seed,label) XOR criome_mask), so ouranos-mirrored ciphertext is insufficient to read without criome, or does that over-couple the zero-knowledge mirror to the key custodian's availability and weaken the offline-first posture?
- Default routing granularity: per-store KEK only (coarse, cheap) as the default with per-record DEKs reserved for stores with a stated per-record erasure requirement — but how does the guardian decide at intake time which stores need per-record erasure, and can that decision be migrated later without re-encrypting the whole store?
- Cross-store reference rendering policy in the materialized view: how exactly should a dangling/undecryptable reference surface — a typed 'exists-but-not-readable' sentinel vs a typed 'erased/redacted' sentinel vs silent omission — and should these two cases (no-key vs shredded) be distinguishable in the view at all, or does distinguishing them itself leak existence?
- Store-id entropy and derivation: what is the concrete scheme for blake3-of-key-derived-secret store ids, how are they distributed to legitimately-discovering instances without a public registry, and how does the Privacy magnitude bind to id-entropy (does the most sensitive tier require the id itself to be a held secret, and how is that id handed to a new authorized instance)?
- How does the existing already-designed branch/fork/merge/rebase-over-the-log interact with per-store generations and concurrent recipient-set changes (BeeKEM-style conflict keys / p2panda strong-removal) — specifically, does a merge across two branches with divergent recipient sets produce a well-defined recipient set, and what is the conflict-resolution rule?
- Two store classes (Data-Encryption log class vs Message-Encryption ephemeral class): is the ephemeral/Double-Ratchet class actually in scope for Spirit at all, or does every Spirit record belong to the readable log class — and if ephemeral is in scope, what routes a record there vs the log, and how does it interact with the authoritative-log/checkpoint guarantees?

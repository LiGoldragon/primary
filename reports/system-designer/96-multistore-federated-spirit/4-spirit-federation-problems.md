# Spirit as a Federation of Key-Scoped Stores: New Problems and Best-Known Resolutions

## Frame

The psyche direction (Spirit `dun9`) is the GoPass mental model lifted into Sema: private component state as **multiple key-gated stores**, each encrypted to a recipient set, where key possession *is* the access boundary *and* the crypto-shred erasure unit, and a Spirit instance's view is the union of the stores it can decrypt. This validates well — the model is essentially what GoPass, Tahoe-LAFS, Willow/Meadowcap, Keybase, and p2panda already are — but federation introduces five genuinely new problems beyond what any single one of those systems solves. The deepest finding: **the cleanest prior art is not GoPass at all but Willow + Meadowcap + Private Area Intersection**, because Willow is the only system that treats *store existence itself* as private, which is sub-problem (a) — and GoPass conspicuously does not. The five sub-problems and their best-known resolutions follow; the strong realizations to steal are tagged in the clever list.

## The GoPass-to-Sema baseline (validated, with one correction)

GoPass's `mounts` are exactly the proposal: each mount is a separate repository with its own recipient set, managed by `gopass recipients add/remove --store=<mount>`, and the view is the union of mounts whose keys you hold ([gopass features](https://github.com/gopasspw/gopass/blob/master/docs/features.md)). The model maps cleanly. But GoPass also exposes the **load-bearing correction the psyche already anticipated**: removing a recipient and re-encrypting (`gopass fsck --decrypt`, [PR #2373](https://github.com/gopasspw/gopass/pull/2373)) **does not erase** — the removed recipient may have cached plaintext or pulled the prior ciphertext from git history, and the old recipient-key still decrypts every commit it ever had ([gopass #1083](https://github.com/gopasspw/gopass/issues/1083)). So **revocation (stop future access by re-wrapping the data key) and erasure (crypto-shred so nobody decrypts) are different operations** — a distinction GoPass blurs and the Sema design must keep crisp because the log commits over *ciphertext*: re-wrapping the data key for a new recipient set leaves all historical ciphertext readable by the old key; only shredding the data key erases. This is the single most important thing GoPass teaches by getting it half-wrong.

## (a) Store-existence-as-metadata — does knowing a store EXISTS leak?

Yes, and GoPass leaks it badly: mount points, `.public-keys/<ID>`, and the git tree are plaintext, so anyone with the repo sees every store's name, recipient set, and shape even with zero decryption keys. For Spirit mirrored to a zero-knowledge ouranos server this is the central question, and **Willow's Private Area Intersection (PAI) is the strongest known answer** ([Willow PAI](https://willowprotocol.org/specs/pai/index.html), [Meadowcap](https://willowprotocol.org/specs/meadowcap/index.html)). The Willow realization: **read-access enforcement lives at the replication layer, not the data model** — "peers voluntarily withhold Entries from those lacking valid read capabilities... this prevents knowledge that data exists." Two peers run **private set intersection over *fragments* of their read capabilities**, so each learns only the namespaces/subspaces/paths they *both* already hold a cap for; a store you have no key for is **not enumerable** — you cannot learn it exists ([Willow PAI](https://willowprotocol.org/specs/pai/index.html), [jzhao overview](https://jzhao.xyz/thoughts/Willow-Protocol)). NamespaceIds and SubspaceIds are "sufficiently long and random-looking," and path components are encrypted per-subspace so paths are "unguessable." This yields the **registry tiering** the prompt asks for: existence-private stores have random opaque IDs and are discoverable *only* via PSI against a held key (mutual-knowledge gate); a lower tier of merely access-controlled (but existence-public) stores can sit in an enumerable catalog. The mapping to our basis: store IDs should be blake3 of a key-derived secret (not a human name), so the ouranos server and any keyless instance see only opaque, unlinkable store handles — existence is itself behind the key.

## (b) Atomicity / consistency across stores

A write or query spanning two key-scoped stores cannot be a global ACID transaction — the stores have different recipient sets, may live on different instances, and the ouranos server cannot coordinate across ciphertext it can't read. The prior art is unanimous: **abandon cross-store atomicity for per-store-atomic + saga/eventual consistency** ([saga pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/saga)). Each store's append-only log gives local atomicity (one log append is atomic); a cross-store operation becomes a **saga of per-store local transactions with compensating actions**, and the system is **causally consistent, not linearizable**. The deeper realization from local-first research: **"merge what you can, fork what you can't"** ([Schiefer/Litt/Jackson, PaPoC 2022](https://crdt.tech/papers.html)) and **causal consistency is provably sufficient to maintain referential integrity** ([Ensuring referential integrity under causal consistency](https://arxiv.org/pdf/1803.03482)) — a reference CRDT keeps cross-entity links valid without global transactions. This dovetails with the branch/fork/merge-over-log we already designed: a cross-store write is a fork that merges per-store, never a 2PC. Where a cross-store invariant truly must hold synchronously, the only honest answer is to **co-locate that data in one store** — atomicity is a reason to choose store boundaries, which feeds directly into routing (e).

## (c) Per-store guardian / schema / branch / migration vs shared

The federation forces a **two-layer split**: a *shared* meta-policy (what kinds of stores may exist, the universe of types, the IntakePolicy classifier itself) versus *per-store* enforcement (each store's recipient set, its branch state, its schema version, its migration cursor). Tahoe-LAFS shows the per-container discipline: each dirnode carries its own caps and its **children's rwcaps are encrypted with the parent dirnode's writecap**, giving "transitive readonlyness" ([dirnodes spec](https://github.com/tahoe-lafs/tahoe-lafs/blob/master/docs/specifications/dirnodes.rst)) — schema/policy travels *with* the encryption boundary, not in a global table the keyless server can read. p2panda makes per-store membership a **CRDT over a DAG of add/remove/promote ops with "strong removal"** ([p2panda group encryption](https://p2panda.org/2025/02/24/group-encryption.html)) — i.e. each store's *guardian state is itself a replicated log*, which is exactly our per-component-policy-over-log design generalized to per-store. The resolution: the IntakePolicy/guardian is **shared as code/meta-signal** (one classifier definition) but **instantiated and versioned per store**; schema migration runs **per store, lazily, on the store's own branch**, because a global migration would require decrypting every store at once (impossible — and forbidden by the daemon-never-parses-config rule). Migrations that cross stores are sagas (b).

## (d) Per-store vs per-record granularity (envelope keys)

This is the granularity tradeoff, and the prior art is decisive: **use envelope encryption to decouple the access/recipient boundary from the erasure unit** ([crypto-shredding granularity](https://oneuptime.com/blog/post/2026-02-17-how-to-set-up-crypto-shredding-for-gdpr-right-to-erasure-compliance-in-google-cloud/view)). The standard three-tier KEK → DEK → EDEK pattern: a per-*record* DEK encrypts the entry, and that DEK is wrapped by the store's KEK (the recipient-set key). This gives **both granularities at once**: the *store key* is the access + recipient boundary (re-wrap to change recipients, "key rotation must not require re-encryption of existing data"), while the *per-record DEK* is the fine-grained crypto-shred unit (drop one DEK to erase one entry without touching the store). Per-store-only erasure is too coarse (right-to-erasure of one fact shouldn't nuke a whole store); per-record-only keys explode key count ("millions of users → millions of keys", "unacceptable performance degradation... a key per row"). The realization to steal: **per-store for access, per-record envelope DEK for erasure** — and crucially, because our log commits over ciphertext, shredding a record's DEK leaves the blake3 hash-link chain and BLS checkpoints *intact* (the commitment is over ciphertext that becomes permanently opaque), so erasure doesn't break the log's integrity. **Lazy revocation** ([Backes/Cachin, CMU](https://www.cs.cmu.edu/~alina/papers/lazyfs-sisw.pdf); [Fu 1999](https://eprint.iacr.org/2005/334)) closes the loop: on recipient removal, mark the store's KEK "dirty" and re-wrap only on next write rather than eagerly re-encrypting — but **lazy revocation is revocation, not erasure** (the old key still reads old ciphertext), which re-confirms the (a)/baseline distinction.

## (e) Privacy magnitude → store ROUTING

Today Spirit's Privacy magnitude is a per-record threshold the guardian compares against. In the federation it becomes a **routing/assignment policy**: the IntakePolicy classifier no longer gates a single store with a threshold, it **selects which store an entry lands in** — Privacy magnitude maps to a recipient-set (= store), so `Zero`-privacy public intent routes to a broadly-shared (even existence-public) store, while high-privacy records route to narrow-recipient, existence-private stores discoverable only by PSI (a). This is precisely **Willow's three-dimensional placement** (subspace/path/timestamp) generalized: the classifier computes a *point* (which subspace/store) rather than a *gate* (above/below threshold) ([Willow](https://jzhao.xyz/thoughts/Willow-Protocol)). The realization: **routing replaces thresholding** — instead of "is this private enough to keep," it's "which key-scoped store does this belong to," and the magnitude becomes the routing key. This also resolves an old wart: a per-record threshold can't express "private *to a specific recipient set*," but store-routing can, because the store *is* the recipient set. The guardian must be **fail-closed on ambiguity** (route to the narrowest store when uncertain) and routing must be **deterministic and append-only-auditable** so the same record can't leak by landing in two stores of different privacy.

## Synthesis: what to build

The federation is real prior art, not invention — but **no single system solves all five**. Steal: **Willow PAI** for (a) existence-privacy and (e) point-routing; **saga + causal-consistency reference CRDTs** for (b); **p2panda's CRDT-membership-as-log + Tahoe's cap-carries-schema** for (c); **envelope KEK/DEK + lazy revocation** for (d). The one discipline every system enforces and Sema must too: **revocation ≠ erasure** — re-wrapping a data key stops future access but the old key reads old ciphertext forever; only shredding the (per-record envelope) DEK erases. Because our log commits over ciphertext with blake3+BLS, crypto-shred erasure is *integrity-preserving* by construction, which is a genuine advantage over GoPass-over-git (where erasure means rewriting history).

## 1. Dolt internals — prolly trees, the noms chunk store, and why git's blob problem disappears

### Prolly trees (probabilistic / content-defined-chunking Merkle B-trees)
Dolt stores every SQL table as a **prolly tree**: a B-tree-shaped index that is simultaneously a Merkle tree. The data structure originates in **Noms** (Attic Labs). Each table is a sorted map from primary key to row, laid out in tree nodes ("chunks"). The defining trick is how node boundaries are chosen — not by a fixed fan-out, but by **content-defined chunking (CDC)**: a rolling hash runs over the *sorted keys*, and wherever the hash falls below a threshold a chunk boundary is cut. Dolt normalizes chunk sizes around **~4 KB average** using a CDF-based boundary probability `(CDF(end) - CDF(start)) / (1 - CDF(start))` (an improvement over Noms' raw geometric distribution, motivated by OLTP use). Each chunk is then hashed (SHA-512 truncated to 20 bytes) to produce its **content address**, internal layers map "highest key in chunk → child content address," and the process repeats until a single **root hash** remains. Source: [DoltHub Prolly Tree docs](https://www.dolthub.com/docs/architecture/storage-engine/prolly-tree), [Prolly Trees blog](https://www.dolthub.com/blog/2024-03-03-prolly-trees/), [How to Chunk Your Database into a Merkle Tree](https://www.dolthub.com/blog/2022-06-27-prolly-chunker/).

Two properties fall out for free:
- **History independence**: the same final dataset produces the same tree (and same root hash) regardless of insert/update/delete order, because boundaries depend only on content, not on operation history.
- **Structural sharing**: chunks are stored in a content-addressed block store, so any chunk shared between two versions (or two branches) is stored exactly once.

### Why a small change rewrites only a few chunks (NOT git's whole-blob problem)
Because boundaries are keyed on *content*, a single-cell update is **copy-on-write on one leaf chunk**, then a recompute of the content addresses of just that chunk's ancestors up to the root. The docs put it precisely: "every edit to a table in Dolt is a minimum of 4 KB multiplied by the depth of the tree, with some small probability of writing multiple chunks." Crucially Dolt's rolling hash considers **only keys, not values**, so a fixed-size value change *never* shifts a boundary; and the probability a single-byte change *does* shift a boundary is ~**1/4096 (0.02%)**. Contrast git: any byte change to a file recomputes the entire blob's hash, producing a brand-new whole-file object — git then relies on after-the-fact zlib + heuristic delta-compression in packs, which works poorly on reshuffled binary formats. Prolly trees make the "only rewrite what changed" property *structural and a priori*, not a compression afterthought. (Sources as above.)

### The noms chunk store (NBS) — verified on the local embedded Dolt
Inspected at `/home/li/primary/.beads/embeddeddolt/primary/.dolt/noms/`:
- **`manifest`** (149 bytes): `5:__DOLT__:<rootHash>:<...>:00000000000000000000000000000000:vvvv...:130734` — table-file-format version `5`, the `__DOLT__` schema tag, the current **root chunk hash**, and the list of table files. Multiple `nbs_manifest_<n>` siblings are atomic-swap staging copies.
- **`vvvvvvvvvvvvvvvvvvvvvvvvvvvvvvvv`** (138 MB): the **append-only journal table file**. Its header begins `\0\xff` followed by a 20-byte chunk address, then chunk records tagged `RTVL` / `ADRM` / `DCMT` — the journal records new chunks as they're written.
- **`journal.idx`** (3.8 MB): the journal index mapping chunk addresses → offsets, header `\0\xff` + 20-byte address.
- **`oldgen/`**: where the "newgen" journal is **conjoined/compacted** into immutable older-generation table files (currently empty here — young store).
- NBS table-file format is three sections: ChunkRecords (the data), an Index (prefix map of address-first-8-bytes + ordinals, lengths array, address suffixes), and a fixed Footer with chunk count. `Contains` is an O(log N) binary search on the prefix map; `Retrieve` sums preceding lengths for the byte offset. Source: [DoltHub Block Store docs](https://www.dolthub.com/docs/architecture/storage-engine/block-store).

### Commit graph, diff/merge, remotes — verified locally
- **Commit graph**: `dolt log` on the beads DB shows a real chain of content-addressed commit ids (`g29e1vqfu5i4g3te3lotk3odjr96r41p` → `8kks2...` → `tdiv2v3...`), with `refs/heads/main` and `remotes/origin/main` refs in `repo_state.json`. A commit is a content-addressed object pointing to a root prolly tree plus parent(s) — same DAG model as git, but the leaves are rows, not files.
- **Cheap diff**: diff walks the two root hashes top-down and **skips any subtree whose content address matches**, descending only where hashes differ — O(d) in the size of the difference, not O(n) in table size. Verified: `dolt diff` between two adjacent commits of the ~60-column `issues` table reported only the two changed cells (`status: open→closed`, `close_reason`) on one row, plus one `events` insert and one `metadata` cell — genuinely cell-level.
- **Merge** is three-way at the cell level: structural sharing lets Dolt find the common ancestor subtrees and only reconcile the chunks that diverged; non-overlapping cell edits merge without conflict.
- **Remotes** transfer only **novel chunks**: clone/push/pull diff the chunk sets by content address and ship what the other side lacks (dedup is automatic from content addressing). The local store also shows **dolt-as-git-remote**: `.dolt/git-remote-cache/<hash>/repo.git/` is a bare git repo whose `refs/dolt/data/...` and `refs/dolt/blobstore/...` point at packed Dolt chunks; a decompressed object is a noms tree referencing child chunk addresses (`02uq0qld5vch7r3ne5k0uo1sr5dvjsg9`), and `repo_state.json` lists remote `origin = git+ssh://git@github.com/.../primary.git`. So Dolt can ride a plain git host as a chunk transport.
- **Content-addressed archive backups**: `/home/li/primary/.beads/backup/` holds 288 `*.darc` files (86 MB), base32-hash-named, with `DOLTARC` footer magic — Dolt's **archive format**, content-addressed chunk bundles that are the on-disk embodiment of structural sharing for off-machine recovery.

## 2. Why naive `git add component.redb` is bad (and why a daily snapshot is still worth it)

`redb` stores the whole database as **one file: a collection of mmap'd, copy-on-write B+trees** (a header + regions split into pages; a table-tree, per-table data-trees, and a pending-free-tree, all COW). Source: [redb design.md](https://github.com/cberner/redb/blob/master/docs/design.md). When you commit a write, redb allocates **new pages** for the modified path and frees old ones for later reuse — so even a one-row change relocates pages and rewrites internal pointers across the file. This defeats git two ways: (a) git addresses the **entire file as one blob** — any change is a brand-new ~full-size object; (b) git's delta-compression heuristic finds little reusable structure because COW page reshuffling scrambles byte offsets across versions (binary formats are "very poor at information preserving when everything gets scrambled"). The realistic outcome: roughly one near-incompressible blob per snapshot — a 10 MB DB changed 5 times ≈ 50 MB stuck permanently in history. (Empirically git's zlib delta can sometimes recover *some* — one SQLite test packed two 864 KB revisions to 329 KB — but it's unreliable and gets worse as the file churns.) Sources: [SQLite-in-git HN](https://news.ycombinator.com/item?id=38110286), [ongardie SQLite-in-Git](https://ongardie.net/blog/sqlite-in-git/).

**But a daily snapshot is still a cheap correctness floor.** It is dumb, atomic, and impossible to get wrong: copy the file, commit it. It needs no schema awareness, survives any rkyv-layout or schema-version drift, and gives a guaranteed point-in-time restore even if the typed migration story is mid-flight. The bloat is bounded and offline; correctness and recoverability are not negotiable. The lesson is *not* "never snapshot" — it's "don't mistake a snapshot for version control": snapshots give you restore points, not cheap diff/merge/blame. Note: beads itself does NOT git-add the Dolt file — `.beads/.gitignore` ignores `dolt/`, `*.db`, `backup/`, and instead exports an `interactions.jsonl` / `.darc` content-addressed backup, exactly because the binary-in-git path is bad.

## 3. Prior art worth stealing (one paragraph each)

**Datomic** — the database is an **immutable, append-only set of datoms** `(entity, attribute, value, transaction, op)` where `op ∈ {assert, retract}`. Writing is the pure function `d/with(db, tx-data) → db'` that grows the value by **accretion**; nothing is ever mutated in place. Every transaction gets a strictly monotonic logical `t` (and wall-clock `txInstant`), so `d/as-of` time-travels to any past state and `d/history` exposes every assertion/retraction across time. Worth stealing: the *operation log IS the database*; "current state" is a materialized view over the log; time-travel and audit are free. Source: [Datomic transaction model](https://docs.datomic.com/transactions/model.html), [Retract](https://docs.datomic.com/client-tutorial/retract.html). **signal-sema is already nearly here** — `SemaOperation = {Assert, Mutate, Retract, Match, Subscribe, Validate}` (`signal-sema/src/operation.rs`) plus `sema-engine`'s `CommitLogEntry { commit_sequence: CommitSequence, snapshot: SnapshotIdentifier, operations: NonEmpty<CommitLogOperation> }` (`sema-engine/src/log.rs`) is a Datomic-style monotonic-sequence accretion log. The one divergence: Sema adds a first-class `Mutate` (Datomic models update as retract+assert) — a deliberate ergonomic choice the engine notes lets identified families avoid "simulating mutation through retract-plus-assert."

**irmin** (OCaml, MirageOS) — a library for **mergeable, branchable, content-addressed stores** built on git's design principles (clone/push/pull/branch/rebase) but for arbitrary application data, with **user-defined value types** and pluggable backends (`irmin-mem`, `irmin-git` bidirectional git bridge, `irmin-pack` — a space-optimised on-disk store inspired by git packfiles). Worth stealing: the cleanest demonstration that "git semantics" is a **library over a content-addressed store parameterized by your own typed merge function**, not a monolith — and that user-defined custom merge resolvers can drive automatic, high-throughput conflict handling. Source: [irmin GitHub](https://github.com/mirage/irmin), [MirageOS intro](https://mirage.io/blog/introducing-irmin).

**TerminusDB** — a document-graph DB that **never mutates**: every commit appends an **immutable, content-addressed delta layer** (added + removed RDF triples) in a **succinct data structure** (~13 bytes/triple on billion-triple sets), and a query resolves current state by traversing the layer stack. It exposes branch/diff/merge/clone/time-travel like git-for-structured-data, and runs periodic **delta-rollup** (squash-like) operations to flatten the stack for read performance while retaining history. Worth stealing: the **layer-stack + periodic rollup** pattern — an append-only delta log that stays read-fast by compacting cold layers (directly analogous to Dolt's journal→oldgen conjoining and a model for compacting a Sema operation log). Source: [TerminusDB explanation](https://terminusdb.org/docs/terminusdb-explanation/), [GitHub](https://github.com/terminusdb/terminusdb).

**"git-as-database" patterns** (Dolt-as-git-remote, sqlite-diffable, dump-to-text) — the recurring lesson is **decompose the binary into content-addressed pieces and let git move the pieces**, never the monolith. Dolt's `refs/dolt/data` chunk-packing (seen in the local cache) is the mature version; the naive end is text-dumping for line diffs. The principle: pick a *chunk* granularity git can dedup, and a *projection* (text or typed log) for human/tool diff.

## 4. Design lessons for OUR case (strict-typed rkyv + redb + hard schema migration)

**The hard constraint that picks the architecture: record TYPES change on every schema migration.** A pure prolly-tree/content-addressed-chunk store (Dolt/noms) is type-agnostic at the storage layer — it diffs *bytes of rows*, and a schema change is "just" a new table schema recorded in the commit. That works for Dolt because its rows are dynamically-typed SQL cells. **It does not fit a strict-typed rkyv store**, where the on-disk bytes ARE a specific Rust type's archived layout: change the type and old chunks become undecodable, so byte-level structural sharing across a migration boundary buys nothing (the bytes don't share). Conversely, a **pure append-only typed operation log** (Datomic) fits the typed/accretion model beautifully but says nothing about how to read v3 bytes after the code only knows v4.

**Recommendation: a hybrid — an append-only typed operation log layered over a content-addressed-*schema* substrate.** This is exactly the model `sema`/`sema-engine` already name as their endpoint, and the research validates it:
- **Today (realization step, already shipped):** `Sema::open_with_schema(path, SchemaVersion)` writes the schema version on first open and **hard-fails on mismatch** (`sema/ARCHITECTURE.md` §Versioning). Migration today is a coordinated rebuild — delete old DB, recompile against new schema, or run a one-off migrator. The two-submodule From-chain (`historical` reproducing old rkyv types → `current_shape` → `From` chain) is the migrator's body. This is the cheap, correct floor; the daily file snapshot (§2) backstops it.
- **Eventually (the right shape, and the lesson):** identify a schema by the **hash of its Sema source** — equal hash ⇒ equal schema by construction, retiring the manual `SchemaVersion` integer. A runtime then holds **multiple schema versions concurrently**: chunks/records archived under v3's hash decode through v3's typed shape, v4's through v4's, and **migration is a reducer** — a typed function `v3-record → v4-record` that runs over the v3 archive to produce a v4 archive, both coexisting under different schema-hash addresses until v3 is retired (`sema/ARCHITECTURE.md` lines 235-256). This is **Datomic's "schema-as-data"** (the schema is itself queryable facts, evolves by accretion) fused with **Dolt's "schema-in-the-commit"** (each commit/snapshot carries the schema identity that decodes its records) — and irmin's "typed merge as a user-supplied function" generalizes to "typed migration as a user-supplied reducer."

**Concrete lessons to carry into the design:**
1. **Make the operation log the source of truth, current state a materialized view** (Datomic, TerminusDB, and already sema-engine's `CommitLogEntry` over a monotonic `CommitSequence`). This gives time-travel, audit, and replay for free, and makes `Mutate`/`Retract` first-class rather than destructive — important because redb's COW page-reclaim means `sema`'s kernel `remove` is *irreversible* (`sema/INTENT.md`: "Removal is irreversible at the kernel layer"), so recoverable deletion must live in the log layer above, not the storage kernel.
2. **Schema identity must travel inside the commit/snapshot, not as ambient global state.** Dolt records the table format in the manifest and schema in each commit; the lesson for rkyv is to tag each archived record (or each table-file generation) with the **schema-hash that produced it**, so a future binary knows which `From`-chain to run. A single global `SchemaVersion` (today's mechanic) is the degenerate one-version-at-a-time case of this.
3. **Migration = reducer, run lazily or as a compaction pass.** Mirror Dolt's journal→oldgen conjoining and TerminusDB's delta-rollup: keep old-schema records readable under their hash, run the `From`-chain reducer to rewrite them into the new schema during a compaction, and only then retire the old archive. This makes a schema migration a *background re-encode*, not a stop-the-world rebuild.
4. **Don't chase byte-level structural sharing across migration boundaries** — it's a mirage for typed rkyv (the whole point of prolly trees is type-agnostic byte sharing, which a hard type change breaks anyway). Get the *logical* equivalent instead: structural sharing *within* a schema version (content-address the rkyv chunks if cheap diff/sync is wanted), plus reducer-based bridging *across* versions.
5. **Keep the text edge (NOTA) as the diffable projection**, the way git-as-database patterns dump-to-text and Dolt offers SQL diff: the strict rkyv bytes are for the machine; NOTA is the human/tool-diffable view of the same typed records — already the workspace doctrine.

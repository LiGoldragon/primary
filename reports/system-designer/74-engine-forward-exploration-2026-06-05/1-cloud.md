# Cloud two-contract triad port — current state and move-forward map

Area: `cloud`, `signal-cloud`, `meta-signal-cloud` (under
`/git/github.com/LiGoldragon/`). Read-only mapping for the engine-forward
exploration. All citations verified against source on 2026-06-05.

## Summary verdict up front

The cloud triad is **the most advanced two-contract port in the stack** and
the only daemon area where the generic triad runner is already proven driving
two contracts end-to-end. But there is a **two-implementation split**: a live
hand-written `Store` path that the running daemon actually serves, and a fresh
schema-generated `SchemaRuntime` engine that is built and tested but **NOT
wired to any socket**. The remaining work is convergence (retire `Store`, serve
`SchemaRuntime`) and a missing two-listener daemon primitive — not greenfield.
The foundation under the schema/runner path is stable enough to build on now;
the daemon-orchestration glue is the part still on shifting sand.

## 1. CURRENT STATE — landed vs scaffold (file:line evidence)

### What actually exists and works

**Both Signal wire contracts are real, hand-written, and complete.**

- `signal-cloud/src/lib.rs:327-338` — ordinary contract via `signal_channel!`:
  exactly two peer operations `Observe(Observation)` + `Validate(Validation)`,
  reply arms `Observed / Validated / RequestUnsupported / RequestRejected`. This
  matches the intent (vnnx): working, peer, read-only Observe+Validate. The
  whole record vocabulary (capabilities, zones, records, redirects, plans,
  validation findings) is defined `signal-cloud/src/lib.rs:40-325`.
- `meta-signal-cloud/src/lib.rs:155-166` — meta contract via `signal_channel!`
  with the **8 owner-only mutations**: `RegisterAccount, RotateCredential,
  SetPolicy, PreparePlan, PrepareProjection, ApprovePlan, ApplyPlan,
  RetireAccount`. The intent (vnnx) named "8 account/plan mutations"; this is
  exactly 8 and matches.
- Record types are **shared correctly across the two contracts**: meta's
  `lib.schema` and the cloud Nexus/SEMA schemas import `signal-cloud` roots
  (`Provider`, `DomainName`, `Plan`, `ZoneListing`, etc.) — e.g.
  `cloud/schema/sema.schema:30-32` pulls `signal-cloud:lib:Provider` and
  `meta-signal-cloud:lib:CredentialHandle`. This is the "sharing record types"
  half of vnnx, realized.

**Both contract crates emit Signal/Nexus/SEMA-shaped generated schema.**

- `signal-cloud/schema/lib.schema` + `meta-signal-cloud/schema/lib.schema` are
  authored 4-section NOTA schema sources (imports / input roots / output roots /
  body). They generate `src/schema/lib.rs` carrying the `@generated by
  schema-rust-next` marker (`signal-cloud/src/schema/lib.rs:1`,
  `Input`/`Output` enums at lines 307/314). The generation is wired through the
  shared driver: `signal-cloud/build.rs` calls
  `GenerationPlan::wire_contract(...)` and publishes the schema directory via
  `CargoSchemaMetadata::new("signal-cloud").emit_schema_directory(...)` plus
  `links = "signal-cloud"` (`signal-cloud/Cargo.toml`). This is the
  asschema-removal-done world: `schema/*.schema -> src/schema/*.rs` directly, no
  `.asschema`.

**The daemon-owned Nexus + SEMA planes exist as real schema + generated Rust.**

- `cloud/schema/nexus.schema` — the daemon's decision/effect language:
  `NexusWork [SignalArrived SemaReadCompleted SemaWriteCompleted
  EffectCompleted]` and `NexusAction [CommandSemaRead CommandSemaWrite
  CommandEffect ReplyToSignal Continue]`, plus the `EffectCommand`
  (`CloudflareObserveZones / ObserveRecords / ApplyPlan`). It imports the two
  contract `Input`/`Output` roots AND the local SEMA roots
  (`cloud/schema/nexus.schema:8-21`). This is exactly the (lc2r) shape: Nexus
  plane schema lives inside the daemon crate, imports the wire contracts.
- `cloud/schema/sema.schema` — durable-state plane: `SemaReadInput / SemaWriteInput`
  roots, `AccountPolicyTable`, `PlanTable`, `StoredPlan`, `StateMarker`
  (commit-sequence + digest). The 8 write inputs mirror the meta contract's 8
  operations (`cloud/schema/sema.schema:46-54`).
- `cloud/build.rs` consumes both contract schema dirs from Cargo metadata
  (`DependencySchema::from_cargo_metadata("signal-cloud"...)` and `meta-signal-cloud`,
  lines 49-61), targets `daemon_runtime` generation, and freshness-checks the
  checked-in `src/schema/{nexus,sema}.rs`. No hard-coded checkout paths —
  matches the INTENT.md constraint. If a contract's metadata is missing the
  build **warns and skips** rather than failing (`build.rs:81-86,105-110`).

**The generic triad runner is real and the cloud engine plugs into it.**

- `triad-runtime` crate exists and is depended on by `cloud` (Cargo.toml:41).
  It exposes `Runner`, `RunnerEngines`, `NextStep`, `ContinuationLimit`,
  `ContinuationExhausted` (`triad-runtime/src/lib.rs:29-32, runner.rs`). This is
  the (7ca4) generic runtime runner, extracted.
- The generated `NexusEngine` trait (`cloud/src/schema/nexus.rs:733-774`) is the
  (rpr5) author surface: the daemon author implements `apply_sema_write`,
  `observe_sema_read`, `run_effect`, `budget_exhausted_reply`, and `decide` —
  the 3 plane engines + effect handler + budget reply. The `execute` default
  method builds a `triad_runtime::Runner`, wraps the engine in a generated
  `NexusRunnerAdapter` (`nexus.rs:776-820`), and drives it. **The adapter is
  generated glue** — `NexusAction::into_runner_next_step` (`nexus.rs:721-731`)
  maps generated actions onto `triad_runtime::NextStep`. This is rpr5 realized:
  authors write engine methods, the runner adapter is emitted.
- `cloud/src/schema_runtime.rs` is the fresh engine implementation (the bp8k
  "write from desired shape" path). It implements `NexusEngine` AND `SemaEngine`
  on one `SchemaRuntime` struct: decides ordinary→SemaRead, meta→SemaWrite,
  maps SEMA completions back to Signal replies (`schema_runtime.rs:299-374`).
- **Verified working**: `cargo test --offline --test schema_runtime` passes 2/2
  (`cloud/tests/schema_runtime.rs`): an ordinary capability observation flows
  Signal→Nexus→SEMA→reply, and a meta registration flows
  Signal→Nexus→SEMA-write→reply, both through the generated runner via
  `runtime.execute(...)`. The crate builds clean offline.

### What is scaffold / placeholder / NOT wired (brutally honest)

**The running daemon does NOT use the schema engine or the runner.** This is the
single most important finding. `cloud/src/daemon.rs:24` constructs
`Arc::new(Mutex::new(Store::new()))` and serves it; `Store` is the ~1300-line
hand-written legacy implementation in `cloud/src/lib.rs:448-1288` that calls the
contracts' hand-written `Operation`/`Reply` API (`handle_ordinary_request` /
`handle_owner_request`). `grep triad_runtime` in `daemon.rs` and the bins
returns **nothing** — the daemon never touches the runner. So:
- The **live** path = legacy `Store` + hand-written `signal_channel!` API. It is
  the production-shaped one (real Cloudflare DNS via `flarectl`, diff-aware
  plans, approval ceremony — `lib.rs:1105-1172`).
- The **fresh** path = `SchemaRuntime` + generated planes + runner. It is
  correct and tested but **serves no socket** and has **stub effects**:
  `run_effect` returns empty listings / echoes the identifier
  (`schema_runtime.rs:316-328`), plan preparation/approval/apply all return
  `RequestRejected` (`schema_runtime.rs:186-197`), state is in-memory `Vec`s,
  there is no Cloudflare IO and no durable storage.

**`SchemaRuntime` is an in-memory triage engine, not the real provider engine.**
The real provider logic (Cloudflare zone/record listing, plan diffing, apply,
credential verification) lives ONLY in legacy `Store`. The fresh engine has the
control-flow skeleton but none of the provider substance ported over. So the
"implement the engine, do not stop at scaffolding" emphasis (545o/4tas) is
**partially met**: the runner wiring and SEMA-state triage are implemented; the
provider effects and durable state are NOT.

**No two-listener daemon primitive exists.** `triad-runtime/src/daemon.rs` only
offers `SingleListenerDaemon` / `BoundSingleListenerDaemon` (one
`UnixListener`). The cloud daemon needs to run TWO listeners (ordinary + owner)
over ONE shared engine — the intent (vnnx): "runtime imports both + runs two
listeners." The legacy `daemon.rs` does this by hand
(`run_ordinary_listener` + `run_owner_listener` threads, `daemon.rs:34-43`),
but there is no generic/generated two-listener runner in `triad-runtime`. This
is a real generator/runtime blocker for the orchestrated target.

**`PrepareProjection` / projection lowering exists only in legacy.** The
`signal-domain-criome` projection→DesiredState lowering (the domain-criome
integration, ARCHITECTURE.md slice 9) lives in `cloud/src/lib.rs:120-260` on the
legacy types. The fresh engine's `decide_meta_input` routes `PrepareProjection`
to a SEMA write that returns `RequestRejected` (`schema_runtime.rs:189-191`).

**Concept schemas are stale stubs.** `signal-cloud.concept.schema` and
`owner-signal-cloud.concept.schema` are v0.1 `Status Concept` placeholders
(`(Status Concept)`) carrying a different, older vocabulary than the real
`lib.schema`. They are vestigial and should be deleted, not maintained.

## 2. MOVE-FORWARD WORK ITEMS (ordered) + 3. STABILITY VERDICT

Ordered toward the running-orchestrated-system target (a live `cloud-daemon`
serving both contracts through the generated runner, supervisable by persona).

### Item A — Add a two-listener daemon runner to `triad-runtime` [SAFE-NOW]
**What:** Add a `TwoListenerDaemon` (or generalize to N listeners) alongside
`SingleListenerDaemon` in `triad-runtime/src/daemon.rs`: two `UnixListener`s
(ordinary + owner) feeding ONE shared `DaemonRuntime`/engine behind a mutex,
each stream decoded against its own contract's frame codec. **Repo/file:**
`triad-runtime/src/daemon.rs`. **Size:** M (~150-250 lines; the single-listener
shape and the legacy hand-rolled two-thread pattern in `cloud/src/daemon.rs`
are both templates). **Depends on:** nothing — pure addition.
**Verdict [SAFE-NOW]:** the `DaemonRuntime` trait, `ListenerError`, and
socket-prep helpers are stable and generic; adding a second listener is additive
and the wire contracts that define the two frame shapes are landed and frozen.
This is the missing-runner-primitive blocker the psyche wants named — it is
buildable now without rework risk because it sits on the stable
single-listener/socket machinery.

### Item B — Port real provider effects into the fresh engine's `run_effect` [PREP→SAFE-NOW for structure]
**What:** Move the Cloudflare provider IO (zone/record listing, plan apply,
credential verification) from legacy `Store` (`cloud/src/lib.rs:730-1197`) into
the `SchemaRuntime::run_effect` handler so `CloudflareObserveZones /
ObserveRecords / ApplyPlan` (`cloud/schema/nexus.schema:40-43`) actually call
Cloudflare. **Repo/file:** `cloud/src/schema_runtime.rs` + reuse
`cloud/src/cloudflare*.rs`. **Size:** L. **Depends on:** the Nexus effect
vocabulary (already landed) and the existing `cloudflare` module (reusable
as-is). **Verdict [PREP]:** the EffectCommand/EffectResult schema is stable
(SAFE to build against), and the `cloudflare.rs` provider client is reusable
verbatim, but the **Nexus decide-loop's effect routing is incomplete** — today
`decide` turns `EffectCompleted` into a bare `Continue`
(`schema_runtime.rs:348-350`) rather than threading the observed
records/zones into a SEMA write or a reply. That decide-loop wiring must be
finished first, so structure now, finalize the control flow as Item C lands.

### Item C — Implement SEMA write/read substance for plans + projections [PREP]
**What:** Replace the `RequestRejected` stubs for `PreparePlan / PrepareProjection
/ ApprovePlan / ApplyPlan` (`schema_runtime.rs:186-197`) with real logic:
diff-aware plan preparation, the `StoredPlan`/`AccountPolicyTable` tables from
`cloud/schema/sema.schema:64-67`, projection lowering (port from
`lib.rs:120-260`). **Repo/file:** `cloud/src/schema_runtime.rs`. **Size:** L.
**Depends on:** Item B (apply needs provider effects), and the SEMA table
schema (landed). **Verdict [PREP]:** the SEMA state schema (tables, markers) is
stable and the legacy algorithms are a direct port source, but plan-apply is
entangled with provider effects (Item B) and with whether persistence lands
(Item E), so structure the table-backed state now, finalize apply after B.

### Item D — Wire the daemon to serve `SchemaRuntime` through the runner; retire `Store` [WAIT→then SAFE]
**What:** Make `cloud/src/daemon.rs` construct `SchemaRuntime`, wrap it in the
two-listener runner (Item A), and route each incoming frame through
`NexusEngine::execute(...)` instead of `Store::handle_*_request`. Then delete
the legacy `Store` and the hand-written `signal_channel!` call paths once the
fresh engine reaches parity. **Repo/file:** `cloud/src/daemon.rs`, `cloud/src/lib.rs`,
`cloud/src/bin/cloud-daemon.rs`. **Size:** M for the wiring, L for the deletion.
**Depends on:** A, B, C (cannot retire `Store` until the fresh engine has
provider + plan parity). **Verdict [WAIT]:** blocked on B+C reaching feature
parity — flipping the daemon to the stub engine today would regress the live
Cloudflare path. The **wiring** to the runner becomes SAFE-NOW the moment Item A
lands and can be done against a still-stubbed engine behind a feature flag, but
the **`Store` deletion** waits on parity.

### Item E — Durable SEMA persistence (redb) behind the SEMA engine [WAIT]
**What:** Back `AccountPolicyTable` / `PlanTable` with durable storage instead of
in-memory `Vec`s. **Repo/file:** `cloud/src/schema_runtime.rs` (+ a storage
module). **Size:** L. **Depends on:** the sema-engine persistence layer that
ARCHITECTURE.md:67-69 explicitly defers ("intentionally deferred because the
current engine still pulls the deprecated `signal-core` dependency").
**Verdict [WAIT]:** named blocker — the shared `sema-engine` persistence is on
the deprecated `signal-core` dependency and is part of the still-shifting
foundation. The SEMA table SCHEMA is stable (so the data shapes won't churn),
but the persistence MECHANISM is not yet safe to build on. Keep the store
boundary small (it already is) and wait.

### Item F — Delete vestigial concept schemas [SAFE-NOW]
**What:** Remove `signal-cloud/schema/signal-cloud.concept.schema` and
`meta-signal-cloud/schema/owner-signal-cloud.concept.schema` (stale `Status
Concept` v0.1 stubs superseded by the real `lib.schema`). **Size:** XS.
**Depends on:** nothing. **Verdict [SAFE-NOW]:** they are dead; the real
`lib.schema` files are the live source and are frozen.

## 4. THE ONE highest-leverage first step

**Item A — add the two-listener daemon runner to `triad-runtime`.** It is the
single missing primitive standing between "the runner is proven in a unit test"
and "the runner can actually be the daemon's serving loop for BOTH contracts."
It is [SAFE-NOW] (additive, on stable single-listener + frozen-wire-contract
foundation, zero rework risk), it is the named generator/runtime blocker the
psyche asked to surface, and it unblocks Item D (the whole orchestrated-daemon
flip). Every other item is engine substance that can proceed in parallel, but
without the two-listener runner the cloud daemon cannot be the generated-runner
daemon at all — so this is the leverage point.

## Foundation-stability note (why these are not sand)

Stable and safe to build on now: (1) **both Signal wire contracts** —
hand-written, complete, frozen, the 2+8 operation split matches intent exactly;
(2) **the runner shape** — `triad-runtime::Runner`/`RunnerEngines`/`NextStep`
and the generated `NexusEngine` trait + `NexusRunnerAdapter` glue are landed and
test-proven driving two contracts; (3) **asschema removal is done** — generation
is `schema/*.schema -> src/schema/*.rs` via the shared driver with no `.asschema`
and no hard-coded paths; (4) **the plane separation** — Signal in contract
repos, Nexus+SEMA inside the daemon crate (lc2r), is physically realized and
builds. Still shifting (the [WAIT] items): the **sema-engine persistence layer**
(stuck on deprecated `signal-core`) and the **daemon-orchestration glue** (the
two-listener runner does not exist yet, and persona-supervised orchestration of
this daemon is entirely unbuilt here).

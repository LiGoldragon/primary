# Prior art: local-first, CRDTs, and patch theory

## Frame: what this cluster teaches a single-writer log

The brief asks for a reusable library that lets any component version-control its durable Sema state — server-backed, atomic, surviving schema churn — and the accepted direction is "version the LOG, not the store": make the commit log payload-bearing, hash-linked, authoritative, and demote the redb tables to a rebuildable materialized view. That sentence is, almost word-for-word, the central architectural lesson of the modern local-first / CRDT world. The systems below all converged on the same shape from a different starting point (concurrent multi-writer text editing), and the most recent of them (eg-walker, 2024) arrived exactly where the brief points: **an authoritative append-only event log of lightweight operations, plus a materialized view that is a discardable cache replayed from the log.**

The crucial filter for us: every system here exists to *merge concurrent writers*. We are single-writer per store (one redb file, one engine handle, one daemon). So the CRDT *merge machinery* — YATA tie-breaks, pushout conflict resolution, transient-CRDT replay on concurrent branches — is mostly a non-fit. What transfers is the **substrate**: content-addressed hash-linked operation logs, the log/view split, the sync protocols that ship a suffix efficiently, and the compaction-vs-sync tension that Beelay's Sedimentree directly answers. We should mine the substrate and discard the merge theory.

## Automerge — content-addressed change DAG, columnar log, Bloom-filter sync

### What it is and core model
A JSON-like CRDT library (Rust core, JS/other bindings) for arbitrary structured documents. Internal representation is a monotonically growing **OpSet**; merging two replicas is set union of operations. Every operation has an `opID` = Lamport clock `(actorId, counter)`. Changes (commit-like atomic bundles of ops) form a **directed acyclic graph**; the DAG "heads" (the set of changes with no successors) uniquely identify a document state. ([automerge.org/docs/reference/glossary], [automerge.org storage docs])

### How it makes state durable / content-addresses
This is the most directly relevant system to our brief. A **change hash is the 32-byte SHA-256 of the chunk-type byte (`0x01`) ++ chunk-length ++ chunk-contents** of that change serialized as a change chunk. Each change carries an explicit array of dependency hashes (its DAG parents), so changes form a **hash-linked DAG** — exactly the "prev digest" / hash-link envelope field in the brief's typed replay envelope. Each chunk also carries a 4-byte checksum = first four bytes of that same SHA-256, a cheap integrity probe that doesn't require recomputing the full hash. ([automerge-binary-format-spec])

Storage is **columnar**: rather than row-per-op, each op *field* becomes a column, then each column is delta-encoded / run-length-encoded / DEFLATE-compressed. A *change chunk* holds one change; a *document chunk* aggregates many changes (with an actor registry and a heads index mapping change hash → columnar position for random access). ([automerge-binary-format-spec], [automerge storage docs])

### Schema evolution story
Weak and worth noting as a *gap we must not inherit*. Automerge documents are schemaless JSON-shaped; there is no typed-schema version concept and no first-class migration entry. "Schema change" is just more ops that add/rename/remove keys, and old changes referencing old keys remain in the DAG forever. There is no `SchemaTransition(v_old_hash -> v_new_hash, reducer)` analogue. This is the opposite of our world, where each rkyv schema version is a genuinely different, byte-fragile type set and a migration is a first-class typed log entry.

### Reusable-library packaging
Genuinely a library, not a service: `automerge` (Rust core) + `automerge-repo` (batteries-included sync/storage toolkit) + pluggable `StorageAdapter` (any key/value store supporting prefix-range queries; documented as safe for concurrent Repos over the same storage) and `NetworkAdapter`. Genericity is achieved by being **schemaless** — one generic document type holds arbitrary JSON. That is precisely the stringly-typed-generic-store failure mode our discipline forbids (`skills/abstractions.md`: the type must carry meaning). Automerge buys reusability by *giving up* perfect specificity. We cannot copy that mechanism; we need genericity over *typed* record families.

### Sync protocol (transfers well)
Two peers exchange **heads + a Bloom filter** of the change hashes they hold (≈10 bits/commit in the prototype). The receiver uses the filter to find which of its changes the peer is missing and ships only those; round trips scale with the longest head-to-common-ancestor path, not history length. Based on Kleppmann's hash-graph-sync work (arXiv:2012.00472). ([kleppmann.com bloom-filter-hash-graph-sync], [automerge sync docs]) This is a concrete, proven answer to "ship the log suffix since the peer's last head" — though for single-writer linear logs a Bloom filter is overkill; a simple "send everything after sequence N / after head-hash H" suffices, and the Bloom machinery only earns its keep under branching.

### Server / durability model (Beelay + Sedimentree — directly answers our compaction problem)
**Sedimentree** is Automerge's newest answer to exactly the brief's "compaction breaks naive suffix sync" worry. It models the commit DAG as **strata** (compacted segments): a stratum references a start and end commit hash with intermediate **checkpoints**, and *omits the change hashes of the interior runs it compacts*. This keeps granular history queryable without storing every hash. Peers holding differently-compacted views reconcile by aligning on shared checkpoint boundaries. **Beelay** wraps this for a sync *server*: it syncs whole *collections* of linked documents, uses RIBLT (rateless invertible Bloom lookup tables) to reconcile which docs differ with bandwidth proportional to the actual diff, and — critically — the server holds an **encrypted reachability index and operates over opaque data without needing full document state in memory**. ([github.com/automerge/beelay/blob/main/docs/protocol.md], [automerge.org/blog/automerge-repo])

This is the closest external prior art to our checkpoint protocol (identity, schema-hash inventory, covered range, digests, retained-suffix policy, pruned-head behaviour). The strata-with-checkpoints structure is a literal template for our checkpoint design: a checkpoint is the anchor that lets a peer who only kept the compacted suffix still sync against a peer holding the granular tail. Note the divergence we should make deliberately: Beelay's server is *zero-knowledge* (opaque/encrypted) so it can host untrusted multi-tenant docs; our server (ouranos as backup ingest target) is first-party and trusted, so we can keep digests and schema-hash inventory in the clear and let the server validate the chain.

## Yjs / Yrs — Lamport-pair operation log, state-vector delta sync, no content hashes

### What it is and core model
The dominant production CRDT for collaborative editing (Yrs is the Rust port). A list-CRDT implementing **YATA** ("Yet Another Transformation Approach"). Each `Item` (a run of inserted content) has `id = (clientID, clock)` — a Lamport pair — plus `origin`/`originRight` neighbour references that make concurrent inserts at the same position resolve deterministically. The `StructStore` keeps items per-client in insertion-clock order (binary-searchable by id). ([docs.yjs.dev/api/internals], [yjs INTERNALS.md])

### Durability / content-addressing
Notable as the **counter-example to content-addressing**: Yjs identifies everything by `(clientID, clock)` Lamport pairs, **never by content hash**. There is no Merkle/hash DAG; integrity rests on the integer clocks alone. Deletions are a state-based **delete set** (a bitfield/run encoding), recording *that* an item is deleted but not when or by whom. Garbage collection replaces deleted item content with a lightweight `GC` marker storing only the run length. ([docs.yjs.dev/api/internals], [yjs INTERNALS.md])

### Sync protocol (the state-vector idea transfers cleanly)
Two-step delta sync: peer A sends its **state vector** (`map clientID → clock`, i.e. "highest op I've seen from each writer"); B computes the minimal missing update by clock comparison and ships only the delta, in a compact custom binary encoding (v1/v2). A 77k-deletion benchmark compresses to 4.5 KB. ([docs.yjs.dev/api/internals], [yjs INTERNALS.md]) For our single-writer store the "state vector" degenerates to **one integer — the peer's last known CommitSequence** — which is exactly the brief's "ship the log suffix since the peer's last head," and confirms that the brief's choice to keep `CommitSequence` as the ordered cursor (digests sit *beside* it) is the right shape: a monotonic per-writer counter is all a single-writer delta needs; a hash is not orderable and cannot answer "what's after N."

### Schema evolution / library packaging
No schema concept (shared types: Y.Map, Y.Array, Y.Text), so nothing transfers on migration. Packaged as a library, generic only over a fixed set of shared CRDT types — again schemaless-generic, not typed-family-generic. Same non-fit as Automerge for our genericity mechanism.

## Pijul + darcs — patch theory, the pristine-as-CRDT, and a copy-on-write KV backend

### What they are and core model
Distributed VCS built on **patch theory**: a repository state is *a set of patches*, and patches that don't depend on each other **commute** (can be reordered). darcs (Haskell, 2003) pioneered this; **Pijul** (Rust) reformulated it on a sound categorical basis (Mimram & di Giusto: merge = a pushout in a category where files are objects and patches are arrows). ([pijul.org/model], [pijul.org/manual/why_pijul], [cs.tufts.edu theory-of-patches])

### How they version / represent conflicts
darcs's fatal flaw is directly instructive: conflicting patches **never commute** in darcs, and resolving them can be **exponential in the number of conflicting patches** — real two-line merges taking hours. Pijul's fix: represent the working state (the **pristine**) as a **graph of byte-chunks with edges** that is itself a **CRDT** — conflicting patches *always* commute because the conflict is represented *in the graph* (extra vertices/edges) and only *detected* afterwards by inspecting the graph, never resolved eagerly. Each patch is content-addressed by a hash. ([pijul.org/model], [pijul.org/faq], [wikibooks Understanding Darcs/Patch theory])

### Durability backend (this is the most transferable Pijul artifact)
Pijul stores its pristine and change index in **Sanakirja** — an on-disk **transactional, ACID, copy-on-write B-tree key-value store in Rust**, explicitly designed as a generic ACID allocator in a file on which higher data structures (ropes, R-trees, radix trees) are built. Its standout feature is **O(log n) `fork_db`** — cheap branchable/clonable tables via transactional reference counting — and it benchmarks 20–50% faster than LMDB on graph-storage loads. ([pijul.org/posts/2021-02-06-rethinking-sanakirja], [crates.io/crates/sanakirja], [docs.rs/sanakirja]) Sanakirja is *the same class of substrate as redb* (single-file, COW B-tree, ACID, mmap-ish). The relevant lesson is the layering: Pijul keeps the **patches (the log) as the authoritative content-addressed store** and treats the **pristine graph as a cache** ("This graph acts as a 'cache' of applied patches") — the log/view split again, from the VCS direction. Sanakirja's cheap `fork_db` is a feature redb lacks (redb 4.x has no online-snapshot path in use per the brief); if cheap branching ever became a requirement, Sanakirja is named prior art, but adopting it would mean leaving redb, which is out of scope.

### Schema evolution / library packaging
No data-schema evolution story (it versions source text, not typed records). Packaged as an application + the reusable Sanakirja crate; the patch algebra (`libpijul`) is a library but is generic over *text/byte graphs*, not over arbitrary typed records — so its genericity mechanism doesn't transfer, only Sanakirja does.

## Eg-walker (2024) — the system that already is "version the log, not the store"

### What it is
Event Graph Walker, by Joseph Gentle (diamond-types) and Martin Kleppmann, arXiv:2409.14252, "Better, Faster, Smaller." The most important entry for us because it independently validates the brief's exact architecture.

### Core model — this *is* our direction
It stores an **event graph**: a DAG of **lightweight operations** (`insert(pos, char)`, `delete(pos)`) each linked to its causal **parents** — i.e. an append-only operation log with hash/version parent links, and *no* per-element CRDT metadata persisted in steady state. The materialized document is **just an indexed sequence**. CRDT machinery is **constructed transiently only when merging concurrent branches**, by replaying the relevant subgraph, then **discarded after the merge**. Result: an order of magnitude less steady-state memory than state-based CRDTs and orders-of-magnitude faster document load from disk, while merging long-running branches stays fast. ([arxiv.org/abs/2409.14252], [loro.dev/docs/concepts/event_graph_walker])

### Why it matters to us
This is the strongest external evidence that **"authoritative lightweight op-log + rebuildable materialized view"** is not a speculative direction but the current state of the art. The single most useful insight for our single-writer case: eg-walker only pays the CRDT/replay cost **on concurrent branches**; **on a linear (single-writer) history there is no concurrency and the "merge" is a trivial append.** That is our common case. So for us, replay is *pure schema migration replay*, not conflict resolution — the brief's `SchemaTransition` reducer plus checkpoint is the right and *only* replay we need, and we get to skip the entire transient-CRDT apparatus that makes eg-walker complex. We are eg-walker with the concurrency removed and typed schema transitions added.

## Where local-first ideals land for us

The Ink & Switch "Local-first software" manifesto (Kleppmann, Wiggins, van Hardenberg, McGranaghan, 2019) frames the *why*: data you own, works offline, long-term preservable, multi-device, with the network as an enhancement not a dependency. ([inkandswitch.com/essay/local-first]) The brief's remote-durability levels (local-committed / queued-for-mirror / server-committed with a named RPO; **the local view must never lag — read-after-write — only the remote mirror lags**) are a single-writer specialization of the local-first stance: the local redb commit is authoritative and synchronous; the server mirror is an asynchronous durability tier. This matches local-first's "local is the source of truth, sync is best-effort enhancement," and is the *opposite* of a server-of-record (cloud) model. Worth stating explicitly because it justifies why we keep the same-redb-transaction free atomicity for local commit (log and view cannot diverge) and only the *mirror* is allowed to lag.

## The experiment matrix (same-file vs separate-file log) seen through prior art

The psyche wants both tested. Prior art splits cleanly on this axis and is worth tabulating in the eventual design report:

- **Same-file (log table inside the component's `.sema`, written in the same redb txn as the view):** today's behaviour, and it buys *free atomicity* — log and materialized view commit together and cannot diverge, which is exactly redb's single-writer ACID guarantee. Closest analogue: Automerge document chunks bundling changes + heads index in one artifact, and Pijul's pristine + change index both living in one Sanakirja file. Risk: compaction/pruning of the log fights the view's storage lifecycle inside one file; Sedimentree's strata show compaction wants its own lifecycle.
- **Separate-file log (log as its own append-only artifact alongside the view):** matches Automerge-repo's `StorageAdapter` (changes stored as individually-hashed chunks, snapshots separate) and Beelay/Sedimentree's stratum files. Buys independent compaction, easy suffix-shipping to the server (the file *is* the backup unit), and a log that survives a corrupt/rebuilt view. Loses the same-transaction atomicity — now you need a two-phase / write-log-then-apply discipline (repository-ledger's atomic spool — temp-write+rename, commit-then-move-to-processed — is our in-house template for exactly this, and is already event-shaped). The torn-write window between "log appended" and "view updated" is the thing to design against.

Neither side is free. Same-file = atomicity-for-free, compaction-friction. Separate-file = compaction/sync-friendly, atomicity-must-be-engineered. Both are represented in shipping prior art, which is why testing both is reasonable rather than wasteful.

## Genericity: how the prior art resolves (against us) the central tension

The brief names the central tension: generic over arbitrary record families **without** degenerating into a stringly-typed generic-record store. The CRDT cluster resolves this tension by **choosing the wrong side for us** — Automerge and Yjs are *schemaless*; their genericity *is* "one generic document, fields are strings." That is precisely what `skills/abstractions.md` forbids. So the prior art's lesson here is a **warning, not a template**: do not buy reusability by erasing types. The eg-walker / log-of-typed-ops shape points the better way — the log is generic over an *operation/entry trait*, and each component supplies its own typed record families and its own typed `SchemaTransition` reducers; the library owns the *envelope, chaining, checkpoint, and replay-driver* nouns (an owning `CommitLog`, `Checkpoint`, `SchemaTransition`, `ReplayCursor`, `MirrorAck` — real data-bearing types), while the *payload type* stays the component's concern. That keeps the method-on-data-bearing-type discipline (no ZST namespaces, no free functions) and keeps perfect specificity at the payload edge. The genericity lives in the *log substrate traits*, not in a universal record type — the inverse of how Automerge/Yjs do it.

## Reference: our anchors the prior art maps onto

- Our SHA-256-equivalent already exists in-house but isn't wired into sema: criome's blake3 `ObjectDigest::from_bytes` (signal-criome/src/lib.rs:55-57) is the same content-addressing move as an Automerge change hash; spirit's `state_digest` (store.rs:902-915, blake3 over committed state) is a hand-rolled, store-level version of what a per-entry log digest would systematize. These are the local seeds the library would absorb.
- Our `CommitLogOperation` (log.rs:63-68 — `operation, table_name, key`, metadata-only, no payload) is the pre-payload ancestor of Automerge's payload-bearing change; making it payload-bearing + hash-linked is the "version the log" step.
- `SnapshotIdentifier(u64)` (snapshot.rs:19) and `CommitSequence` (the ordered cursor in `CommitLogEntry`, log.rs:15-19) are our Lamport-counter analogue — Yjs confirms a monotonic counter is the right *cursor* and a hash is the right *identity*, and they must coexist (the brief's "digests sit beside the monotonic markers"), exactly as Automerge keeps both opID Lamport counters *and* change hashes.

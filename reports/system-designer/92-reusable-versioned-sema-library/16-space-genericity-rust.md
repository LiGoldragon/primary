# Design space: reusable in Rust without a stringly-typed generic-record store

## What this report maps

The psyche wants ONE reusable library (Spirit `j487`) that lets any component opt into version-controlled, server-backed, atomic, schema-churn-surviving Sema state (Spirit `29pb`). The hard part is not the wire or the server. It is *genericity*: the library must work for arbitrary record families across many components, while `skills/abstractions.md:229-232` forbids a generic-record / stringly-typed / string-tagged-dispatch store, and the no-free-functions rule forces every verb onto an owning noun. This report surveys how mature reusable VC/storage libraries achieve content-agnostic reuse, then enumerates the concrete Rust shapes OUR library could take, each scored against that central tension. It maps options; it does not pick one.

The first finding from the code is that we are not starting from zero genericity — we are starting from *half* genericity, and the half that exists is the half the discipline warns against.

### The seam that already exists in sema-engine

`sema-engine` is already generic over the record type at the call site: `register_table<RecordValue>(TableDescriptor<RecordValue>)` and `register_identified_table<RecordValue>` return a phantom-typed `TableReference<RecordValue>` (`table.rs:25-71`, `engine.rs:60-92`), and every verb (`Assertion<RecordValue>`, `Mutation<RecordValue>`, `Retraction<RecordValue>` in `mutation.rs`) carries the record type through. The bound that makes any type storable is the blanket trait `EngineStoredValue` (`record.rs:80-104`): a type is engine-storable iff it is `rkyv::Archive + Clone + Serialize` with a deserialisable, byte-checkable archive. This is *derive-based per-type codec* genericity — the rkyv lineage — and it is fully typed.

But the moment data leaves the type system, genericity collapses to a string. The `Catalog` stores only `TableRegistration { table_name: String }` (`catalog.rs:13-37`), matched by `table_name() == name.as_str()`. The commit log records `CommitLogOperation { operation, table_name: String, key: Option<RecordKey> }` (`log.rs:63-68`) — metadata only, no payload, no schema identity, NOT replayable. So the existing seam is typed at the API and stringly-typed at the catalog/log boundary. The reusable VC library's job is precisely to push a *typed* identity (a schema-hash selector) down to where today there is a bare string, WITHOUT inventing a `GenericRecord` blob to carry it. That is the whole design problem in one sentence.

## Prior art: how reusable VC/storage libraries get content-agnostic

### Irmin — ML functors over (Contents, Hash, Branch, Path, Node, Commit)

Irmin is the closest prior art to `j487`: a library, not a database, that gives *any* OCaml type a git-like versioned store. It achieves reuse with **functors** — `Irmin.Make` is parameterised over a small set of module signatures: a `Contents` module (your value type plus a 3-way `merge`), a `Hash`, a `Branch`, a `Path`, and the backend (`Irmin_mem`, `Irmin_git`, `irmin-pack`). The store is a Merkle DAG of typed blobs; the user supplies the content type and its merge function, the functor supplies commits, branches, history, and sync. The lesson that *applies to us*: content-agnostic reuse is achieved by parameterising over a *fixed, small set of named roles* (content, hash, merge), not by erasing the content to bytes. Rust's analogue of a functor over a content signature is a generic `VersionedStore<R>` bounded by a `RecordFamily` trait — the option block below. The lesson that *does not apply*: Irmin's headline feature is mergeable, branchable, conflict-resolving state (3-way merge for collaboration). Our system is single-writer-per-store, append-only, Datomic-shaped; we want linear history and replay, not branch-merge. We can drop the `merge` leg of the functor entirely, which simplifies the trait substantially.

### Datomic — schema-as-data, the universal `[entity attribute value tx op]` datom

Datomic stores everything as immutable 5-tuples; the schema is itself data (attributes are entities). This is maximal content-agnosticism: one physical shape carries every domain. It maps onto our commit-log-as-truth direction almost exactly — a payload-bearing, append-only, time-indexed log of facts is a datom log. The lesson that *applies*: time/sequence is the spine, facts accrete, the "current view" is a fold (this is exactly "version the log, not the store"). The lesson that *does NOT apply*, and is the sharpest hazard for us: Datomic's universality is bought by erasing types — every value is a member of one `Object`-like value union keyed by a runtime attribute identifier. That IS the stringly-typed generic-record store `abstractions.md:229-232` forbids. We want the Datomic *log topology* without the Datomic *value erasure*. The schema-hash selector is how we keep the topology while restoring the type: instead of "attribute id 42 means this is a string," we say "schema-hash `h` selects this exact rkyv type set as the decoder."

### Git — a tiny fixed closed set of typed object kinds (blob/tree/commit/tag)

Git is content-agnostic over file *contents* (any bytes) but rigidly typed over *structure*: exactly four object kinds, each with a fixed grammar, all content-addressed by hash. This is the opposite strategy from Datomic: not "one universal record," but "a small closed sum of structural kinds, and the domain payload is opaque inside the blob." The lesson that *applies*: a closed, small enum of *log-entry kinds* (Assert / Mutate / Retract / SchemaTransition / Checkpoint) is exactly Git's blob/tree/commit/tag move, and it satisfies perfect specificity at the envelope level because the kinds are named, not stringly-dispatched. The lesson that *does NOT apply*: Git makes the domain payload an opaque blob (a file is bytes). If we copy that wholesale, our *payload* becomes the erased byte blob `abstractions.md` warns about — typed envelope, untyped cargo. The ground truth already rejected git-as-substrate (double content-addressing); the design-relevant residue is git's *typed-kinds-with-hashes* envelope discipline, not its opaque-blob payload.

### sled / redb — generic `K/V`, genericity by total content-erasure

redb (`Table<K, V>`) and sled are generic over key/value by treating both as `&[u8]` (redb adds a `RedbValue`/`Key` trait for typed access). This is the lowest, most reusable layer and it is *correctly* content-agnostic because a storage engine's job IS to not know what it stores. `sema` already sits here: `Table<K, V: Archive>` (`sema/src/lib.rs:15,231`) wraps redb with an rkyv codec and a version-skew guard. The lesson that *applies*: total erasure is fine at the *kernel* (sema) — a B+tree should not know about record families. The lesson that *does NOT apply*: total erasure is exactly wrong at the *VC library* layer, because the VC library's whole value is knowing enough about each entry to replay it into typed state across schema versions. The byte-blob seam belongs strictly below the VC library, never inside it.

### serde / rkyv derive — per-type monomorphised codecs

`#[derive(Archive)]` generates a bespoke, monomorphised, zero-copy codec per type. This is the most type-preserving genericity mechanism available and the one we already depend on (`EngineStoredValue`, `record.rs:80`). The lesson that *applies*: derive-based codecs give us per-family typing for free, with no runtime tag — the type IS the codec. The hazard: rkyv archives are schema-FRAGILE (any field add/remove/reorder changes byte layout, no silent back-compat). So a monomorphised codec is bound to *one* schema version; replaying a log that spans versions needs *one decoder per (family, schema-version)* — which is precisely why the replay envelope needs a schema-hash selector, and why each historical `SpiritStoreV1..V7` already exists as a frozen type set (`production_migration.rs:89-116`). The derive mechanism alone cannot span versions; it needs a registry of versioned decoders on top.

## The Rust shapes our library could take

Each option is scored on: (a) does it honour perfect specificity / no stringly-typed fallback; (b) does every fn land on an owning noun; (c) how the typed replay envelope stays typed across families; (d) what the missing owning noun is.

### Option A — a `Versioned` / `Replayable` trait that each record family implements

Each storable type implements a trait carrying its own schema identity and replay rule, e.g. an associated `const SCHEMA_HASH` (or assoc-fn) plus an `apply(self, view) -> view` style fold. The engine is generic over `R: Replayable` and the log entry is `LogEntry<R>`.

- Specificity: HIGH. No string tag — the schema-hash is a const on the type; dispatch is monomorphised, the type carries the meaning. Honours `abstractions.md:229-232`.
- Owning noun: the trait methods live on the record family type itself (good), but the *replay/fold* logic wants an owner that is not the record — the missing noun is something like a `Reducer` or `Projection` per family (the thing that folds a log entry into the materialised view). This is the Irmin `Contents.merge` leg, minus merge.
- Envelope stays typed: only within a single `R`. A log that is homogeneous per table is fine; a single log interleaving *many* families (the realistic case — spirit has Entry plus referents) cannot be `Vec<LogEntry<R>>` for one `R`. This is the option's hard wall: trait-bound genericity is per-type, the log is multi-type. It forces either one log per family (fragments atomicity) or erasure at the collection boundary (Option C/D).
- Verdict to watch: clean for single-family stores, breaks down exactly where the existing commit log already lives — one interleaved log per component.

### Option B — a generic `VersionedStore<R: RecordFamily>` (the Irmin-functor analogue)

A `VersionedStore` struct parameterised by a `RecordFamily` trait that bundles the value type, its key, its schema-hash, and its decoders. `RecordFamily` is the Rust spelling of an Irmin content signature. One `VersionedStore<SpiritEntryFamily>`, etc.

- Specificity: HIGH at the type level; the family trait names exactly the family. No generic-record.
- Owning noun: FOUND — `RecordFamily` is the missing owning noun the verb is asking for, and `VersionedStore<R>` owns the log/view/checkpoint logic generically. This is the cleanest answer to "a library of generic types and traits must still find owning nouns": the noun is the family, the generic store hangs methods off it.
- Envelope stays typed: same multi-family wall as A. A component with N families needs N `VersionedStore`s or a heterogeneous container, reintroducing the erasure question at the seam between stores and the *single* commit log that must remain in one redb transaction for free atomicity (`log.rs:15-19`, the same-file atomicity property). This is where the same-file-vs-separate-file experiment bites: per-family `VersionedStore`s map naturally onto separate logs/files; a single shared atomic log wants Option C/D.
- Verdict to watch: best per-family ergonomics; in direct tension with the one-redb-file-per-component, one-atomic-log invariant.

### Option C — a registry keyed by schema-hash with associated-type / boxed decoders

A `FamilyRegistry` maps `SchemaHash -> &dyn Decoder` (or an enum of decoders). The log is a homogeneous `Vec<ReplayEnvelope>` where each envelope carries a `SchemaHash` selector and a payload; replay looks up the decoder by hash and produces a typed value into the right family's view.

- Specificity: MEDIUM, and this is the knife-edge. The schema-hash is a *typed selector into a closed set of registered decoders*, NOT an open stringly-typed dispatch — IF the registry is a closed sum (enum) over the component's known families, it honours specificity (it is Git's closed-kinds move). IF the registry is an open `HashMap<SchemaHash, Box<dyn Decoder>>` with a fallback, it degenerates toward the forbidden generic-record store. The boundary between honouring and violating `abstractions.md` runs exactly through "closed enum of families" vs "open map with default." The report flags this as the single most important line to hold.
- Owning noun: `FamilyRegistry` and `SchemaHash` are owning nouns; the decoder is a trait impl. Survives the no-free-fn rule.
- Envelope stays typed across families: this is the ONLY option that cleanly supports one interleaved atomic log over many families — which is what the existing log already is and what same-file atomicity wants. The payload is bytes-on-disk but never bytes-in-the-type-system: it is decoded immediately on read via the hash-selected typed decoder, so the *resting* representation is erased but the *operating* representation is fully typed. Whether "erased at rest, typed in motion" satisfies the spirit of perfect specificity is a genuine open question for the design phase, not a settled yes/no.
- Verdict to watch: the registry is where Datomic's topology and rkyv's typing are reconciled; it is also where the system is one careless `Box<dyn>` away from the forbidden store.

### Option D — trait objects (`Box<dyn ...>`) vs monomorphised generics, orthogonal to A/C

This is not a fourth design but the dispatch axis cutting across A-C. Trait objects (`dyn Decoder`, `dyn Replayable`) buy a heterogeneous log and runtime extensibility at the cost of an erased type behind the v-table and a hard collision with rkyv (rkyv's archived types and `CheckBytes` bounds in `EngineStoredValue` `record.rs:80-92` are not naturally object-safe — `for<'serialize>` HRTB serializer bounds and associated `Archived` types resist `dyn`). Monomorphised generics keep full typing and zero-cost dispatch but cannot put two families in one `Vec`. The realistic shape is a *hybrid*: a closed `enum LogEntryKind` (monomorphised, named variants per family — the specificity-honouring choice) rather than `Box<dyn>` (the specificity-eroding choice). The schema-hash then selects an enum variant, not a v-table. This keeps it as Git's closed-kinds rather than an open plugin registry.

### Option E — sealed `RecordFamily` trait + generated closed enum (the synthesis the code is pointing at)

Combine B's per-family trait with C's single-log registry by making `RecordFamily` a **sealed** trait (only the component's own families implement it) and generating, per component, a closed `enum ComponentRecord { Entry(SpiritEntry), Referent(...), ... }` whose variants are exactly the sealed families. The log is `Vec<ReplayEnvelope>` where the envelope's schema-hash selects an enum variant; the enum is a typed closed sum, not a blob.

- Specificity: HIGH. Sealed trait + closed enum is the maximal-specificity way to be generic — the set of families is fixed and named at compile time per component; no open dispatch, no fallback. This is exactly the "closed sum of structural kinds" Git lesson plus the rkyv per-variant codec.
- Owning noun: `RecordFamily` (sealed) is the trait; the per-component closed enum is the owning noun for the interleaved log; `ReplayEnvelope`, `SchemaHash`, `Checkpoint`, `CommitSequence` (kept beside the hash per ground truth) are the supporting nouns.
- Envelope stays typed across families: yes, via the closed enum — decode selects a variant, replay folds the variant into the materialised view. The schema-hash is the *decoder selector* the design direction calls for, realised as a variant selector rather than a string.
- Cost to watch: the closed enum is component-specific, so the *reusable* part is the trait + envelope + registry + checkpoint machinery (generic over `R: RecordFamily`), while the closed enum is generated per component (likely a derive/macro — and macro output must emit into `impl` blocks per the no-free-fn-in-schema rule). This splits the library cleanly into "reusable generic mechanism" (the `j487` deliverable) and "per-component closed instantiation" (generated). It also means a new schema version is a NEW variant / NEW sealed family + a `SchemaTransition` log entry (`v_old_hash -> v_new_hash`, reducer) — exactly the frozen `SpiritStoreV1..V7` pattern (`production_migration.rs:89-116`) but generated and replayable instead of hand-rolled.

## The central tension, restated as a placement question

`abstractions.md:229-232` (perfect specificity, no generic-record, no string-tagged dispatch) and the no-free-function rule together demand that the library find an *owning noun* for genericity rather than an erased bag. The survey says the owning noun is some spelling of **`RecordFamily`** (Irmin's content signature, sealed), and the missing-but-implied nouns are **`Reducer`/`Projection`** (the fold from log entry to view — Irmin's merge leg minus merge), **`SchemaHash`** (the typed decoder selector that replaces the catalog's bare `table_name: String`, `catalog.rs:13-37`), and **`ReplayEnvelope`** (the typed-in-motion, erased-at-rest log entry). The genuinely undecided question — the one this report deliberately leaves open — is *where on the erasure spectrum the resting log payload sits*: fully monomorphised per-family (Options A/B, breaks the single atomic log), or hash-selected into a closed sum (Options C/E, keeps the single atomic log but rests as bytes). "Erased at rest, typed in motion via a closed sealed sum" is the candidate that most plausibly satisfies both the discipline and the same-file atomicity invariant — but whether resting-bytes honours the *spirit* of perfect specificity is a design call for the kernel owners, not this map.

## The same-file vs separate-file experiment, in genericity terms

The psyche wants both tested. The genericity choice and the file choice are coupled, and the report should make the coupling explicit so the experiment is fair:

- **Same-file log** (log table in the component's one `.sema`, written in the same redb txn as the view — today's free atomicity, `log.rs:15-19`) strongly favours a *single interleaved log over many families* → Options C/E (closed-enum / hash-selected envelope). Per-family monomorphised stores (A/B) fight this because they want N logs.
- **Separate-file log** (a dedicated log file the view is rebuilt from) relaxes the single-transaction constraint and makes per-family `VersionedStore<R>` (Option B) more natural, at the cost of the atomicity that same-file gets for free (log and view can now diverge under crash; needs an explicit commit-then-publish protocol like repository-ledger's spool: temp-write+rename, commit-then-move-to-processed).

So the experiment is not just "two file layouts" — it is "closed-enum single-log genericity (C/E) vs per-family generic-store genericity (B)," and the file layout is the forcing function. Both experiments should carry the same record families (spirit's Entry + referents are the honest multi-family test case) so the multi-family wall in A/B actually shows up.

## Where this lands

The library splits, on every option, into a *reusable generic core* (envelope, schema-hash, checkpoint, durability levels, sequence cursor) and a *per-component instantiation* (the families, their decoders, their closed enum or their N stores). `j487`'s "build once, opt in per component" is satisfied by the core; specificity is satisfied by the instantiation being closed and sealed rather than open and stringly-keyed. The kernel inversion ("version the log, demote tables to a view") is *orthogonal* to the genericity choice — it can be done under any of A-E — and remains the sema/sema-engine owners' call, not decided here.

Kind: sub-agent report (system-designer, inherited from orchestrator per Spirit 920).

## Operation-by-operation parity

The production daemon's eight operation roots are declared at
`signal-persona-spirit/spirit.schema:7-15` and re-declared in Rust
as variants of `Operation` (emitted by `signal_channel!([schema])` at
`src/lib.rs:944`). spirit-next's operation roots are the `Input` enum
declared in `schema/lib.schema:2` and emitted into Rust as
`Input` at `src/schema/lib.rs:286-293`.

| `Record` | `Entry` (Topics, Kind, Description, Certainty) | `Record` | `Entry` (Topics, Kind, Description, Magnitude) | partial | name + arity match; Magnitude/Certainty rename + variant gap (see below) |

| Production variant | spirit-next variant | Match |

| `Clarification` | `Clarification` | yes |

Same vocabulary, same order, same arity. Five discriminants 0..=4.

### `Magnitude` (= production `Certainty` alias)

| Production variant | Discriminant | spirit-next variant | Discriminant | Match |
|---|---|---|---|---|
| `Minimum` | 0 | `Minimum` | 0 | yes |
| `VeryLow` | 1 | `VeryLow` | 1 | yes |
| `Low` | 2 | `Low` | 2 | yes |
| `Medium` | 3 | `Medium` | 3 | yes |
| `High` | 4 | `High` | 4 | yes |
| `VeryHigh` | 5 | `VeryHigh` | 5 | yes |
| `Maximum` | 6 | `Maximum` | 6 | yes |
| `Zero` | 7 | (absent) | — | **gap** |

Production at `signal-sema/src/magnitude.rs:1-15` uses `#[repr(u8)]` with
`Zero` placed last (= 7) "so persisted rkyv discriminants for the original
seven variants stay stable" — that is the discriminant-stability discipline
of Spirit 1249 manifested in source. spirit-next at
`src/schema/lib.rs:275-283` derives the discriminant order from the schema
declaration `Magnitude (Minimum VeryLow Low Medium High VeryHigh Maximum)`
(`schema/lib.schema:46`). The 0..=6 prefix matches by accident of equal
prefix ordering; spirit-next has no `Zero` rung. Adding `Zero` to spirit-next
must be done by appending (not by prepending) to preserve discriminant
stability if the same redb file were ever to be read by both.

### `TopicSelection` / `MatchKind` vs `TopicMatch`

Production carries TWO types: `MatchKind` (a 3-variant enum) and
`TopicSelection` (a `{ match_kind, topics }` record). spirit-next
collapses these into one `TopicMatch` enum-with-payload.

| Production shape | spirit-next shape | Notes |
|---|---|---|
| `MatchKind::Any` + `topics: []` | (no equivalent — `Query.topic_match` cannot represent "match anything") | gap: spirit-next requires choosing `Partial` or `Full` |
| `MatchKind::Partial` + `topics: [a,b]` | `TopicMatch::Partial(Topics([a,b]))` | yes |
| `MatchKind::Full` + `topics: [a,b]` | `TopicMatch::Full(Topics([a,b]))` | yes |

`signal-persona-spirit/src/lib.rs:381-475` vs
`spirit-next/src/schema/lib.rs:237-241`. The encoded NOTA differs:
production says `(Partial [a b])` as an enum head with a vector arg;
spirit-next emits `(Partial (Topics [a b]))` because `Topics` is a wrapping
newtype — but `NotaTransparent` would collapse it. Need to check
`nota-next` semantics for newtype handling versus production's
`NotaTransparent`.

### `ObservationMode` / `Mode`

Production: `SummaryOnly`, `WithProvenance` (`signal-persona-spirit/src/lib.rs:317-321`).
The `NotaDecode` impl at lines 332-343 also accepts legacy
`DescriptionOnly` (mapped to `SummaryOnly`) as compatibility input.

provenance; there is no narrower / wider observation mode toggle.

### `RecordIdentifierSelection`

Production: `Exact(RecordIdentifier)`, `Range(RecordIdentifierRange)`

queries (`(Range (1050 1060))`) are impossible on the next side.

### `Observation` / `Subscription` / `SubscriptionToken`

Production `Subscription`, `SubscriptionToken`, `SubscriptionSnapshot`,
`SubscriptionOpened`, `SubscriptionRetracted` (`src/lib.rs:882-909`):
spirit-next has zero subscription types or events.

### `Presence`

Production `Presence` (`signal-persona-spirit/src/lib.rs:347-353`):
`Active`, `Absent`. Used by `PresenceView` and `StateObserved`.
spirit-next: no `Presence`, no `PresenceView`, no presence state
machine.

### `ValidationError` (next-side only)

spirit-next at `src/schema/lib.rs:199-203`: `EmptyTopic`,
`EmptyDescription`, `EmptyQueryTopic`. Production carries validation

codec errors at parse time rather than as a typed reply.

### `UnimplementedReason` (production-only)

`NotBuiltYet`, `IntegrationNotLanded` (`src/lib.rs:914-917`). The wire
carries this as `(RequestUnimplemented (NotBuiltYet))` for operations
declared in the schema but not implemented on the daemon side
(notably `Tap`/`Untap`). spirit-next has no equivalent.

## Compatibility analysis

### Wire-level compatibility — none

A client built against production cannot speak to a spirit-next daemon
and vice versa, for four independent reasons:

1. **Disjoint operation-root vocabularies.** Production accepts
   `State`, `Record`, `Observe`, `Watch`, `Unwatch`, `Remove`,
   `ChangeCertainty`, `Tap`, `Untap`. spirit-next accepts `Record`,
   `Observe`, `Lookup`, `Count`, `Remove`. The intersection is
   `Record`, `Observe`, `Remove`. Sending `Observe` to spirit-next with
   the production payload (`(Observe (Records (...)))`) parses as
   `Observe` with body `(Records (...))` rather than `Observe Query`
   — payload-shape mismatch, parse failure.
2. **Disjoint codec libraries.** Production uses `nota-codec` (commit
   `ee90eef` cited in `skills/spirit-cli.md:141`); spirit-next uses
   `nota-next` via the `nota-text` feature. The encoded representations
   of `Topics`, `Description`, and the bracket-string handling are
   plausibly compatible but not verified; they evolved independently.
3. **Wire-framing protocol drift.** Production's signal channel is
   wired via the `signal_channel!([schema])` macro from `signal-frame`.
   spirit-next has its own `SignalTransport` (`src/transport.rs:48-86`)
   with a 4-byte length prefix (`LENGTH_PREFIX_BYTE_COUNT = 4`,
   `transport.rs:12`). Whether the production daemon uses the same
   4-byte prefix or a different framing is not verified in this audit;
   that is sub-agent 2's territory.
4. **Disjoint sockets and slots by design.** The deployment-slot
   discipline (`skills/spirit-cli.md` §"Deployment slots") states each
   daemon owns its own socket pair and redb. `spirit-next` already
   sets `SPIRIT_NEXT_SOCKET=/tmp/spirit-next.sock` by default
   (`src/bin/spirit-next.rs:32-33`) — explicitly a different namespace
   from production's `PERSONA_SPIRIT_SOCKET`. The side-by-side model
   is already physically separating the two; the wire-level gap is
   acceptable for that reason.

### Source-level compatibility — partial

Where names match, structures often do not match:

- `Kind` enum: identical (5 variants, same order). Wire-level
  encoded compatibility likely (same `NotaEnum` PascalCase tokens).
- `Magnitude` 0..=6 prefix: identical order, identical discriminants —
  but the production `Zero = 7` rung is absent from spirit-next, and
  spirit-next does not use `#[repr(u8)]` (relies on default Rust
  discriminant assignment in the generated code at `src/schema/lib.rs:275-283`).
- `Topic`, `Description`, `RecordIdentifier`: same conceptual newtypes
  over the same inner types, but production wraps them with
  `NotaTransparent` (encoded as bare tokens or bracket strings)
  while spirit-next has `pub struct Foo(pub String)` with the
  `nota-next` encoder. Encoding likely compatible at the bracket-string
  level but not verified.
- `Entry`: same four fields in the same order
  (`Topics`, `Kind`, `Description`, `Magnitude/Certainty`). The
  rkyv-archived form is plausibly compatible — the rkyv discriminant
  stability depends only on field order and leaf-type discriminants;
  the leaf types match through index 6 of `Magnitude`. **If a record
  was written by production with `Certainty = Zero`, spirit-next
  cannot decode it** (unknown discriminant 7).

### Source-level INCOMPATIBLE (drift)

- `RecordsObserved` reply payload: production returns
  `Vec<RecordSummary>` where `RecordSummary = { identifier, topics,
  kind, description, certainty }` (5 fields). spirit-next returns
  `ObservedRecords { record_set: RecordSet, database_marker }` where
  `RecordSet = Vec<Entry>` and `Entry` has NO `identifier` field
  (4 fields). A spirit-next `Observe` reply identifies records by
  position in the vector, not by id. **Records returned by spirit-next's
  `Observe` cannot be referenced by `Lookup` from the same reply
  payload — the identifier is gone.** That is a substantive design
  divergence, not a rename.
- `RecordAccepted`: production = bare `RecordIdentifier`,
  spirit-next = `SemaReceipt { record_identifier, database_marker }`.

## Recommendations — minimum to bring spirit-next to parity for daily use

The user accepts incremental progress and side-by-side deployment.
Daily-use parity for the unsuffixed `spirit` command means: capturing
new intent and observing recent records. Per `skills/intent-log.md`
the captured record needs at minimum a topic vector, kind, description,
and magnitude. Per `skills/spirit-cli.md` the daily-use observations
are the `(Partial [topic] None Any Recent SummaryOnly)` shape and
the `(RecordIdentifiers ((Range (N M)) SummaryOnly))` shape.

The minimum delta that lets spirit-next replace the unsuffixed `spirit`
slot is:

### Required for capture parity

1. **Add `Zero` to `Magnitude`.** Append as the eighth variant in
   `schema/lib.schema:46` so the 0..=6 discriminants stay stable.
   Cited evidence: `signal-sema/src/magnitude.rs:1-15`. Spirit 1249.
2. **Add a `State` operation root** taking a `StatementText` (= bare
   `String`). Cite `signal-persona-spirit/src/lib.rs:355-358` and
   `spirit.schema:8`. Even a minimal implementation that classifies
   to the sema engine as an `Assert` observation suffices for daily use.
3. **Stamp records with `RecordedTime`.** Add `Date`, `Time`,
   `RecordedTime` to the schema. The daemon stamps date/time itself
   (`skills/spirit-cli.md:148-149`); no client-supplied timestamp.
   Persist these alongside `Entry` in the redb table.

### Required for observation parity (daily-use subset)

4. **Add `RecordIdentifier` to the `Entry` returned by `Observe`** — or
   introduce a `RecordSummary` parallel to production's, with the
   identifier in field 0. Without this the daily-use observation
   replies are unusable as input for follow-up `Lookup` / `Remove` /
   `ChangeCertainty` operations.
5. **Add `RecordedTimeSelection`** with at minimum the variants
   `Any`, `Recent`, `Shallow`, `Deep`, `VeryDeep`, `Since`, `Until`,
   `Between`. The qualitative depths are the dominant agent-side
   observation idiom (`skills/spirit-cli.md:201-207`).
6. **Add a topic-selection `Any` mode** — either as a third
   `TopicMatch` variant (`Any` carrying no topics) or by switching
   `Query.topic_match` to `Option<TopicMatch>` where `None` means
   "any topic". Daily-use observations frequently want "all records
   in the last day across every topic"; today's spirit-next requires
   naming a topic.
7. **Add `CertaintySelection`** with `Any`, `Exact`, `AtMost`,
   `AtLeast`. The removal-candidate review band depends on
   `(Exact Zero)`.
8. **Add `ObservationMode`** with `SummaryOnly` and `WithProvenance`
   variants. The `WithProvenance` mode is what surfaces the
   daemon-stamped date/time to agents.

### Required for record-maintenance parity

9. **Add `ChangeCertainty (CertaintyChange)` operation root** and
   `CertaintyChanged` reply. Cited: `signal-persona-spirit/src/lib.rs:373-376`
   and `:783-786`. Without it the `Zero`-nomination workflow
   (`skills/spirit-cli.md:179-186`) does not work.

### Deferrable (not blocking daily use)

10. `Watch` / `Unwatch` + subscription substrate — the CLI is
    single-call (`skills/spirit-cli.md:243-247`) so subscriptions
    are agent-library territory. Defer.
11. `Tap` / `Untap` — placeholder no-op in production. Defer.
12. `TopicsObserved`, `QuestionsObserved`, `StateObserved` /
    `PresenceView` — useful but not blocking for intent capture.
13. `RequestUnimplemented` — a typed envelope for "I see this root
    but haven't built it"; can be subsumed under the existing
    `Error (ErrorReport)` channel as a transitional measure.

### One unique-to-spirit-next decision to make

`Lookup` and `Count` are extras. `Lookup` cleanly maps onto
production's `(Observe (RecordIdentifiers ((Exact N) Mode)))` so the
two stacks can converge without removing it. `Count` is new — there
is no production equivalent — keep it.

The eight items 1-9 above are sufficient and necessary for spirit-next
to take over the daily-use surface; the deferred items can land
incrementally inside the side-by-side slot without affecting the
unsuffixed `spirit` cutover criterion.

## Parity verdict (one sentence)

spirit-next as of commit `7c350679` is a structurally disjoint daemon
sharing only the `Record`, `Observe`, and `Remove` operation root NAMES
with production v0.3.0 (deployed commit `df09280a` + signal contract
`4c7b51ff`); the payloads, replies, selector enums, codec library, and
the `Magnitude::Zero` rung all differ, so wire-level compatibility is
zero and daily-use parity needs nine specific schema additions plus a
record-identifier surfacing fix before the unsuffixed `spirit` slot
could be cut over without losing capture/observation function.

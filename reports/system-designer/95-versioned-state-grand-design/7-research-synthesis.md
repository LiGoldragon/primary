# 7 — Research synthesis: design recommendations (adopt / reject / clever)

# 95 — Versioned-state grand design: synthesis and design recommendations (corrected)

This is the synthesis layer. Six web-verified research chapters extend report 92's general survey into the new dimensions the psyche escalated: full DVCS semantics over the typed database, per-component intake/merge/rebase policy, verifiable BLS-signed history, zero-downtime migration, and one consistent blake3 + criome-BLS crypto basis. This report ties them into a single coherent design and states structural recommendations precisely enough for the orchestrator to draw. It does not redraw; it specifies what to draw. This revision applies twelve corrections (eleven point-fixes plus six completeness gaps) verified against the live criome schema and current upstream specs.

The one-sentence thesis, sharpened from report 92's "version the log, not the store": **because our typed event log already carries each change's diff, branch / fork / rebase / disjoint-key-merge collapse to one primitive — replay a log suffix onto a target head through a per-component policy function — and the only structure that ever needs to exist beyond the append-only hash-linked DAG is (i) a transient per-key overlap index built during a same-key merge and discarded after, and (ii) an optional Merkle structure over the *checkpoint state* (never the log) for diffing two distant branch states whose shared log was pruned.** Everything else in the design is naming — except the unification's one genuine exception: rebase and merge are *both* instances of the shared replay-fold primitive but, in our order-dependent re-digesting model, they are *distinct operations with distinct results*, not the single operation Pijul's commutativity makes them.

## A. The branch / fork / merge / rebase model, defined precisely

The six chapters converge, independently and from different donor lineages (eg-walker "replay events between the LCA and discard the scaffold", jj "the operation log is authoritative and conflicts are first-class", Datomic "a branch is a movable cursor, `d/with` appends a private suffix", TerminusDB "rebase = rerun commits on the current head"), on one vocabulary. Pijul contributes the *conflict-as-a-state* principle and the observation that rebase and merge share a substrate — but **not** its "rebase == merge" identity, which is a theorem of commutativity we deliberately forgo (Section "Positioning" and correction below). Fix the vocabulary, because the donors use these words inconsistently.

The substrate: a per-component **hash-linked DAG** of typed `ReplayEnvelope` entries, each carrying `(op, family/schema-hash, record-key, payload, prev-digest, sequence)`. Report 92 had a hash-linked *chain*; the only structural change branching forces is promoting it to a **DAG** so a merge entry can name two parents. This is minimal and cheap.

- **State** = the materialized redb view obtained by folding (`Reducer::apply`) a prefix of the log. The store stays a discardable cache exactly as report 92 established; DVCS does not change that.
- **Branch** = a named movable cursor `(name → head-digest)` into the DAG. Not a copy of anything. `main` and `feature` are two pointers; entries are shared by content address up to divergence. Creating a branch is O(1). Advancing a branch cursor is **not** unconditional: it is a compare-and-swap against the expected prior head (`(name, expected_old_digest → new_digest)`), git's ref-update CAS — see the concurrency gap below — so concurrent lane writes cannot lost-update the head.
- **Head representation** = a **frontier**: the minimal set of DAG heads identifying a version. We take Loro's frontier *concept* (Loro itself identifies a frontier by `OpId` = peer-id + counter) and instantiate it with our **blake3 head digests**, since our heads are content-addressed. A single-parent head is a one-element frontier; a just-merged head is a one-element frontier over the merge entry; the multi-element case appears only mid-construction.
- **Fork** = a branch that diverges by appending a private suffix the origin never sees. Mechanically identical to branch at creation; the word marks intent (independent evolution, possibly a different policy). Structural sharing is free because the log is append-only and content-addressed.
- **Merge** = produce a new head whose state reflects *both* divergent suffixes since the common ancestor (the merge base / LCA in the segment DAG). The merge entry has **two parents** recorded *in a fixed canonical order* `(parent_into, parent_from)`; **the existing suffixes are not re-digested**. The result is the deterministic fold `base → S_into → policy(S_from)`. Because intake is order-dependent, the canonical `(into, from)` ordering is **load-bearing for convergence**: two honest peers merging the same two heads must pick the same order or diverge. The merge entry's blake3 covers `(parent_into_digest, parent_from_digest, the ordered resolution record, the resulting state-digest)`.
- **Rebase** = re-apply one branch's suffix `S_from` onto a *different* base, producing a *new linear suffix with new digests* (each entry's prev-digest changes, so its blake3 changes — the git SHA cascade). History is rewritten; the DAG stays single-parent for that segment.

**The two operational primitives underneath all of it:** (1) find the common ancestor in the segment DAG; (2) replay a suffix of typed diffs through a per-entry policy onto a target head. Everything else is which suffix, which target, and whether you keep both parents (merge) or rewrite (rebase). Merge and rebase are the *same fold* over *different (range, base)* with *different identity-handling* — they are not the same *operation* and do not produce the same result.

### The guardian-style intake policy, made precise

This is the single cleverest fit in the design and the orchestrator should foreground it. Define one trait family — a **default implementation plus a per-component override** — invoked at *exactly one place*, the replay-of-suffix loop:

- `IntakePolicy::admit(base_state, entry) -> Admit | Reject(reason) | Transform(entry')` — called per entry during rebase and during the from-suffix leg of merge. Default impl: `Admit`. Spirit's override: route through the **guardian**, which is *already* how Spirit admits/rejects/transforms a live write. **The guardian is not a new merge algorithm; it is the existing write-admission function, now also invoked on replayed entries.** A rebased entry is therefore admitted on identical terms to an originally-authored one — rebase-through-guardian is free reuse, not new machinery.
- `ConflictPolicy::resolve(base_value, into_value, from_value) -> Resolved | Reject | KeepConflict` — called *only* when the transient overlap index reports both suffixes wrote the same record-key. This is Irmin's typed `merge ~old t1 t2` functor signature exactly, bound per component instead of per content-type. Default impl: a declared rule (last-writer-by-sequence, or hard-fail-as-typed-conflict).

**The two legs are independent.** Same-key overlap triggers `ConflictPolicy`; the guardian (`IntakePolicy::admit`) runs on **every** from-entry regardless of key disjointness. This matters for the correctness claim below.

The hard invariant on both, forced by Datomic's documented failure (you cannot branch from the past because `d/with` computes retractions against the *unfiltered live* db): **every intake/merge/rebase decision must be a pure function of the base state being rebuilt and the entry — never of the live head.** Violate this and forking from an old checkpoint silently corrupts. This is a correctness rule, not a guideline, and must be a tested property (fork from a pruned-past checkpoint, append, assert no live-head leakage).

The signature is the Datomic transaction-function shape (`[db-before, entry] → verdict`), the dispatch rides the same closed-sum schema-hash selector that already chooses the decoder (so policy dispatch costs nothing new), and the trait is sealed (report 92's knife-edge: closed-sum, no open `Box<dyn>` fallback).

## B. Where structural trees (prolly / MST) do and do NOT help

The load-bearing distinction the whole design turns on, and which most of the industry conflates: **the LOG is not the STATE, and they want different structures.**

| | The LOG (event log) | The materialized STATE (checkpoint / view) |
|---|---|---|
| "Diff between two of these" | The **suffix** — already present, no computation | Must be *computed* by comparing two large values |
| Branch | Shares a prefix for free; a branch *is* fork-point + suffix | A whole second value; naive diff is O(state) |
| Right structure | A hash-linked DAG (what we already have) | A Merkle Search Tree **only when states are large and distant** |

**Within one branch's history the log wins outright and needs no tree.** A prolly tree's celebrated property — compare two trees, get O(differences) — is something the log gets by construction: the difference between revision N and N+k *is literally entries N+1..N+k*. There is nothing to diff; you already stored the diff. This is why report 92's rejection of prolly trees *for the log* was correct and **remains correct permanently** — and the branching requirement does *not* revive them.

A structural tree earns its cost in **exactly one place**: diffing or transferring two *distant materialized checkpoint states* whose shared log has been pruned by compaction (the cross-branch, cross-peer case merge introduces). And there, the recommendation is specific: use a **Merkle Search Tree (MST), not a Dolt prolly tree** — but for the honest reasons, not inflated rhetoric. Report 92's objection ("no mutable sorted keyspace to re-chunk") was specific to the **log** and applies to *neither* prolly nor MST on the keyed **checkpoint state**, which genuinely *does* have a sorted mutable keyspace. So MST does not "survive an objection prolly failed" — the objection never applied to the checkpoint use for either structure. Among structures for the keyed checkpoint, prefer MST for the concrete properties it earns: order-preserving keys (Auvolat/Taïani, SRDS 2019 — better anti-entropy locality than Dynamo-style key-hashing), a simpler hash-determined depth (leading-zero-bits of `blake3(key)` / fanout — note this still has the same boundary-shift-on-collision *class* as content-defined chunking, just at key granularity), native page-range anti-entropy, and a clean single-record inclusion proof that is the natural home for BLS attestation. MST is hash-agnostic, so blake3 swaps directly into the depth function.

**The strongest possible witness for keeping the tree off the diff path:** ATProto — the most MST-native system in production. In **Sync v1.1 (proposal 0006, developed 2024; relay rollout and the operative blog post May 2025)** it moved repo-sync **diffing/catch-up** from MST-tree-diff to a **CAR stream ordered by causal write order**, while **retaining per-commit MST proof blocks in firehose `#commit` messages** for inductive verification. So MST did not *leave* — what moved to write-causal order is the *diff* computation; the tree stays for *proofs*. A Merkle-native system kept the tree only for proofs and used the write-ordered log for the diff. That is precisely our thesis, validated from the strongest witness — but only stated accurately.

Two further sharp boundaries the orchestrator should mark:

- **Structural sharing collapses across a schema migration.** rkyv re-encodes every record on a `SchemaTransition`, re-chunking the whole tree; history-independence holds only *within* one schema version. So an MST checkpoint helps intra-schema distant-state diff and *not at all* across a migration — exactly the boundary the typed `SchemaTransition` already draws.
- **HAMT is over-engineering today.** Useful only to shard a single very-large family for partial fetch; our component sizes (Spirit's intent store is small) don't need it. File it as "available if a family gets huge," not a design element.

## C. Conflict-free vs policy-mediated merge, per family

The design consciously picks *per `RecordFamily`* between two incompatible philosophies of automated merge, because both are just different replay functions over the same DAG:

- **Convergence-by-construction (CRDT / patch theory).** Concurrent ops always commute to one deterministic state; there is never a "merge failed," but you cannot *reject*. The semantic conflict is resolved by a fixed rule chosen at type-design time.
- **Convergence-by-decision (3-way / guardian).** Detect divergence, surface a conflict set, run a policy that can *admit / reject / transform*. This is the only model that can reject — which Spirit's guardian demands.

The clever move: a typed event log lets us choose which applies per family. The honest per-shape verdict:

- **Disjoint-key writes** (our overwhelming common case): merge is **free and conflict-free only under the default `Admit` policy** (or any policy whose verdict is independent of cross-key aggregate state). This is a scoped claim, and the scope is the correction. Replaying two suffixes that touch different record-keys is order-independent *for a null policy* — no three-way comparison, no Merkle diff. **But under a stateful guardian it is not free:** admitting from-entry `E` against base-state-after-`S_into` can yield a different verdict than admitting `E` against the bare base, even for a disjoint key (a guardian can reject on aggregate state, referential checks, counts, invariants — exactly Spirit's case). So for a stateful guardian the from-suffix must still be replayed entry-by-entry through `admit()` against the *evolving merged base*. "No same-key collision" eliminates the `ConflictPolicy` leg, **not** the `IntakePolicy` leg. The structural payoff stands at the data layer: we never *compute* a diff — the suffix is the diff, already typed, already in hand. **Dolt's 2025 fast-merge must still walk the prolly trees to *recover* node-level patches before merging** (verified: dolthub.com 2025-07-16; post-2025 that recovery scales with the change, not the table — so the contrast is "they walk to recover; we stored them," not "their merge is table-sized"). We skip the recovery step entirely.
- **Concurrent same-key writes** (the *only* real same-key conflict): needs a **transient per-key overlap index** built from the two suffixes and **discarded after the merge** — eg-walker's "temporary CRDT built during merge, thrown away after," made trivial for us because our keys are explicit record identifiers, so the index is a plain `key → (branch, last-writer-entry)` hash map, not a sequence CRDT with positional tombstones. The index carries all three sides (base/into/from, Dolt's conflict-storage idea) so `ConflictPolicy` can make a real three-way decision, not just last-writer-wins.
- **Sequence-shaped families** that want convergent merge need a 2023-era interleaving discipline (Fugue's maximal-non-interleaving); naive RGA produces interleaving garbage. Only relevant if a family is genuinely sequence-shaped.
- **Tree/move families** converge via Kleppmann's highly-available move (undo-newer / apply-old / redo, with a per-move cycle-check predicate) — which is *itself* event-log replay with a per-op validity predicate, i.e. the guardian generalized. This is the proof-of-concept that "replay with a per-family predicate converges."
- **Invariant-bearing families** (`balance >= 0`, unique id, valid reference) **cannot be made conflict-free by construction** — CRDTs guarantee convergence to the *same* state, not that the state satisfies an invariant. These families *must* surface a guardian decision; trying to auto-merge them is the cargo-cult to reject.

**Conflict-as-a-state** (Pijul's pushout-completion insight, rediscovered by 2024-2026 CRDTs and realized in jj's first-class conflicts): when a convergent family genuinely cannot pick a winner, the default policy materializes a **typed conflict record** (a family variant) rather than crashing, and a later guardian/human entry resolves it as just another logged event. This also means **a merge/rebase never blocks at the log level** — it always succeeds, emitting a typed `Conflict` / `Rejected` / `Divergence` entry the policy resolves out-of-band. That is the natural home for a guardian rejection during rebase. See the rebase-continuation gap below for what "never blocks" means mid-suffix.

## D. Verifiable + BLS-signed history

Our hash-linked log is tamper-evident but not yet efficiently *verifiable to an untrusted peer*. The transparency-log world solved this; the upgrade is small and largely standard — but be precise about what criome already provides versus what this design must *build*, because the synthesis previously overstated reuse.

**What criome provides today (verified against `signal-criome/src/schema/lib.rs`):** the signature-scheme *tag* enum `SignatureScheme::{Bls12_381MinPk, Bls12_381MinSig}` (lines 98-100); the per-component authorization-policy axis `AuthorizationPolicyClass::{SimpleSelfSigned, ComplexQuorum}` (215-217); the threshold newtype `RequiredSignatureThreshold(Integer)` (84); the cosignature-outcome state machine `SignatureAuthorizationResult::{SingleSignature, RequiredSignaturesSatisfied, PendingSignatures, ...}` (195-197); and the solicit verbs `SignatureSolicitation` / `SignatureSolicitationRoute` (445/458). There is **no** `SignatureAuthorizationMode` type — the synthesis invented that name; the real per-component axis is `AuthorizationPolicyClass`.

**What is NOT in criome today (verified by source scan — grep for aggregate/proof-of-possession/PopVerify/FastAggregate returns nothing):** any aggregate-verify, any proof-of-possession, any same-message cosignature aggregation. The aggregation leg is **aspirational, a build item, not extant reuse.** The previous "where criome's BLS-over-Ed25519 bet pays off" phrasing was doubly wrong: "BLS-over-Ed25519" is incoherent (BLS12-381 and Ed25519 are different curves/schemes; criome carries a `SignatureScheme` tag, not "BLS over Ed25519"), and it presented unbuilt machinery as paid-off.

Recommended layering:

- **Keep the blake3 hash-chain on every append** (cheap local sequential integrity, free in the redb txn — report 92's most valuable property, untouched).
- **Maintain a Crosby-Wallach / RFC 6962 Merkle history tree over the same entries** (leaf = entry blake3 digest, leaf index = `CommitSequence`). Complementary, not redundant: the chain gives cheap *local* sequential integrity; the tree gives cheap *remote* O(log n) inclusion and consistency proofs, converting ouranos from a trusted blob store into an *untrusted prover*. Report 92's "digest beside the cursor, not as the cursor" rule survives intact.
- **Sign the tree HEAD, not every entry.** One BLS checkpoint signature attests an unbounded prefix because inclusion proofs chain each entry to the signed root. The latency-sensitive daemon write path stays signature-free; BLS cost lands only at checkpoint cadence. The checkpoint is a C2SP-shaped signed note (origin, tree size = `CommitSequence`, root hash = blake3 Merkle root); our richer fields (schema-hash inventory, pruned-head policy) embed the note rather than cram into it.
- **Per-component attestation policy** maps onto criome's *real* contract: `AuthorizationPolicyClass::{SimpleSelfSigned, ComplexQuorum}` for the policy axis, `RequiredSignatureThreshold(Integer)` for the threshold, `SignatureAuthorizationResult` for the cosignature outcome state machine, `SignatureSolicitation`/`SignatureSolicitationRoute` for the solicit verb. Default is `SimpleSelfSigned` (single trusted writer); `ComplexQuorum` is opt-in for components/branches that admit untrusted contributors. **This is the per-component policy at a *second, orthogonal* layer**: the guardian is a *content-admission* policy on intake/rebase; the BLS attestation threshold is a *history-integrity* policy on checkpoints. Two distinct hooks on the same log.
- **Same-message BLS aggregation is the clever fit — and a build item.** When N witnesses cosign the *same* checkpoint root, `FastAggregateVerify` collapses them to one signature verified in ~2 pairings regardless of N. **Sizes, stated correctly for the chosen variant:** with `Bls12_381MinPk` the aggregate checkpoint **signature is 48 bytes** (signatures on G1) and **public keys are 96 bytes** (on G2). If a 96-byte signature was actually wanted (e.g. to keep witness-roster public keys at 48 bytes), that is `Bls12_381MinSig`, not MinPk — pick one and state pk-size and sig-size together; the prior "MinPk … 96-byte signature" conflated the two. The MinPk/MinSig axis is independent of the proof-of-possession choice. **Mandatory must-build:** same-message aggregation is insecure against the rogue-key attack unless defended — `FastAggregateVerify` + `PopVerify` on every cosigner key per draft-irtf-cfrg-bls-signature. This whole aggregation-plus-PoP leg is **new machinery this design adds**, not reuse.
- **ouranos serves tiles + the signed checkpoint statically** (dumb, CDN/locally-cacheable, no engine — honouring report 92's "ingest is a dumb append endpoint"). The client computes proofs. But tiles are tuned for random-access proof-serving to many clients; our dominant case is one consumer re-syncing a contiguous suffix, where a raw byte-suffix copy is cheaper. **Adopt tiles for the proof / per-record / peer-verification surface; keep the cheap byte-suffix for bulk catch-up.**

## E. Zero-downtime migration as a branch

Classic online-migration tooling (gh-ost shadow tables, pgroll dual-write triggers, Stripe four-phase backfills) exists almost entirely to migrate *a mutable store that is the source of truth and cannot be torn down*. We do not have that constraint — our store is a derived view. So the correct primitive is simpler and is a branch:

**`SchemaTransition`-as-typed-log-entry → side-branch replay → equivalence-verified checkpoint → atomic head-swap.**

1. The `SchemaTransition` is itself a typed, hash-linked, BLS-signed log entry carrying `(v_old_hash → v_new_hash, reducer)`; the reducer *is* the backfill transform, expressed as data attached to the migration (pgroll's `up`/`down` made first-class).
2. **Replay onto a side branch**: rebuild the redb view by folding the existing log under the new decoder/reducer. No shadow table, no dual-write triggers — the store is regenerated, so there is nothing to dual-write.
3. **Verify state-equivalence at a post-migration checkpoint** — Stripe's Scientist discipline *inverted*: a *total offline* comparison of old-view vs new-view checkpoints (both content-addressed, so equivalence can be a hash relation over shared subtrees), BLS-attested.
4. **Atomic head-swap** — import gh-ost's all-or-nothing-or-revert cut-over *invariant* (drop the MySQL mechanics): flip the authoritative head old→new in one step; on any failure the old head stays authoritative. Daemons self-resume from persisted SEMA against the current head.

The agent-era shift that makes this the *practical* path: "rewrite the reducer and replay all history on a branch" was historically clean-but-impractical because authoring and verifying a correct deterministic reducer over all history was expensive. Agents now author and verify deterministic reducers fast, and determinism (state is a fold over the log) is the enabling contract — verify it at the boundary.

**The dual-decode window** (pgroll expand/contract + TerminusDB replayable-migration) makes *cross-schema merge* possible and is mandatory given rkyv's brittleness: during a transition both schema-hashes are decodable at once (the closed enum carries old+new sealed families), readers select by schema-hash, and "contract" drops the old family once no entry or branch references it. If two branches diverged across a schema change, the merge policy decodes each by its own schema-hash, projects the older through the `SchemaTransition` reducer so the branches become comparable, then merges on the common schema.

**The one genuinely expensive operation, bounded.** When the shared log was *pruned* and only checkpoints remain, the merge base *is* a checkpoint state, and projecting one whole checkpoint through a `SchemaTransition` reducer is an **O(state) full re-fold**, not O(suffix) — re-introducing the very full-state cost the log was meant to avoid, and it must itself be deterministic and attested. This is the design's single expensive path. Bound it: the live-branch-base retention rule below must be **schema-aware** — forbid pruning below any live branch's divergence point *even across a schema boundary* — so cross-schema-across-pruned-history merge cannot silently demand an unbounded re-fold of a discarded state.

**The unification to highlight:** migration, rebase, and merge-intake are **one operation — a policy-parameterized fold over a log range producing a new branch** — differing only in source range and policy function. (This is the *true* unification; it is not the Pijul rebase==merge *identity*, which we forgo.)

## F. The consistent blake3 + BLS crypto layer

One basis, no per-component divergence, honouring constraint `x0ja`:

- **blake3 for ALL content addressing**: every log-entry digest, every prev-digest DAG link, Merkle history-tree leaves and nodes, checkpoint root hash, MST node hashes (swap blake3 into the hash-determined-depth function; MST is hash-agnostic). No third hash.
- **criome BLS for ALL signing and attesting**: checkpoint heads (signed tree heads), witness cosignatures, equivalence attestations on a migration, branch-head attestations. BLS aggregation does double duty — compress a chain of head-attestations into one signature, and collapse an N-witness quorum to constant size/verify-cost (the aggregation+PoP machinery being a build item per Section D).
- **One basis, double duty:** the blake3 hash chain proves *integrity*; the same chain under one BLS-signed checkpoint proves *authority* and authorizes trusted-server ingest.
- **The forward-looking seam, stated honestly.** Keep the checkpoint note algorithm-agnostic — verified that the C2SP signed-note format takes "one or more signature lines" and "Verifiers MUST ignore signatures from unknown keys" — so a post-quantum cosignature can be *added beside* BLS later without reformatting. **But acknowledge the spec has moved:** the current C2SP tlog-checkpoint spec says logs **SHOULD use ML-DSA-44** to sign the checkpoint (MAY use any algorithm). So we are **diverging from C2SP's recommended algorithm**, and the honest reason is precise: BLS aggregation/threshold is the load-bearing property C2SP's per-signature model does *not* give us; ML-DSA-44 would fork the basis and lose aggregation. We reuse C2SP's *algorithm-agnostic signed-note envelope*, not its recommended algorithm. BLS is the *aggregation-aligned* choice, not the *standard-aligned* choice — and BLS is not post-quantum (~117-120-bit classical), which is acceptable only for the trusted-tailnet threat model. Do **not** adopt ML-DSA-44 now (low urgency for that threat model; it forks the basis), but build the envelope so it can be added beside BLS without reformatting.

## G. Six completeness items the design must close

These are gaps the prior synthesis left silent; each needs a named mechanism, not a wave.

1. **Garbage collection / reachability.** The DAG-plus-movable-cursors model never said what keeps an entry alive or reclaims orphaned suffixes after a branch delete or an abandoned rebase. State it: **branch refs (frontiers) are GC roots; reclamation is mark-from-frontiers** (Dolt's newgen/oldgen generational GC — walk reachable-from-refs, pack survivors, flip manifest — is the reference). "Live-branch-base retention" is a *compaction floor* (when not to prune), not a GC story. Decide explicitly whether ouranos (a dumb append endpoint) GCs at all or grows unbounded; default: ouranos is append-only and never GCs, the local store GCs against its own frontiers.
2. **Rebase continuation semantics.** Merge handles same-key conflict (overlap index + `ConflictPolicy`); rebase re-applies `S_from` onto a new base where individual entries may now conflict with entries already there. "Always succeeds, emit typed entries" must specify the *continuation* policy. Adopt jj's model: a mid-suffix `Rejected`/`Conflict` entry **records and continues** — it does *not* halt the rebase. For an order-dependent guardian this is consequential: the recorded conflict/rejection becomes part of the base the subsequent entries see, so continuation is deterministic only if the recorded entry is canonical.
3. **Branch-cursor concurrency control.** With many lanes sharing stores, two agents advancing the same branch head is a lost-update unless cursor advance is a **compare-and-swap** `(name, expected_old → new)`. redb single-writer-per-file gives this locally; cross-peer (ouranos advancing a head it also serves) and multi-branch-per-file updates need the explicit CAS rule. State it as the cursor contract, not an O(1) free advance.
4. **Bitemporality, placed not deferred.** Valid-time vs system-time is orthogonal to branching — but Spirit is an intent log with explicit lifecycle (supersession, retirement, forgetting). Close the gap in one sentence so implementers don't reach for a branch to model belief-revision: **valid-time lives as a payload field inside the typed record, never as a branch.** A fork that revises history and a record that asserts a different belief-time are different things; the latter is a normal logged entry.
5. **Cross-schema-pruned merge cost, bounded** — covered in Section E item "one genuinely expensive operation": schema-aware retention floor; it is the single O(state) path and must be attested.
6. **Meta-evolution of the VC format itself.** `SchemaTransition` is the migration primitive for component *record* families — but the VC layer's *own* rkyv types (`ReplayEnvelope`, `Checkpoint`, the merge-entry shape, the frontier representation, the BLS checkpoint note) will also evolve. A change to `ReplayEnvelope`'s layout is a migration of the **log format**, which cannot be expressed as a `SchemaTransition` entry *inside that same log* (chicken-and-egg). This is report 92's two-guards distinction (format-identity guard vs schema-version guard) that the grand design collapsed to one. Name it: **the envelope/checkpoint format carries its own format-version tag and a format-level migration path, distinct from per-family `SchemaTransition`.** The format-level migration reads the old envelope format and writes a fresh log in the new envelope format — it is a whole-log rewrite, the rare event, gated and attested.

## Positioning against the state of the art

The novel thing — undelivered by any single donor — is a **compile-time-typed, schema-versioned record DVCS with a pluggable per-component merge/intake/rebase policy over a fragile zero-copy format, on one blake3 + BLS crypto basis.** Every donor gives up exactly one leg: CRDTs and Pijul and git-blobs give up types; Datomic and XTDB give up branching; CRDTs bake in one fixed resolution (give up pluggable policy); text DVCSs give up the typed-DB shape. eg-walker independently arrived at our exact log-is-authoritative / view-is-discardable / merge-state-is-transient architecture and is EuroSys-2025 recognized — our typed-closed-sum + per-component-policy variant is the contribution nobody ships. The assembly is ours.

**The one framing we do NOT import: Pijul's "rebase == merge" identity.** That identity is a *theorem* (Mimram/di Giusto pushout theory) that holds *because* patches commute and a change's identity is order-independent. Our design simultaneously (a) builds an **order-dependent** log where (b) rebase **re-digests** the suffix (new prev-digest → new blake3) while merge keeps both parents and does **not** re-digest. Under (a)+(b), rebase and merge are demonstrably **different operations with different results and different identities** — the opposite of Pijul's theorem. Taking "rebase==merge framing" would be importing a result whose precondition (commutativity) we explicitly reject. What we *do* take from Pijul is the *conflict-as-a-state* principle. The genuine unification is the Datomic/jj/Dolt one: both rebase and merge are instances of *replay-suffix-through-policy* — the shared *fold*, not the shared *result*.

## Clever ideas (distilled)

- Rebase-through-guardian: the guardian is not a new merge algorithm — it is the existing write-admission function, now also invoked on replayed entries, so a rebased entry is admitted on identical terms to an authored one (the single cleverest fit).
- One primitive, several jobs: replay-suffix-through-policy is the shared FOLD under rebase, merge-intake, crash-recovery, remote-mirror-ingest validation, and schema migration — they differ only in source range and policy function. (The fold is shared; the RESULT/identity is not — rebase re-digests and is single-parent, merge keeps both parents — so this is a shared-primitive unification, NOT Pijul's rebase==merge identity, which we forgo.)
- The diff-carrying log means we never RECOVER a diff: Dolt's 2025 fast-merge must still WALK the prolly trees to recover node-level patches before merging (even though post-2025 that walk scales with the change, not the table); we stored the typed suffix, so the suffix already IS the patch set — strictly stronger than making the store diffable.
- Only one real same-key conflict needs structure, and only a TRANSIENT key->writer hash map carrying all three sides (base/into/from), built from the two suffixes and discarded after merge (eg-walker's throwaway scaffold) — trivial because our keys are explicit record identifiers, no positional tombstones. The IntakePolicy/guardian leg is independent of this and runs on every from-entry regardless of key disjointness.
- Sign the tree HEAD, not every entry: one BLS checkpoint signature attests an unbounded prefix via inclusion proofs, keeping the latency-sensitive write path signature-free. Same-message FastAggregateVerify then collapses an N-witness quorum to one Bls12_381MinPk signature of 48 bytes (signatures on G1; public keys 96 bytes on G2) verified in ~2 pairings — but this aggregation+proof-of-possession leg is NEW machinery to build, not extant criome reuse.
- ATProto — the most MST-native system in production — kept its Merkle tree only for per-commit PROOF blocks in firehose messages but in Sync v1.1 (proposal 0006, 2024; relay rollout May 2025) moved repo-sync DIFFING onto a causal-write-ordered CAR stream: a Merkle-tree-native system used the write-ordered log to carry the diff and the tree only to verify, validating our whole thesis from the strongest possible witness.
- Two orthogonal per-component policy layers on the same log: the guardian is a content-admission policy on intake/rebase (default Admit + Spirit override); the BLS attestation threshold is a history-integrity policy on checkpoints, mapped onto criome's REAL AuthorizationPolicyClass::{SimpleSelfSigned, ComplexQuorum} + RequiredSignatureThreshold(Integer) — both customizable, default + override.
- For the keyed CHECKPOINT STATE only (never the log), prefer a Merkle Search Tree over a Dolt prolly tree for the properties it actually earns — order-preserving keys with better anti-entropy locality (Auvolat/Taiani SRDS 2019), hash-determined depth, native page-range anti-entropy, clean single-record inclusion proof for BLS attestation. (Report 92's no-mutable-keyspace objection was about the LOG and applies to neither structure on the keyed state — MST does not 'survive an objection prolly failed.')
- Canonical merge ordering is load-bearing for convergence: because intake is order-dependent, the merge entry records (parent_into, parent_from) in fixed order and the result is the deterministic fold base -> S_into -> policy(S_from), so two honest peers merging the same two heads recompute the identical head — the per-merge analogue of the virtual-merge-base determinism the ort rule needs.
- We diverge from C2SP's now-recommended ML-DSA-44 checkpoint signature ON PURPOSE: BLS aggregation/threshold is the load-bearing property C2SP's per-signature model does not give us, so we reuse C2SP's algorithm-agnostic signed-note ENVELOPE (one-or-more signatures, verifiers MUST ignore unknown-key signatures) while keeping BLS for the aggregation-aligned (not standard-aligned) choice — and the envelope lets a PQ cosignature be added beside BLS later without reformatting.

## Adopt

- Promote the hash-linked log from a CHAIN to a DAG (a merge entry names two parents IN FIXED CANONICAL ORDER (parent_into, parent_from) plus a recorded resolution and resulting state-digest) — the only structural change branching forces, minimal and cheap.
- One trait pair invoked at exactly one place (the replay-of-suffix loop): IntakePolicy::admit(base_state, entry) -> Admit|Reject|Transform with default Admit and Spirit's guardian as the override; ConflictPolicy::resolve(base,into,from) (Irmin's merge ~old t1 t2 signature) called ONLY when the transient overlap index reports a same-key collision. The two legs are independent: same-key overlap triggers ConflictPolicy; the guardian/IntakePolicy runs on EVERY from-entry regardless of key disjointness.
- Hard invariant: every intake/merge/rebase decision is a pure function of the base state being rebuilt and the entry, NEVER the live head (Datomic's cannot-branch-from-the-past trap) — make it a tested property: fork from a pruned-past checkpoint, append, assert no live-head leakage.
- Disjoint-key merge is free and conflict-free ONLY under the default Admit policy (or any policy independent of cross-key aggregate state); under a stateful guardian the from-suffix must still be replayed entry-by-entry through admit() against the evolving merged base, which MAY reject/transform a disjoint-key entry. Scope the 'free merge' claim accordingly.
- Merge and rebase are DISTINCT operations sharing one fold: merge keeps both suffixes and adds a two-parent head with no re-digesting; rebase rewrites onto a new base and re-digests the suffix (the unavoidable O(|suffix|) blake3 cascade, single-parent). Do NOT claim rebase==merge (that is a Pijul commutativity theorem whose precondition we reject). Prefer merge over rebase when mirror bandwidth matters, since rebase re-ships the whole suffix.
- A merge/rebase never blocks at the log level — it always succeeds, emitting a typed Conflict/Rejected/Divergence entry the policy resolves out-of-band (jj first-class conflicts). Rebase continuation policy: a mid-suffix Rejected/Conflict entry RECORDS-AND-CONTINUES (jj model), it does not halt; for the order-dependent guardian the recorded entry becomes part of the base subsequent entries see, so it must be canonical.
- Maintain a Crosby-Wallach/RFC-6962 Merkle history tree over the same entries (leaf = entry blake3 digest, index = CommitSequence) for O(log n) remote inclusion/consistency proofs, keeping the blake3 hash-chain for free local integrity — complementary, not redundant.
- Checkpoint = C2SP signed note (origin, tree size, blake3 root) signed with Bls12_381MinPk (aggregate signature 48 bytes on G1, public keys 96 bytes on G2) — or Bls12_381MinSig if 48-byte witness keys matter more, state pk+sig sizes together. Same-message FastAggregateVerify + mandatory PopVerify proof-of-possession on every cosigner key is NEW machinery to BUILD (not extant in criome, which today carries only the SignatureScheme tag, AuthorizationPolicyClass, RequiredSignatureThreshold, SignatureAuthorizationResult, and SignatureSolicitation types). Per-component attestation policy via the real AuthorizationPolicyClass::{SimpleSelfSigned, ComplexQuorum} + RequiredSignatureThreshold(Integer).
- Zero-downtime migration = SchemaTransition-as-typed-log-entry (reducer = backfill transform) -> side-branch replay -> total offline equivalence-verified BLS-attested checkpoint -> gh-ost-style atomic all-or-nothing-or-revert head-swap; dual-decode window (both schema-hashes live) enables cross-schema merge.
- If and only if two DISTANT checkpoint states must be diffed without their shared log, use a Merkle Search Tree keyed by (family, record-key) with blake3 in the depth function — NOT a Dolt prolly tree. (Report 92's objection was about the LOG and applies to neither structure on the keyed checkpoint state.)
- Checkpoint retention floor must be SCHEMA-AWARE: refuse to compact past the oldest live branch's merge-base even across a schema boundary (or snapshot the base into the checkpoint). Cross-schema-across-pruned-history merge is the single O(state) full-re-fold operation and must itself be deterministic and BLS-attested — bound when it can occur.
- Handle criss-cross/multiple-LCA topologies with a git-merge-ort-style virtual merge base: recursively merge the ancestors through the same policy into a synthetic base before three-way-merging the heads — a correctness requirement once components branch and re-merge.
- Garbage collection: branch refs (frontiers) are GC roots; reclamation is mark-from-frontiers + pack survivors (Dolt newgen/oldgen). 'Live-branch-base retention' is a compaction FLOOR, not a GC story. Default: the local store GCs against its own frontiers; ouranos is append-only and never GCs.
- Branch-cursor advance is a compare-and-swap (name, expected_old_digest -> new_digest), not an unconditional O(1) write — redb single-writer gives it locally; cross-peer and multi-branch-per-file updates need the explicit CAS rule.
- The VC METADATA format (ReplayEnvelope, Checkpoint, merge-entry, frontier, BLS note) carries its OWN format-version tag and a format-level migration path (whole-log rewrite, gated+attested), distinct from per-family SchemaTransition — a format-identity guard separate from the schema-version guard (report 92's two-guards distinction).
- Head representation is a frontier (Loro's concept; Loro uses OpIds) instantiated with our blake3 head digests since heads are content-addressed; valid-time lives as a payload FIELD inside the typed record, never as a branch.

## Reject (cargo-cult to avoid)

- Prolly trees / content-defined chunking FOR THE LOG: the log is append-only with no random-mutable sorted keyspace to re-chunk, and its diff is its suffix — the per-write tax buys nothing on an append. Report 92's rejection holds permanently; branching does NOT revive it. (Structures help only the distant-checkpoint-diff case, and even there MST is the better variant — for the properties it earns, not because it 'survives' an objection that never applied to the keyed state.)
- Pijul's rebase==merge IDENTITY as an adopted framing: it is a commutativity THEOREM (Mimram/di Giusto pushout) requiring order-independent change identity. Our log is order-DEPENDENT and rebase re-digests while merge keeps both parents, so rebase and merge are DISTINCT operations with distinct results/identities. Take Pijul's conflict-as-a-state principle; reject the rebase==merge identity AND the pristine/conflict-graph categorical substrate.
- Git's snapshot-and-rediff merge: git stores trees and recovers changes by diffing snapshots — the entire diff-recovery step we skip because we store the diffs. Git proves the DAG+replay model is sound but is the least clever donor for our shape.
- General CRDT auto-merge as the resolver (Automerge/Yjs/Loro as a dependency): bakes ONE fixed resolution semantics into the data type, contradicting per-component-policy, and reaches reusability by erasing type (schemaless JSON) — report 92's forbidden genericity path. Implement the eg-walker IDEA natively over our typed closed-sum; Loro/diamond-types are reference to READ, not crates to depend on.
- TerminusDB triple/RDF + succinct rank-select encoding, and Irmin/CRDT type-erasure genericity: all reach reusability by erasing type. Adopt their merge/rebase mechanics (delta-layer = log entry, rollup = checkpoint, replayable migration makes branches comparable); reject the genericity substrate.
- Datomic/XTDB single-authoritative-timeline assumption: time-travel and replay but architecturally hostile to branch/fork (Datomic explicitly cannot branch the past; d/with is throwaway speculative). Branching comes from the git/Dolt/jj lineage — do not import the single-timeline assumption. Bitemporality (valid-time vs system-time) is orthogonal and lives as a payload FIELD, never a branch; don't conflate it with branching.
- Threshold / multi-party tBLS quorum (drand/Filecoin) and durability quorum (rqlite/openraft multi-target ingest): solve distributed trust among many parties; ouranos is single and trusted. Use plain BLS signing+aggregation for the checkpoint; ComplexQuorum stays OPT-IN, default SimpleSelfSigned. Witness networks are low-value in a single-psyche single-tailnet deployment.
- Shadow/ghost tables and store-level dual-write triggers (gh-ost mechanics, pt-osc, pgroll triggers, Reshape, Stripe Hadoop backfill, binlog tailing): all migrate a mutable store IN PLACE; our store is a derived view rebuilt from the log, so a parallel shape inside the store is wasted scaffolding. Import only gh-ost's atomic cut-over INVARIANT and pgroll's transform-as-migration-data IDEA.
- Branching/forking from an arbitrary PAST point with a policy/reducer that reads the live head: the Datomic corruption trap — a bare log offset into a shared mutable index is not a self-contained branch point. Branch points must be content-addressed checkpoints; forbidden unless every policy/reducer is a pure function of the base being rebuilt.
- A general structural merge engine for arbitrary conflicts (Dolt-style cell-merge engine): cargo-cult. The only real conflict is concurrent same-key write; everything else is order-independent replay (under a stateful guardian, still entry-by-entry admit, but no structural merge engine). Build the narrow transient three-sided overlap index + ConflictPolicy, not a general engine.
- HAMT as the versioned structure: not history-independent in the ordered-diff sense, so wrong for diffing two states by key range; only shards a single huge family for partial fetch, which our component sizes don't need. File as 'available if a family gets huge.'
- ML-DSA-44 post-quantum checkpoint signature NOW (despite C2SP's current SHOULD-ML-DSA-44 recommendation), and STARK proof aggregation as a BLS successor: both fork the single blake3+BLS basis and lose BLS aggregation/threshold today. We DELIBERATELY diverge from C2SP's recommended algorithm to keep aggregation, reusing its algorithm-agnostic signed-note envelope so PQ can be added BESIDE BLS later. Do not adopt now (low-urgency for the trusted-tailnet threat model).
- Sparse Merkle tree / verifiable map (CONIKS-style) for the keyed VIEW: needed only when branch/merge concretely requires authenticated view diffs or non-membership proofs; pure per-commit overhead for single-writer single-branch operation. Adopt only when the branching requirement actually lands.
- Tile-structured fetch for BULK suffix sync: tiles are tuned for random-access proof-serving to many clients; our dominant case is one consumer re-syncing a contiguous suffix, where a raw byte-suffix copy is cheaper. Use tiles only for the proof/per-record/peer-verification surface.
- The claim that disjoint-key merge is unconditionally free: reject it as stated. It is free ONLY under the default Admit policy; under Spirit's stateful guardian the IntakePolicy leg runs on every from-entry and may reject/transform a disjoint-key entry on aggregate/referential/invariant grounds. Disjointness eliminates the ConflictPolicy leg, not the IntakePolicy leg.
- The phrasing 'criome's BLS-over-Ed25519 bet' and any claim that same-message aggregation / FastAggregateVerify / proof-of-possession already exist in criome: BLS12-381 and Ed25519 are different schemes (incoherent phrasing), and source scan finds NO aggregate-verify, NO PoP, NO cosignature aggregation in signal-criome/criome — only the SignatureScheme tag and the policy/threshold/result types. Treat the entire aggregation+PoP leg as a build item, not reuse; and use the real type name AuthorizationPolicyClass, not the invented SignatureAuthorizationMode.

## Open questions

- Does resting-bytes (payload + schema-hash on disk, typed only in motion) honour the SPIRIT of perfect specificity, or is it the forbidden generic-record store wearing a typed coat? Report 92 flagged this as a genuine open call; the per-component closed-sum decode is the proposed answer but the knife-edge is real.
- Where does the branch/merge/policy layer LAND — a new crate, a sema-engine primitive, or a layer beside the kernel handle? Report 92 left home-layer undecided; the IntakePolicy/ConflictPolicy/MergePolicy noun set sharpens but does not resolve it.
- Is the default ConflictPolicy last-writer-by-sequence, or hard-fail-as-typed-conflict? Last-writer silently loses a concurrent write; typed-conflict is safe but forces resolution. The choice may itself be per-component.
- How aggressively must checkpoints be emitted to keep eg-walker's worst case (deeply interleaved branches with no critical version) out of reach, and does aggressive emission interact badly with the live-branch-base retention rule (you cannot compact past any live branch's base)?
- Should the guardian's IntakePolicy gain a learned/cached decision layer (git rerere as the guardian's decision-log -> training flywheel from report 585), and if so does that violate the pure-function-of-base-state invariant (a learned policy reads history, not just the base)?
- Cross-schema merge requires projecting the older branch through the SchemaTransition reducer to a common schema — is the reducer guaranteed invertible/lossless enough to make a three-way merge meaningful across a schema boundary, or does some migration shape make cross-schema merge undefined?
- Multiple-LCA criss-cross is rare for a single-psyche coordination store — do we implement the virtual-merge-base recursion now, or refuse multi-LCA merges explicitly (fail loudly rather than silently pick one base) until a real case appears?
- Does the equivalence verification on migration (old-view vs new-view checkpoint comparison) need a defined semantic equivalence relation beyond hash-equality, given that a migration that intentionally CHANGES state (not just re-encodes it) will never hash-match?

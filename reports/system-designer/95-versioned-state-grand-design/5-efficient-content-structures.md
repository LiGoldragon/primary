# Efficient content-addressed structures for a typed event-log VCS: where prolly trees, MSTs, HAMTs and succinct state earn their cost — and where the log already wins

## Frame: the question report 92 could not yet ask

Report 92 rejected prolly trees as Dolt-by-analogy cargo-cult, and it was right *for the design it surveyed*: an append-only log with **no merge** and a single writer. Its prior-art matrix repeatedly tagged 3-way merge as "does not apply (single-writer)" (Irmin, Datomic, the CRDT family). The new psyche requirements delete that premise. Branching, forking, rebasing and merging are now first-class, with a per-component intake/merge/rebase policy. The honest re-examination this report owes is therefore narrow and sharp: **now that merge is in scope, is a structural content-addressed tree justified — and if so, exactly where?**

The answer that the whole report defends: a structural tree (prolly tree / Merkle Search Tree) is justified in **exactly one place** in our design — diffing or transferring two *distant materialized states* (checkpoints) — and is **still unnecessary for the log itself**, because an append-only hash-linked log already gives every property a structural tree is invented to provide *within a single branch's history*. The cleverest part of our case is that the log and the checkpoint want *different* structures, and most of the industry conflates them. ATProto's 2024 sync redesign — which kept its Merkle Search Tree for *state* but moved *sync* off MST-diffing and onto a causal-ordered write log — is the live proof that even a Merkle-tree-native system reaches for the log when it wants a diff.

## The load-bearing distinction: the LOG is not the STATE

Two artifacts, two cost models. Keeping them apart is the entire analysis.

| | The LOG (event log) | The materialized STATE (checkpoint / redb view) |
|---|---|---|
| Shape | Append-only sequence of typed, hash-linked diff entries | Full current value of every record family |
| "Diff between two of these" | The **suffix** — already present, no computation | Must be *computed* by comparing two large values |
| Branch | Shares a prefix **for free**; a branch *is* a fork point + a suffix | A whole second value; naive diff is `O(state)` |
| Transfer to a peer | Ship the bytes after the peer's last head | Ship the whole thing, or diff-then-ship-delta |
| What makes it cheap | Linearity + content-addressed links (blake3) | **Structural sharing** in a content-addressed tree |
| Right structure | A hash-linked list (what we already have) | A prolly tree / MST **only when states are large and distant** |

Within one branch's history the log wins outright. A prolly tree's celebrated property — "compare two trees, get `O(differences)`" — is a property the log gets *by construction*: the difference between revision N and revision N+k **is literally the entries N+1..N+k**. There is nothing to diff; you already stored the diff. This is why report 92's rejection was correct and remains correct *for the log*. The structural tree only starts to earn its keep when you must compare or transport two states whose common ancestor is far away and whose intervening log you do **not** have — the cross-branch, cross-peer case that merge introduces.

## Prolly trees / content-defined chunking — re-examined with merge in scope

**What it is.** A B-tree whose node boundaries are set by a rolling hash over content, not fixed fanout; each node content-addressed into a block store. The defining property is **history-independence**: a given key-value set has exactly one tree shape regardless of insert order, so identical subtrees share one address and structural-equality is a single root-hash compare ([DoltHub prolly-tree docs](https://docs.dolthub.com/architecture/storage-engine/prolly-tree); [People Keep Inventing Prolly Trees](https://www.dolthub.com/blog/2025-06-03-people-keep-inventing-prolly-trees/)).

**How it does merge (the genuinely new 2024-2025 detail beyond report 92).** Dolt's "fast merge" (July 2025) merges by walking *nodes*, never rows. Phase 1 walks each branch against the common ancestor producing key-range "patches"; phase 2 merges the two patch streams, recursing into children **only where ranges overlap**; phase 3 applies patches — and in the best case "applying a patch means simply inserting the node into the merged tree, without ever needing to load or inspect the node's children." Benchmark: 5M-row branches merged in 0.01s vs 9s, a 1000x win, because cost scales with *structural differences*, not changed rows ([Leveraging Prolly Trees for Efficient Table Merging](https://www.dolthub.com/blog/2025-07-16-announcing-fast-merge/); the six-step base algorithm in [Three-way Merge in a SQL Database](https://www.dolthub.com/blog/2024-06-19-threeway-merge/)).

**Where it fits us — and where it still does not.** This is the precise dividing line:

- It does **not** help the log. The log is append-only; its diff is the suffix; there is no random-mutable sorted keyspace to re-chunk. Report 92's verdict stands unchanged here, and the write tax (`(1 + k/w)·log_k(n)` per write) buys nothing on an append.
- It **does** help one specific operation that merge newly introduces: computing the diff between two *checkpointed states* whose shared log has been pruned. If branch A and branch B were both checkpointed and the intervening log is gone (compaction is mandatory — Dolt's whole journal→oldgen GC exists for this reason), the only way to diff them in `O(differences)` rather than `O(state)` is to have stored each checkpoint as a content-addressed structural tree. **This is the single place a prolly tree earns its cost in our design.**
- The hard caveat that keeps this honest: our state is **rkyv**, and report 92 already established that for rkyv "essentially every schema change is a full-rewrite event." A prolly tree over rkyv records is *history-independent only within one schema version*; a `SchemaTransition` re-encodes every record and re-chunks the entire tree, collapsing all structural sharing across the migration. So a prolly-tree checkpoint helps **intra-schema** distant-state diff and **not at all across a migration** — exactly the boundary our typed `SchemaTransition`-as-log-entry already draws.

## Merkle Search Trees (MST) — the cleaner structural tree for our case

**What it is.** The structural tree we should actually reach for if we reach for one. An MST is a Merkle-ized search tree whose *depth for a key is deterministic from the key's hash* — count leading zero bits of the hash and divide by the fanout factor — so the tree shape is a pure function of the key set, fully history-independent, with **no rolling-hash window and no boundary-shift ripple** ([Merkle Search Trees, Auvolat & Taïani, SRDS 2019](https://github.com/domodwyer/merkle-search-tree); ATProto's instantiation below). It was designed precisely as a *state-based CRDT anti-entropy primitive*: two replicas exchange compact page-range summaries and reconcile in `O(differences)`.

**How ATProto uses it — and the 2024 lesson that is the heart of our report.** Bluesky's AT Protocol stores each repository as a signed-commit-over-an-MST: a key like `app.bsky.feed.post/9adeb165882c` lands at MST depth = ⌊leading-zero-bits(SHA-256(key)) / 2⌋ (fanout 4); the signed commit carries `did`, `version`, `data` (CID of the MST root), and `rev` (a TID acting as a **logical clock**, monotonic) ([AT Protocol Repository spec](https://atproto.com/specs/repository); [repo README](https://github.com/bluesky-social/atproto/blob/main/packages/repo/README.md)). Then the decisive move: **in v1.1 they stopped diffing MSTs to sync and switched to a CAR-file stream ordered by the causal order of writes** — the MST still exists for *state and proofs*, but it is "no longer the main mechanism for sync." Sync follows the *log of writes*, not a tree diff. That is our thesis, validated by the most MST-native system in production: **keep the tree for verifiable state, use the log for the diff.**

**Where it fits us.** MST beats prolly trees for our checkpoint because (a) hash-determined depth removes the rolling-hash boundary-ripple write tax that report 92 flagged as the reason to reject prolly trees, and (b) its native idiom is *page-range anti-entropy between two peers* — exactly the ouranos-mirror "diff my checkpoint against yours" case. If we ever Merkle-structure a checkpoint, an MST keyed by `(family, record-key)` is the better choice than a Dolt-style prolly tree, and it composes cleanly with blake3 (swap SHA-256 for blake3 in the depth function; the algorithm is hash-agnostic). The ATProto **proof chain** — a compact inclusion proof for one record against the signed root — is the mechanism that lets ouranos verify a single record without the whole tree, and is the natural home for criome BLS attestation over the root.

## HAMT — useful as a checkpoint *index*, not as the versioned structure

**What it is.** A hash-array-mapped trie: a hashmap stored as a trie keyed by successive slices of a key's hash, giving ~`O(1)` lookup with no rehashing and copy-on-write structural sharing on update. IPFS uses a HAMT over merkledag nodes to shard large UnixFS directories; fanout is tunable (IPFS uses 256) ([go-unixfs HAMT](https://github.com/ipfs/go-unixfs/blob/master/hamt/hamt.go); [Grokipedia: HAMT](https://grokipedia.com/page/Hash_array_mapped_trie)).

**Where it fits — narrowly.** A HAMT is *not* history-independent in the strong ordered-diff sense an MST is (it is a hash trie, not a sorted search tree), so it is the wrong tool for *diffing two distant states by key range*. But it is the right tool for one sub-problem: a content-addressed, structurally-shared **map** when a checkpoint must shard a very large family into addressable sub-blocks for partial fetch (fetch only the records under one hash-prefix). For our component sizes this is almost certainly **over-engineering** — Spirit's intent store is small — so HAMT is filed as "available if a family ever gets huge," not as a design element. Naming it honestly: HAMT is cargo-cult for us today.

## Succinct structures (TerminusDB) — the immutable-delta-layer model maps; the encoding does not

**What it is.** TerminusDB decomposes documents into RDF triples held in **succinct data structures** (bit-vectors with rank/select, wavelet-tree-style dictionaries) inside content-addressed, immutable **delta layers**: a base layer plus stacked layers each carrying added-subject/object/predicate dictionaries and a negative-edge set for deletions; queries run in-memory against the succinct form (~13 bytes/triple on billion-triple graphs); periodic **delta rollup** compacts the stack ([TerminusDB explanation](https://terminusdb.org/docs/terminusdb-explanation/); [Wikipedia: TerminusDB](https://en.wikipedia.org/wiki/TerminusDB); succinct-structures blog returned HTTP 522 on fetch — cited from the search abstract and the docs).

**What maps and what does not.** The *architecture* is almost exactly ours and is independent confirmation of the direction: immutable content-addressed layers (≈ our log entries / checkpoints), a stack of deltas as the source of truth, queries against a materialized form, and **rollup as compaction** (≈ our checkpoint bounding replay). The Git-like branch/merge built **on top of immutable delta layers** is a second existence proof — alongside Irmin — that branch/merge does not require a mutable store. What does **not** map: the succinct *encoding itself* (rank/select bit-vectors over a triple universe) is tuned for read-time graph traversal of RDF; our payloads are typed rkyv records under a closed-sum decode, not triples, and our read path is redb. We take the **layered-immutable-delta + rollup** idea and reject the succinct triple encoding — clean separation of concept from mechanism.

## Merkle DAGs and the commit graph — the part that is simply correct for us

The commit DAG (noms/Dolt/Irmin/Git/ATProto all share it) is the model report 92 already endorsed and the new requirements lean on hard. Branching = a second head pointer into the same content-addressed DAG; forking = the same with an independent ref namespace; the shared prefix is shared *storage* by content-addressing, not copied. blake3-linking each log entry to its parent already makes our log a Merkle chain; promoting it to a **DAG** (an entry may name two parents) is the minimal change that makes *merge* representable: a merge commit is a log entry with two parent links and a recorded resolution. This is non-negotiable and cheap, and it is where blake3 (content address of every entry and checkpoint) and criome BLS (a signature/attestation over a DAG node's hash, aggregatable across a history) sit naturally — one consistent crypto basis, no per-component divergence, mirroring the transparency-log "sign the checkpoint" discipline below.

## The new dimension report 92 never covered: per-component intake / merge / rebase POLICY

This is where the genuinely clever combination for *our* system lives, and no surveyed VCS has it as a typed, swappable hook — they hardcode one merge policy. The psyche's own example names the mechanism: Spirit has a **guardian** that admits / rejects / transforms entries, and "a rebase there routes each entry through the guardian." Three prior-art pieces give us the exact shape:

- **Rebase = event replay through a reducer (eg-walker).** Gentle & Kleppmann's eg-walker (EuroSys 2025) is "version the log" arrived at independently *with* merge: store the event graph on disk, keep only current state in memory, and to merge a divergent branch **replay its events** against the current state, retaining just the original operation descriptions (no CRDT metadata) ([Eg-walker paper](https://arxiv.org/abs/2409.14252); [Kleppmann's writeup](https://martin.kleppmann.com/2025/03/30/eg-walker-collaborative-text.html); [Loro's EG-Walker docs](https://loro.dev/docs/concepts/event_graph_walker)). Rebase in our design is *exactly* this: take a branch's log suffix and replay each typed entry onto a new base — and **the replay step is the policy hook**. Default policy = apply the entry's diff. Spirit's override = route each entry through the guardian, which returns Accept / Reject(typed reason) / (and, if we extend it) Transform. The guardian is already a binary admission gate returning a typed verdict (report 585); rebase-through-guardian is the same call, looped over a suffix.

- **The typed signature of the hook = a Datomic transaction function.** Datomic's transaction function is a pure `[db-before, args] → tx-data` — it transforms/validates a proposed change against the *current database value* before it commits ([Datomic transaction functions](https://docs.datomic.com/transactions/transaction-functions.html)). That is precisely the type our policy hook wants: `fn intake(state_before, entry) -> Verdict { Admit(entry) | Reject(reason) | Transform(entry') }`. A **default impl** (Admit) plus a **per-component override** (Spirit's guardian) is the Rust trait-with-default-method shape, owned by the component's `RecordFamily` — not a free function, not a ZST namespace.

- **rerere = cached merge-policy decisions.** Git's reuse-recorded-resolution records how a conflict was resolved and replays it when the same conflict recurs during a rebase ([git-rerere](https://git-scm.com/docs/git-rerere)). For us this is the guardian's **decision-log → training flywheel** (report 585 §3) generalized: a component's policy may *learn* from past Admit/Reject/Transform decisions and replay them deterministically. That closes the loop between "policy hook" and "the guardian gets better over time."

Pijul's patch theory is the road **not** taken here, and naming why is the anti-cargo-cult: Pijul makes patches commute so merge is a categorical pushout with no 3-way base ([Pijul theory](https://jneem.github.io/pijul/); [Pijul FAQ](https://pijul.org/faq)). It is elegant, but it buys order-independence by constraining what a patch *is* — and our whole point is that intake is **not** order-independent: the guardian's verdict on an entry *depends on the state it is replayed against*. We want ordered replay-through-policy, which is eg-walker/Datomic-shaped, not Pijul-shaped. Pijul is cargo-cult for a policy-gated log.

## Verifiability — transparency-log discipline is the missing crypto frame

The new "verifiable history" angle has a mature, directly-applicable model report 92 didn't cover: **Certificate-Transparency-style logs**. A log periodically emits a signed checkpoint (Signed Tree Head); **witnesses** counter-sign a new checkpoint *only after verifying a Merkle consistency proof that it is an append-only extension of the previous one*; any two checkpoints are reconcilable by a consistency proof logarithmic in log length ([Transparency logs](https://transparency.dev/articles/logs-a-verifiable-transport-layer/); [RFC 6962](https://www.rfc-editor.org/rfc/rfc6962.html); [transparency-dev/witness](https://github.com/transparency-dev/witness)). Map onto us: **ouranos is a witness.** When it ingests our log suffix it verifies the blake3 hash-chain consistency and counter-signs the new head with **criome BLS** — and BLS *aggregates*, so a chain of head-attestations compresses to one signature ([BLS multi-signatures, Boneh et al.](https://crypto.stanford.edu/~dabo/pubs/papers/BLSmultisig.html)). This gives append-only-history verification for free over machinery we already need (suffix shipping + checkpoints), with the one consistent blake3-for-addressing / BLS-for-attestation basis the design mandates.

## Zero-downtime migration at the post-agent frontier

The brief says target the best end-shape because zero-downtime complex migrations are routine. The mature pattern is **expand/contract (parallel change)** with versioned schemas kept simultaneously live: pgroll keeps old and new schema versions addressable at once via *views over the same physical table*, with triggers dual-writing during backfill, then contracts ([pgroll guide](https://neon.com/guides/pgroll); [expand-contract pattern](https://systemdr.systemdrd.com/p/database-schema-migrations-with-zero)). The clever mapping for us: our `SchemaTransition`-as-typed-log-entry **is** expand/contract expressed in the log — the transition entry names `(v_old_hash → v_new_hash, reducer)`, the reducer is the backfill transform, and replay-from-checkpoint *is* the backfill. Two schema versions coexist for the window where some peers' suffixes are pre-transition and the local checkpoint is post-transition; the per-family schema-hash selector (already in our design) is the "which view do you read" switch pgroll implements with `search_path`. We get pgroll's coexistence property structurally because the log already carries the transition as a first-class, replayable event.

## Bottom line for our design

One log, one checkpoint, and they want different structures. The log stays a blake3-linked DAG — promoted from a chain to a DAG so merge commits are representable — and needs **no** prolly tree or MST, because its diff is its suffix; report 92's rejection holds for the log permanently. A Merkle Search Tree (not a Dolt prolly tree) is justified **only** for the materialized checkpoint, **only** when two distant branch states must be diffed or transferred without their shared log, and **only within a single schema version**. Merge/rebase is event replay (eg-walker) through a typed per-component policy hook shaped like a Datomic transaction function, defaulting to "apply the diff" and overridable per component — Spirit's guardian being the worked example. Verifiability is transparency-log witnessing with ouranos as the witness and aggregatable criome BLS over checkpoint heads. The succinct/HAMT machinery is available if a family ever gets huge, and is over-engineering until then.

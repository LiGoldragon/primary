# Branch / Fork / Merge / Rebase on a Typed Event Log: DVCS Semantics Mapped onto Per-Component Diff-Carrying Logs

## Frame: what report 92 left open, and the new question

Report 92 (`reports/system-designer/92-reusable-versioned-sema-library/`) surveyed the *durability* axis — how to make the log authoritative, the store a rebuildable view, with checkpoints, a remote mirror, and a typed schema-hash decoder. It deliberately treated everything as **single-writer**: Datomic-shaped, "no merge machinery." It even rejected prolly trees as cargo-cult on those grounds.

The new psyche requirement changes the regime: full DVCS semantics — **branch, fork, rebase, merge** — over the typed log, with a **per-component intake/merge/rebase policy** (Spirit's guardian admits/rejects/transforms each entry). That re-opens exactly the question report 92 closed. This report extends into that dimension and answers the KEY QUESTION precisely. It does not re-survey general prior art (report 92 owns that); it isolates the *branch/merge/rebase mechanics* of each donor and maps them onto a diff-carrying log with a per-entry policy hook.

The headline finding: for an append-only typed event log where **the diff IS the log suffix**, three of the four DVCS verbs collapse to cursor arithmetic plus replay-through-policy, and structural state trees (prolly/Merkle-of-the-materialized-state) are **not required** for them. Only one verb — *conflict detection on merge of genuinely concurrent writes to the same key* — needs structure, and even there the structure is a cheap derived index, not a second authoritative store. This is the clever core, and it is exactly where the 2024 state of the art (eg-walker) independently landed.

## The four verbs, defined precisely for a diff-carrying log

Fix vocabulary first, because the donors use these words inconsistently. Our substrate: a per-component hash-linked log of typed `ReplayEnvelope` entries, each carrying `(op, family/schema-hash, key, payload, prev-digest, sequence)`. A **state** is the materialized view obtained by folding a prefix of the log. The fold is `Reducer::apply`, the per-component policy is the new seam.

- **Branch** = a *named movable cursor* into the commit-DAG of log segments. Not a copy of anything. `main` and `feature` are two `(name -> head-digest)` pointers; the log entries themselves are shared by content address up to the divergence point. Creating a branch is O(1) — write one pointer. This is *exactly* Datomic's "connections are like branches, database values are like commits" and Git's ref model; both confirm a branch is a pointer, never a clone.
- **Fork** = a branch whose head is taken and then *diverges* by appending entries that the origin never sees. Mechanically identical to branch at creation; the word marks intent (independent evolution, possibly a different policy). A fork is "hold cursor at H, append a private suffix."
- **Merge** = produce a new head whose state reflects *both* divergent suffixes since the common ancestor (the merge base / LCA in the segment DAG). For a diff-carrying log this is: take base `B`, suffix `S_into` (base→into-head), suffix `S_from` (base→from-head); the merge result is `fold(B, then S_into, then replay-S_from-through-policy)`. The merge commit has **two parents** in the DAG and a content-addressed head over both suffixes.
- **Rebase** = re-apply one branch's suffix `S_from` onto a *different* base `into-head`, producing a *new linear suffix* with new entry digests (the prev-digest of each re-applied entry changes, so its content hash changes — the cascade git documents). The DAG stays single-parent; history is rewritten. For us rebase is "replay `S_from`, entry by entry, through the target's intake policy, onto the target head."

The two operational primitives underneath all of merge and rebase are therefore **(a) find the common ancestor in the segment DAG** and **(b) replay a suffix of typed diffs through a per-entry policy onto a target head**. Everything else is naming.

## The KEY QUESTION answered: do merge/rebase need structural state trees?

**No, for everything except concurrent same-key conflict detection — and yes there, but only as a derived index, never a second authoritative store.** Decompose:

### Rebase needs no structural state tree at all
Rebase is pure suffix replay. We already carry each change's diff as a typed log entry; re-applying it onto a new base is `for entry in S_from { policy.intake(entry, target_state)?; target = Reducer::apply(target, entry') }`. No materialized-state Merkle tree is consulted. The *only* structural object touched is the **commit-DAG of segments** (to locate the base and to chain new prev-digests). This is the Pijul insight in our terms: because each unit is a self-contained change, replay onto a new context is well-defined. The cost is re-digesting every re-applied entry (new prev-digest ⇒ new blake3 ⇒ the cascade), which is O(|S_from|) hashing — the genuine, unavoidable rebase tax, identical to git's.

### Merge with non-overlapping keys needs no structural state tree
If `S_into` and `S_from` touch disjoint key sets, the merged state is just `apply(apply(base, S_into), S_from)` — replay both suffixes, order-independent on disjoint keys. No three-way cell comparison, no Merkle diff. This is the common case and it is *free* on a diff-carrying log: we already have both suffixes as typed entries; we do not need to diff two materialized states to discover what changed, because the log already told us. **This is the structural advantage of versioning the log instead of the store**: Dolt must *compute* the diff between two prolly trees (its fast-merge 2025 work is entirely about making that diff cheap via chunk-hash skipping); we never compute a diff — the suffix is the diff, already typed, already in hand.

### Only concurrent same-key writes need structure — and only a derived index
The one thing replay-of-suffix cannot answer by itself: *did `S_into` and `S_from` both write the same key, and if so which wins / is it a conflict?* Pure linear replay would just let the later-applied suffix clobber, silently losing the other side. To detect overlap you need, per key, the set of `(branch, last-writer-entry)` since the base. That is a **key→head-writer index over the suffix**, which is O(|suffix|) to build and is *discarded after the merge* — it is not the authoritative store, it is eg-walker's "temporary CRDT, built during merge, thrown away after." So the precise answer: merge needs a *transient* structural index keyed by record-key, scoped to the two suffixes, never a persistent state tree.

This is the exact shape eg-walker (2024, Gentle & Kleppmann, EuroSys '25) proved out for text: the event graph is authoritative and on disk; a CRDT is materialized *only* to compute a merge, only over operations after the lowest common ancestor (the "critical version" partition), and discarded immediately — using 1-2 orders of magnitude less memory than a persistent CRDT and merging up to ~160,000× faster than OT on large divergence. Our case is strictly easier than text because our keys are explicit record identifiers (no positional tombstone/character-ID problem that makes text hard), so the transient index is a plain hash map keyed by record key, not an eg-walker sequence CRDT.

**Conclusion to the key question:** for our shape, *replay-of-suffix-through-policy is sufficient for branch, fork, rebase, and disjoint-key merge*; concurrent-same-key merge additionally needs a transient per-key overlap index built from the suffixes and a per-component conflict policy. We never need a persistent Merkle/prolly tree of the materialized state. The materialized redb view stays a discardable cache exactly as report 92 already established — DVCS does not change that.

## Where each donor's mechanism maps (branch/merge/rebase specifically)

### Git commit-DAG + rebase + three-way merge
The reference model. Branch = ref pointer; merge = find LCA via the commit DAG (the `ort` strategy, default since ~2021, replacing recursive; multiple merge-bases are themselves recursively merged into a synthetic base), then three-way diff base↔ours, base↔theirs, apply both; rebase = re-apply commits one by one onto a new base, **recomputing every SHA because the parent hash is an input to the child's hash** — the documented cascade. *Maps:* our segment-DAG plays the role of the commit DAG; our prev-digest chain is the parent-hash cascade, so our rebase re-digesting is the same forced cost. *Does not map:* git diffs *snapshots* to recover changes (it stores trees, not diffs); we store the diffs, so we skip git's entire diff-recovery step on merge. Git is the proof the DAG+replay model is sound; it is the *least* clever donor because it throws away the per-change diff we keep.

### Dolt — prolly-tree three-way merge (and 2025 fast-merge)
Dolt stores tables as content-addressed prolly trees (chunked ~4KB, structurally shared across branches). Merge finds the merge-base commit, then does a *cell-level* three-way merge; conflicts are stored by primary key in a side prolly-tree carrying base/ours/theirs. The 2025 fast-merge makes diffing tree-structural: equal subtree hashes are skipped, so merge cost scales with *changed tree nodes*, not row count (~1000× on their benchmark). *Maps as a warning, not a pattern:* Dolt's cleverness is making the *store* diffable because it versions the store. **We version the log, so we get the diff for free and Dolt's entire prolly machinery is solving a problem we don't have.** Report 92 already rejected prolly trees as cargo-cult for the single-writer case; the branching requirement does *not* revive them — the suffix-is-the-diff property is strictly stronger than "make the store cheap to diff." The one transferable idea is *conflict storage by key carrying all three sides* (base/into/from): our transient overlap index should carry the same triple so a per-component policy can do a real three-way decision, not just last-writer-wins.

### Irmin — the three-way-merge functor
Irmin parameterizes the store over a `Contents` signature that *must* provide a `merge` function: `merge ~old t1 t2 -> result`, computing the LCA and calling the user's three-way merge, conflict-or-value. This is the single most important structural donor for the policy requirement: **merge is a typed, user-supplied function bound to the content type, not a fixed algorithm in the engine.** *Maps directly:* this is our per-component merge/rebase policy. Irmin's `merge ~old t1 t2` becomes our `Reducer`-adjacent `MergePolicy::resolve(base_value, into_value, from_value) -> Resolved | Rejected | Transformed`, and the guardian is precisely a `MergePolicy`/`IntakePolicy` impl. Irmin is single-writer per branch but supports merging branches via this functor — exactly our setup. The *clever* combination: Irmin binds merge to the *value type*; we bind intake/merge/rebase to the *component* via a sealed trait with a default impl plus per-component override (Spirit overrides with the guardian). Irmin proves typed, per-content merge is real and shipped; we generalize "per content type" to "per component policy, default + override."

### TerminusDB — branch/rebase/merge over immutable delta layers
Closest production system to our exact shape. Every commit is an **immutable delta layer** (added/removed triples); a commit-graph holds branch pointers, parents, and enables time-travel/squash/reset/rebase. **Rebase = "commits made by different authors are rerun on the current database"** — literal replay of commits onto a new head, our definition exactly. Layer rollup (compacting a run of delta layers) is their checkpoint analogue and only needs to inspect a *subset* of layers. *Maps directly:* delta-layer = our log entry (a typed diff); commit-graph = our segment-DAG; their rebase-as-rerun = our suffix-replay; their rollup = report 92's checkpoint. TerminusDB is the existence proof that "rebase = replay typed diffs onto a new base over an immutable-layer log" is production-viable. *Does not map:* their triple/stringly substrate (RDF) is the genericity path report 92 forbids; we keep typed closed-sum families instead. The transferable, production-proven core: **migration and rebase are the same kind of object** — a named, classified, replayable per-commit op. Our `SchemaTransition` entry and a rebased domain entry are both just "typed diffs replayed through policy."

### Datomic / XTDB / the "no branch" lineage
Datomic has **no branches by design** — transactions serialize into one linear schedule; a single global index. You *simulate* a branch with `d/with` (speculative write: takes a db value, returns a new db value with extra datoms, in-memory, never sent to the transactor) plus Datomock (put that value in an atom, keep `with`-ing). So **a Datomic "branch" is: hold a database value (= a commit), append a private suffix in memory.** That is precisely our fork primitive. *The sharp failure mode* (Daniel Janus, 2025): **you cannot branch from the past.** `d/with` computes its new datoms (especially *retractions*) against the **unfiltered current** db, so forking from an `as-of` historical value yields retractions of values that don't exist in the filtered view — internally inconsistent. *Maps as our load-bearing warning:* if our `IntakePolicy`/`Reducer` ever computes an entry's effect (e.g. a tombstone, a "supersede prior" op, a guardian transform) against the *live head* instead of *the base state being replayed onto*, rebase/branch-from-a-past-checkpoint silently corrupts. **Rule forced by Datomic's mistake: every intake/merge/rebase decision must be a pure function of (base state being built, entry), never of the current live head.** XTDB v2 (2024) is the counter-move: it *adds* branching to the temporal model via trie-branching across LSM levels with UUIDv4 ids — proof that branching a temporal/append store is current, real, and being actively built (not exotic), but its mechanism (LSM trie branching) is for a store-versioned system; we get branching more cheaply because we version the log.

### Fossil
Authoritative content-addressed artifacts + a *rebuildable* relational cache (`rebuild`), set-reconciliation sync. Branches are just named leaves in the artifact DAG. *Maps:* `rebuild` is report 92's "demote tables to a view," and branching adds nothing new to Fossil's storage — a branch is metadata over the same content-addressed artifact set. Confirms branching is free when the store is a rebuildable view over content-addressed history; it adds no new requirement on the substrate.

### Pijul — patches commute, so rebase ≡ merge
Pijul's claim: patches are designed to commute (theory related to CRDTs); **"rebase and merge are the same operation, or rather it doesn't matter which you do"**; conflicts are a *representable state*, not a failure, and are resolved by a later change once and never recur; merge is a pushout in the category of files-and-patches. *Maps as the aspirational ceiling and a precise boundary:* IF our diffs over disjoint keys commute (they do — applying two record writes to different keys is order-independent), THEN for that case our rebase and merge genuinely *are* the same replay and produce the same state — we inherit Pijul's best property for free on the common case. BUT Pijul achieves *full* commutativity only by enriching every patch with conflict-state structure (a graph, not a flat diff); our entries are flat typed diffs. So we get commutativity exactly where keys are disjoint and *lose* it on concurrent same-key writes — which is why those, and only those, need the transient overlap index. **The clever framing Pijul gives us: don't implement merge and rebase as two code paths. Implement one "replay-suffix-through-policy" primitive; merge is replay that keeps the base's suffix and adds a second parent, rebase is replay that rewrites onto a new base. One engine, two callers.**

### eg-walker / Patchwork (the 2024-2026 frontier)
eg-walker (2024): event graph authoritative on disk, transient CRDT built only to merge, only over post-LCA operations, discarded after; materialized doc is a cache. Ink & Switch Patchwork (2024-2025): "universal version control" primitives — lightweight branches, diff views, merge-back-or-discard — over Automerge documents, aiming at branches/merge for *arbitrary* app data, not just text. *Maps as direct validation of our whole design:* eg-walker is "version the log, materialize a discardable view, build merge structure transiently" arrived at independently and award-recognized — it is our exact architecture for the merge operation. Patchwork is the proof that the *industry frontier* is generalizing branch/merge to typed app data with per-document policy, which is our requirement. The divergence: both are CRDT/schemaless (Automerge) for genericity — the path report 92 forbids; we keep typed closed-sum families and a per-component policy instead of a universal schemaless CRDT. We are at the frontier; our typed-discipline variant is the novel contribution.

## The per-component policy seam, made precise

The brief's guardian example crystallizes the design. Define one trait family (default impl + per-component override), invoked at exactly one place — the replay-of-suffix loop:

- `IntakePolicy::admit(base_state, entry) -> Admit | Reject(reason) | Transform(entry')` — called per entry during rebase and during the from-suffix leg of merge. Default impl: admit. Spirit's override: route through the guardian (admit/reject/transform), which is *already* how Spirit's guardian works on live writes — rebase reuses the identical guardian, so a rebased entry is admitted on the same terms as an originally-authored one. This is the single cleverest fit in the whole design: **the guardian is not a new merge algorithm; it is the existing write-admission function, now also invoked on replayed entries.**
- `ConflictPolicy::resolve(base_value, into_value, from_value) -> Resolved(value) | Reject | KeepConflict` — called only when the transient overlap index reports both suffixes wrote the same key. Default impl: a declared rule (last-writer-by-sequence, or hard-fail). Irmin's `merge ~old t1 t2` is this signature exactly.

Both are pure functions of *the state being built* and the entry/values — never the live head (the Datomic rule). A rebase is `replay(S_from, onto=into_head, via=IntakePolicy)`; a merge is `replay(S_from, onto=into_head, via=IntakePolicy)` plus the overlap-index pass through `ConflictPolicy`. One primitive, parameterized by policy. The policy is a sealed trait so genericity stays typed (report 92's knife-edge: closed-sum, no open `Box<dyn>` fallback).

## Failure modes, named (rejecting cargo-cult)

- **Re-digesting cascade on rebase is unavoidable and O(|suffix|).** Each re-applied entry gets a new prev-digest ⇒ new blake3 ⇒ new entry identity, cascading like git's SHA rewrite. This is correct, not a bug, but it means rebasing a long divergent suffix re-hashes the whole suffix and *invalidates the remote mirror's copy of those entries* — the mirror must ingest the rebased suffix as new content, and the old suffix becomes garbage. Budget mirror bandwidth for rebase = re-ship the suffix. (Merge does *not* re-digest the existing suffixes — it only adds a merge head — so prefer merge over rebase when mirror bandwidth matters.)
- **Branching from a past checkpoint is the Datomic trap.** If a policy/reducer computes effects against the live head, forking from an old checkpoint corrupts. Forced rule: replay effects against the base being rebuilt. Test this explicitly (fork from a pruned-past checkpoint, append, verify no live-head leakage).
- **Concurrent same-key write is the *only* real conflict** and needs the transient overlap index + `ConflictPolicy`; everything else is order-independent replay. Do not build a general structural merge engine — that is cargo-culting Dolt's prolly-merge for a problem the log already solved.
- **Compaction interacts with branching.** A checkpoint that prunes the log head below a branch's base orphans the branch (the base state is gone). Checkpoint policy must pin the merge-base of every live branch (refuse to compact past the oldest live branch base), or carry a base-state snapshot in the checkpoint. Report 92's checkpoint protocol must add "live-branch-base retention" to its retained-suffix policy.
- **Multiple merge bases** (criss-cross merges) — git's `ort` synthesizes a merged base recursively. For us this is rare (mostly linear forks of a coordination store) but the segment-DAG LCA must handle it or refuse it explicitly rather than silently picking one base.
- **Schema divergence across a branch.** If `from` ran a `SchemaTransition` the `into` branch hasn't, replaying `S_from` onto `into` must replay the transition first (the migration is just a typed entry in the suffix — TerminusDB's "migration and commit are the same object" pays off here). The schema-hash selector makes each replayed entry self-describing, so a rebase across a schema boundary decodes correctly entry-by-entry. This is a place our typed log is *stronger* than Dolt (whole-table-rewrite on schema change) — the transition rides the suffix.

# DVCS Semantics over a Typed Event Log: the 2024–2026 Frontier and the Clever Combination

## Scope and how this extends report 92

System-designer report 92 (`reports/system-designer/92-reusable-versioned-sema-library/`) already surveyed the general prior art for "version the log, not the store" — Dolt/Noms/prolly, Datomic family, Git/Mercurial/Fossil, Irmin/IPLD, TerminusDB, event-sourcing/CQRS, WAL-shipping, CRDT/patch theory, serialization evolution, CDC/backup — and converged on a payload-bearing, hash-linked, authoritative log with a rebuildable redb view, genericity via a sealed `RecordFamily` + closed-enum decode by schema-hash. It deliberately set aside three-way merge because every system it surveyed was single-writer.

The new psyche requirement — full DVCS semantics (branch / fork / rebase / merge) over the typed database, with **per-component customizable intake / merge / rebase policy** (default impl + per-component override; Spirit's guardian admits/rejects/transforms each entry) — is exactly the gap report 92 left open. This report covers only the new dimensions: the merge/branch/rebase frontier, the 2024–2026 state of the art that did not exist or was not load-bearing in 92, and the clever combination for a typed event log whose entries already carry each change's diff.

The single most important framing: **we are not building a CRDT, and we are not building Datomic.** We are building a *git/Dolt/Pijul-shaped DVCS* whose objects happen to be typed rkyv records on a blake3-linked log, whose merge resolver is *per-component pluggable*, and whose server is trusted. That combination — typed-DB + DVCS-merge + pluggable policy + trusted-mirror + one consistent blake3/BLS crypto basis — does not exist as a shipped system anywhere. Each leg exists; the assembly is ours.

## The merge/branch/rebase frontier (the dimension report 92 skipped)

### eg-walker (2024, EuroSys 2025): replay-between-LCA, build ephemeral merge state, discard

Eg-walker (Gentle & Kleppmann) is the load-bearing 2024 idea. It stores edits as an immutable DAG of events, each event being a *lightweight index-based operation* (insert(pos,char) / delete(pos)) plus a unique id and parent ids — **not** a fat CRDT structure with per-character unique ids. The materialized document text is cached on disk; CRDT metadata is *not* persisted.

The clever part is the merge. To merge two diverged branches it does **not** replay all history. It finds the most recent *critical version* (a point where everything before causally precedes everything after — effectively the lowest common ancestor of the divergence), and replays only the events after it, building a **temporary** internal CRDT that simultaneously tracks two versions ("prepare" = events as originally generated, "effect" = events applied so far). Concurrent operations are reconciled by *retreating* and *advancing* events as the walk crosses concurrency boundaries, then the temporary state is **discarded**. Result: 1–2 orders of magnitude less steady-state memory than persistent CRDTs, O(n log n) merge versus OT's O(n²), and a document that took an hour to merge under OT merges in 24 ms ([arXiv:2409.14252](https://arxiv.org/html/2409.14252v1), [Loro's eg-walker writeup](https://loro.dev/docs/concepts/event_graph_walker)).

This is the exact shape of *our* merge. Our log already carries each change's typed diff (the "lightweight op"). Branch merge = walk the two branch tips back to their common-ancestor checkpoint, replay the typed diffs through an ephemeral reconciliation pass, materialize a new head view, and persist only the *resulting* typed entries (not the reconciliation scaffold). We get eg-walker's "version the log, build merge state transiently, discard it" property for free because our log is already op/diff-bearing.

### jj (Jujutsu): conflicts are first-class history objects; rebase never blocks

Jujutsu's defining move is that **a conflict is a first-class value that can live in history**. If a rebase produces a conflict, the conflict is *recorded in the rebased commit* and the rebase *succeeds*; you resolve it later, whenever. Every operation (merge, rebase, amend, bookmark move) is captured in an **operation log** that makes the whole repo state reversible ([jj-vcs/jj](https://github.com/jj-vcs/jj), [jj conflicts docs](https://jj-vcs.github.io/jj/latest/conflicts/)). Editing a commit auto-rebases all descendants.

This is the policy primitive we need. Spirit's guardian "admits, rejects, or transforms" — that is precisely a per-component conflict/intake *resolver*, and jj proves the right architecture: **the merge/rebase operation always succeeds at the log level; an unresolved conflict is a typed entry the policy resolves out-of-band.** A rebase through Spirit routes each replayed entry through the guardian; a guardian *rejection* is not a hard failure of the rebase, it is a recorded `Rejected`/`Divergence` entry. jj also validates our own substrate choice: jj is already our VCS, its operation-log + first-class-conflict model is the closest existing thing to what we want, and "the log is authoritative, every op is reversible" is its core thesis.

### Pijul: deletion is a label change, not a removal; conflicts are graph states

Pijul models a file as a graph of chunks with *alive*/*dead* labels on edges; a deletion never removes a vertex, it relabels edges (a deleted-but-still-referenced vertex is a "zombie"). A conflict is simply a *graph state* (two alive vertices with no path between them, or a cycle) rather than an error condition. Merge is a **pushout** in a category where files are objects and patches are arrows, which makes merge **associative and commutative** — patch order genuinely does not matter, eliminating whole classes of spurious conflicts ([Pijul model](https://pijul.org/model/), [Pijul theory](https://pijul.org/manual/theory.html), [A Categorical Theory of Patches](https://www.researchgate.net/publication/258521116_A_Categorical_Theory_of_Patches)).

Two transferable ideas, one rejected substrate. Transferable: (1) **tombstone-as-relabel** — our log already does this (a retire/supersede is a typed entry, not a byte deletion), so we inherit Pijul's "history is append-only; deletion is data" property; (2) **conflict-as-state** — a merge that can't auto-resolve produces a *typed Conflict record* the component's policy interprets, not a wedged operation. Rejected: the line-graph/CRDT substrate itself — we are a typed record store, not a text-line graph, so we do not adopt Pijul's vertex model, only its discipline that merge order should not matter where the policy declares operations commutative.

### Dolt fast-merge (2025): merkle-diff only touches the differing subtrees

Dolt's July-2025 fast merge is an asymptotic-not-constant improvement. Because tables are prolly trees (content-addressed B-trees with structural sharing — identical 4 KB chunks stored once across branches), merge walks both trees against the common ancestor and **only recurses into subtrees whose content-address differs**; identical subtrees are skipped entirely. Three phases: patch generation (find key-ranges present on a branch but not the ancestor), patch merging (interleave the two patch streams, recursing only where ranges overlap), patch application (splice non-overlapping patches directly when node boundaries align). 5M rows/branch merged in ~0.01 s vs ~9 s for the row-enumeration approach — ~1000× ([Dolt fast merge](https://www.dolthub.com/blog/2025-07-16-announcing-fast-merge/), [structural sharing study](https://www.dolthub.com/blog/2024-04-12-study-in-structural-sharing/), [database branches](https://www.dolthub.com/blog/2024-09-18-database-branches/)). Cell-level three-way merge resolves non-overlapping changes silently; same-cell changes become conflicts with `ours`/`theirs` strategies ([Dolt merges](https://docs.dolthub.com/sql-reference/version-control/merges)).

For us, the prolly tree itself stays *rejected* (report 92's call — we are append-only Datomic-shaped, not random-mutable-keyspace Dolt-shaped). But the *merge-by-merkle-diff* idea ports onto our hash-linked log at a coarser grain: blake3 links + Sedimentree-style checkpoints let a merge/sync skip log segments whose head digests already match, recursing only into the diverged suffix. We get "only examine what might differ" without adopting prolly trees, because the checkpoint chain is already a (linear, not B-tree) merkle structure.

### git merge-ort + virtual merge base (recursive multi-LCA): our criss-cross story

merge-ort replaced git's recursive strategy and is the default since 2.34 (the `recursive` name became an alias in 2.50, June 2025). Two relevant facts. (1) Performance: rewritten for rename detection, 500×–9000× faster on rename-heavy merges/rebases ([merge-ort.h](https://github.com/git/git/blob/master/merge-ort.h), [Palantir on git's merge machinery](https://blog.palantir.com/optimizing-gits-merge-machinery-1-127ceb0ef2a1)). (2) Correctness under criss-cross: when two heads have **multiple** best common ancestors (no unique LCA — a real possibility the moment branches merge into each other), merge-ort recursively merges the ancestors into a single **virtual merge base** and three-way-merges against that ([git merge-strategies](https://git-scm.com/docs/merge-strategies)).

This is a correctness requirement we must inherit, not an optimization. Once components branch and re-merge, our event DAG gets criss-cross topologies with no single LCA. The replay-between-LCA merge (eg-walker) must therefore be "replay-between-*virtual*-LCA": when multiple ancestors exist, recursively merge them first (through the same per-component policy) to synthesize the base, then merge the two heads against it. Skipping this produces spurious conflicts/divergences exactly where git's old recursive strategy did.

### Semantic / pluggable merge drivers (Mergiraf, weave, git custom drivers): the policy-hook precedent

git's `gitattributes` + `merge.<driver>` mechanism is the canonical proof that **merge resolution is pluggable per path**: register a driver, point file-globs at it, git hands the driver three temp files (`%O` base, `%A` ours, `%B` theirs) and the driver writes the result ([gitattributes](https://git-scm.com/docs/gitattributes), [custom merge driver example](https://github.com/Praqma/git-merge-driver)). The 2024–2025 frontier makes these drivers *structure-aware*: Mergiraf parses base/left/right with tree-sitter and matches syntax trees via GumTree to resolve conflicts git can't ([Mergiraf](https://codeberg.org/mergiraf/mergiraf), [LWN](https://lwn.net/Articles/1042355/)); weave matches *semantic entities* (functions/classes by name+type+scope, not AST position) and resolves 31/31 benchmark merges vs git's 15/31 ([weave](https://github.com/ataraxy-labs/weave)).

This is the architectural template for the per-component policy. Our `MergePolicy` is the typed, in-process analogue of a merge driver: instead of three temp files it receives `(base: ComponentRecord, ours: ComponentRecord, theirs: ComponentRecord)` — already-decoded typed records — and returns `Merged(record)` | `Conflict(typed)` | `Reject` | `Transform(record)`. The default impl is a generic last-write/structural merge; the per-component override is the guardian. Crucially, *because our records are typed and already decoded by schema-hash*, our policy operates on domain values, not text — the semantic-merge frontier's whole goal — by construction, not by re-parsing.

## The verifiable / efficient-sync frontier

### Sedimentree (Ink & Switch / Automerge Beelay): deterministic hash-boundary strata + checkpoints

Sedimentree recursively compresses commit-DAG ranges into **strata** layered by age — old commits in large low strata, recent commits in small high strata. The clever mechanism is **deterministic boundary selection by hash**: interpret a commit hash as a number, count trailing zero nibbles, and that determines its stratum level (four trailing zeros → a level-4 boundary). Because the hash is immutable, *peers with different histories independently agree on the same boundaries for shared segments* — no negotiation. Strata retain interior **checkpoint** hashes at smaller-stratum boundaries so containment ("stratum x supports stratum y") is decidable from boundary+checkpoint hashes alone. Peers reconcile by exchanging *minimal* sedimentrees (boundaries only, checkpoints stripped to save megabytes on large docs), then recursively requesting only the sub-ranges that differ. It works over **encrypted** strata — the server forwards opaque chunks it cannot read ([sedimentree.md](https://github.com/automerge/beelay/blob/main/docs/sedimentree.md), [Beelay talk](https://www.youtube.com/watch?v=neRuBAPAsE0), [Subduction](https://github.com/inkandswitch/subduction)).

This is the closest external template for our checkpoint + suffix-sync + compaction protocol, and report 92 already flagged it. The new, sharper takeaway for the *branching* requirement: Sedimentree's deterministic hash boundaries give us **stable, content-defined checkpoints that double as merge LCAs**. The eg-walker "critical version" and the Sedimentree "stratum boundary" are the same object for us — a checkpoint is simultaneously (a) a replay floor, (b) a sync reconciliation point, and (c) a candidate merge base. One mechanism, three jobs. Our deliberate divergence stays the same: ouranos is *trusted*, so we skip Beelay's zero-knowledge-encrypted-server machinery and keep BLS-signed plaintext strata.

### Transparency-log checkpoints + BLS: signing history, not just hashing it

The verifiable-log literature (Certificate Transparency, Go's tlog, the 2025 LegoLog) all share one shape: a Merkle tree over the log, a periodically **signed checkpoint** (signed tree head) that lets a client verify inclusion in *logarithmic* work instead of replaying the whole log, plus inclusion/consistency proofs ([research!rsc tlog](https://research.swtch.com/tlog), [LegoLog](https://eprint.iacr.org/2025/1234.pdf), [CT how it works](https://certificate.transparency.dev/howctworks/)). BLS aggregate/threshold signatures (Boneh–Lynn–Shacham) are the production crypto for *compressing many signatures into one* and for *threshold attestation* — drand and Filecoin run threshold-BLS at scale; 2024–2025 work (Mysten's subset-optimized aggregation, NIST's tBLS treatment) keeps t=64 aggregation in the ~8 ms range ([BLS Wikipedia](https://en.wikipedia.org/wiki/BLS_digital_signature), [Mysten subset-optimized BLS](https://www.mystenlabs.com/blog/new-bls-aggregation-for-proof-of-stake), [NIST tBLS](https://csrc.nist.gov/csrc/media/presentations/2025/stppa8-threshold-bls/images-media/stppa8-tbls.pdf)).

The clever combination for us: **the blake3 hash chain proves integrity; the BLS-signed checkpoint proves authority and attests history.** A checkpoint is not just "covered range + schema-hash inventory" (report 92) — it is a *BLS-signed* tree head over the blake3-linked segment. That gives three properties at once: (1) ouranos verifies a pushed log suffix's authority with one signature check before sealing it WORM; (2) a fork carries the signed checkpoint of its base, so a merge can *prove* the two branches share an ancestor without re-shipping it; (3) criome's existing BLS basis signs the checkpoints, so there is exactly one signing scheme and one hashing scheme across every component — the "one consistent basis, no per-component divergence" mandate is satisfiable with zero new crypto. (Caveat below: don't over-reach into threshold-BLS quorum.)

## The zero-downtime / post-agent-frontier dimension

### pgroll / Reshape expand–contract: two schema versions live simultaneously

The post-agent design brief says target the best end-shape because zero-downtime complex migrations are now routine. The Postgres frontier shows what "routine" concretely looks like: pgroll (and its predecessor Reshape) implement **expand–contract** so that *two schema versions are served at once*. Expand adds only backward-compatible structure (new physical column + triggers that dual-write old↔new); backfill populates the new shape; the old and new app versions both read their own view; contract removes the old shape once nothing reads it. Rollback is instant because the old version never left until contract ([pgroll](https://github.com/xataio/pgroll), [expand-contract](https://xata.io/blog/pgroll-expand-contract), [Neon pgroll guide](https://neon.com/guides/pgroll)).

This maps cleanly onto our `SchemaTransition` + decode-by-schema-hash. A migration is already a typed log entry (`SchemaTransition(v_old_hash → v_new_hash, reducer)`); pgroll adds the operational frontier we should adopt: **during a transition both schema-hashes are decodable at once** (the closed-enum carries both the old and new sealed `RecordFamily`), readers select by schema-hash, and the "contract" step is dropping the old family from the enum once no entry or branch references it. For us the dual-decode window is also what makes a *cross-schema merge* possible: if `ours` is on the new schema and `theirs` on the old, the merge policy decodes each by its own schema-hash, projects through the reducer (TerminusDB's exact "replay migrations so branches become comparable" — see below), then merges on the common schema. rkyv's brittleness (report 92: essentially every schema change is a full-type-set transition) makes the dual-decode window mandatory, not optional.

### TerminusDB: migration-as-replayable-op so branches become comparable

TerminusDB tracks the schema migration **on each commit** as named, classified, replayable operations with data-rewrite rules, specifically so that "branches can be made comparable" — you can replay one branch's migrations onto another to obtain a common schema before merging a change-request ([TerminusDB schema migration guide](https://terminusdb.com/docs/schema-migration-reference-guide/), [schema migration blog](https://terminusdb.com/blog/schema-migration/)). It runs on immutable content-addressed delta layers with periodic rollup compaction.

This is the production existence-proof that *migration and merge compose*: a `SchemaTransition` on our log is exactly TerminusDB's per-commit replayable migration, and "replay migrations so branches become comparable" is precisely how we reconcile two branches that diverged across a schema change. The triple/stringly substrate stays rejected (report 92); the *migration-as-replayable-op-that-makes-branches-mergeable* idea is directly load-bearing for the new branching requirement.

## Where the frontier does NOT fit our event-log shape

- **Datomic / XTDB immutable timeline ≠ branching.** Both are append-only single-timeline immutable DBs. Datomic explicitly *cannot branch the past* — `with`/speculative-writes give a throwaway in-memory db value, `as-of` filters, but neither forks history ([Daniel Janus: you can't branch Datomic from the past](https://blog.danieljanus.pl/2025/04/22/datomic-forking-the-past/)). XTDB v2 has only an append-only `SYSTEM_TIME` line plus bitemporal `VALID_TIME` ([XTDB](https://xtdb.com/), [XTDB immutability walkthrough](https://docs.xtdb.com/tutorials/immutability-walkthrough/part-4.html)). **Lesson:** the immutability tradition gives us replay and time-travel but is *architecturally hostile* to the branch/fork the psyche now requires. We get branching from the git/Dolt/Pijul/jj lineage, not the Datomic lineage — do not import Datomic's "single authoritative timeline" assumption into the merge design. (Bitemporality is a separate, optional concern; valid-time is orthogonal to branching and we should not conflate them.)

- **Prolly trees / Dolt storage substrate.** Report 92 already rejected this; the 2025 fast-merge work does not change the verdict. Prolly trees pay off for a *random-mutable keyspace* with cell-level diff; our log is append-only and Datomic-shaped. We take the merge-by-merkle-diff *idea* (skip identical segments) at checkpoint grain, not the prolly-tree *substrate*.

- **CRDT auto-merge as the resolver (Automerge/Yjs/Loro).** automerge-repo's pluggable storage/network adapters and Loro's frontiers-vs-version-vectors + shallow-snapshot history truncation are good engineering ([automerge-repo](https://github.com/automerge/automerge-repo), [Loro versioning](https://loro.dev/docs/advanced/version_deep_dive)), and **frontiers** (a compact set of DAG-head hashes) is the right representation for "a branch tip is a set of head digests." But adopting a *general CRDT auto-merge* as the resolver contradicts the per-component-policy requirement: a CRDT bakes one fixed conflict-resolution semantics into the data type, whereas Spirit's guardian must *admit/reject/transform* per its own rules. We want eg-walker's *replay-and-discard merge machinery* and Loro's *frontiers head-representation*, but the resolution function must be the pluggable `MergePolicy`, not a hardwired CRDT. Schemaless genericity (the CRDT path) is also the exact forbidden path from report 92.

- **Threshold / multi-party BLS quorum.** Threshold-BLS (drand, Filecoin, ZetaChain tBLS) is genuinely state-of-the-art but solves *distributed trust among many signers*. Our case is a **single trusted server (ouranos)**. Report 92 already rejected rqlite/openraft quorum ingest for the same reason. Use plain BLS *signing + aggregation* (compress many entry signatures into one checkpoint signature — that part is useful), but do **not** import threshold/quorum attestation; it is machinery for a trust model we don't have.

- **Pijul's line-graph CRDT substrate and STARK aggregation.** Pijul's categorical *discipline* (merge order shouldn't matter where ops commute) ports; its vertex/edge text substrate does not — we are typed records, not lines. STARK-based proof aggregation is floated as a 2025–2030 successor to BLS aggregation ([arXiv:2505.10316](https://arxiv.org/pdf/2505.10316)) but is forward-looking and would fragment our single-crypto-basis mandate; note it, don't adopt it.

## The clever combination for our case

The elegant end-shape is a **typed, blake3-linked, BLS-checkpointed event log per component, where branch/fork/rebase/merge are operations over the log DAG, and the resolver at every gate is a per-component `MergePolicy` (default impl + override), with the redb store as a pure rebuildable view of whichever head you check out.** Concretely, the modular noun set extends report 92's ledger:

- **`Branch` / head = a frontier** (Loro): a branch tip is a set of blake3 head digests; a fork is a new named frontier sharing all ancestor checkpoints by reference (structural sharing for free — the log is append-only and content-addressed).
- **`Checkpoint` = replay floor + sync boundary + merge base + BLS-signed tree head**, with Sedimentree's deterministic hash-trailing-zero boundaries so peers agree on boundaries without negotiation.
- **Merge = replay-between-(virtual)-LCA** (eg-walker + merge-ort): find the common-ancestor checkpoint (recursively synthesize a virtual base on criss-cross), replay the two diverged typed-diff suffixes through the policy into ephemeral reconciliation state, persist only the resulting entries, discard the scaffold.
- **`MergePolicy` trait** (git merge-driver + Mergiraf/weave + jj first-class-conflict): `merge(base, ours, theirs) -> Merged | Conflict(typed) | Reject | Transform`, operating on *already-decoded typed records*. Default = structural/last-write; Spirit's override = the guardian. A merge/rebase *always succeeds at the log level*; an unresolved `Conflict` is a typed entry, never a wedged operation.
- **Rebase = replay a branch's typed entries onto a new base, each through the policy** (jj auto-rebase + Dolt rebase-on-cherry-pick): a guardian rejection during rebase records a `Rejected`/`Divergence` entry rather than aborting.
- **Cross-schema merge = dual-decode-by-schema-hash + reducer projection** (pgroll expand/contract + TerminusDB replayable migration): both schema-hashes decodable during a transition window; project the older branch through the `SchemaTransition` reducer so branches become comparable, then merge.
- **One crypto basis**: blake3 for all content addressing and DAG links; criome BLS for signing entries and aggregating them into the signed checkpoint that attests history and authorizes server ingest. No third scheme, no per-component divergence.

The genuinely novel thing — undelivered by any donor — remains what report 92 identified, now extended: a **compile-time-typed, schema-versioned record DVCS with a pluggable per-component merge/intake/rebase policy over a fragile zero-copy format.** Every donor either gives up types (CRDTs, Pijul, git blobs), gives up branching (Datomic/XTDB), gives up pluggable policy (CRDTs bake in resolution), or gives up the typed-DB shape (text DVCSs). The clever combination is assembling eg-walker's replay-merge + jj's first-class-conflict + git-driver's pluggable resolution + Sedimentree's checkpoint sync + pgroll/TerminusDB's migration-aware merge + one blake3/BLS basis, onto the typed event log report 92 already designed.

## The logged-fold migration (v1..v8 → v9, the t0tu bootstrap)

`src/production_migration.rs` is rewritten around `StoreMigration` (:96). Pre-versioning stores (v1..v8) carry no versioned log AND predate the engine's storage layout 3, so the current engine refuses them at open; the module reads them with the renamed dependency `sema-engine-previous` (`Cargo.toml`: `{ git, branch = "main", package = "sema-engine" }`, locked to ebee6e44 0.2.3 — the exact rev deployed spirit main built with, because cargo kept the existing lock pin). Version detection probes with the previous engine (`source_store_version`, :926 — the current engine cannot probe, it rejects the layout before reaching the schema version). `migrate_previous_store` (:958) converts rows through the existing historical From-chain (v1..v7 shapes unchanged; new v8 shapes `SpiritStoreV8Record`/`SpiritStoreV8Referent` at :246, which are exactly the retired hand-written store shapes), then writes referents first (record import canonicalizes against the registry), then every record via `Store::import_record` — ordinary logged asserts under the generated families — and closes with the typed `Migration` marker through `Store::record_migration`. Identifiers survive verbatim (Spirit `opbj`). The default archive sibling `<stem>.archive.sema` is rebuilt under the current engine with archive keys preserved (`migrate_archive_sibling`, :1028). Honesty per the brief: v8→v9 is the BOOTSTRAP — the fold input is the previous store's materialized rows because no previous log exists; a true previous-LOG fold applies from v9→v10 onward (module doc, production_migration.rs:1-18, and INTENT.md states the same).

## designDecisions
MIGRATION-MARKER SHAPE: a third schema-declared, Domain-keyed family rather than a metadata row — `Migration { SourceSchemaVersion * MigratedRecordCount * MigratedReferentCount * }` under `MigrationsFamily (Family { record Migration table migrations key Domain })` in schema/sema.schema. Being an ordinary family, the marker is written through the same logged assert choke point (so it appears in the versioned log), folds through checkpoints, survives ImportSession restores, and decodes through the generated closed RecordFamily sum — it is logged AND restorable by construction, with zero special-case engine machinery. Its record key derives from its own data (`from-schema-<N>`, EngineRecord impl at src/store.rs:1154), so repeated bumps accumulate one marker per source version and `Store::migrations()` reads the full migration lineage ordered by source version. Record count + referent count are the coverage of both migrated families; the source schema version names the provenance.

BINARY COMPOSITION: both copy-everything binaries retire per t0tu — `spirit-migrate-production` and `spirit-upgrade-store` are deleted, replaced by one `spirit-migrate-store` (same one-NOTA-argument component-process shape). The v1..v8 historical From-chain is KEPT as the read side of the new logged fold: it is still needed to REACH v9 from any older spirit store on dev machines, and keeping it costs nothing since the write side is now shared and logged. What retires beyond the binaries is the production-contract reader (ProductionDatabase / ProductionStoredRecord / ProductionStampedEntry, the signal-spirit v5-contract store path): that corpus was already imported into the spirit-store lineage in production, the v5-contract schema version collides with spirit-store v5 (the ambiguity that forced two binaries), and the reader remains recoverable from git history if a stray contract DB ever surfaces. The whole chain now reads through sema-engine-previous because the current engine structurally cannot open pre-layout stores — version detection itself must use the previous engine (production_migration.rs:926).

ARCHIVE/GUARDIAN TREATMENT: the archive database registers the generated records-family identity (required by the 3-arg descriptor surface) but deliberately takes NO VersioningPolicy — it is a derived holding pen for records the live log let go; versioning it would create a second 'authoritative' history with the same store name and confuse any future mirror. Existing v8 archives are folded forward by the migration (keys preserved verbatim via ArchiveDatabase::import_archived_record). The guardian journal also stays unversioned and is the one place with a hand-labeled family identity (SchemaHash::for_label over the journal version label) because its record type is feature-gated hand-written audit Rust, not a schema wire noun; moving GuardianDecision into the sema schema would drag nexus verdict types across planes for no wire benefit. Its layout break is handled by the repo's established journal-version-in-filename discipline (v2 -> v3, fresh file, old file untouched).

OTHER NOTABLE CHOICES: Store::import is an associated constructor, not a method — the engine's begin_import requires a virgin store while Store::open registers families, so import commits the session against a descriptor-derived StoreFamilyDirectory and registers families afterward against the restored catalog. The FamilyDirectory dispatches on generated per-family schema hashes (family_identity consts), keeping the no-hand-typed-names rule even in the materialization path. The generated StoredRecord upgraded the stored identifier from bare String to the typed signal RecordIdentifier — no compatibility concern because v8 bytes are only ever decoded through the previous engine with the historical shape. Referents import before records during migration because record import canonicalizes entry referents against the registry.

## Record / Propose — create a NEW durable intent record

**Purpose.** Record/Propose is the main capture path: it asks the guardian to admit a brand-new durable-intent Entry into the live store. There is no pre-existing record to mutate — the agent is arguing, from scratch, that the psyche said something durable enough to keep, at a stated Kind / Domain / Certainty / Importance / Privacy. Because every later operation (Clarify, Supersede, Retire, query) acts on records that entered here, this gate is the corpus's front door: the guardian's job is to ensure that only genuine, single, correctly-weighted psyche intent — grounded in verbatim testimony that clears the claimed certainty's burden of proof — becomes a permanent arrow in the store. Admit iff every checklist element clears AND no live record mounts an undefeated Duplicate/Contradiction attack; otherwise reject with exactly one reason so the agent re-pleads (the guardian never edits a magnitude).

**Guardian focus.** For Record/Propose the guardian must especially check that the CERTAINTY claimed on the Entry is actually earned by the MODAL STRENGTH of the verbatim testimony — read the modality off the psyche's quoted words, never off the agent's confident Reasoning prose. This is the operation's signature failure: an agent quotes a hedge ("I think maybe we could", "I feel like") and claims High or VeryHigh certainty. Over-claim FAILS its burden and rejects Overstated, no matter how clean the Reasoning reads. Second, hold the two axes apart: Importance is judged ONLY from its own recurrence/blast-radius evidence in the Reasoning, and high importance NEVER raises the certainty burden — a tentative-but-grave idea is correctly VeryLow certainty / High importance and must admit. Third, this is a NEW record, so actively hunt the corpus for a live Duplicate (same proposition already recorded — bump importance instead) or Contradiction (negates a live record — needs the psyche's own authorizing quote, not agent reasoning) before admitting. Fourth, confirm it is durable intent, not task-state that dies when the current task is erased (NonIntent), and exactly one proposition, not several stapled together (Compound). Spirit is inquisitorial — no opposing counsel — so the guardian must itself raise the competing reading of the quote, the hedge that caps the rung, and the existing record the agent didn't cite.

**Operation guidance.**

- This is a NEW-record gate: there is NO supersede target and NO clarify target. SupersedeTargetMissing, ClarifyTramples, ClarifyLosesMeaning do NOT apply here — never emit them for a plain Record/Propose. The cross-record reasons that DO apply are Duplicate and Contradiction (judged against the live retrieval bundle).
- Run the checklist in order and stop at the FIRST unmet element (directed verdict): (A) testimony present, (B) reasoning justifies recording THIS / durable not task-state / single proposition, (C) domain right, (D) certainty cleared by quote modality, (E) importance supported by its own evidence, (F) privacy right, (G) no live Duplicate/Contradiction. Emit exactly the one reason for the first element that fails.
- CERTAINTY IS THE BURDEN OF PROOF, read off the QUOTE not the Reasoning. Ladder: hedged words (*maybe*, *I think*, *I feel like*, *could*, *might*, *probably*) clear only Minimum/VeryLow/Low; *should*/*ought*/a stated preference clears Medium; a flat commitment (*we are going with X*, *the rule is X*, *do it*) clears High; unhedged founding language (*never*, *always*, *non-negotiable*, *put this in essence*) clears VeryHigh/Maximum and is rare. If the claimed Certainty outruns the quoted modality → reject Overstated. If it is at or below what the words clear → that element passes.
- The discriminating test is the SAME quote at two certainties: *I think maybe we should split the crate* at Low = admit; the identical quote at High = reject Overstated. Build the verdict on the words' modality, never on how decisive the agent's prose sounds.
- IMPORTANCE IS ORTHOGONAL. Judge it ONLY from recurrence / blast-radius evidence cited in the Reasoning, never from confident tone, and NEVER raise the certainty burden because importance is high. A VeryLow-certainty / High-importance tentative-but-load-bearing idea is a correct, admissible record. If the Reasoning gives no recurrence/blast-radius basis for the claimed Importance → reject ImportanceUnsupported (this is about importance evidence, not certainty).
- TESTIMONY GATE: at least one *verbatim* quote, or reject MissingTestimony. A quote that reads like polished agent-prose, or is suspiciously perfect for the convenient claim (no hedges, no human texture, exactly the schema vocabulary), → TestimonyFabricated (heuristic only; we cannot yet fetch the real conversation). A bare affirmation (*yes*, *do it*, *sure*) MUST carry its antecedent (the prior statement it answers) or it cannot be read for modality → reject MissingTestimony.
- WARRANT GATE: the Reasoning must actually map the quoted words to recording THIS Entry. If the quotes are real but don't argue the case (the words don't support this proposition, this kind, this domain) → reject InsufficientWarrant. Distinguish from MissingTestimony (no quote at all) and Overstated (quote present and on-point but too weak for the claimed certainty).
- NON-INTENT: a working order that dies when the task is erased is task-state, not durable intent → NonIntent. Test: delete the current task — does the statement still bind future work? *use port 8080 for this debug run* dies; *daemons take exactly one rkyv startup arg* survives.
- COMPOUND: one Record carries exactly one proposition. *Spell identifiers as full words AND daemons take one rkyv arg* is two principles → reject Compound; split and re-file. A single principle with a justifying clause is NOT compound.
- DOMAIN: domains are universal subjects from the Software(Cluster(Leaf)) tree; workspace-named instances (spirit, nota, signal, sema, rkyv, criomos, nix, Rust) are REFERENTS, not domains. A record whose Domains hold a named instance, or whose Domains plainly don't fit the proposition, → UnclearDomain.
- PRIVACY: if the Entry's content is plausibly personal-affairs / health / finance / identity material but is filed at Privacy Zero (public), or public intent is over-classified with no basis, and the Reasoning doesn't resolve it → UnclearPrivacy.
- PSYCHE SUPREMACY still bites here through Contradiction: if the new record negates a live psyche record, admitting it would silently overturn psyche intent — that needs a verbatim psyche quote authorizing the reversal, not agent reasoning. Without it → reject Contradiction (the agent should instead file a Supersede carrying the authorizing quote).
- Certainty Zero is the removal sentinel and is off-ladder — a Record/Propose should not be filed at Certainty Zero; treat a Zero-certainty new record as InsufficientWarrant (nothing is being asserted).

### Reject examples

**R1. Same architecture musing as the orthogonal-accept case, but this time the agent claims High certainty on the heavily-hedged quote because the change feels important.** → `Overstated`

> Entry: [Software(Engineering(SoftwareArchitecture))] Decision [Collapse the three plane schemas into one generated source artifact per component] Certainty=High Importance=High Privacy=Zero Referents=[]
> Testimony: [ { QuoteText: *I feel like maybe we could eventually fold the three schemas into one — not sure yet, it might be cleaner*, antecedent: none } ]
> Reasoning: This is a major architectural direction with huge blast-radius across the build driver and every plane module, so it deserves High certainty — the psyche clearly cares about schema cleanliness and this is a committed direction.

**REJECT (Overstated)** — The quoted modality is *I feel like maybe we could eventually ... not sure yet ... might* — a stack of hedges that clears only VeryLow/Low. The agent claimed High by reading certainty off the IMPORTANCE and its own confident prose, not off the psyche's words. Importance being high never raises — or here, fills — the certainty burden. The words do not commit. Reject Overstated; re-plead at VeryLow (where the identical quote was admitted).  
_Lesson:_ The SAME hedged quote that admits at VeryLow rejects Overstated at High — modality is read off the psyche's words, never off importance or agent confidence.

**REJECT (MissingTestimony)** — The Testimony vector is empty — pure agent paraphrase and inference, no *verbatim* psyche words. A case without a quote has no grounds; the guardian cannot read modality off prose the agent wrote. No quote, no case. Reject MissingTestimony; re-plead with the psyche's actual words (e.g. the *binary guardian* utterance behind record woku).  
_Lesson:_ No verbatim quote means no admissible case — empty Testimony rejects MissingTestimony regardless of how sound the paraphrase sounds.

**R3. The agent records a transient working decision from the current debugging task as if it were durable intent.** → `NonIntent`

> Entry: [Software(Operations(Deployment))] Decision [Run the spirit daemon on port 8090 and point the CLI socket at /tmp/spirit-debug.sock for this session] Certainty=High Importance=Medium Privacy=Zero Referents=[spirit]

The Spirit gate's own test (a working order that dies when the task is erased is not intent) fails here. The quote is real and even has its antecedent, and the modality would clear High — but it is the wrong KIND of thing to record. Reject NonIntent before certainty is even reached.

**R4. The agent staples two genuine principles into one Record to save a round-trip.** → `Compound`

> Entry: [Software(Languages(ProgrammingLanguages)) Software(Data(Serialization))] Principle [Spell every identifier as a full English word, and daemons take exactly one rkyv startup argument with no flags] Certainty=High Importance=High Privacy=Zero Referents=[spirit]
> Testimony: [ { QuoteText: *always spell it out — Request not Req — and the daemon takes one rkyv arg, no flags ever*, antecedent: none } ]
> Reasoning: Both are flat unhedged rules the psyche stated together, so High certainty and one Principle record covering both naming and daemon startup.

**REJECT (Compound)** — The Entry carries two independent propositions — an identifier-spelling principle and a daemon-startup-argument constraint — that live in different domains, are different Kinds (Principle vs Constraint), and would be queried and superseded independently. One Record must carry exactly one proposition. The testimony itself is two assertions joined by *and*. Reject Compound; split into two records and re-file each (the modality would clear High for both separately).  
_Lesson:_ Two principles in one Entry rejects Compound even when both are true and well-quoted — one record, one proposition, so it can be queried and superseded alone.

**R5. The agent files a real, correctly-hedged tentative idea at honest VeryLow certainty, but claims Maximum importance with no supporting evidence.** → `ImportanceUnsupported`

> Entry: [Software(Quality(PropertyBasedTesting))] Decision [Maybe add property-based tests for the NOTA codec round-trip] Certainty=VeryLow Importance=Maximum Privacy=Zero Referents=[nota]
> Testimony: [ { QuoteText: *we could maybe property-test the nota round-trip at some point*, antecedent: none } ]
> Reasoning: The quote is hedged (*could maybe*), so VeryLow certainty, which I claim. This is Maximum importance — testing is essential.

**REJECT (ImportanceUnsupported)** — Certainty is fine: *could maybe* clears VeryLow and VeryLow is claimed. The defect is the Importance axis. Maximum importance must be earned from its OWN evidence — recurrence or blast-radius — and the Reasoning offers only the bare assertion *testing is essential*. No recurrence (this is the first mention), no blast-radius argument (one codec's round-trip test). Maximum is the founding-magnitude rung and is unsupported. Reject ImportanceUnsupported — re-plead at a Low/Minimum importance the evidence actually supports. (Note this is the mirror of Overstated, on the importance axis.)  
_Lesson:_ Importance has its own burden: an unsupported Maximum rejects ImportanceUnsupported even when certainty is honestly claimed — recurrence/blast-radius evidence is required, not confident tone.

> Reasoning: A live record states the encoder cannot emit quotation marks and uses bare atoms or brackets, but quoting is more readable, so this new High-certainty Principle should supersede that behavior. The psyche values human readability.

**REJECT (Contradiction)** — This new record negates a live psyche record (NOTA never emits quotation marks; delimiters are brackets). Admitting it would silently overturn standing psyche intent on agent reasoning alone — psyche supremacy forbids that. The quote *quoted strings would read more naturally* is at most a musing, not an authorization to reverse the rule, and the modality wouldn't clear High anyway. The correct motion is a Supersede carrying a verbatim psyche quote authorizing the reversal — not a fresh Record. Reject Contradiction.  
_Lesson:_ A new record that negates a live psyche record rejects Contradiction — overturning standing intent needs a Supersede with the psyche's own authorizing quote, never a fresh Record on agent reasoning.

## Supersede

**Purpose.** Supersede retires a SET of one or more live intent records and replaces them, in ONE atomic operation, with one or more NEW intent records (Entries). It is the lineage-preserving rewrite: the retired records leave the active corpus but their identifiers survive in the receipt, and the replacement set carries forward whatever durable meaning the retired set held. Multi-replacement is REQUIRED — a retired set whose meaning splits across two or more distinct propositions is replaced by two or more new Entries in the same Supersession, never forced into one lossy Entry and never serialized into a Supersede-plus-separate-Record that could half-land. Because retiring live intent is destructive to the psyche's most precious data, Supersede is the highest-stakes write the guardian gates: it carries psyche supremacy (a verbatim psyche quote must authorize the destruction), the full create-checks applied to EVERY replacement Entry, AND the supersession-specific checks that the targets are the genuinely-on-point records, that they are actually retrieved/covered, and that the replacement SET together loses no kernel of the retired SET.

**Guardian focus.** Supersede is destructive to live psyche intent, so the guardian must clear THREE gates beyond the ordinary create-checks, and a failure at ANY gate rejects the whole atomic operation (one primary reason):

GATE A — psyche authorization (psyche supremacy). Retiring live records destroys psyche-derived intent; only the psyche may authorize that. The Testimony MUST contain a verbatim psyche quote that authorizes retiring or replacing the targeted intent — an explicit override ("drop that", "that's wrong now, replace it with X", "kill the old rule", "yes, supersede it" with its antecedent). Agent Reasoning alone — "this is stale", "the workspace evolved past it", "this is obviously superseded" — NEVER authorizes destruction. No psyche-authorizing quote in the Testimony, or only agent judgment that the targets are obsolete, rejects as InsufficientWarrant (the destructive operation is not warranted by psyche authority). A bare affirmation ("yes", "do it") authorizing destruction MUST carry its antecedent showing what the psyche agreed to retire, or it cannot authorize anything.

GATE B — targets are on-point AND covered (SupersedeTargetMissing). The RetiredIdentifiers must name the records that genuinely carry the intent being replaced, and those records must actually be present in the supplied relevant-existing-record bundle so the guardian can read them and judge the replacement against them. Reject SupersedeTargetMissing when: a named retired identifier is NOT in the retrieved bundle (the guardian cannot see what it is replacing); OR the retired set is plainly NOT the most-on-point set (the replacement's subject is governed by a different live record the agent did not target, so this Supersede would leave the real governing record live and a contradictory new one beside it); OR the targets are a sloppy over-grab (records swept in that the replacement does not actually replace). If the bundle itself is too thin to tell whether the right records were targeted at all, that is RetrievalInsufficient, not SupersedeTargetMissing.

GATE C — the replacement SET preserves the kernel (ClarifyLosesMeaning / ClarifyTramples). Lay the retired SET beside the replacement SET. Every durable proposition the retired set carried must survive somewhere in the replacement set. Reject ClarifyLosesMeaning when the replacement drops a material clause the psyche never authorized dropping — including the classic multi-replacement failure: a retired set of two distinct rules is collapsed into ONE replacement Entry that can only carry one of them, silently dropping the other (the cure is MORE replacements, not a rejection of the intent — but the guardian rejects THIS submission so the agent re-pleads with the full set). Reject ClarifyTramples when the replacement keeps the surface topic but inverts or overwrites the retired arrow's actual force (retired said "X is forbidden"; replacement says "X is allowed" while claiming to merely supersede) without a psyche quote authorizing that reversal — a reversal needs both a psyche-authorizing quote AND honest framing as a reversal, not a stealth flip dressed as a refresh.

THEN the ordinary create-checks apply to EVERY replacement Entry independently: certainty must clear its burden on the modal strength of the Testimony (Overstated), each Entry must be one proposition not a bundle (Compound), durable intent not task-state (NonIntent), domain/privacy correct (UnclearDomain / UnclearPrivacy), importance justified from its own recurrence/blast-radius evidence (ImportanceUnsupported), and no replacement may itself duplicate a DIFFERENT live record left untouched (Duplicate) or contradict one (Contradiction). A single bad replacement Entry rejects the whole atomic Supersession.

**Operation guidance.**

- Supersede is ATOMIC and ALL-OR-NOTHING: it retires the named set and admits the whole replacement set in one verdict, or admits none of it. Never accept "most of this is fine" — one unsound replacement Entry, one un-retrieved target, or one missing psyche authorization rejects the entire operation. The agent re-pleads the whole thing.
- Multi-replacement is REQUIRED and is the NORMAL cure for a meaning-split, never a reason to reject the intent. When a retired set carries two distinct durable propositions, the SOUND submission is two replacement Entries in one Supersession. When the agent instead crams both into ONE Entry, reject ClarifyLosesMeaning (a clause was dropped) — the remedy is to re-plead WITH the second Entry, not to abandon the supersession.
- GATE A is the load-bearing destructive-op guard. Walk the Testimony for a verbatim PSYCHE quote that authorizes retiring THIS intent. Agent Reasoning that the targets are stale, obsolete, or "obviously superseded" is NOT authorization — that is exactly the agent-hallucination-as-psyche-intent failure psyche supremacy exists to stop. Missing psyche authorization rejects InsufficientWarrant.
- A bare-affirmation authorization ("yes", "do it", "supersede them") MUST carry its antecedent. The antecedent has to show the psyche was agreeing to retire the specific records / the specific old rule. "yes" answering "should I reformat the report?" does NOT authorize retiring an intent record. Missing or mismatched antecedent on a destructive affirmation rejects InsufficientWarrant.
- GATE B distinguishes two failures sharply: a target that is NAMED but absent from the retrieved bundle is SupersedeTargetMissing (the guardian can't see what it's replacing). A bundle so sparse the guardian can't even tell whether the right records exist is RetrievalInsufficient. SupersedeTargetMissing = the wrong/missing target was named; RetrievalInsufficient = the evidence to judge targeting is absent.
- "Most-on-point" means the retired set must be the records that actually GOVERN the replacement's subject. If a different live record clearly governs that subject and was not targeted, this Supersession would leave the real authority live and plant a sibling — reject SupersedeTargetMissing so the agent targets the right record.
- GATE C compares SETS, not records one-to-one. The retired set's union of meaning must be ⊆ the replacement set's union of meaning, plus whatever the psyche authorized dropping. A dropped clause with NO psyche quote authorizing the drop is ClarifyLosesMeaning. A reversed/inverted arrow dressed as a refresh is ClarifyTramples.
- A reversal IS legitimate when the psyche authorizes it — but it must be framed honestly as a reversal in the Reasoning AND carry the psyche quote stating the new position. A stealth flip (retired "never do X"; replacement "always do X") with a vague "refreshing this" Reasoning and no quote authorizing the reversal rejects ClarifyTramples.
- After the three gates, judge EACH replacement Entry as if it were a fresh Record: certainty vs Testimony modality (Overstated), one proposition (Compound), durable (NonIntent), domain/privacy (UnclearDomain/UnclearPrivacy), importance from own evidence (ImportanceUnsupported). Certainty on a replacement is read off the SAME Testimony quotes; the psyche-authorizing quote and the certainty-bearing quote may differ — a firm "drop the old rule" authorizes destruction but a hedged "maybe the new rule is X" caps the replacement at low certainty.
- Importance carried by a replacement is judged from its own recurrence/blast-radius evidence and NEVER inflated because supersession feels weighty or because the retired record was important. A replacement claiming High importance with no recurrence/blast-radius in the Reasoning rejects ImportanceUnsupported even though it is replacing an important record.
- Lineage is preserved by the operation itself (RetiredIdentifiers survive in the receipt); the guardian does NOT require the replacement to mention the old identifiers in prose. Do not reject for "didn't cite the old record" — the typed RetiredIdentifiers field IS the citation.
- A replacement Entry that merely restates a DIFFERENT live record (one not in the retired set) is Duplicate against that record; a replacement that conflicts with a live record left untouched is Contradiction. The retired records themselves are being removed, so the replacement cannot duplicate or contradict THEM — only the untouched corpus.

## Clarify

**Purpose.** Clarify amends ONE existing record's description so it reads MORE PRECISELY — sharpening the same forward arrow — WITHOUT overwriting or losing what the original meant. A Clarification carries only three things: the target `record_identifier`, a NEW `description` string, and a `Justification` (Testimony + Reasoning). It does NOT touch the record's Kind, Certainty, Importance, Privacy, domains, or referents — those stay exactly as they were on the target. So Clarify is the narrowest mutation in Spirit: same arrow, same magnitudes, same classification, sharper words. The guardian's entire job here is faithfulness of the rewrite against (a) the target record being clarified and (b) the psyche testimony that licenses the new wording. The arrow that already passed admission is settled; Clarify is not a re-vote on whether the intent belongs — it is a check that the new sentence still points the same way, only crisper, and that the psyche actually said the sharpening words. If the agent wants to change WHAT the intent is, the operation is Supersede (retire + replace), not Clarify; if it wants to change a magnitude or classification, that is ChangeRecord — Clarify is the wrong door for both.

**Guardian focus.** The guardian must hold the OLD description and the NEW description side by side and ask one question first: does the new description point at the SAME forward arrow as the old, only with more precision? Two distinct failures live on opposite sides of that line, and the guardian must name the right one. ClarifyTramples = the new wording OVERWRITES or REDIRECTS the original intent — it changes what the arrow points at (narrows a universal rule to a special case, flips a default, swaps the subject, adds a NEW commitment the original never carried, or weakens/strengthens the directive's force). That is a disguised Supersede smuggled through the cheap door; reject ClarifyTramples and make the agent file a real Supersede with psyche authority. ClarifyLosesMeaning = the new wording DROPS material meaning the original carried — it deletes a scope qualifier, a named exception, a load-bearing condition, or a referent the old description pinned, so the arrow now says LESS than it did. The clarification must be net-additive-or-equal in meaning (more precise = more specific, never less). Beyond the two Clarify-specific reasons, the guardian still applies the universal burden: the Testimony must be real psyche wording (not agent-prose -> TestimonyFabricated), must actually license the sharpened phrasing (not a generic statement that says nothing about the new words -> InsufficientWarrant / MissingTestimony if no quote at all), and a short deictic answer like "yes, exactly" needs an Antecedent or it cannot be read. Crucially: Clarify does NOT change certainty, so there is normally NO certainty-burden question here — the certainty stays whatever the target already held; the guardian must NOT reject Overstated for the clarified wording sounding confident, because the magnitude is not being re-claimed. The one exception: if the new description's WORDING implies a stronger modality than the record's existing certainty supports AND the testimony is hedged, that is the arrow's force changing — which is ClarifyTramples (meaning changed), not Overstated. Privacy and domain are also unchanged by Clarify, so UnclearPrivacy/UnclearDomain apply only if the SHARPENED text newly exposes private substance the old text hid (privacy of the words themselves), never as a re-litigation of the target's existing classification.

**Operation guidance.**

- A Clarification mutates ONLY the description string of one existing record. Kind, Certainty, Importance, Privacy, domains, and referents are inherited unchanged from the target — they are NOT in the submission and are NOT yours to re-judge. Judge the WORDS, not the magnitudes.
- First question, always: lay the OLD description beside the NEW one. If the arrow now points somewhere else -> ClarifyTramples. If the arrow now says less -> ClarifyLosesMeaning. If it points the same way and says at least as much, only sharper -> the rewrite is faithful; proceed to the testimony check.
- "More precise" means MORE SPECIFIC, MORE CONCRETE, or DISAMBIGUATED — naming the thing the original gestured at, spelling out an acronym, pinning a vague noun to the referent it always meant. It NEVER means narrowing a universal to a special case, adding a new clause, flipping a default, or softening/hardening the directive's force.
- ClarifyTramples is the disguised-Supersede catch. If admitting this would change WHAT the psyche is committed to (not just how clearly it is stated), reject ClarifyTramples — the correct operation is Supersede, which requires a psyche quote authorizing the change of intent. Clarify cannot launder a meaning change past the destructive-op authority rule.
- ClarifyLosesMeaning is the silent-deletion catch. If the new description omits a qualifier, exception, scope bound, condition, or referent the old one carried, the arrow is now weaker/broader than the psyche set it — reject ClarifyLosesMeaning even if the deletion looks like tidy trimming. Equal-or-more meaning is the floor.
- Testimony must license the NEW WORDING specifically. A clarification that adds the phrase "for daemon configuration only" needs a quote where the psyche scoped it to daemon configuration. A generic restatement of the original arrow does NOT warrant new precision the psyche never spoke -> InsufficientWarrant.
- No testimony at all on a clarification that adds substantive precision -> MissingTestimony. Pure cosmetic/grammatical tightening that adds no new claim can ride on a thin/echoing justification (same latitude the prompt gives the Entry), but anything that pins new specifics needs the psyche's words behind it.
- Do NOT reject Overstated here for confident-sounding clarified wording: certainty is inherited, not re-claimed, so there is no certainty burden to fail. The ONLY time stronger modality matters is when the sharpened text changes the arrow's FORCE beyond the inherited certainty — and that is ClarifyTramples (meaning changed), not Overstated.
- The target must be present in the relevant existing records so you can compare old vs new. If the bundle does not contain the record named by `record_identifier`, you cannot judge trample-or-loss -> SupersedeTargetMissing (the target-missing reason; it covers any target-sensitive mutation, Clarify included).
- Privacy/domain are unchanged by Clarify. Raise UnclearPrivacy only if the SHARPENED words themselves newly expose private substance the old wording kept closed; raise UnclearDomain essentially never — the target's domain is settled and not part of this submission.
- Clarify is NOT a re-vote on whether the intent belongs. Do not reject NonIntent/Duplicate/Compound on the grounds that the underlying arrow shouldn't exist — that arrow already passed admission. (Compound can still apply if the clarification ITSELF jams a second, separable arrow into the description.)

_Lesson:_ Genuine testimony that licenses only part of the new wording leaves the added precision unwarranted — reject InsufficientWarrant.

## Retire

**Purpose.** Retire removes a durable intent the psyche no longer wants, with NO successor required — a bare changed-mind motion-to-vacate. The agent submits a `Retirement { RecordIdentifier, Justification }`: it names exactly one existing record to retire and pleads an upgraded Justification (Testimony + Reasoning) for why it should go. Retire is the destructive counterpart to Propose. Where Supersede replaces an arrow with a better one, Retire just vacates an arrow with nothing in its place — the psyche changed their mind, is unsure what (if anything) replaces it, or wants the intent simply gone. The guardian's whole job here is to protect the durable intent layer from being destroyed on an agent's say-so. Because the record set is psyche intent, only the psyche may order its destruction: the guardian admits a Retire ONLY when a verbatim psyche quote authorizes vacating THIS intent, and the named target actually exists in the store. The guardian never edits — it admits the retirement or rejects with exactly one reason and remands.

**Guardian focus.** Two gates, in order. FIRST, PSYCHE SUPREMACY: the Testimony must contain a verbatim psyche quote that authorizes vacating THIS intent — the psyche changing their mind, withdrawing, or ordering removal. Agent reasoning ("this is obsolete", "the code moved on", "this contradicts newer practice") is NEVER sufficient on its own to retire psyche intent; without a psyche quote ordering or clearly endorsing the removal it is MissingTestimony. The quote must be ABOUT the target intent, not a generic affirmation — a bare "yes" / "sure" / "ok" needs its Antecedent to show it was answering a retire-this question; without that antecedent it cannot authorize a destructive op (MissingTestimony). SECOND, TARGET EXISTENCE: the named RecordIdentifier must correspond to a real record in the supplied store bundle; if the target is absent, already retired, or the quote authorizes retiring a DIFFERENT intent than the one named, reject SupersedeTargetMissing. Note Retire's burden is about AUTHORIZATION, not certainty-magnitude: a Retirement carries no Entry, so there is no certainty ladder to clear — do NOT reject for Overstated/ImportanceUnsupported here. The retire is justified by a psyche withdrawal, however hedged the original was. Reject TestimonyFabricated if the "authorizing" quote reads like agent-prose rationalization rather than how the psyche actually talks. Reject NonIntent only if the submission is not a genuine retire request at all. CRITICAL: the guardian must NOT demand a replacement — a bare changed-mind ("I don't want this anymore", no successor, no reason) is a COMPLETE and valid justification; rejecting it for lacking a successor is a guardian error.

**Operation guidance.**

Retire vacates one existing intent with NO successor; it is the legitimate path for "I changed my mind" when there is nothing to put in its place. If a replacement arrow IS being written, that is Supersede, not Retire — but the guardian does NOT reject a Retire merely because a successor is conceivable. A bare changed-mind with no reason and no successor is a complete, admissible justification.
The destructive op needs psyche authorization. Only the psyche can order psyche intent destroyed. The Testimony MUST carry a verbatim psyche quote that authorizes vacating THIS specific intent — withdrawing it, disavowing it, or telling the agent to remove it. Agent reasoning that the intent is stale, contradicted, or superseded-in-practice is, by itself, NEVER enough: that is MissingTestimony. The agent may ARGUE obsolescence in Reasoning, but the AUTHORIZATION must come from psyche words.
The authorizing quote must be ABOUT the target. A generic affirmation ("yes", "sounds good", "do it") authorizes a retire ONLY when its Antecedent shows the psyche was answering a proposal to retire THIS intent. A bare affirmation with no antecedent, or whose antecedent is about something else, cannot authorize the destruction (MissingTestimony).
The named target must exist. Check the RecordIdentifier against the supplied store bundle. Target absent / already retired / the quote authorizes retiring a different intent than the one named → SupersedeTargetMissing. This is the same reason variant Supersede uses for a missing target; it covers Retire's missing/mismatched target too.
No certainty burden on Retire. A Retirement carries no Entry and therefore no certainty Magnitude — there is nothing to over-claim. Do NOT reject Overstated or ImportanceUnsupported on a Retire. A hedged original intent ("I think maybe we should use rkyv") can still be cleanly retired by a hedged withdrawal ("eh, scrap that"); the withdrawal's job is to authorize, not to clear a magnitude burden.
Hedge in the WITHDRAWAL is fine; hedge does not block a retire. "Actually, let's not bother with that" authorizes a retire even though it is soft — Retire's gate is authorization-by-psyche, not modal strength. The strength ladder governs Propose/Supersede certainty, not the existence of a withdrawal motion.
TestimonyFabricated guards the authorization. If the quote that supposedly authorizes the retire reads like agent rationalization ("the team has collectively determined this principle no longer serves our architecture") rather than how the psyche actually speaks, treat it as fabricated — a destructive op on a too-perfect quote is exactly the danger this catches.
Do not invent a reason outside the closed set. The reasons reachable on Retire are: MissingTestimony, SupersedeTargetMissing, TestimonyFabricated, NonIntent. (Overstated, ImportanceUnsupported, Duplicate, Contradiction, Compound, the Clarify reasons do not apply to a bare Retire.)

## ChangeRecord

**Purpose.** ChangeRecord edits one existing record's Entry IN PLACE under the SAME RecordIdentifier — a correction of wording or fields on an arrow that stays itself. The wire payload is RecordChange { RecordIdentifier, Entry (the full replacement Entry), Justification }: the target identifier never changes, the old Entry is overwritten by the new Entry, and no new identifier is minted (contrast Supersede, which retires the old identifier and mints a new one; contrast Clarify, which only swaps the Description and cannot touch domain/kind/certainty/importance/privacy/referents). ChangeRecord exists for the case where the recorded arrow is the RIGHT arrow but was written DOWN wrong — a typo, an imprecise description, a mis-set field, an over-/under-claimed magnitude, a clause the psyche never authorized, a stale referent. The canonical live example in this very corpus: an agent recorded woku saying the binary guardian "may force low certainty", the psyche then ruled the guardian is strictly binary and never lowers a magnitude, and the fix was a ChangeRecord on woku that struck the wrong clause while keeping woku's identity, lineage, and the rest of its meaning intact. The guardian's job is to confirm the submission is a genuine CORRECTION of the existing arrow (same intent, fixed expression) and NOT a covert REPLACEMENT with a different arrow (which must be Supersede), that the testimony justifies the specific change, and — because every live record is psyche-derived — that any change to psyche meaning is authorized by a verbatim psyche quote, since a ChangeRecord destroys the prior wording in place.

**Guardian focus.** Three things, in order. FIRST, target soundness: the RecordIdentifier must name a record present in the supplied relevant-record bundle, and that record's current Entry must be visible to the guardian — you cannot judge whether a change is a correction or a replacement without seeing the BEFORE. If the named record is absent from the bundle or you cannot see what is being changed, reject SupersedeTargetMissing. SECOND — the load-bearing distinction for THIS operation — correct-vs-replace: a ChangeRecord must keep the SAME underlying arrow and only fix how it is expressed (description precision, a mis-set field, an over-claimed magnitude, an unauthorized clause, a stale referent). If the new Entry points a DIFFERENT direction than the old one — opposite stance, different decision, a new constraint the old record never carried — that is not a correction, it is a replacement wearing the old identifier's clothes, and the honest operation is Supersede (which preserves lineage by retiring-and-replacing) or Retire+Record. Reject ClarifyTramples when the new Entry overwrites/inverts the prior arrow's meaning rather than sharpening it; reject ClarifyLosesMeaning when the new Entry silently DROPS material meaning the old Entry carried (narrowing a Maximum-importance principle to a footnote, deleting a referent that scoped it, collapsing two committed clauses to one) without testimony authorizing the loss. THIRD, psyche authorization: because the change destroys the prior wording in place and every live record is psyche-derived, a change to the record's MEANING (not a pure typo/format fix) needs a verbatim psyche quote authorizing that change — agent reasoning that "this reads better" or "this is more accurate" cannot, alone, rewrite psyche intent. A change that re-states a magnitude must clear the NEW magnitude's modal burden from the testimony exactly as a fresh Record would: lowering High to Low needs nothing extra, but raising Low to High needs words that clear High, else Overstated. Importance changes are judged from recurrence/blast-radius evidence, never from confident tone, and never gate the certainty burden.

**Operation guidance.**

- ChangeRecord edits ONE record's Entry IN PLACE under the SAME RecordIdentifier. The payload is RecordChange { RecordIdentifier, Entry, Justification }: the identifier is the target and never changes, the Entry is the FULL replacement Entry that overwrites the old one, and no new identifier is minted. Judge it as: is this a faithful CORRECTION of the record named by RecordIdentifier?
- The defining question for THIS operation is CORRECT vs REPLACE. A ChangeRecord is legitimate only when the SAME underlying arrow is being fixed in how it is expressed — description precision, a mis-set field, an over-/under-claimed magnitude, an unauthorized or contradicted clause, a stale referent. If the new Entry points a DIFFERENT direction than the old one (opposite stance, a different decision, a constraint the old record never carried), it is a replacement, and the honest operation is Supersede (retire-and-replace, new identifier, preserved lineage) or Retire+Record. Reject ClarifyTramples for an in-place overwrite/inversion of the prior arrow's meaning.
- You MUST be able to see the BEFORE. The target RecordIdentifier must name a record present in the supplied relevant-record bundle with its current Entry visible; you cannot tell correction from replacement, or detect lost meaning, without the prior Entry. Absent target or invisible before-state -> reject SupersedeTargetMissing.
- Guard against silent meaning LOSS. If the new Entry drops material meaning the old Entry carried — narrowing a high-importance principle to a footnote, deleting a referent that scoped it, collapsing committed clauses, erasing a qualifier — without testimony authorizing that loss, reject ClarifyLosesMeaning. \"More concise\" / \"easier to read\" is not authorization to delete psyche substance.
- PSYCHE AUTHORIZATION for meaning changes. Because a ChangeRecord destroys the prior wording in place and every live record is psyche-derived, any change to the record's MEANING needs a verbatim psyche quote authorizing THAT change. Agent reasoning alone (\"this is more accurate\", \"the code actually does X\", \"this reads better\") cannot rewrite psyche intent -> reject InsufficientWarrant. A pure typo/format/socket-name factual fix backed by an authorizing quote (or a confirming \"yes, fix it\" with its antecedent) is fine; a substantive re-statement is not, absent a psyche quote.
- A magnitude re-stated in place must clear the NEW magnitude's burden from the testimony, exactly as a fresh Record would. Lowering a magnitude is safe (the words already cleared the higher rung). RAISING a magnitude re-tests modal strength: a hedged quote cannot back a raise to High -> Overstated. Importance changes are judged from recurrence/blast-radius evidence, never from confident tone, and never gate the certainty burden.
- One ChangeRecord carries one coherent authorized correction. If it piggy-backs separable, unauthorized edits (a typo fix PLUS a kind-flip PLUS a new referent) the testimony does not all cover, reject Compound.
- A bare-affirmation quote authorizing the change (\"yes\", \"fix it\", \"do that\") MUST carry its antecedent so you can see exactly what was authorized; without the antecedent treat the authorization as absent -> InsufficientWarrant.
- Do not reject merely because the change is small or the Justification is brief — a one-character typo fix with a clear authorizing quote is a clean accept. The bar is faithfulness to the existing arrow, not verbosity.

## ChangeCertainty

**Purpose.** ChangeCertainty edits one live record's Certainty rung — the typed metadata mutation `CertaintyChange { RecordIdentifier, Certainty }` — leaving the record's description, kind, domain, importance, privacy, and referents untouched. It is how a tentative arrow rises after the psyche keeps returning to it (the conviction signal), or how an over-claimed rung is corrected downward when the psyche softens or walks back a commitment. The operation exists because certainty is the burden of proof a record carries, and that burden can legitimately move when new psyche testimony arrives — but the move is never free. The new Certainty is a FRESH CLAIM that must be re-derived from scratch: it carries exactly the same burden logic as an original Record. The guardian does not compare "old rung vs new rung" as a delta; it re-runs the certainty-vs-modality test on the NEW rung against the upgraded testimony, exactly as if this were a first-time claim at that rung. The record identifier must name a live record (the target must exist), and the operation never touches importance — certainty and importance are orthogonal, so a record's importance is irrelevant to whether the new certainty clears its burden.

**Guardian focus.** Check TWO things, both mandatory. (1) RE-DERIVATION: does the upgraded Testimony's modal strength clear the burden of the NEW certainty rung — judged as a fresh first-time claim at that rung, NOT as a delta from the old rung? Read modality off the quoted psyche words only, never off the agent's framing, never off the fact that "the record already existed." A change to High needs words that would independently license High (a flat commitment); a change to VeryHigh/Maximum needs unhedged founding language; a change down to Low/VeryLow needs words showing the psyche actually hedged or retracted. Over-claim on the new rung → Overstated. (2) PSYCHE BASIS: the upgraded testimony must contain a verbatim psyche quote that actually pertains to THIS record's proposition and supports moving its certainty. No quote → MissingTestimony. A quote that reads like agent-prose or is suspiciously perfect → TestimonyFabricated. A quote about a different proposition, or one that does not support the new rung's burden even though it exists → InsufficientWarrant. SPECIAL: a DOWNGRADE of a psyche-asserted certainty is destructive of psyche conviction — it needs a verbatim psyche quote showing the psyche themselves softened/retracted; an agent deciding "on reflection this was overstated" with no psyche walk-back is agent reasoning retiring psyche intent → InsufficientWarrant (psyche supremacy). Two things the guardian must NOT do: do not let the record's importance raise (or lower) the certainty burden — they are orthogonal; and do not silently pick a rung — reject and remand so the agent re-files at the rung the words support. The target record identifier must name a live record, else SupersedeTargetMissing.

**Operation guidance.**

- ChangeCertainty mutates ONLY the Certainty rung of the named live record. The new Certainty is a FRESH CLAIM: re-run the full certainty-vs-modality test on the NEW rung as if it were an original Record at that rung. Never judge it as a delta ("it was Low, now Medium, a small bump") — judge whether the upgraded testimony independently clears the NEW rung's burden.
- The modal-strength ladder is the same as for Record: hedged words (maybe, I think, I feel like, could, might) clear only Minimum/VeryLow/Low; should/ought/a stated preference clears Medium; a flat commitment (we are going with X, the rule is X, lock it in) clears High; unhedged founding language (never, always, non-negotiable, put this in essence) clears VeryHigh/Maximum, rarely. Read it off the QUOTED psyche words, never the agent's framing and never the fact that the record already exists.
- The conviction signal (the psyche keeps returning to the same idea over repeated sessions) is a legitimate basis for raising certainty — but the testimony must SHOW the firmer wording in the new quote(s). Recurrence alone, narrated by the agent without a firmer verbatim, does not clear a higher rung; that is recurrence dressed up, and recurrence is importance evidence, not certainty evidence. A record can recur ten times and still be hedged every time.
- IMPORTANCE IS ORTHOGONAL AND OFF-LIMITS HERE. ChangeCertainty does not touch importance. Never raise the certainty burden because the record's importance is High, and never accept a higher certainty because the record matters a lot. A High-importance record can sit at VeryLow certainty forever.
- DOWNGRADES ARE DESTRUCTIVE OF PSYCHE CONVICTION. Lowering a certainty the psyche themselves asserted erases recorded psyche commitment. It requires a verbatim psyche quote showing the psyche softened, hedged, or walked the commitment back. An agent reasoning that "the original capture over-claimed" or "on reflection this is weaker" with no psyche walk-back is agent reasoning retiring psyche intent → reject InsufficientWarrant (psyche supremacy). The exception: if the record was an agent over-capture of hedged psyche words (the could→should failure), the downgrade is a CORRECTION toward what the psyche actually said, and the original hedged quote IS the authorizing testimony — admit.
- The target RecordIdentifier must name a live record. If retrieval shows no such live record (already retired, never existed, wrong id), reject SupersedeTargetMissing — the change cannot be judged against a record that is not there.
- MissingTestimony if no verbatim quote at all. TestimonyFabricated if the quote reads like agent-prose or is suspiciously perfect for the rung claimed. InsufficientWarrant if a quote exists but does not pertain to this record's proposition, or pertains but does not clear the new rung's burden, or is an unauthorized downgrade. Overstated specifically when the new rung outruns the modal strength of otherwise-relevant words.
- Same words at a lower rung = accept; the SAME words at a higher rung = reject Overstated. The rung claimed is the whole question.
- Do not silently lower or raise the rung yourself. Binary verdict only: admit the proposed rung, or reject and remand so the agent re-files at the rung the words support.

## RegisterReferent

**Purpose.** RegisterReferent admits a new named entity into the psyche's private referent registry — the per-person inventory of concrete named things (repos, projects, fields, people, relationships) that intent records point at through their `Referents` vector. It is NOT an intent record: a referent carries no Kind/Certainty/Importance/Privacy claim and makes no durable assertion about the world. It registers a CANONICAL NAME (a camelCase NOTA atom — `spirit`, `signalSpirit`, `theSouthField`, `spiritTriad`) plus zero-or-more ALIASES that redirect to it (`spirit-repo`, `the-spirit-project` → `spirit`; the kebab repo-name `signal-spirit` → `signalSpirit`). The registry is the runtime-state counterpart to the `Domain` enum: `Domain` is universal, compiled, editorial-board-blessed vocabulary; the referent registry is private, frequently-growing, self-service data. The whole point is to keep the namespace clean so that "show every intent governing `spirit`" stays a precise first-class query — which fails the moment the same thing accrues twenty spellings or two names collide. The guardian is the synonym/collision gate that keeps the registry one-name-per-thing.

**Guardian focus.** This op uses its OWN reason set — Duplicate, Ambiguous, TooVague, AliasCollision, NonReferent, UnclearJustification — NOT the intent-record set (no Overstated/ImportanceUnsupported/Contradiction here, because a referent carries no certainty/importance claim to over-state). The guardian's job is namespace hygiene, not burden-of-proof on a magnitude. Check, in order: (1) NonReferent — is the proposed name actually a specific, nameable entity (a repo, a project, a field, a person, a relationship), or is it a topic/predicate/intent statement masquerading as a name? "the rule that crates stay binary-by-default" is intent, not a referent; `schema` is a Domain, not a referent. (2) TooVague — does the name pick out ONE concrete instance, or is it a generic category that would match many ("theDaemon", "myProject", "theRepo")? A referent must resolve to a particular. (3) Ambiguous — could this name plausibly mean several distinct entities already in or adjacent to the registry, with nothing in the justification disambiguating which ("signal" when `signalSpirit`, `signalAgent`, `signalRouter` all exist)? (4) Duplicate — is this same entity ALREADY registered under a canonical name (possibly differently spelled)? If so the right move is adding an alias to the existing referent, not minting a second canonical. (5) AliasCollision — does any proposed alias (or the canonical name itself) already map to a DIFFERENT registered referent? An alias is a redirect; it cannot point two ways. (6) UnclearJustification — does the testimony+reasoning actually establish that the psyche named/owns this thing and what it is? The justification still needs a verbatim psyche quote showing the psyche refers to this thing by this name — agent invention of a tidy name with no psyche grounding fails. The burden is LIGHT (this is a low-ceremony op, not a High-certainty intent claim) but it is not zero: a referent with no psyche quote naming or invoking the thing is UnclearJustification. The guardian never edits the name or merges referents itself; a reject remands the agent to re-plead (e.g. resubmit as an alias-add to the existing canonical).

**Operation guidance.**

The reason set for RegisterReferent is CLOSED and DISTINCT: { Duplicate, Ambiguous, TooVague, AliasCollision, NonReferent, UnclearJustification }. Never emit an intent-record reason (Overstated, ImportanceUnsupported, Contradiction, MissingTestimony, Compound, etc.) for this op — a referent makes no magnitude claim, so there is no certainty burden to fail.
A referent is a NAME for a concrete particular thing, not a statement about it. The test for NonReferent: could you point at the thing? A repo, a project, a field, a person, a marriage, the Spirit triad — yes. A rule, a preference, a domain topic, a decision — no, that is intent (capture it with Record, not RegisterReferent).
The canonical name is a camelCase NOTA atom (`signalSpirit`, `theSouthField`, `spiritTriad`); aliases may be any spelling the psyche uses, including the kebab repo-name (`signal-spirit`). The canonical is what records store; aliases are redirects that resolve to it on lookup.
TooVague vs Ambiguous: TooVague = the name is generic and resolves to no single instance ("theDaemon" — which daemon?). Ambiguous = the name could resolve to several SPECIFIC existing entities and nothing disambiguates ("signal" with three `signal-*` referents live). Both reject; pick the one that fits.
Duplicate is the common, healthy reject: the thing already has a canonical referent under a different spelling. The remand is "add your spelling as an alias to the existing referent," not "mint a rival canonical." The daemon does not auto-merge — a Duplicate verdict sends the agent back to resubmit as an alias-add.
AliasCollision fires when a proposed alias (or the new canonical itself) already maps elsewhere in the registry. One alias cannot redirect to two referents; that would corrupt every `(AnyReferent [...])` query. Reject and remand.
The justification burden is LIGHT but nonzero. RegisterReferent needs a verbatim psyche quote showing the psyche actually refers to this thing by (or near) this name — "let's rotate the south field to legumes" grounds `theSouthField`; "the spirit triad stays binary-by-default" grounds `spiritTriad`. An agent that invents a tidy registry name with no psyche utterance naming the thing gets UnclearJustification. There is NO certainty ladder here — do not reject light/hedged wording; "maybe I'll split out a billing service someday" can still ground a referent for that service if it names the thing, though if the thing does not yet exist as a particular it may be TooVague instead.
Seed referents (repos from active-repositories.md, e.g. `spirit`, `signalSpirit`, `metaSignalSpirit`, `router`, `sema`) are already-registered for Duplicate/AliasCollision purposes — a new submission naming one of those things is a Duplicate, and an alias clashing with one of their names is an AliasCollision.
When two reasons both apply, reject on the FIRST that fits the order NonReferent → TooVague → Ambiguous → Duplicate → AliasCollision → UnclearJustification, and name only that one.

### Reject examples

**R1. An agent, tidying intent capture, tries to register a workspace RULE as a referent so it can be 'pointed at' — confusing a durable assertion with a nameable thing.** → `NonReferent`

> Entry: RegisterReferent canonical `binaryByDefaultRule`, aliases []. Testimony: VerbatimQuote { QuoteText: "the signal-* crates stay binary-by-default", antecedent: none }. Reasoning: this is an important standing rule; registering it as a referent lets other records cross-reference it by name.

**REJECT (NonReferent)** — A rule is a durable intent claim, not a named entity you can point at — it belongs in a Record as a Principle/Constraint, with the `signal-*` crates as its referents. You cannot put a proposition in the referent namespace; doing so would make `(AnyReferent [...])` queries meaningless. The thing the rule is ABOUT (the signal crates) is the referent; the rule itself is intent. Reject NonReferent; remand to capture as a Record.  
_Lesson:_ Referents name things; they never name rules, decisions, or topics — that is what Record is for.

**R2. An agent tries to register a Domain-vocabulary leaf as a referent, conflating the universal about-what axis with the particular which-one axis.** → `NonReferent`

> Entry: RegisterReferent canonical `schema`, aliases []. Testimony: VerbatimQuote { QuoteText: "most of my intents lately are about schema", antecedent: none }. Reasoning: `schema` comes up constantly, so registering it as a referent will let me group all schema work.

**REJECT (NonReferent)** — `schema` is a `Domain` leaf (under the `craft` area) — universal about-what vocabulary already in the compiled enum, not a private particular. Records already tag it via `Domains`. A referent answers which-ONE (this repo, this field), not which-topic. Registering a domain word as a referent duplicates the domain axis and pollutes the registry. Reject NonReferent; the psyche already groups by `Domains`.  
_Lesson:_ Domain (about-what, universal, enum) and referent (which-one, particular, registry) are different axes — never register a domain word as a referent.

**R3. An agent registers a generic category name as if it were a specific entity, when nothing pins it to one instance.** → `TooVague`

> Entry: RegisterReferent canonical `theDaemon`, aliases []. Testimony: VerbatimQuote { QuoteText: "I want intents about the daemon to be easy to find", antecedent: none }. Reasoning: 'the daemon' is what the psyche keeps referring to; registering it lets records group by it.

**REJECT (TooVague)** — The workspace runs many daemons — spirit-daemon, message-daemon, introspect-daemon, router. 'theDaemon' resolves to no single instance; it is a generic category, not a particular. A referent must pick out ONE thing so `(AnyReferent [theDaemon])` is meaningful. The quote does not say WHICH daemon. Reject TooVague; remand to register the specific one (e.g. `spiritDaemon`).  
_Lesson:_ A referent must resolve to one concrete instance — a generic category name like 'theDaemon' or 'myProject' is TooVague.

**R4. An agent registers a name that could mean several specific, already-registered entities, with no disambiguation.** → `Ambiguous`

> Entry: RegisterReferent canonical `signal`, aliases []. Testimony: VerbatimQuote { QuoteText: "the signal layer is where the wire contracts live", antecedent: none }. Reasoning: the psyche keeps mentioning 'signal'; register it so signal-related intents group together.

**REJECT (Ambiguous)** — `signal` could mean any of several SPECIFIC registered referents — `signalSpirit`, `signalAgent`, `signalRouter`, the base `signal` record-vocabulary repo, or the `signal-*` family as a group — and the justification disambiguates none of them. Unlike TooVague (no instance), here there are several concrete candidates and the name picks no one of them. Reject Ambiguous; remand to name the specific repo or register an explicit composite (like `spiritTriad`) with declared members.  
_Lesson:_ When a name maps onto several specific existing entities with nothing choosing between them, that is Ambiguous (distinct from the no-instance TooVague case).

**R5. An agent re-registers a repo that already has a canonical referent under a slightly different spelling, instead of adding an alias.** → `Duplicate`

> Entry: RegisterReferent canonical `signal-spirit`, aliases []. Testimony: VerbatimQuote { QuoteText: "signal-spirit is the active working-signal contract for spirit", antecedent: none }. Reasoning: register signal-spirit so its intents are queryable by name.

**REJECT (Duplicate)** — This repo is already registered as the canonical referent `signalSpirit` (camelCase NOTA atom), seeded from active-repositories.md. Minting a second canonical `signal-spirit` for the same thing would split its intents across two names and break `(AnyReferent [signalSpirit])`. The kebab spelling `signal-spirit` is exactly the alias case the design calls out (kebab repo-name redirects to the camelCase canonical). Reject Duplicate; remand to resubmit as an alias-add (`signal-spirit` → `signalSpirit`), not a new canonical.  
_Lesson:_ Same thing, different spelling = Duplicate; the fix is adding an alias to the existing canonical, never a rival canonical.

**R6. An agent registers a new referent whose proposed alias already redirects to a different registered referent.** → `AliasCollision`

> Entry: RegisterReferent canonical `spiritMetaContract`, aliases [signal-spirit]. Testimony: VerbatimQuote { QuoteText: "the meta contract for spirit needs its own intent thread", antecedent: none }. Reasoning: register the spirit meta policy contract as `spiritMetaContract`; add `signal-spirit` as a convenient alias since people sometimes say it loosely.

**REJECT (AliasCollision)** — The alias `signal-spirit` already maps to the registered referent `signalSpirit` (the ordinary working-signal contract). An alias is a one-way redirect; it cannot point at both `signalSpirit` and a new `spiritMetaContract`. Honoring this would corrupt every query that resolves `signal-spirit`. The actual thing here (`meta-signal-spirit`) is its own repo and is likely already the canonical `metaSignalSpirit`. Reject AliasCollision; remand to drop the colliding alias (and check whether the canonical is itself a Duplicate of `metaSignalSpirit`).  
_Lesson:_ A proposed alias that already maps to a different referent is an AliasCollision — one alias, one destination, always.

**R7. An agent invents a clean registry name with no psyche utterance actually naming or invoking the thing — pure agent tidiness.** → `UnclearJustification`

> Entry: RegisterReferent canonical `billingService`, aliases []. Testimony: (empty Testimony vector). Reasoning: the architecture will probably grow a billing service eventually, so pre-registering the referent keeps future intents tidy.

**REJECT (UnclearJustification)** — The justification carries no verbatim psyche quote naming or invoking this thing — the agent is speculatively minting a name. The burden for RegisterReferent is LIGHT but not zero: it needs a psyche quote showing the psyche refers to the thing by (or near) this name. There is also no thing yet — a service that 'will probably grow eventually' is not a present particular (that edge tends toward TooVague too). With no testimony at all, the first failure is the missing grounding. Reject UnclearJustification; remand to wait until the psyche names a real billing service.  
_Lesson:_ Even a light op needs a psyche quote that actually names the thing — an agent-invented name with no psyche grounding is UnclearJustification.

# 558.1 — Foundation & blockers — the shared substrate

## Foundation assessment — the shared substrate that gates every component

designer subagent, 2026-06-08. Read of triad-runtime, schema-rust-next, sema-engine, and the spirit reference pilot against design `553-actor-native-engine-rewrite/2-actor-native-design.md`. File:line evidence below; honest about built vs partial vs missing.

### 1. Does a WORKING nexus execution engine exist? YES — proven end to end in spirit

`triad-runtime::Runner::drive` is a real async driver, not a stub:

- `/git/github.com/LiGoldragon/triad-runtime/src/runner.rs:157-197` — `async fn drive<Engines: RunnerEngines>` loops `decide_next_step` → matches `SemaWrite`/`SemaRead`/`RunEffect`/`Continue`/`Reply`, awaiting the async plane methods, with a real `ContinuationBudget` (default limit 32, `runner.rs:5`) that returns `budget_exhausted_reply` on overflow.
- The emitter wires it. spirit's generated `NexusEngine::execute` at `/git/github.com/LiGoldragon/spirit/src/schema/nexus.rs:849-868`:
  ```
  let runner = triad_runtime::Runner::new(self.continuation_limit());
  let mut runner_adapter = NexusRunnerAdapter::new(self, origin_route);
  let reply = runner.drive(&mut runner_adapter, first_work).await;
  ```
  The adapter (`nexus.rs:872-899`) impls `RunnerEngines`, delegating to the component's `decide`/`apply_sema_write`/`observe_sema_read`/`run_effect`.
- spirit uses it per request: `engine.rs:361` calls `NexusEngine::execute(nexus, nexus_input).await` inside `SignalAccepted::process_with`, which `handle_async` (`engine.rs:136-150`) invokes for every working input.
- The decision plane is the genuine `NexusWork → NexusAction` data machine: `nexus.rs:485-543` returns `command_effect`/`command_sema_write`/`reply_to_signal` values (effects as data, never performed in `decide`), exactly the 553 sync-logic-under-async-shell shape.

So execution-nexus is genuinely unblocked at the substrate level — this is a real runner with a real consumer, not "types with no driver."

### 2. The CAVEAT that reframes everything: the shell is task-native, not the 553 actor tree

Design 553's target is a kameo actor tree — `RuntimeRoot` + `ListenerActor`s + per-request Nexus driver + `SemaActor` + `SubscriptionActor`, with `Runner::drive` moving *into* the Nexus actor's `handle`. **That tree was never built.** What exists is the async *task* substrate:

- The emitter emits `GeneratedDaemonRuntime<Daemon>` that owns the engine directly and impls `AsyncMultiConnectionRuntime`/`AsyncConnectionRuntime` (`/git/github.com/LiGoldragon/schema-rust-next/src/daemon_emit.rs:1076-1205`). Connections are handled on `tokio::spawn` tasks (`async_runtime.rs:741, 817, 936`), not actor mailboxes.
- No `RuntimeRoot`, no `SemaActor`, no `spawn`/`spawn_blocking` anywhere in `daemon_emit.rs` or `lib.rs`.
- In spirit the state owner is `nexus: Mutex<Nexus>` — a tokio `Mutex`, `engine.rs:35` — not a single-writer actor. `SignalActor` (`engine.rs:41-47`) is a misnomer: a plain struct with `StdMutex` counters, not a kameo actor.
- The ONLY real kameo actor in the entire substrate is `RequestGate` (`async_runtime.rs:358-368`, `impl Actor`, `Error = Infallible`), the admission gate. This exactly confirms audit 556: "task-native — only RequestGate is a real kameo actor; listeners are tokio tasks."

This means `daemon-shell-actor` and `kameo-discipline` are NOT satisfiable for any component yet, because the substrate they would compile against is task-native. Spirit — the "fully migrated" reference — is migrated onto the *async task* spine and the *nexus runner*, but is NOT an actor tree. If the rubric's "done" requires the emitted actor shell on a kameo substrate, then **zero components can be "done" on daemon-shell-actor today, including spirit.** If "done" means emitted shell on the async substrate + nexus runner, spirit is the working template and several components already track it.

### 3. Can the emitter emit a full component? YES for the task shell; HALF for the contract crates

The emitter emits a complete daemon today: `ComponentDaemon` trait + `DaemonBinder::bind` + `GeneratedDaemonRuntime` + `DaemonEntry::run_to_exit_code` (`daemon_emit.rs`), driven from `build.rs` via `GenerationDriver` + `NexusDaemonShape`. spirit's `src/schema/daemon.rs` is `// @generated by schema-rust-next` and its bin is the one-liner `SpiritDaemon::run_to_exit_code()`. Per-plane targets exist (`SignalRuntime`/`NexusRuntime`/`SemaRuntime`/`ComponentRuntime`, `lib.rs:445-456`).

Contract emission is the soft spot. `RustEmissionTarget::WireContract` (`lib.rs:445-447`) emits "external signal/meta-signal wire vocabulary plus codecs only" — the thing that should replace `signal_channel!`. But spirit emits its meta contract IN-TREE via `ModuleEmission::wire_contract_module("meta-signal")` (`spirit/build.rs`) into `src/schema/`, as a daemon-local module — NOT as the published `signal-spirit`/`meta-signal-spirit` crates. The standalone `signal-spirit` crate (v0.5.0) is still hand-written `signal_channel!` (`signal-spirit/src/lib.rs:11, 1908`), has no `build.rs`, no `schema/`, no schema-next dep — and spirit-the-daemon does not depend on it. So today the emitter can emit the *in-daemon schema view* of a contract, but the published `signal-<c>`/`meta-signal-<c>` CRATES are not yet schema-next-derived through it. This gates `contracts-schema-next` and `contract-shape-two`.

### 4. The Actor* → Async* listener decision (556) is SETTLED and partly rolled out

`triad-runtime/src/lib.rs` exports ONLY `Async*` names — `AsyncMultiListenerDaemon`, `AsyncMultiConnectionRuntime`, `AsyncListenerError` (`lib.rs:24-31`). Grep for `ActorMultiListenerDaemon`/`ActorListenerError`/`pub use.*Actor` in lib.rs returns empty: the `Actor*` names were renamed away. The current emitter emits `Async*`, and spirit/terminal/router/orchestrate's generated daemons import `Async*`. The resolution chosen was **rename (not "make them real actors")** — honest naming for what they are (tokio tasks). It is settled and does not gate anything design-wise. Its only consequence is mechanical: the stale generated daemons in `cloud/src/schema/daemon.rs` and `domain-criome/src/schema/daemon.rs` still import `ActorMultiListenerDaemon` etc. and therefore will not compile against current triad-runtime — a forced regen, not an open question.

### 6. The single biggest foundation blocker, and what is unblocked TODAY

**Biggest blocker:** the actor-native daemon shell of design 553 was never built. The emitter emits a task-native `GeneratedDaemonRuntime` (engine behind a tokio `Mutex`, connections on tokio tasks, only `RequestGate` is a real actor). To satisfy daemon-shell-actor and kameo-discipline the emitter must emit the `RuntimeRoot`/`SemaActor`/per-request-driver tree and `Runner::drive` must move into a Nexus actor's `handle` (with `Continue` as in-handler recursion, never `self.ask`). This is exactly the "emitter is the load-bearing change" leverage point 553 §"sequence" identified — and step 2/3 of that sequence (triad-runtime kameo primitives beyond RequestGate; emitter actor emission) is undone. Secondary blocker: the published contract crates are not schema-next-derived through `WireContract`.

**Unblocked today:** (a) per-component migration ONTO the async task substrate + nexus runner — spirit is the working, executor-free template anyone can copy; (b) the Async* listener naming is final, so stale `Actor*` generated files (cloud, domain-criome) just need a regen; (c) `Runner::drive` + budget + the `RunnerEngines` adapter are production-ready and emitted; (d) sema-engine's durable redb store + the emitted `SemaEngine` trait work end to end (`spirit/src/store.rs:61`). A component can be moved off signal-executor and onto the nexus runner today without waiting for the actor tree — that is the honest "migrated" bar the current stack supports.

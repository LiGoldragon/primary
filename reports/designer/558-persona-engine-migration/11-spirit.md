### Migration path for spirit

spirit is already the to-be on the daemon shell, nexus runner, SEMA storage, binary boundary, naming, contract-shape, and docs — it is the canonical copy source. The one substantive gap in the exemplar itself is **kameo-discipline**: promote `Nexus`/`Store` from a `tokio::sync::Mutex` single-flight to a real kameo single-writer state actor (ask-not-tell, RestartPolicy, redb off the async handler via spawn_blocking), ideally emitted by schema-rust-next so the pilot inherits it. The second, lighter gap is **contracts-schema-next**: the standalone `signal-spirit`/`meta-signal-spirit` crates are still `signal_channel!`, so either migrate them to schema-next or formally bless spirit's in-crate schema as the shared source. With those landed, the real work is the **production cutover**: rebuild `persona-spirit` on the emitted `ComponentDaemon` shell + nexus runner, delete `signal-executor` / `signal-sema` / `signal-engine-management` (folding lifecycle into `meta-signal-spirit`), retire the persona-* naming and the enforcing executor test, and migrate live SEMA state. Effort: medium — the to-be exists and is proven; the gating items are the kameo state-actor emission and then the executor removal on the production source.

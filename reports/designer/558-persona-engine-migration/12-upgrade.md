## upgrade — migration assessment

**Role:** runtime leg of the upgrade triad — schema/version migration orchestration, the migration catalogue, adjacent-version handover, and eventual policy/quarantine state. Currently skeletal: both binaries are placeholders.

**Overall readiness: partial. Effort: large.**

All three triad repos are present (`upgrade`, `signal-upgrade`, `meta-signal-upgrade`). The schema-next groundwork is unusually far along *as types* — the daemon's own `src/schema/lib.rs` is a clean `@generated by schema-rust-next` emission with full nexus + SEMA roots and zero `signal_channel!`, and `tests/generated_schema.rs` exercises the whole `NexusWork -> NexusAction -> SEMA -> Output` projection. But none of it is load-bearing: the daemon is a hand-written placeholder, live execution still runs through `signal-executor`, and there is no actor at all. The docs are commendably honest about exactly this gap.

### Side notes

- `src/execution.rs:241` `pub fn first_reply(...)` is a free function (re-exported from lib.rs:30) — violates the method-only Rust rule and should become an associated/trait method when the executor path is rewritten.
- The generated schema even contains a daemon/actor scaffold (`ActorStartFailure`, `on_start`/`on_stop` at `src/schema/lib.rs:3910+`) that src/ never consumes — confirming the gap is wiring, not emission.

### Migration path for upgrade

upgrade is a "types-ahead, runtime-behind" component: the schema-next contract types and the nexus/SEMA projection are emitted and tested, so the work is wiring, not design. The sequence is (1) switch `build.rs` to emit a `NexusDaemonShape` daemon shell and replace the placeholder bin with `<UpgradeDaemon as DaemonEntry>::run_to_exit_code()` on triad-runtime's actor substrate; (2) finish the contract cutover by deleting `signal_channel!` from both `signal-upgrade` and `meta-signal-upgrade` so the daemon imports generated channel/reply types; (3) introduce a real kameo state actor owning the catalogue/event-log/quarantine on a sema-engine single-writer store (redb off-handler via `spawn_blocking`, RestartPolicy declared); and (4) drive requests through the generated `NexusWork -> NexusAction` runner via `Runner::drive`, then delete `execution.rs`'s `signal-executor` path and its manifest dep and rewrite `tests/runtime.rs` to assert nexus routing. Blockers are upstream: schema-rust-next must emit upgrade's daemon shell, and triad-runtime's actor-native substrate plus the generated nexus runner must be the confirmed target before signal-executor can leave.

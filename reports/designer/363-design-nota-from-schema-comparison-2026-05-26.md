# 363 — Design exploration: nota from schema

*Designer-assistant parallel-implementation track per intent record 823,
adopting the methodology in `skills/double-implementation-strategy.md`.
Challenges /361 §4's wider recursion-floor cut by building an empirical
proof of the narrower cut — `nota.schema` actually driving Rust emission
for codec types.*

## §1 Frame

Per intent record 746 (Maximum, 2026-05-26): *"NOTA itself is
schema-derived. NOTA's own grammar and types are described by a schema
(the foundational schema). Everything NOTA-related is schema-derived from
that point downward; the schema generates the Rust code that interprets
NOTA."*

Operator's track (`/199` → `/200` → `/202`) settled empirically on a
WIDER recursion-floor cut: NotaCore stays hand-authored Rust;
`nota.schema` exists for documentation but the codec is hand-written.
`/361` §4 accepted this as the practical baseline and marked the narrower
cut as aspirational pending empirical proof.

This report IS the empirical proof. Per the double-implementation
strategy (records 815, 817, 818), the designer track produces a
comparison artifact. The result either tightens the cut (the schema-
driven emission is feasible enough to replace hand-authored code) or
confirms the wider cut is correct (the recursion floor is genuinely
where operator placed it).

The verdict below is: **partially feasible**, with a clear empirical
boundary. Details follow.

## §3 The narrower-cut feasibility verdict

**Partially feasible.** Specifically:

| Sub-question | Verdict | Evidence |
|---|---|---|
| Can codec TYPES emit from `nota.schema`? | YES | 31 declarations emit cleanly: 4 unit enums, 3 tagged enums, 9 newtypes, 15 tuple structs. `nota-emitted` compiles. |
| Is emission deterministic? | YES | `emitter_output_is_deterministic` test pins byte-identical re-runs. |
| Can the kernel itself emit from schema? | NO | Pure circularity — the kernel must read schema-source BEFORE emission can run. The kernel is the recursion floor and must be hand-authored. |
| Can byte-level recognition policy emit? | NO | The mapping from byte patterns (e.g. "apostrophe inside brackets indicates string") is policy, not types. The schema declares the result kind (`InlineString` vs `Vector`) but not the predicate that chooses between them. |
| Can the kernel→emitted lifter emit? | NO (with caveat) | The schema describes BOTH kernel's internal types AND emitted types, but the mapping between them is its own logic. A future schema could declare lift rules; this design doesn't try. |
| Is the result MATERIALLY BETTER than hand-authored nota-codec? | NO (in current form) | The emitted types replace the type DECLARATIONS but not the lexer/decoder logic (~1500 LOC of which is byte-level work). |

**Verdict in one sentence**: the narrower cut emits TYPES from the schema
successfully, but the recursion floor remains structurally where operator
placed it — the lexer + delimiter recogniser are necessarily hand-
authored. The wider cut is empirically correct on the byte-recognition
front; the narrower cut is empirically correct on the TYPE-shape front.

## §5 What the kernel boundary actually is

The kernel crate weighs 726 LOC with comments. Stripping comments and
blank lines, the substantive code:

| Component | Approx LOC | What it does |
|---|---|---|
| Error enum + Display | ~75 | 10 named-variant error cases |
| Token + TokenKind | ~30 | 13 token kinds + byte spans |
| Node + NodeKind + queries | ~70 | 9 node kinds + 6 query methods |
| `Kernel::lex` byte recogniser | ~180 | The bracket-vs-string disambiguation, line comments, byte literals, numbers, identifiers |
| `NodeWalker` parser | ~160 | Token-stream-to-Node walker for the three delimiter pairs |
| Total non-comment | ~515 LOC | |

**Of this ~515 LOC, very little is type-declaration work** — types are
~100 LOC (declarations only), so ~400 LOC is BYTE-RECOGNITION POLICY
that doesn't appear in the schema and would need a schema-declared policy
language to emit from. That policy language doesn't exist; designing it
is materially harder than the type-emission problem.

The cut sits where the design-track placed it because:

1. The schema can describe TYPES declaratively (`Name (VariantA VariantB)` or
   `Position [u32]`).
2. The schema can NOT describe BYTE-RECOGNITION POLICY declaratively
   without a separate sub-language for declaring lexer states +
   disambiguation rules.

If a future iteration adds that sub-language, the kernel COULD shrink
toward ~50 LOC (just the schema-loading bootstrap). That's a meaningful
research project; it isn't what this design exploration tried.

## §6 What emits from nota.schema

The 31 declarations the emitter produces, by kind:

- **Unit enums (4)**: `Delimiter`, `BlockKind`, `IdentifierClass`, `SymbolKind`, `StringForm`.
- **Tagged enums (3)**: `Node` (9 variants), `Classification` (4 variants), `ParseError` (10 variants), `ParseResult` (2 variants), `LiteralKind` (8 variants), `AtomKind` (7 variants), `TokenKind` (13 variants). [Counted as variants-with-payloads enums; `LiteralKind`/`AtomKind`/`TokenKind` are unit-only but classified as tagged when needed.]
- **Newtypes (9)**: `Position`, `Line`, `Column`, `Text`, `Byte`, `IntegerLiteral`, `FloatLiteral`, `BooleanValue`, `BytesValue`, `TokenStream`, `Document`.
- **Tuple structs (15)**: `Span`, `SourcePosition`, `SourceSpan`, `Token`, `MapEntry`, `IdentifierToken`, `StringValue`, `DelimiterPair`.

The emitter respects schema declaration order (per `/195`'s order-
preservation requirement). Determinism is pinned by
`emitter_output_is_deterministic`.

What's NOT emitted:

- The byte-level lexer (the bracket-vs-string disambiguation, line
  comments, byte literals, numbers, identifiers).
- The `NodeWalker` parser logic (token-stream-to-Node).
- The `Lifter` mapping from `kernel::Node` to `nota_emitted::Node` (the
  schema describes BOTH families but not the mapping; could a future
  schema declare lift rules? Yes — out of scope).
- Method bodies of `Node::is_record()`, `Node::children()`, etc. (these
  could emit from a schema-declared "queries" section; the design
  doesn't try).
- Display impls, Error impls, std-trait derivations (the emitter
  attaches a fixed `#[derive(Debug, Clone, PartialEq)]`; configurable
  derives are not yet schema-declared).
- The token-kind classifier (mapping a byte to which `TokenKind` to
  emit — this is byte-recognition policy, not type declaration).

## §7 Open questions surfaced (do-not-infer)

Per record 735, gaps are surfaced rather than closed by inference:

1. **What language would describe byte-recognition policy?** The
   simplest path is a state-machine declared in the schema (state
   transitions on byte ranges). The schema-schema would need
   primitives for byte ranges and predicates. Distinct from the type
   surface; needs psyche review.
2. **Should the kernel→emitted lifter be schema-declared?** The
   schema currently declares two type families (kernel internal +
   emitted) but no mapping between them. A mapping section in
   schemas (e.g. `{ kernel::Identifier emitted::Identifier(IdentifierClass Text) }`)
   would express the lift policy declaratively. Worth?
3. **Does `nota-emitted` graduate to a real codec, or stay as
   type-only fixtures?** The current emitted crate is type
   declarations only. Adding the Display impls, the readers, the
   writers requires more emission machinery. The point of this
   exploration was feasibility, not completeness.
4. **Identifier classification — schema rule or kernel built-in?**
   The lifter currently classifies identifiers (`PascalCase` /
   `CamelCase` / `KebabCase` / `LowerSnake`) via hand-authored
   pattern logic. The schema declares the enum but not the
   classification predicate. Should it? See record 800's
   qualifies_as_* discussion.
5. **The byte-vs-character boundary.** Kernel works on bytes
   throughout (`source.as_bytes()`). The schema speaks of `u8` and
   `String`, but the parse-time invariants assume ASCII delimiters.
   How does Unicode interact with the qualified-symbol classifier?
6. **`Result`-as-tagged-enum.** The schema declares
   `ParseResult ((Ok Node) (Err ParseError))`, which emits as a
   normal enum (not `std::result::Result`). Should the emitter
   detect Result-shaped enums and emit `pub type ParseResult = Result<Node, ParseError>;` instead?

These are gaps in the design where the prototype made the smallest
reasonable interpretation. None are blockers for the feasibility
verdict.

## §9 Recommendation

**Adopt a HYBRID position**, narrower than `/361` §4 but wider than
record 746's maximal reading:

1. **Keep operator's wider cut for byte recognition.** The lexer + tree
   walker stay hand-authored in `nota-next`. The empirical finding here
   confirms `/200` §"Slice 4" is right.
2. **NARROW the cut for TYPE DECLARATIONS.** Adopt schema-driven
   emission for the codec's TYPE surface — `Node`, `Token`, error
   variants, `MapEntry`, classification enums, etc. Asschema's
   `TypeDeclaration` (operator's `/200` §"Slice 5") becomes the input;
   `emit_schema!` generates the type declarations directly from it.
   Treat byte-recognition policy as an EXPLICIT residual on the kernel
   side, documented as a recursion-floor marker.
3. **Defer the byte-recognition sub-language.** That's a follow-on
   research direction; the cost/benefit isn't proven and the kernel as
   it stands is ~500 LOC of well-understood code.

This recommendation is consistent with `/361` §12's empirical-vs-
aspirational status table: the "narrower cut" row moves from
🔵 ASPIRATIONAL to ⚪ PARTIAL — type emission demonstrated, byte
recognition still hand-authored.

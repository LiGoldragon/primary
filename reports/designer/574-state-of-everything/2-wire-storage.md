# Storage + wire foundation

> A clean, disciplined foundation tier: eight hand-written library/contract crates (no real daemons, one kameo actor total in triad-runtime), all NOTA going through genuine typed nota-next codecs — the fake-NOTA count is zero cluster-wide. The two real liabilities are (a) ~57 free-function violations concentrated almost entirely in signal-frame (wire helpers in src/frame.rs + the proc-macro emit/validate crate), and (b) uniform schema staleness: every repo carries a vestigial `(Status Concept)` `.concept.schema` stub that no longer drives anything and predates today's terse-form grammar.

## Cluster: Storage + wire foundation

Eight hand-written foundation crates — storage kernel, engine, wire spine, vocabulary, proc-macros, and the daemon runtime. **No repo in this cluster is a daemon.** The whole tier is libraries/contracts that consumer components (signal-criome, signal-cloud, …) link. There is exactly **one real kameo actor in the entire cluster** (`RequestGate` in triad-runtime). NOTA handling is **clean cluster-wide — zero fake-NOTA**. The liabilities are concentrated, not diffuse: signal-frame owns ~47 of the cluster's ~57 free-function violations, and every repo carries an inert, pre-terse-grammar `.concept.schema` stub.

### Dependency / encoding posture

Every consumer is on the **old nota-next encoding, and that is safe**: nobody pins `d8862b6` (the bare-safe string-projection bump). The cluster splits into two pin groups — `f0e435a` (signal, signal-sema, version-projection; **8 behind**) and `ae5c25c` (signal-frame, triad-runtime; **4 behind**). `ae5c25c` is precisely the rev schema-rust-next deliberately holds to isolate the encoding bump, so those two are intentional, not stale. The four post-`ae5c25c` commits (`16493c8` bracket-string render, `2719624` pipe escaping, `027e18a` bare-safe atoms, `d8862b6` doc) form the encoding change and are **unreached by any consumer here** — no mismatch, but also no adoption of the new projection yet.

`flake.lock` never disagrees with `Cargo.lock` on foundation crates in this cluster: these are crane builds whose `flake.lock` pins only the nix toolchain (`nixpkgs`/`fenix`/`crane`); the foundation-crate revs live exclusively in `Cargo.lock`. `signal-derive` has no `flake.lock` at all. Minor cross-consumer drift worth a single `cargo update`: signal-frame is pinned `440dadd` (2 behind) by sema-engine but `7330505` (3 behind) by triad-runtime; signal-sema is HEAD in sema-engine but `4f75d1c` (1 behind) in signal.

### Schema state — uniform stale-stub pattern

All eight repos that have a schema carry exactly one `<name>.concept.schema` marked `(Status Concept) (Version 0 1)` (triad-runtime has none — correct for pure runtime infra). **None drive code generation** (no `build.rs`, no `src/schema/*.rs`, no `include!`) — the Rust source is the live source of truth and the concept stubs are inert intent placeholders. They predate today's terse grammar: sema/sema-engine repeat pre-52ro self-tags (`(Fact Fact)`, `(Record Record)`); signal-frame hand-declares `Bytes (String)` (pre-yp29 reserved leaf). The only forward-pointing schema artifact is signal-sema's `magnitude.schema.nota` — a terse single-line `(Magnitude Minimum VeryLow … Zero)` enum, the model to extend. There is **no live split nexus/sema/signal.schema anywhere in this cluster**; the concept-vs-triad duality resolves entirely to "concept = stale stub, code = live."

### Free-function violations (~57 total, ~47 in signal-frame)

The cluster splits sharply: **sema, sema-engine, signal, signal-sema, version-projection, and triad-runtime have ZERO module-level fns.** All violations live in signal-frame and signal-derive.

**signal-frame/src (11 — the highest-value fixes; real runtime work on types that already exist):**
- `src/frame.rs:156` `encode_archive`, `:165` `length_prefix`, `:176` `strip_length_prefix`, `:191` `short_header_from_archive`, `:200` `short_header_from_length_prefixed`, `:204` `encode_frame_archive`, `:215` `strip_short_header` — wire-codec helpers that are `ShortHeader`/frame associated fns escaped to module scope (the `encode()`/`encode_length_prefixed()` methods right below them delegate in).
- `src/caller.rs:105` `parent_executable`, `:109` `parent_start_time`, `:114` `start_time_ticks` — `/proc` readers that belong on `Caller`/`ProcessIdentifier`; `start_time_ticks` is additionally a hand-rolled proc-stat parser.
- `src/namespace.rs:32` `classify` — redundant free fn that `NamespaceSection::classify` (line 42) merely wraps; the body should live in the method.

**signal-frame/macros (36) + signal-derive (7) — proc-macro helpers (lowest severity):** `emit.rs`/`validate.rs` free fns over the data-bearing `ChannelSpec`/`*BlockSpec` (e.g. `emit(&ChannelSpec)` → `ChannelSpec::emit(&self)`), and signal-derive's `struct_shape`/`enum_shape`/`lower_*`/`unwrap_path` over `&syn` types. Violations by the letter of the rule, but the conventional proc-macro token-builder idiom doing compile-time work — counted for honesty, not flagged as runtime hazards. The `#[proc_macro]` entrypoints (`signal_channel`, `legacy_signal_channel`, `derive_schema`) are the legitimate `fn main`-equivalents and are **not** violations.

### Fake-NOTA: none (clean cluster-wide)

Every `to_nota`/`from_nota` in the cluster is legitimate: a `#[derive(NotaEncode/NotaDecode)]`, a hand-written `impl NotaEncode` that builds through `nota_next::Delimiter::wrap([...])` and delegates to children's `.to_nota()` (the correct pattern for generic enums `PatternField<T>`, `Request<Payload>`, `MutateOperation`/`Slot<Node>` that the derive can't cover), or the signal-frame emitter (`macros/src/emit.rs:789` `emit_payload_enum_codec`) generating exactly that delegating shape — the explicitly-not-flagged emitter case. The single hand-rolled codec, version-projection `ContractVersion::to_nota_literal` (`version.rs:90`), emits a `#hex` **atom** for a 32-byte leaf via `format!("{byte:02x}")` — a leaf→atom hex codec, **not** paren-record assembly, so not fake-NOTA. It is, however, the prime **lm84/yp29 migration target**: a hash identifier should now be a newtype-over-`Bytes` (`Digest Bytes` / `(Bytes 32)`) using schema-rust-next's `FixedBytes<N>` hex prelude (`eca4028`) rather than a hand-hexed `[u8;32]`.

### What most deserves the psyche's attention

1. **signal-frame's src/ free fns (11)** are the only free-function violations doing real runtime work on types that already exist — fold the `frame.rs` wire helpers onto `ShortHeader`/the frame types and the `caller.rs` `/proc` readers onto `Caller`. Highest-value cleanup in the cluster.
2. **signal/src/schema.rs is self-declared dead code** ("role under review… no obvious consumer… retirement tracked under bead `mentci-next-4v6`") — a compile-time `KindDescriptor` mechanism superseded by sema-resident runtime records, and the reason-to-exist of signal-derive rides on it. Decide retire vs repurpose.
3. **version-projection's `ContractVersion`** is hand-hexed `[u8;32]` — migrate to the `Bytes` leaf now that schema-rust-next ships the `FixedBytes<N>` hex prelude.
4. **Uniform concept-schema staleness** — eight inert `(Status Concept)` stubs in pre-terse grammar drive nothing; either retire them or promote to real terse schemas (signal-sema's `magnitude.schema.nota` is the template).

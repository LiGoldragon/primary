# Audit #7 — Code bulk and repetition: where does the code actually end up?

Psyche deep-audit of the schema-derived stack, report 7. This one is
purely a measuring exercise: *how much* code exists, *where* it lives,
how much is **hand-written** versus **machine-emitted**, and where the
stack is paying a copy-paste tax. Every number below is pasted from a
command I actually ran; the command and its directory are named at each
table. No estimates, no rounding-from-memory.

## Method and what counts as "generated"

Three distinct generation mechanisms live in this stack, and the audit
has to keep them apart because they land code in three different places:

1. **OUT_DIR generation** (the canonical contract shape). A
   `build.rs` lowers `schema/<name>.schema` and writes
   `<name>_generated.rs` into `$OUT_DIR` (under `target/`). The crate's
   hand-written `src/lib.rs` pulls it in with
   `include!(concat!(env!("OUT_DIR"), "/..."))`. The generated file is
   **never committed** — it only exists after a build. `signal-spirit`
   is the reference case.

2. **Committed-into-`src/schema/` generation** (the daemon shape).
   `spirit/build.rs` writes `spirit/src/schema/{signal,nexus,sema}.rs`
   into the source tree. These files ARE committed and carry a
   `// @generated by schema-rust-next` banner. Without an env var the
   build *checks* their freshness; with `SPIRIT_UPDATE_SCHEMA_ARTIFACTS=1`
   it *rewrites* them.

3. **Committed test-fixture generation** (inside `schema-rust-next`).
   The generator's own test suite commits golden `*_generated.rs` /
   `*.generated.rs` files so a diff catches output drift.

Everything in `src/` that is NOT one of those three is **hand-written**.
Tools: `tokei 14.0.0` (`which tokei` → `/home/li/.nix-profile/bin/tokei`),
plus `find … | xargs wc -l`. Cargo is `~/.nix-profile/bin/cargo`.

The split that matters: of 23698 lines of committed `src/`, **2758

generated lines into `target/` at build time (measured below).

So the generated-code *story* of the stack is concentrated, not diffuse:
the machinery to generate (schema-next + schema-rust-next = 12722 hand
LOC) dwarfs the generated output that is actually committed anywhere
(2758 LOC in spirit + 6157 LOC of committed test fixtures, measured
below). **The stack is mostly the engine that does the generating, not
the generated artifacts.**

## The amplification ratio: schema source → generated Rust

`signal-spirit` is the clean measurement because it is the minimal
one-operation contract. Its schema source, with comments stripped
(`grep -vE '^\s*(;;|$)' signal-spirit/schema/signal-spirit.schema`):

```
{}
[
  (Input (Record Entry))
  (Output (RecordAccepted RecordIdentifier))
]
{
  Topic [Text]
  Description [Text]
  RecordIdentifier [Integer]
  Entry [Topic Description]
}
```

That is **11 declaration lines** (4 type definitions + 1 input + 1
output + structural brackets). I built it (`cargo build` in
`/git/github.com/LiGoldragon/signal-spirit`, exit 0) and measured the
emitted file:

```
$ wc -l target/debug/build/signal-spirit-*/out/signal_spirit_generated.rs
  232 .../signal-spirit-5840adf8bf7eacd3/out/signal_spirit_generated.rs
  232 .../signal-spirit-85dd06b937256757/out/signal_spirit_generated.rs
```

(Two build-hash directories, byte-identical: `diff` over the two →
`IDENTICAL`.) **11 declaration lines amplify to 232 lines of Rust — a
~21× expansion.** What those 232 lines contain (`grep -c` on the file):
7 `pub struct/enum`, 12 `impl` blocks, 7 `derive` lines, 17 functions.
Head of the file:

```rust
// @generated by schema-rust-next

pub type Text = String;
pub type Integer = u64;

#[derive(Clone, Debug, PartialEq, Eq)]
pub enum NotaDecodeError {
    Parse(String),
    ExpectedSingleRoot { found: usize },
    ExpectedDelimited { type_name: &'static str, delimiter: &'static str },
    ...
}

impl std::fmt::Display for NotaDecodeError { ... }
impl std::error::Error for NotaDecodeError {}
```

This is the value proposition stated as a number: every contract repo
hand-writes ~10 schema lines and a ~30-line `build.rs`, and gets ~230
lines of encode/decode/error machinery for free. The hand-written cost
in `signal-spirit` is `src/lib.rs` (119) + `build.rs` (34) + schema (32)
= 185 lines, against 232 generated; for richer contracts (measured
below) the generated half grows far faster than the hand-written half.

### The full pipeline has an intermediate artifact

The chain is not schema→Rust directly; there is a lowered intermediate.
`spirit/schema/` holds both stages
(`wc -l /git/.../spirit/schema/*`):

```
   71 signal.schema     (human-facing DSL)
    5 signal.asschema   (lowered NOTA — see note)
```

The `.asschema` reports as 5 "lines" but is really one dense NOTA object
per section: section 1 (imports/exports), section 2/3 (input/output
roots), section 4 (the entire type namespace on a single line). Head of
`spirit/schema/signal.asschema`:

```
(spirit:signal [0.1.0])
[]
[]
[(Record (Some (Plain Record))) (Observe (Some (Plain Observe))) ...]
[(RecordAccepted ...) (RecordsObserved ...) ...]
[(Public SourcePath (Alias (SourcePath String))) (Public Topic (Alias (Topic String))) ... ]
```

So the real generation graph is **`.schema` (DSL) → `.asschema` (lowered,
canonical, NOTA) → `.rs` (emitted)** — `schema-next` owns the first
arrow, `schema-rust-next` owns the second. The `.asschema` is the
artifact the generator actually consumes; the `.schema` is sugar for
humans.

## REPETITION, part 1 — the `build.rs` copy-paste tax

There are **31 contract repos** (`ls -d signal-* meta-signal-*` minus the
two `worktrees`/`mockup` scratch dirs). Of the ones that generate via
`build.rs`, the audit found **four different `build.rs` API styles
co-existing** — and within the newest style, near-total duplication.

Which generation API each `build.rs` calls
(`grep -hE 'RustEmitter|GenerationPlan|GenerationDriver|CargoSchemaMetadata' <repo>/build.rs`):

| Repo(s) | `build.rs` LOC | API style |
|---|---|---|
| `signal-spirit` | 34 | `RustEmitter.emit_file(&asschema)` — unit-struct method, oldest |
| `signal-cloud`, `meta-signal-cloud` | 33 | `GenerationDriver::new(GenerationPlan::wire_contract(…))` + `CargoSchemaMetadata` |
| `signal-domain-criome`, `meta-signal-domain-criome` | 24–25 | `CargoSchemaMetadata` only — shortest |
| `spirit` | (in 60-line file) | `GenerationPlan::new(…).with_module(ModuleEmission::…)` — daemon variant |
| `signal-upgrade`, `meta-signal-upgrade` | **264** | `RustEmitter::new(RustEmissionOptions…)` + 260 lines of hand-rolled witness scaffolding |

Two facts jump out.

**(a) The generator's own public API is not stable.** `signal-spirit`
calls `RustEmitter.emit_file(&asschema)` — `RustEmitter` used as a unit
struct. `signal-upgrade` calls `RustEmitter::new(RustEmissionOptions::…)`
— `RustEmitter` now a constructed type taking an options object, with new
methods `emit_file_from_nota_path` / `emit_file_from_binary_path`. Same
type name, incompatible call shapes, both live in committed `build.rs`
files. A contract author copying an existing `build.rs` gets a different
API depending on which neighbour they copied.

**(b) The 264-line `build.rs` is duplicated almost verbatim.** `diff
signal-upgrade/build.rs meta-signal-upgrade/build.rs` (in
`/git/github.com/LiGoldragon`):

```
11c11
<         crate_name: "signal-upgrade",
---
>         crate_name: "meta-signal-upgrade",
13c13
<         update_environment_variable: "SIGNAL_UPGRADE_UPDATE_SCHEMA_ARTIFACTS",
---
>         update_environment_variable: "META_SIGNAL_UPGRADE_UPDATE_SCHEMA_ARTIFACTS",
74c74
<             .expect("lower signal-upgrade schema source");
---
>             .expect("lower meta-signal-upgrade schema source");
100c100
<                 "signal-upgrade schema must emit src/schema/lib.rs, found {}",
---
>                 "meta-signal-upgrade schema must emit src/schema/lib.rs, found {}",
```

**Four lines differ out of 264.** The other 260 lines — `SchemaBuild`,
`CheckedInAsschemaArtifact`, `GeneratedAsschemaArtifactFiles`,
`GeneratedSchemaSourceFile`, the `GeneratedFileArtifactWitness` trait,
`CheckedInSchemaSource`, and all their `impl` blocks — are byte-identical
copy-paste between two repos. This is build-orchestration scaffolding
(load schema → lower → write artifacts → assert checked-in freshness →
emit Rust → witness NOTA-vs-binary equivalence) that has no business
being re-pasted per repo; it belongs as a single reusable entry point in
`schema-rust-next::build` (which is exactly what the 33-line
`GenerationDriver`/`GenerationPlan` shape already is — the newer 264-line
hand-roll is a *regression* away from the shared helper). The
`signal-cloud`/`meta-signal-cloud` pair shows the same 1:1 duplication at
33 lines (`diff` → 3 lines differ, all crate-name/env-var), but there the
duplication is cheap because the heavy lifting is already inside
`GenerationDriver`.

**The fix surface is clear: every `build.rs` in the stack should be the
~10-line `GenerationDriver`/`CargoSchemaMetadata` shape, and the 264-line
hand-roll should be deleted in favour of it.** This audit does not apply
the fix (report-only), but the duplication is unambiguous and quoted
above with real diff output.

## REPETITION, part 2 — the `tests/round_trip.rs` skeleton

`round_trip.rs` is the standard contract test file. **12 of them define
the same `fn round_trip_request` / `round_trip_reply` helper**
(`grep -lE 'fn round_trip_request' signal-*/tests/round_trip.rs … | wc -l`
→ 12: signal-criome, signal-harness, signal-sema-upgrade, signal-system,
signal-orchestrate, signal-persona-spirit, signal-mind, signal-introspect,
signal-terminal, signal-upgrade, meta-signal-upgrade, signal-router).
Wider shared-scaffold counts across `signal-*/tests/round_trip.rs`:

| Shared element | `grep -l … | wc -l` |
|---|---|
| `Frame::new(...)` + `decode_length_prefixed` scaffold | 16 |
| `use nota_codec::{Decoder, Encoder, NotaDecode, NotaEncode}` (identical import) | 15 |
| `fn round_trip_request` / `round_trip_reply` helper | 12 |
| `…daemon_configuration_round_trips_through_rkyv` test | 6 |

The `round_trip.rs` files are large and visibly templated. `diff
signal-harness/tests/round_trip.rs signal-introspect/tests/round_trip.rs`
shows the shape: identical *skeleton*, differing *payload*. Both define:

```rust
fn round_trip_request(request: <Contract>Request) {
    let frame = Frame::new(FrameBody::Request { exchange: …, request: request.into_request() });
    let bytes = frame.encode_length_prefixed();
    let decoded = Frame::decode_length_prefixed(&bytes).expect("decode");
    match decoded.into_body() { FrameBody::Request { request, .. } => { … } … }
}
```

— byte-for-byte the same control flow (encode → length-prefix → decode →
match → assert verb/payload), with only the contract's type names and the
operation payloads swapped. The same is true of the
`<contract>_daemon_configuration_round_trips_through_rkyv` and
`…_through_nota_text` test pair: the harness and introspect versions
differ only in the config struct's field names. This is hundreds of lines
of test scaffolding re-typed per contract. Per-file `round_trip.rs` LOC
(`wc -l`, the templated ones):

```
signal-mind            1371      signal-terminal         864
signal-persona-spirit   729      signal-orchestrate      648
signal-criome           648      signal-system           545
```

The candidate fix (not applied here) is a generated or macro-provided

## REPETITION, part 3 — `Cargo.toml` shape and the rkyv feature string

`Cargo.toml` across signal contracts is a near-perfect template. `diff
signal-harness/Cargo.toml signal-introspect/Cargo.toml`:

```
2c2
< name         = "signal-harness"
---
> name         = "signal-introspect"
5c5
< rust-version = "1.85"
---
> rust-version = "1.88"
7,8c…   (repository + description lines)
12c12
< name = "signal_harness"
---
> name = "signal_introspect"
20a21
> signal-message = { … }
```

Name, lib-name, description, `rust-version` (drifting: 1.85 vs 1.88 vs the
`signal-introspect` value — itself a small inconsistency), and the
dependency list differ; the *structure* is identical. The most concrete
copy-paste artifact is the rkyv feature list, which is the same 5-element
string in **68 `Cargo.toml` files**:

```
$ grep -rl 'features = ["std", "bytecheck", "little_endian", "pointer_width_32", "unaligned"]' */Cargo.toml | wc -l
68
$ grep -rl 'unsafe_code = "forbid"' */Cargo.toml | wc -l
70
```

68 repos repeat `rkyv = { version = "0.8", default-features = false,
features = ["std", "bytecheck", "little_endian", "pointer_width_32",
"unaligned"] }` verbatim; 70 repeat the `unsafe_code = "forbid"` lint
line. A workspace-level dependency table or a shared `[workspace.dependencies]`
would collapse all 68. Today a change to the rkyv feature set (e.g.
flipping `pointer_width_32` to `_64`) is a 68-file edit.

## Where the bulk lives — the one-paragraph answer

The schema-derived stack's code mass is **concentrated in two
hand-written engine repos** — `schema-next` (8147 `src/` LOC, the DSL →
`asschema` lowering) and `schema-rust-next` (4575 `src/` LOC, the
`asschema` → Rust emitter, home of the stack's single biggest hand file
at 3462 LOC). Those two are 12722 of 20940 hand-written core `src/` lines
(**61%**). The NOTA layer (`nota-codec` 2880 + `nota-next` 2517 = 5397) is
the next mass. The *generated* output that actually persists is small and
localised: 2758 committed lines in `spirit/src/schema/` plus 6157 lines of
committed test fixtures in `schema-rust-next`. Contract repos themselves
are tiny by design (`signal-spirit` = 119 hand `src/` lines → 232 emitted,
a 21× amplification). **The expensive, repeated thing is not the generated
code — it's the per-repo build/test/manifest scaffolding around each
contract**: a 264-line `build.rs` duplicated 4-lines-different across two
repos, ~18 hand-typed `round_trip.rs` files sharing one skeleton, and a
5-element rkyv feature string copy-pasted into 68 `Cargo.toml` files. That
scaffolding is the stack's real repetition tax, and all three pieces are
mechanically de-dupable.

## Things I could not measure in this environment

Nothing in the task was un-runnable: `tokei`, `cargo`, the deployed
`spirit` CLI, and all repos under `/git/github.com/LiGoldragon` were
present and responsive. Two scoping notes: (1) I built `signal-spirit`,
`spirit`, and `schema-rust-next` (all exit 0) to materialise OUT_DIR
output and confirm the freshness-check; I did not rebuild every one of the
31 contract repos, so the OUT_DIR generated-LOC amplification is measured
on `signal-spirit` only. (2) The deployed-vs-source contract gap I tripped
over (`Observation` vs `TopicMatch`) is real but belongs to audit #1; I
report it here only as the side-effect that proved the runtime executes
generated decode code.

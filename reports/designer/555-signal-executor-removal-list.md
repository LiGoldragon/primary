designer, 2026-06-08. The dependency-and-code removal list the psyche asked
for: what must be *removed and replaced* in each component to honestly call
it "migrated to the new engine." Anchored in Spirit `ng1x` (a migration is
incomplete until the superseded dependency and execution path are removed —
wrapping is not migrating), `v3um` (a manifest line is not the migration:
every source path, test witness, and architectural assertion that used the

at a time, smallest first, tested before moving on). Supersedes the
signal-executor paragraph in `554`.

## Is signal-executor the only culprit?

**On the dependency axis: yes.** signal-executor is the single old-execution
library left in the four components. There is no separate `executor` crate,
and every other git dependency they carry — `sema-engine`, `signal-frame`,
`signal-sema`, `schema-rust-next`, `schema-next`, `nota-next`,
`triad-runtime`, `version-projection`, `sema` — is current stack. So the
manifest answer is one line per component.

**But the culprit is a code pattern, not a line.** What signal-executor
supplies is an entire execution shape each component hand-wires:

| signal-executor surface | what it does | nexus replacement |
|---|---|---|
| `Lowering` (impl per component) | contract operation → component-local commands | the schema-emitted nexus `decide` / lowering |
| `CommandExecutor` (impl per component) | execute commands atomically, correlate effects → reply | SEMA apply + effect handling inside the nexus runner |
| `Executor::new(...).execute(request)` | the driver loop | the generated `NexusWork → NexusAction` runner (553) |
| `ObserverSet` / `ObserverChannel` / `FrameObserverBridge` | publish operation/effect events | the schema-emitted subscription/observer |

A migration that deletes the `Cargo.toml` line but leaves the
`impl CommandExecutor`, the `Executor::new` wiring, the tests that assert
that path, and the ARCHITECTURE.md prose claiming "executes through
signal-executor" has not migrated anything — it has broken the build. Per
`v3um` the unit of removal is: **manifest + source + test witnesses +
architectural assertions.**

One false positive to avoid: orchestrate's `INTENT.md`/`ARCHITECTURE.md`
mentions of "executor capacity", `agent_executors`, and `spawn_plans` are
the orchestration *domain* (agent execution capacity), not the
signal-executor library. They stay.

## The gating dependency

orchestrate and upgrade already carry the generated nexus *types*;
repository-ledger and persona-spirit do not carry even those. But none of the
four can complete until the **nexus execution engine of design 553** (the
actor-native `NexusWork → NexusAction` runner that actually drives requests)
is real — the generated `NexusWork`/`NexusAction` enums are the data shape,
not the driver. So the honest sequencing is: land the 553 nexus runner, then
migrate smallest-first per `r310` — orchestrate or upgrade first (types
already present, driver swap only), repository-ledger next, persona-spirit
last (generate nexus + invert the enforcing invariant + rewrite the
architecture doc).

## signal-sema: not a culprit — keep the vocabulary, fix the misleading name

The operator flagged `signal-sema` as an adjacent cleanup; the psyche asked
what it actually is and observed the name is misleading. Investigation
confirms the psyche: `signal-sema` is kept, but its **name lies about its
role.**

**What it actually is.** `signal-sema` is **Layer 3** of the ratified
three-layer signal model (designer `238`/`239`/`246`/`248`): a pure library of
the *payloadless* cross-component classification vocabulary. It owns
`SemaOperation` (the closed set `Assert / Mutate / Retract / Match / Subscribe
/ Validate`, with the `OperationClass` Write/Read/Stream/Validation
projection), `SemaOutcome` (`Asserted / … / NoChange`), `SemaObservation` (one
operation class joined with one outcome class), `Magnitude` (the
workspace-universal eight-rung `Zero..Maximum` scale), the read-algebra pattern
primitives (`Bind`, `Wildcard`, `PatternField<T>`), the typed identity values
(`Slot<Payload>`, `Revision`), and the projection traits `ToSemaOperation` /
`ToSemaOutcome` that per-component Command/Effect enums implement so observers
filter cross-component activity by class without knowing payloads.

**Why the name misleads (the psyche is right).** The triad convention is
`signal-<component>` = *the working signal contract to talk to the
`<component>` daemon* (`signal-terminal` ↔ the terminal daemon,
`signal-spirit` ↔ the spirit daemon). By that convention `signal-sema` reads as
"the wire contract to a `sema` daemon / the SEMA store engine." It is neither:
there is no sema daemon it fronts, and it is **not** the contract for
`sema-engine`. It is a cross-cutting vocabulary every component imports. The
word "sema" also collides with SEMA-the-store (`sema-engine`, `.sema`, the
Signal/Nexus/SEMA runtime leg), so the name doubly implies "talk to the sema
store" when the crate is really "the semantic-classification vocabulary." It
sits in the `signal-<component>` namespace while not being a component
contract — a naming-discipline violation by the workspace's own triad rule. It
is the sibling of `signal-frame` (both are the split of the former
`signal-core`: `signal-frame` = frame mechanics, `signal-sema` =
classification vocabulary); `signal-frame`'s name reads cleanly because "frame"
is unambiguous, whereas "sema" overloads.

**Keep it — it is current, not old-executor cruft.** `sema-engine` depends on
it legitimately (`log.rs`/`engine.rs`/`subscribe.rs`/`query.rs`/`mutation.rs`
all reason in `SemaOperation` classes); `Magnitude` is workspace-universal; the
three-layer model is ratified. Do not delete.

**What does change in the four components.** They import the projection traits
(`ToSemaOperation`/`ToSemaOutcome`, `SemaObservation`) specifically to feed the
signal-executor `ObserverSet`. When the executor path moves to the nexus
engine, the hand-wired projection should be **absorbed into the generated
nexus/SEMA route** (the component stops hand-implementing `ToSema*`); the
vocabulary itself stays, imported by the generated code and `sema-engine`. So
the component-side `signal-sema` usage is part of the executor migration; the
crate is not.

**Recommended fix: rename out of the `signal-` namespace.** A name that says
"semantic classification vocabulary," not "contract to the sema engine" —
candidates `sema-vocabulary`, `sema-classification`, `sema-classes`, or, if
the bare word "sema" still pulls toward the store, a non-`sema` word
(`classification` / `semantic-classes`). This is a naming **decision for the
psyche**; it is not captured in Spirit until the psyche chooses, and the rename
is a code-repo change the operator lands (designer supplies the name and
updates the per-repo `INTENT.md`).

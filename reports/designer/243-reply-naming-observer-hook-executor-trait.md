# 243 — Reply naming, observer hook via macro, executor trait

*Three concrete designs for the architecture-level holes from
`/242`: reply variant naming convention (hole 2), observer hook
generated by the same `signal_channel!` macro (hole 4), and an
executor library with a `Lowering` trait (hole 5). Each section
includes a worked example. None of these are landed; all are
designer proposals awaiting psyche approval and a follow-up arc to
implement.*

## 0 · TL;DR

| Hole | Proposed shape |
|---|---|
| 2 (reply naming) | Reply success variants are verb-past-tense (`Submit` → `Submitted`); rejection variants are verb-past-tense + `Rejected` (`SubmitRejected`); when the past-tense collides with a noun (`Announce` → `Announcement`), fall through to the next-best past-tense (`Identified`). |
| 4 (observer hook) | New `observable` block inside `signal_channel!` injects `Observe` / `Unobserve` ops + an `ObserverStream` automatically. Opt-in per channel. |
| 5 (executor) | New crate `signal-executor` (or module in `signal-sema`) defining a `Lowering` trait + an `Executor<L>` struct that handles atomicity, per-op reply mapping, and observer publication. |

## 1 · Hole 2 — Reply variant naming

### The problem

For `operation Submit(Submission)`, five reply names are defensible:

```rust
reply MessageReply {
    Submitted(Receipt),              // past-tense of the verb
    Accepted(Receipt),               // generic outcome
    SubmissionAccepted(Receipt),     // payload-type + outcome
    Receipt(Receipt),                // just the noun
    Stored(Receipt),                 // past-tense of what HAPPENED
}
```

The /241 verb-form rule covers operation roots but says nothing
about reply variants. `signal-persona`'s migration history just
landed one convention (`Launched`, `Retired`, `Identified` for
successes; `LaunchRejected`, `RetireRejected` for failures) but
the convention isn't written down; the next contract author has
to read existing contracts to derive it. Worse, signal-persona
*already* has an inconsistency: `Announce` → `Identified`
because `Announced` collides with announcement-as-noun.

### The proposed rule

Add to `skills/contract-repo.md` §"Public contracts use
contract-local operation verbs" (alternatively to `skills/naming.md`):

> **Reply success variants are verb-past-tense matching the
> operation root.** `Submit` → `Submitted`; `Register` →
> `Registered`; `Launch` → `Launched`; `Retire` → `Retired`.
>
> **Reply rejection variants are verb-past-tense + `Rejected`.**
> `Submit` → `SubmitRejected`; `Register` → `RegisterRejected`.
> Domain-level rejection reasons are payload variants of the
> `Rejected` reply.
>
> **When the verb-past-tense collides with a noun derived from
> the verb,** fall through to the next-best past-tense.
> `Announce` → `Announcement` (noun collision) → use
> `Identified` (the closest verb-past-tense that names what the
> daemon DID after receiving the announcement).
>
> **Lifecycle-shaped verbs (Start / Stop / Drain) may share a
> single `Action*` pair** when the daemon's response shape is
> uniform across them: `ActionAccepted(ActionAcceptance)` /
> `ActionRejected(ActionRejection)`. This is the
> signal-persona precedent.

This collapses the five-options ambiguity into one rule with two
named carve-outs.

### Worked example — signal-persona-spirit

Operations and proposed reply variants:

| Operation | Success reply | Rejection reply |
|---|---|---|
| `State(Statement)` | `Stated(StatementCaptured)` | `StateRejected(StateRejectionReason)` |
| `Record(Entry)` | `Recorded(EntryCaptured)` | `RecordRejected(RecordRejectionReason)` |
| `Query(Selection)` | `Queried(QueryResult)` | `QueryRejected(QueryRejectionReason)` |
| `Watch(Subscription) opens …` | `Watched(WatchOpened)` | `WatchRejected(WatchRejectionReason)` |
| `Unwatch(Token)` | `Unwatched(WatchClosed)` | (no rejection — close is always ok) |

Cleaner than the current ad-hoc mix.

## 2 · Hole 4 — Observer hook via the macro

### The proposal

Add an `observable` block to `signal_channel!`:

```rust
signal_channel! {
    channel Spirit {
        operation State(Statement),
        operation Record(Entry),
        operation Query(Selection),
        operation Watch(Subscription) opens RecordStream,
        operation Unwatch(RecordSubscriptionToken),
    }
    reply SpiritReply { … }
    event SpiritEvent { … }
    stream RecordStream { … }
    observable {
        filter ObserverFilter;
        event OperationReceived;
        event SemaEffectEmitted;
    }
}
```

### What the macro injects

When `observable` is present, the macro adds to the channel:

```rust
// Auto-injected operations
operation Observe(ObserverFilter) opens ObserverStream,
operation Unobserve(ObserverSubscriptionToken),

// Auto-injected stream
stream ObserverStream {
    token ObserverSubscriptionToken;
    opened ObserverSubscriptionOpened;
    event OperationReceived;
    event SemaEffectEmitted;
    close ObserverSubscriptionToken;
}
```

`ObserverFilter` is a contract-author type — declared by the
contract because each contract's filter shape is different.
Examples:

```rust
pub enum ObserverFilter {
    All,
    OnlyOperations { kinds: Vec<SpiritOperationKind> },
    OnlySemaEffects { classes: Vec<OperationClass> },
}
```

`OperationReceived` and `SemaEffectEmitted` are workspace-uniform
event class names — `persona-introspect` can subscribe to the
same vocabulary across every observable channel.

### Three properties this gives

1. **Opt-in per channel.** Contracts without an `observable`
   block produce no observer surface. Lightweight tools and leaf
   utilities skip it.
2. **Workspace-uniform vocabulary** for the event class names,
   variable on the filter shape.
3. **Subscription on the public socket** — observation isn't a
   security concern (no secrets on the wire); subscribing on the
   owner socket would create ambiguity about which owner the
   subscription belongs to.

### Daemon side

The daemon's executor (see hole 5) calls the macro-generated
`publish_operation_received(&op)` after decoding an inbound
operation and `publish_sema_effect_emitted(&effect)` after each
Sema effect commits. The macro emits the publish functions and
the subscriber-set bookkeeping; the daemon wires them into the
executor.

## 3 · Hole 5 — Executor library / `Lowering` trait

### The proposal

New crate `signal-executor` (or module in `signal-sema`):

```rust
pub trait Lowering {
    type Operation: RequestPayload;
    type Reply;
    type RejectionReason;

    /// Single contract op → zero-or-more Sema operations.
    /// Empty Vec means "no state effect" — legitimate for
    /// validation-only or no-op operations.
    fn lower(
        &self,
        operation: &Self::Operation,
    ) -> Result<Vec<SemaOperation>, Self::RejectionReason>;

    /// Build the per-op reply variant from the Sema effects that
    /// were committed.
    fn reply_from_effects(
        &self,
        operation: &Self::Operation,
        effects: &[SemaEffect],
    ) -> Self::Reply;
}

pub struct Executor<L: Lowering, S: SemaEngine> {
    lowering: L,
    sema_engine: S,
    observers: ObserverSet,
}

impl<L: Lowering, S: SemaEngine> Executor<L, S> {
    pub fn execute(
        &mut self,
        request: Request<L::Operation>,
    ) -> Reply<L::Reply, L::RejectionReason> {
        // 1. Lower every op; reject early on lowering failure.
        let mut sema_ops = Vec::new();
        for op in request.payloads() {
            self.observers.publish_operation_received(op);
            match self.lowering.lower(op) {
                Ok(ops) => sema_ops.extend(ops),
                Err(reason) => return Reply::Rejected(reason),
            }
        }

        // 2. Atomic execute. All ops commit or all rolled back.
        let effects = match self.sema_engine.execute_atomic(sema_ops) {
            Ok(effects) => effects,
            Err(error) => return Reply::Rejected(error.into()),
        };

        // 3. Publish Sema effects to observers.
        for effect in &effects {
            self.observers.publish_sema_effect_emitted(effect);
        }

        // 4. Build per-op replies from observed effects.
        let sub_replies: Vec<L::Reply> = request.payloads()
            .iter()
            .map(|op| self.lowering.reply_from_effects(op, &effects))
            .collect();

        Reply::Accepted { per_operation: sub_replies }
    }
}
```

### What this delivers

- **Atomicity guarantee** (closes hole 9): the executor's
  `execute_atomic` is the single binding point. All or none.
- **Per-op reply mapping** (closes hole 8): `reply_from_effects`
  produces one reply per op deterministically; the executor
  positions them.
- **Observer integration** (ties to hole 4): the executor calls
  the macro-generated `publish_*` functions; daemon code never
  has to wire observability manually.
- **Cross-daemon consistency**: every daemon's executor follows
  the same shape; per-component variability is just the
  `Lowering` impl.

### Worked example — signal-persona-spirit's daemon

```rust
struct SpiritLowering { /* state */ }

impl Lowering for SpiritLowering {
    type Operation = SpiritOperation;
    type Reply = SpiritReply;
    type RejectionReason = SpiritRejectionReason;

    fn lower(&self, op: &SpiritOperation)
        -> Result<Vec<SemaOperation>, SpiritRejectionReason>
    {
        match op {
            SpiritOperation::State(statement) => Ok(vec![
                SemaOperation::Assert(self.statement_to_typed_entry(statement)),
            ]),
            SpiritOperation::Record(entry) => Ok(vec![
                SemaOperation::Assert(self.entry_to_typed_entry(entry)),
            ]),
            SpiritOperation::Query(selection) => Ok(vec![
                SemaOperation::Match(self.selection_to_match_plan(selection)),
            ]),
            SpiritOperation::Watch(subscription) => {
                // Watch is purely metadata; no Sema effect.
                self.register_watcher(subscription);
                Ok(Vec::new())
            }
            // ...
        }
    }

    fn reply_from_effects(
        &self,
        op: &SpiritOperation,
        effects: &[SemaEffect],
    ) -> SpiritReply {
        match op {
            SpiritOperation::State(_) => SpiritReply::Stated(
                self.statement_captured_from(effects),
            ),
            SpiritOperation::Record(_) => SpiritReply::Recorded(
                self.entry_captured_from(effects),
            ),
            SpiritOperation::Query(_) => SpiritReply::Queried(
                self.query_result_from(effects),
            ),
            // ...
        }
    }
}

fn main() {
    let executor = Executor::new(
        SpiritLowering::new(),
        sema_engine,
        ObserverSet::new(),
    );
    // ... wire to socket loop ...
}
```

### What `signal-executor` documents

Beyond the trait + struct, the crate's `ARCHITECTURE.md`
documents:

- The atomicity contract — what `execute_atomic` guarantees
  and what it doesn't.
- Failure-mode taxonomy — when lowering fails vs when Sema
  execution fails vs when post-commit publication fails (which
  doesn't roll back).
- Reply correlation — that `reply_from_effects` is called once
  per operation, in request order, with the full effects
  slice (so an op can correlate to its effects).
- Observer publication ordering — operation-received fires
  before lowering; sema-effect-emitted fires after each effect
  commits.

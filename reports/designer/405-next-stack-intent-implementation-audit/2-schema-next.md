# 405/2 — schema-next intent↔implementation audit

*Kind: Sub-agent audit · Layer 1 (schema language) · Topics: schema, macros, asschema, three-schema-types, signal, nexus, sema, intent · 2026-05-28*

*Pinned `main` commit `e0681f2fb038734ef3a00878749ce2870d7c1d51` ("schema: manifest three-plane mail intent"); `git rev-parse HEAD` confirms HEAD is exactly this commit — no drift. `cargo test` is GREEN: 0 unit + 9 `design_examples` + 17 `lowering` + 8 `macro_exploration` + 0 doctest = 34 passing, 0 failing.*

## Verdict

schema-next is a clean, well-tested, position-aware macro engine that lowers a FOUR-position `.schema` document (Imports / Input / Output / Namespace) into an ordered macro-free `Asschema`, dispatching every position through a `MacroRegistry` of `SchemaMacro` trait objects — exactly as records 880/932/933/940 specify, with no sigil scanning and exemplary method-only Rust (no free functions, no ZST namespace holders). BUT the single most important finding is unambiguous: schema-next implements **ZERO of the three schema types as types**. There is ONE generic schema language with ONE engine; the words "Signal", "Nexus", "Sema", and "plane" appear NOWHERE in `src/` (`grep -rni` returns nothing). The three planes survive only as (a) prose in `INTENT.md`/`ARCHITECTURE.md` added by the pinned commit and (b) ordinary namespace declarations named `NexusAction`/`SemaCommand` inside one generic schema in a single test (`design_examples.rs:359`), whose own comment says "until the plane-specific file split lands." Record 982's symmetric-4-position claim is MET *structurally* (the one engine has the 4 positions) but the import/export half is only PARTIAL: imports parse, exports do not exist, and colon-paths are stored-not-resolved. The pinned commit itself is **DOC-ONLY** — 17 lines of prose across two markdown files, zero code. This repo is the most intent-dense in the stack and the most honestly-scoped, but it is one generic plane where intent calls for three.

## Classification per intent item

### Item 1 — Three schema types (Signal / Nexus / SEMA), each its own LANGUAGE — record 982 (High), 964 (Maximum)

**MISSING (as types/engines).** The intent (frame §1.2, workspace `INTENT.md:304-316`) is THREE schema types, *"each has its own engine with its own traits."* schema-next has exactly one engine (`SchemaEngine`, `engine.rs:103`) and one `Asschema` model (`asschema.rs:50`). Hard evidence of absence:

- `grep -rniE "signal|nexus|sema|plane" src/` → no matches anywhere in source.
- `grep -rniE "schema_type|SchemaType|first.record|variant.*signal" src/` → no matches. There is NO routing by schema-type variant.
- The schema-type discriminator that record 964 left OPEN (first-record `(Signal …)`/`(Nexus …)`/`(Sema …)` vs file-extension) is implemented in NEITHER form. Every example schema's first root object is `{}` (the imports map), not a type marker — see `schemas/spirit-min.schema:1`, `schemas/core.schema:1`, `schemas/root.schema:1`, `tests/fixtures/spirit-crate/schema/lib.schema:1`. The engine's `MacroPosition` enum (`macros.rs:8`) has seven variants (RootImports/RootInput/RootOutput/RootNamespace/NamespaceDeclaration/StructFields/EnumVariants) — none is a schema-type selector.

The three planes appear ONLY as ordinary namespace types in one test: `design_examples.rs:359-394` (`design_example_signal_nexus_and_sema_are_schema_declared_planes`) declares `NexusAction`, `NexusResult`, `SemaCommand`, `SemaResponse` as namespace structs/enums inside a single generic schema and asserts they lower as ordinary data. The test's own docstring (`design_examples.rs:351-357`) is candid: *"Nexus and SEMA vocabularies are ordinary schema objects in the namespace until the plane-specific file split lands."* That is the implementation telling you the three types are not yet types. **How many of the three are implemented as schema types: ZERO.** One generic schema language stands in for all three.

### Item 2 — The 4-position shape (Imports/Input/Output/Namespace) — records 933, 940, 982

**MET.** The engine reads exactly four positional root objects and rejects any other count with a typed error. `engine.rs:155-160` gates on `document.holds_root_objects() != 4`; `engine.rs:162-179` lowers position 0 as imports, 1 as input enum, 2 as output enum, 3 as namespace, in declared order. `Asschema` stores precisely these four fields (`asschema.rs:51-57`). `schemas/spirit-min.schema` is the canonical witness read positionally: line 1 `{}` imports, line 2 `(Input ...)`, line 3 `(Output ...)`, lines 4-14 the namespace brace. The wrong-count path is proven both ways in `design_examples.rs:23-48` (3 objects and 5 objects both error with `ExpectedRootObjectCount { expected: 4, .. }`). `root.schema:5` even self-describes the root as `Schema [Imports Input Output Namespace]`, and `lowering.rs:149-156` asserts the lowered root carries exactly those four field references. This is the strongest-met item in the repo.

### Item 3 — Import/export via single-colon namespace — records 902, 952, 982

**PARTIAL, leaning MISSING on the export + resolution halves.** Three distinct sub-claims, with diverging status:

- **Colon-path PARSING and decomposition: MET.** `Name::namespace_segments` splits on `:` (`asschema.rs:15-17`); `local_part` returns the final segment (`asschema.rs:19-24`); `field_name` derives from the local part (`asschema.rs:26-41`). Proven in `design_examples.rs:148-163` (`schema:spirit:Entry` → `["schema","spirit","Entry"]`, local `Entry`) and `lowering.rs:86-113` (colon-qualified names lower end-to-end through a real schema). The single-colon convention mirroring Rust's `::` is honoured — never a double colon.
- **Import DECLARATION capture: MET (shallow).** `RootImportsMacro` (`engine.rs:304-353`) reads the position-0 brace as even key/value pairs into `ImportDeclaration { local_name, source }` (`asschema.rs:107-111`). The spirit-min fixtures use empty `{}` imports, so the populated path is exercised only indirectly.
- **EXPORT surface: MISSING.** `grep -rniE "export" src/ schemas/` returns NOTHING. Record 982 says position 4 (namespace) is BOTH the export surface AND the local-definitions surface; the code treats position 4 purely as local type declarations (`lower_namespace`, `engine.rs:220-236`) and position 0 purely as imports. There is no notion of a type being *exported*; the export half of "imports/exports" is unimplemented. ARCHITECTURE.md line 75-77 (added by the pinned commit) claims "Imports and exports are schema objects too" — this is prose ahead of code.
- **Cross-schema RESOLUTION: MISSING (and self-admitted).** `ImportDeclaration` is only ever *constructed* (`engine.rs:346`), never *resolved* against another schema's namespace — `grep -rn "ImportDeclaration"` shows construct + store + accessor only. `ARCHITECTURE.md:23` states plainly: *"This is intentionally a small loader, not full cross-crate import resolution."* The `SchemaPackage` loader (`module.rs`) maps a colon-qualified module name to a nested file path (`module.rs:44-47`, `module.as_str().replace(':', "/")`) but loads one module's source in isolation; it does not link `signal-frame:Frame` to a foreign schema's definition.

### Item 4 — One recursive struct down to scalars; macros are sugar — records 894, 932, 933, 940

**MET.** Nested struct/enum/newtype definitions are macros applied at known positions, recursing until scalar leaves. The namespace lowers each entry through `MacroPosition::NamespaceDeclaration` (`engine.rs:453`); struct bodies recurse through `StructFields` (`declarative.rs:713-717`); enum bodies through `EnumVariants` (`declarative.rs:742-746`). The recursion bottoms out at `schema_name()` atom leaves (`macros.rs:220-228`). Scalar leaf types (Text, Integer) appear as bare type references in `spirit-min.schema:6-13` and lower as newtypes/struct fields. Single-field structs collapse to `TypeDeclaration::Newtype` (`declarative.rs:727-731`), proven in `lowering.rs:104-112` and `design_examples.rs:74-77`. The macro engine is genuinely a shared substrate — but note it is shared across the seven POSITIONS of ONE schema, not across three schema TYPES (see Item 1). The intent's "shared substrate for all three schema types" is satisfied only in the degenerate sense that there is one type.

### Item 5 — Macros are plain NOTA records dispatched by position+shape via MacroRegistry, NOT sigils — record 880

**MET, and exemplary.** `MacroRegistry::lower` (`macros.rs:180-194`) iterates registered `SchemaMacro` trait objects and dispatches on `schema_macro.matches(object, position)` — a position+shape predicate, with zero sigil scanning. Each macro's `matches` combines position acceptance with delimiter/shape checks: e.g. `RootImportsMacro` requires `position == RootImports && block.is_brace()` (`engine.rs:309-312`); `RootEnumMacro` requires paren + first-child PascalCase symbol (`engine.rs:484-493`). The reserved sigils `~ @ ! ? * =` are NOT consumed (no match for them in `src/`). `tests/macro_exploration.rs` is a dedicated 8-scenario proof of the match-criteria taxonomy from record 932: delimiter-only (`:89`), inner-shape (`:172`), object-count (`:246`), symbol-case pascal/kebab/camel (`:327`), combined criteria (`:426`), first-match-wins by registration order (`:500`), position-aware dispatch (`:592`), and Pair-vs-Block dispatch (`:691`). One honest gap is flagged in code, not hidden: `macro_exploration.rs:9-11` notes dispatch is FIRST-MATCH-WINS by registration order, whereas record 932 names "most specific match" as the intended rule — a documented TODO (cites report 388), classifiable as a minor **PARTIAL** sub-point inside an otherwise MET item.

Note on `$`/`$*` capture sigils: the declarative macro DSL (`builtin-macros.schema`) uses `$Name` for single and `$*Fields` for rest captures (`declarative.rs:431-449`, proven `design_examples.rs:98-134`). These dollar sigils live INSIDE macro-definition templates (the macro *authoring* DSL), not as dispatch sigils scanned in user schemas — so they do not violate record 880, and record 890 (Medium) explicitly proposes the dollar shape. Consistent with intent.

### Item 6 — Authoring shape — records 874, 876, 877, 886, 887 (all Maximum)

**MET, with `core.schema` carrying its exploratory caveat.** Each sub-claim:

- Root NAME omitted, nested objects self-name: **MET.** The reader treats the four root objects as the implicit `Schema` (no `(Schema ...)` wrapper in any `.schema` file); nested defs carry names (`Entry [...]`, `Kind (...)` in `spirit-min.schema:9,12`). `ARCHITECTURE.md:60-61` states the rule.
- Enum = `(QualifiedName (variants...))`: **MET.** `RootEnumMacro` + the declarative `SchemaEnumDefinition` (`builtin-macros.schema:5-7`) handle the paren-name-paren-list form; proven `lowering.rs:46-57`, `design_examples.rs:78-86`.
- Struct = `[fields]`, single-field `[field]` = newtype: **MET.** `SchemaStructDefinition` (`builtin-macros.schema:1-3`); newtype collapse `declarative.rs:727-731`; proven `lowering.rs:105-112`.
- Root SELF-DESCRIPTION (`root.schema`) lowers through the engine: **MET — and this is a real proof, not a dead file.** `schemas/root.schema` exists and `lowering.rs:132-182` (`root_schema_describes_the_schema_root_type`) lowers it through `SchemaEngine::default()` and asserts the `Schema` struct's four fields and the `TypeDeclaration` enum's three variants. Live, passing.
- Minimal built-in `core.schema` FIRST (record 887, Medium, exploratory): **PARTIAL / exploratory-as-intended.** `schemas/core.schema` exists and DOES lower through the engine — `lowering.rs:184-233` (`core_schema_describes_default_builtin_macro_positions`) lowers it and asserts `CoreSchema`'s fields and the seven `MacroPosition` variants. So it is NOT a dead file. BUT it is descriptive, not load-bearing: `core.schema` is a hand-authored schema that DESCRIBES the macro positions; the engine does not BOOTSTRAP from it (the actual macros are registered in Rust at `engine.rs:240-254` plus loaded from `builtin-macros.schema` via `include_str!` at `declarative.rs:15-17`). Record 887 is Medium/exploratory and asks "do we need a built-in core.schema first to read richer schemas" — the answer in code is "core.schema is documentation of the positions, not the reader's bootstrap." Faithful to the exploratory framing; not a divergence.

### Item 7 — Brace `{}` is ALWAYS a key/value map; namespace is pair-style, append-only — record 894

**MET (key/value pair-style); append-only is conceptual-only.** Brace = key/value map is enforced everywhere: imports (`engine.rs:330-335`, even-count gate) and namespace (`engine.rs:416-420`, even-count gate). The namespace is authored pair-style `Name Body`, and the paren form `(Name Body)` is REJECTED — proven twice: `lowering.rs:60-70` (`{ (Entry [...]) }` → `ExpectedEvenMapEntries { found: 1 }`) and `lowering.rs:72-83` (even count of paren objects → `ExpectedSymbol`), plus the positive pair-style path `design_examples.rs:60-87`. Brace-as-enum-sugar at variant positions is also a macro (`BraceEnumVariantsMacro`, `engine.rs:589-630`), odd counts rejected loud (`ExpectedEvenBraceEnumPairs`, proven `design_examples.rs:325-329`). The "DYNAMIC ENUM stored as key/value map, APPEND-ONLY (Cap'n Proto style)" framing from record 894 is honoured in the surface SHAPE (it is a key/value map) but there is no append-only/ordinal-assignment machinery — the namespace is a plain `Vec<TypeDeclaration>` with order preserved (`asschema.rs:56`, `engine.rs:40` "stores declarations in Vec order"). Ordinal-stability / field-number assignment (the load-bearing half of "Cap'n Proto append-only") is not implemented. Minor sub-gap inside a met item.

## Top gaps (ranked)

1. **The three schema types do not exist as types — there is one generic plane.** (Item 1, MISSING; Item 8, DOC-ONLY.) This is THE gap. Record 982/964's whole premise — three schema-type LANGUAGES each with its own engine, routed by a schema-type discriminator — is unimplemented: no Signal/Nexus/Sema token anywhere in `src/`, no first-record variant routing, no per-type engine. The planes live as prose (added by the pinned commit) and as namespace-type naming conventions in one test that admits "until the plane-specific file split lands." Zero of three implemented.

2. **Export surface and cross-schema import resolution are unimplemented.** (Item 3, MISSING halves.) `grep export` finds nothing; ARCHITECTURE.md:23 self-admits "not full cross-crate import resolution." Colon-paths parse and decompose, and import DECLARATIONS are captured, but no type is ever exported and no `crate:module:Type` reference is ever linked to a foreign schema's definition. Record 982 says position 4 is BOTH local defs AND the export surface; only the local-defs half exists.

3. **ARCHITECTURE.md overstates the engine.** (Item 8.) `ARCHITECTURE.md:50` claims the macro engine "lowers all three planes uniformly," and lines 75-77 claim imports/exports are schema objects — capabilities the code lacks. Manifesting intent into repo docs is correct discipline, but the prose reads as describing existing behaviour rather than target behaviour; an auditor reading only ARCHITECTURE.md would over-credit the implementation.

4. **"Most-specific-match" dispatch is FIRST-MATCH-WINS instead.** (Item 5, PARTIAL sub-point.) Record 932 names most-specific-match as the intended dispatch; the registry uses registration order (`macros.rs:186-190`). Documented as a TODO in `macro_exploration.rs:9-11` — honest, but unbuilt.

5. **Append-only / Cap'n-Proto ordinal stability for the namespace is not implemented.** (Item 7 sub-point.) The namespace is a plain order-preserving `Vec`; record 894's "dynamic enum, append-only" framing has no field-number-assignment or ordinal-stability machinery. Surface shape (key/value map) is met; the wire-stability semantics are not.

## What is genuinely strong (so synthesis weights it fairly)

The macro engine itself is faithful, tested, and disciplined: 4-position positional read (Item 2, MET), recursive-struct-to-scalars (Item 4, MET), MacroRegistry position+shape dispatch with a dedicated 8-scenario taxonomy proof (Item 5, MET), brace-as-key/value enforcement with loud rejection of the paren form (Item 7, MET), authoring shape including a LIVE-lowered `root.schema` self-description and `core.schema` (Item 6, MET), and spotless method-only Rust with zero free functions and zero ZST holders (Item 9, MET). 34/34 tests pass. The repo is honestly scoped — its own tests and ARCHITECTURE.md note where the plane split has not landed. The gap is not sloppiness; it is that the engine is one generic plane where the intent (records 964/982) calls for three typed ones, and the pinned commit advanced the DOCS toward three planes without advancing the CODE.

# 1 — current stack map (the as-is)

designer, 2026-06-07. Consolidated from the six-reader grounding workflow
(`w63hfqahs`): runtime, emitter, spirit pilot, kameo target templates,
hand-written holdouts, engine-traits + concurrency intent. This is the
synchronous substrate the actor rewrite replaces.

## The stack at a glance

Two families sit on one shared kernel:

- **Emitted path** — `message`, `spirit`/`spirit-next`, and `cloud`'s emitted
  path are generated by `schema-rust-next`'s `daemon_emit.rs` into each
  component's `src/schema/daemon.rs`, riding `triad-runtime`'s synchronous
  listener traits.
- **Hand-written holdouts** — `lojix` (triad-port), `cloud`-live,
  `repository-ledger` roll their own blocking accept loops.
- **Shared kernel** — `triad-runtime` provides the listener spine
  (`Single`/`MultiListenerDaemon`), the optional concurrency primitive
  (`BoundedWorkers`), and the continuation driver (`Runner::drive`).

## Zero async today — this is greenfield-async-from-zero

`triad-runtime`, `spirit`/`spirit-next`, `cloud`, `repository-ledger`, and
`lojix` have **no tokio and no kameo** in any `Cargo.toml` (confirmed via
`cargo tree`). `triad-runtime` even sets `unsafe_code = forbid` and depends only
on `rustix`/`signal-frame`/`thiserror`. The rewrite introduces an async runtime
where there is none — a Rust 1.85→1.88 toolchain bump (kameo 0.20 requires 1.88)
with workspace fenix lockstep, touching the ~15 crates that depend on
`triad-runtime` at once.

## The de-risking find: three kameo daemons already exist in-workspace

The emitter does not have to invent the shape — it copies established patterns:

- **`harness/src/daemon.rs` + `runtime.rs`** — the canonical blocking-listener →
  kameo bridge: `tokio::runtime::Runtime::new()`, `runtime.block_on(...)`,
  `ActorRef<Harness>.ask(...)`, `stop_gracefully()` / `wait_for_shutdown()`, and
  the kameo 0.20 idiom (`impl Actor { type Args; async fn on_start }`, `impl
  Message<T> { type Reply; async fn handle }`, `#[derive(kameo::Reply)]`).
- **`chroma/src/daemon.rs`** — a `ChromaRoot` root actor holding child
  `ActorRef<StateStore>` / applier refs, with `serve_connection(stream,
  &ActorRef<ChromaRoot>)`. This is the concrete target shape for a daemon actor
  tree.
- **`mind`/persona-mind** — `src/actors/subscription.rs` (subscription actor) and
  `src/actors/store/mod.rs` (the redb store actor, including the documented
  supervised-shutdown caveat at lines 295-307).

Caveat: `mind` uses a **forked** kameo (`persona-lifecycle-terminal-outcome`);
the emitter must target **upstream 0.20** (as `harness`/`introspect`/`terminal`
do), not fork-only APIs.

## The three engine traits (emitted, synchronous)

Emitted by `schema-rust-next/src/lib.rs` (~L2046-2375); golden output in
`tests/fixtures/runner_generated.rs`. The `&mut self`/`&self` split is the
single-writer / parallel-read discipline encoded in the type system:

- **`SignalEngine`** — `triage_inner(&self, Signal<Input>) -> Nexus<Work>` and
  `reply_inner(&self, Nexus<Action>) -> Signal<Output>` (both `&self`, pure
  transforms), plus `on_start`/`on_stop` (sync, returning
  `ActorStartFailure`/`ActorStopFailure`) and no-op trace hooks.
- **`NexusEngine`** — `decide(&mut self, Nexus<Work>) -> Nexus<Action>`
  (the per-step decision), `apply_sema_write(&mut self, …)`,
  `observe_sema_read(&self, …)`, `run_effect(&mut self, …)`,
  `budget_exhausted_reply(&self, …)`, and a default `execute(&mut self, …)` whose
  body builds a `triad_runtime::Runner` and calls `runner.drive(...)` — the
  synchronous continuation loop on the same call stack.
- **`SemaEngine`** — `apply_inner(&mut self, Sema<WriteInput>) ->
  Sema<WriteOutput>` (single-writer) and `observe_inner(&self, Sema<ReadInput>)
  -> Sema<ReadOutput>` (parallel reads).

Crucially, the `_inner` methods are **pure component logic** — they become the
sync surface the actors own and call. `on_start`/`on_stop` already carry
actor-shaped names but are plain method calls, not kameo `Actor::on_start`.

## The continuation driver (Runner::drive)

`triad-runtime/src/runner.rs:149` — `Runner::drive(&self, engines, first_work) ->
Reply` is a same-call-stack loop over `NextStep::{Reply, SemaWrite, SemaRead,
RunEffect, Continue}`, bounded by `ContinuationLimit` (default 32 steps,
`budget_exhausted_reply` on overrun). `RunEffect` calls `run_effect` **inline** —
this is exactly where lojix's multi-minute nix build stalls the worker thread
mid-pipeline.

## The mutex inventory (every shared lock the actor model deletes)

- **spirit** `Engine { signal_actor, nexus: Mutex<Nexus> }` — the central
  `Mutex<Nexus>` locked at 10 sites; plus `SignalActor`'s two `Mutex<Integer>`
  counters, `MailLedger { events: Mutex<Vec<…>> }`, and the emitted
  `EmittedSubscriptions { state: std::sync::Mutex<SubscriptionState> }`.
- **cloud** `Arc<Mutex<Store>>` plus per-field `Mutex<Vec<…>>` interior
  mutability (accounts/policy/plans/…).
- **repository-ledger** `Arc<Mutex<Store>>` shared by both listener threads AND
  the 2-second spool poller — spool ingest and request handling contend on one
  lock.
- **lojix** `Store { state: Mutex<StoreState> }` locked briefly at 10+ sema sites.
- **triad-runtime** `BoundedWorkers` is `Arc<WorkerPermits>` =
  `Mutex<usize>` + `Condvar` counting semaphore + `thread::spawn`-per-task; trace
  recording is `Arc<Mutex<Vec<Event>>>`.

Every one of these is shared-mutable-state-behind-a-lock — the pattern an owning
actor replaces (the mailbox is the lock).

## The emitter is BEHIND the lojix reference

`daemon_emit.rs`'s `GeneratedDaemonRuntime` holds **one shared engine** and runs
`handle_working_stream` **synchronously on the accept thread** — no
`BoundedWorkers`, no per-request engine, no concurrency at all (grep returns zero
`dispatch`/`BoundedWorkers` hits). The `MultiListenerDaemon` accept loop is itself
single-threaded with a `thread::sleep(poll_interval)` busy-poll (10 ms latency
floor). Meanwhile the **lojix** hand-written daemon (which `tj99` says should feed
the emitter) does it right: `handle_stream` offloads each request onto a
`BoundedWorkers` pool (cap 64) and builds a **fresh per-request**
`SchemaRuntime::with_store(self.store.clone())` so the in-flight cursor is never
shared. So the emitted path is *more* serial than the reference — the actor
rewrite must leapfrog the emitter past lojix, not merely match it.

## The hand-written holdouts

- **lojix (the hard case)** — `run_effect → run_nix_build → NixCommand::run` is
  `std::process::Command::new("nix").args(...).output()`: a blocking multi-minute
  `nix build`, plus `ssh`-wrapped remote builds/activations. It buffers the whole
  build log in memory via `.output()` and consumes one `BoundedWorkers` permit for
  the full build.
- **cloud** — two `thread::spawn` blocking-`incoming()` listeners over
  `Arc<Mutex<Store>>`, then `loop { thread::sleep(60s) }`. Serial per socket.
- **repository-ledger** — same skeleton plus `loop { thread::sleep(2s);
  ingest_spool }` sharing the store mutex with the listeners.

## The concurrency intent the topology MUST preserve

Verbatim in `reports/cloud-designer/35-actor-divergence-forensics/3-…`:

- **`2alg`** (Decision High) — concurrent worker model, resolved *against* the
  single-writer serial model; per-request in-flight state; connection-permit cap.
- **`k6w1`** (Principle High) — the bounded thread-per-connection worker model,
  hosted in `triad-runtime` for every triad daemon.
- **`tj99`** (Decision High) — `BoundedWorkers` concurrent serving with a fresh
  per-request engine over a shared store, for the generated emitter.

The single sharpest hazard, named by two readers independently: a naive
**one-actor-per-daemon** mapping (everything through one mailbox) re-serializes
exactly what `2alg`/`k6w1`/`tj99` made concurrent. The topology must keep N
concurrent in-flight requests, and must not serialize reads behind writes.

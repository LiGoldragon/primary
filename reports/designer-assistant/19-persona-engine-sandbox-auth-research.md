# 19 - Persona engine sandbox authentication research

Date: 2026-05-11
Role: designer-assistant

## Verdict

The first full Persona engine sandbox should be a Nix-owned, stateful
test runner launched as a user `systemd-run` service, with an optional
inner `bwrap` mount sandbox for the strict filesystem view.

Use systemd for supervision, cgroup cleanup, credentials, and long-lived
test sessions. Use `bwrap` when the test needs a deliberately tiny
filesystem containing only `/nix`, `/run/current-system`, the sandbox
state directory, and a Wayland socket. Do not start with
`systemd-nspawn`: it is the right later production/container path, but
it is heavier than this test needs and depends on more CriomOS routing
and private-user setup.

The authentication rule should be:

1. Never mount the real harness home/config directory into the sandbox.
2. Pass only narrowly named credential material into a throwaway home.
3. Keep all session history, logs, shell snapshots, caches, and harness
   project state inside the sandbox state directory.
4. Add witnesses that prove the real host session-history files are not
   created, touched, or appended by the test.

## Architecture fit

This test belongs at the Persona apex, not in an individual component
repo. It exercises a full engine: persona-daemon, persona-message,
persona-router, persona-mind, persona-terminal, terminal-cell, and one
or more harnesses. The runner should live in the `persona` repository as
a Nix app/check shape because `persona/ARCHITECTURE.md` already names
the apex repo as the place where development/integration runners compose
multiple components.

The test should create a fresh engine state tree:

- `state/`: Persona component redbs and durable state.
- `run/`: Unix sockets, terminal-cell sockets, engine runtime paths.
- `home/`: fake home for Codex, Claude, Pi, and any other harness.
- `work/`: fixture workspace that agents may read/write.
- `artifacts/`: transcript captures, test logs, and witness outputs.

The engine under test should never receive `/home/li/primary` as its
writeable workspace. It should receive a small fixture repository whose
contents are designed for the test. The goal is to prove orchestration,
spawn, terminal attach, prompt injection, response capture, and
transcript-pointer handling, not to do real work in the primary
workspace.

## Local facts found

Codex CLI:

- `codex login status` reports ChatGPT authentication.
- `codex exec --ephemeral --ignore-user-config` exists.
- `--ignore-user-config` explicitly says auth still uses `CODEX_HOME`.
- The local auth file is `~/.codex/auth.json`.
- Scalar auth fields are `auth_mode`, `tokens.id_token`,
  `tokens.access_token`, `tokens.refresh_token`,
  `tokens.account_id`, and `last_refresh`.
- Session/history files live separately, including
  `history.jsonl`, `session_index.jsonl`, `state_5.sqlite`,
  `logs_2.sqlite`, `log/`, and `shell_snapshots/`.
- A temporary `CODEX_HOME` containing only `auth.json` is enough for
  `codex login status` on the host and inside `bwrap`.

Claude Code:

- `claude auth status` reports Claude.ai subscription authentication.
- On Linux, official docs say credentials are stored at
  `~/.claude/.credentials.json`, or under `$CLAUDE_CONFIG_DIR` when set.
- The credential scalar fields are `claudeAiOauth.accessToken`,
  `claudeAiOauth.refreshToken`, `claudeAiOauth.expiresAt`,
  scopes, subscription/rate-limit metadata, and `organizationUuid`.
- `~/.claude.json` and `~/.claude/file-history/` are state/history/config
  surfaces and should not be mounted into the sandbox.
- A temporary `CLAUDE_CONFIG_DIR` containing only `.credentials.json`
  is enough for `claude auth status` on the host and inside `bwrap`.
  Claude creates a local `.claude.json` in that temporary config
  directory; that is acceptable because it stays in sandbox state.
- `claude -p --no-session-persistence` disables session persistence for
  non-interactive runs.
- `CLAUDE_CODE_SKIP_PROMPT_HISTORY` is documented as the same
  persistence control in any mode.
- `--bare` is not the right default for subscription OAuth because the
  local help says bare mode does not read OAuth/keychain credentials.
  Bare mode should be reserved for API-key or `apiKeyHelper` tests.
- Official docs say `CLAUDE_CODE_OAUTH_TOKEN`, generated by
  `claude setup-token`, authenticates with a Claude subscription and is
  scoped to inference. That is likely the cleanest long-term CI-style
  credential for Claude sandboxes.

Pi:

- Pi supports `PI_CODING_AGENT_DIR`, `PI_CODING_AGENT_SESSION_DIR`,
  `--session-dir`, and `--no-session`.
- Local Pi defaults are `prometheus/qwen3.6-27b`, with enabled local
  or local-fronted models including `prometheus/glm-4.7-flash` and
  `prometheus/qwen3-8b`.
- Pi's auth scalar fields are `openai-codex.type`,
  `openai-codex.access`, `openai-codex.refresh`,
  `openai-codex.expires`, and `openai-codex.accountId`.
- Pi in the sandbox also needs `PI_PACKAGE_DIR` pointed at its Nix-store
  package directory; otherwise it looks for a package under the fake
  home and fails before doing useful work.

Host/runtime:

- `bwrap` is installed at version `0.11.1`.
- `systemd-nspawn` is installed from systemd `260.1`.
- Unprivileged user namespaces work on this host.
- `systemd-run --user` works.
- `systemd-run --user` with `PrivateUsers=yes` works.
- `systemd-run --user` with `PrivateUsers=yes`, `ProtectHome=tmpfs`,
  and `LoadCredential=codex-auth:$HOME/.codex/auth.json` worked as a
  proof that the user service can hide the real home while still seeing
  a loaded credential.
- The Wayland socket is visible at `$XDG_RUNTIME_DIR/$WAYLAND_DISPLAY`
  and can be bind-mounted into `bwrap`.

## Credential model

There are three credential-passing modes. They should be named because
they carry different risks.

`SnapshotCredential`

Copy the current credential file into the sandbox private home before
launch. This avoids writing to host session/config files. It is simple
and worked for `codex login status`, `claude auth status`, and Pi
startup. The risk is OAuth refresh-token rotation: if a harness refreshes
from a copied refresh token, the host credential may become stale. This
must not become the only architecture until a real prompt witness proves
host auth survives a sandbox refresh.

`ReadOnlyHostCredential`

Expose only the credential file read-only. This reduces host mutation
risk. It may fail when a harness wants to refresh or rewrite auth state.
For short smoke tests, it can be useful if the access token is fresh. It
is not durable enough for long full-engine tests.

`IssuedSandboxCredential`

Create a credential specifically for the sandbox and destroy it after
the test. For Claude, the official `claude setup-token` path is promising
because it produces a long-lived OAuth token scoped to inference and does
not save it by default. For Codex, I found `codex login --with-api-key`
and `codex login --with-access-token`, but not yet an official
subscription-equivalent long-lived token flow. This is the unresolved
Codex auth question.

Recommended first implementation:

- Use `SnapshotCredential` for Pi.
- Use `CLAUDE_CODE_OAUTH_TOKEN` if the user creates one; otherwise use
  `SnapshotCredential` for Claude as a local-only witness.
- Use `ReadOnlyHostCredential` or short-lived `SnapshotCredential` for
  Codex only in a witness test, not yet as a settled long-running
  architecture.
- Add a dedicated Codex auth witness that runs a one-prompt sandbox
  session and then proves the host `~/.codex/auth.json` and Codex
  history/session files were not mutated. If that witness shows host auth
  can be invalidated by refresh-token rotation, Codex subscription-auth
  sandboxing needs an issued-token or API-key path.

## Sandbox runner shape

Name the runner `persona-engine-sandbox`. It should be a Nix app, not a
hidden ad hoc script.

The runner should:

1. Create a unique sandbox directory under `$XDG_RUNTIME_DIR` or a
   caller-specified path.
2. Copy or materialize only selected credentials into `home/`.
3. Create an isolated fixture repository in `work/`.
4. Start the full Persona engine with paths rooted under `run/` and
   `state/`.
5. Start terminal-cell and persona-terminal inside the same engine.
6. Start a Pi harness first, because it can use local models.
7. Start Codex and Claude harnesses only when explicitly requested by the
   test profile.
8. Prompt each harness with a deterministic tiny request, such as
   "reply exactly persona-sandbox-ok".
9. Capture transcript pointers and terminal-cell output artifacts.
10. Leave the sandbox alive when invoked in interactive mode so a human
    can attach a Ghostty window to the terminal-cell socket.
11. Tear down the systemd unit and runtime state unless invoked with a
    durable/debug flag.

The systemd wrapper should use properties like:

- `PrivateUsers=yes`
- `PrivateTmp=yes`
- `ProtectHome=tmpfs`
- `ProtectSystem=strict`
- `ReadWritePaths=<sandbox-dir>`
- `LoadCredential=codex-auth:<host codex auth file>` when using
  systemd credential passing
- `LoadCredential=claude-auth:<host claude credential file>` when using
  systemd credential passing

The inner `bwrap` strict profile, when used, should expose only:

- `/nix` read-only
- `/run/current-system` read-only
- the sandbox state directory read/write
- `/proc` and `/dev` as needed for PTY/process operation
- the Wayland socket when a GUI terminal needs to be launched inside the
  sandbox

For display, the cleanest first shape is host-side Ghostty attaching to
a terminal-cell socket in sandbox runtime state. Ghostty then does not
need LLM auth or sandbox home at all. If we need to launch Ghostty inside
the sandbox, bind only the Wayland socket and minimal graphics/font
inputs needed by that terminal.

## Witnesses required

These should be real integration witnesses, not self-contained tests
that test their own fixture logic.

`codex_auth_isolated_status`

Create a temporary `CODEX_HOME` containing only `auth.json`, run
`codex login status`, and assert no Codex session/history files appear in
the host or sandbox except the copied auth file.

`claude_auth_isolated_status`

Create a temporary `CLAUDE_CONFIG_DIR` containing only
`.credentials.json`, run `claude auth status`, and assert any
`.claude.json` write lands in the sandbox config directory, not the host.

`pi_auth_isolated_startup`

Create a temporary `PI_CODING_AGENT_DIR` containing only auth/settings
needed for model selection, set `PI_PACKAGE_DIR` to the Nix package, run
`pi --version` or model listing, and assert no session directory is
created unless explicitly requested.

`wayland_socket_visible`

Bind only `$XDG_RUNTIME_DIR/$WAYLAND_DISPLAY` into the sandbox runtime
dir and assert a GUI process can see the socket. The non-interactive
check should stop at socket visibility; a stateful app can launch
Ghostty for human inspection.

`full_engine_pi_smoke`

Run the real Persona engine stack in a sandbox, start Pi in a
terminal-cell, inject the deterministic prompt through the real
persona-terminal/terminal-cell path, and assert the response appears in
the captured transcript/artifact.

`full_engine_codex_smoke`

Same as the Pi smoke, but with Codex. This witness is the gate for
deciding whether Codex `SnapshotCredential` is acceptable. It must record
before/after metadata for host Codex auth and session-history paths.

`full_engine_claude_smoke`

Same as the Pi smoke, but with Claude. Prefer
`CLAUDE_CODE_OAUTH_TOKEN` if available. If using copied
`.credentials.json`, record before/after metadata for host Claude auth
and history paths.

## Model choices

Default order for this test:

1. Pi local models, especially a small/fast configured model such as
   `prometheus/glm-4.7-flash` or `prometheus/qwen3-8b`, because these
   avoid external quota and are enough for spawn/injection/capture
   testing.
2. Codex `gpt-5.4-mini` with low reasoning for external Codex smoke
   tests.
3. Codex `gpt-5.3-codex-spark` only as an experimental comparison. The
   Codex rate card marks Spark as research preview with non-final credit
   rates, while `gpt-5.4-mini` has explicit lower credit rates than
   `gpt-5.3-codex`.
4. Claude Haiku 4.5 with low effort/minimal turns for Claude smoke tests.

OpenAI's model docs say `gpt-5.4-mini` is lower-latency/lower-cost than
larger GPT-5.4 models, and the Codex rate card lists `GPT-5.4-Mini` at
lower credit rates than `GPT-5.3-Codex`. Anthropic's Haiku 4.5 page says
Haiku 4.5 is available in Claude Code and priced at $1 per million input
tokens and $5 per million output tokens on the Claude Platform.

## Open questions

Codex subscription auth is the only major risk. A copied Codex OAuth
file works for `login status`, but that does not prove it is safe for a
real prompt because a prompt may refresh tokens. The architecture should
not assume copied Codex OAuth is safe until
`full_engine_codex_smoke` proves host auth survives.

Claude has a cleaner long-term answer because official docs expose
`CLAUDE_CONFIG_DIR`, `CLAUDE_CODE_OAUTH_TOKEN`, and
`--no-session-persistence`. The next question is whether we want the
user to generate a dedicated `CLAUDE_CODE_OAUTH_TOKEN` for sandbox tests,
or whether local copied `.credentials.json` is acceptable for
non-CI/manual runs.

Display should start with host-side Ghostty attaching to sandbox
terminal-cell sockets. Launching Ghostty inside the sandbox should be a
separate witness because graphics/font/device bindings can grow quickly.

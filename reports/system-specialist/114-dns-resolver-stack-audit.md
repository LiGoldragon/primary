# DNS Resolver Stack Audit

## Summary

The current desktop resolver path is over-centralized around unbound.
On `ouranos`, every normal DNS lookup goes through local unbound, while
NetworkManager's DHCP-provided DNS is not exposed in `/etc/resolv.conf`.
That makes unbound a single point of failure for the desktop.

The live symptom fits a process-alive resolver failure, not a daemon crash:
unbound is active, systemd will only restart it on process failure, and
the service is not tied to NetworkManager connectivity changes. If the
network drops and returns while unbound keeps running in a bad upstream
state, systemd has no reason to intervene.

The cleaner direction is to split resolver roles:

- desktop / edge nodes use `systemd-resolved` integrated with
  NetworkManager and keep cluster hostnames in `/etc/hosts`;
- router nodes keep a LAN DNS service because they actually serve other
  machines;
- tailnet DNS should not be hardwired through a local forward zone unless
  Tailscale/headscale is active and known healthy.

## Current Declared Shape

CriomOS imports `network/unbound.nix` unconditionally from
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/default.nix:35`.

`network/default.nix` also declares static nameservers:

- `::1`
- `127.0.0.1`
- Cloudflare IPv4
- Quad9 IPv4

That is at
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/default.nix:46`.
The same module already materializes cluster names through
`networking.hosts` from `horizon.exNodes` at
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/default.nix:55`.

`network/unbound.nix` always enables unbound at
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/unbound.nix:107`.
On non-router nodes it listens only on loopback, defined by `listenIPs` at
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/unbound.nix:31`.
Router nodes add the LAN gateway address to the listen set.

The upstream path is DNS-over-TLS to Cloudflare and Quad9:

- provider addresses are declared at
  `/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/unbound.nix:38`;
- `forward-tls-upstream = true` is set at
  `/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/unbound.nix:134`.

The module also forwards `tailnet.<cluster>.criome.` to Tailscale's local
DNS address whenever headscale is enabled, at
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/unbound.nix:122`.

There is a separate `systemd-resolved` shape in
`/git/github.com/LiGoldragon/CriomOS/modules/nixos/network/networkd.nix:52`,
but it is only enabled for `behavesAs.center && !behavesAs.router`.
`ouranos` is currently running NetworkManager, not systemd-networkd, so that
path is not active.

## Live State On Ouranos

Observed on May 11, 2026:

- `unbound.service` is active.
- `systemd-resolved.service` is inactive.
- `NetworkManager.service` is active.
- `systemd-networkd.service` is inactive.
- `resolvconf.service` is active.
- `resolvectl status` cannot connect because resolved is not running.
- `/etc/resolv.conf` is generated by resolvconf and contains only loopback
  nameservers: `127.0.0.1` and `::1`.
- NetworkManager's active Wi-Fi connection has DHCP DNS from the gateway,
  but that does not reach the machine-wide resolver file.
- unbound listens on port 53 only on loopback.
- unbound has `Restart=on-failure`, not a network-health or connectivity
  restart policy.

Recent boot logs show a Wi-Fi activation failure around 10:16, while
unbound did not fail or restart. That matters because the user's observed
manual fix is restarting unbound after connectivity returns. The service
manager is not seeing a failed process.

Tailscale/headscale is also not healthy on this boot:

- `headscale.service` is inactive because a dependency failed.
- `tailscaled.service` is active, but `tailscale status` reports it is
  logged out and cannot reach local headscale.
- `tailscale dns status` reports Tailscale DNS disabled.
- tailnet lookups through the current resolver path time out.

That tailnet state is probably not the whole reason ordinary internet DNS
breaks after link loss, but it proves the local resolver is carrying a
dead split-DNS integration.

## Failure Interpretation

The root design problem is not simply "unbound crashed." The problem is:

1. the desktop has only loopback DNS in `/etc/resolv.conf`;
2. loopback DNS is unbound;
3. unbound uses stateful upstream forwarding over TLS;
4. unbound is not supervised by connectivity state;
5. if unbound remains alive but serves failures after a network flap,
   nothing else can answer ordinary DNS.

This is too much machinery for a desktop client. The desktop already has
NetworkManager, DHCP DNS, and static cluster host entries. Running a local
DoT forwarding resolver in front of all of that is a privacy/security choice
with real recovery cost. Today that cost is paid as "restart unbound by
hand."

The declared fallback nameservers in `network/default.nix` do not currently
protect the live system, because the live resolver file exposes only
loopback.

## Recommended Shape

### Desktop And Edge Nodes

Use `systemd-resolved` as the local resolver and integrate it with
NetworkManager. Let NetworkManager push DHCP DNS into resolved. Keep
cluster static names in `networking.hosts`; that is already generated from
`horizon.exNodes`, and it does not require a DNS daemon.

This would remove unbound from the desktop critical path. If the Wi-Fi
link drops and returns, NetworkManager and resolved are the normal Linux
components designed to update per-link DNS state.

The likely Nix shape is:

```nix
services.resolved = {
  enable = true;
  fallbackDns = [ "1.1.1.1" "9.9.9.9" ];
};

networking.networkmanager = {
  dns = "systemd-resolved";
};
```

The exact committed shape should live in a noun module such as
`network/resolver.nix`, not in an unconditional `network/unbound.nix`.

### Router Nodes

Router nodes still need a LAN-facing DNS service because they hand out DNS
to clients. Keeping unbound there is defensible if we want DNSSEC, DoT, and
local cluster records served to the LAN. If we only need lightweight LAN DNS
plus static records, `dnsmasq` is a plausible simpler router-side service.

The important split is that router needs are not desktop needs. The router
path should be guarded by `behavesAs.router`.

### Tailnet DNS

Do not wire `tailnet.<cluster>.criome.` to Tailscale DNS just because
headscale is enabled. The current machine proves that headscale can be
enabled in config while the live headscale/tailscale path is down. A local
resolver should not install a dead split-DNS dependency into the normal
desktop path.

Tailnet DNS should either:

- be owned by Tailscale/resolved when Tailscale DNS is actually enabled; or
- be disabled until the headscale/tailscale service state is reliable.

## Temporary Mitigation

If we need a short-term improvement before the resolver split lands, add a
NetworkManager dispatcher hook that restarts unbound after connectivity
returns. That would match the manual recovery action.

I would not treat that as the final architecture. It preserves the single
point of failure and adds another script around it.

## Implementation Plan

1. Replace the unconditional unbound import with a resolver module that
   chooses by node role.
2. For non-router NetworkManager nodes, enable `systemd-resolved` and set
   NetworkManager DNS integration to resolved.
3. Keep `networking.hosts` cluster records in `network/default.nix`.
4. Move unbound behind `behavesAs.router`, or replace the router DNS role
   with a lighter DNS-only `dnsmasq` module after checking router needs.
5. Remove or gate the tailnet forward zone until Tailscale/headscale DNS is
   proven live.
6. Add Nix checks that evaluate at least one desktop node and one router
   node, proving:
   - desktop does not enable unbound;
   - desktop enables resolved;
   - router exposes a LAN DNS service;
   - tailnet DNS is absent unless the owning service is enabled.

## Decision

My recommendation is to stop using unbound for desktop nodes. Keep or
replace it only for router nodes. The current system is using a router-grade
resolver shape as the only desktop nameserver, and the failure behavior the
user reports follows directly from that shape.

## Ghost as Publication — inlined from plan 04 §P6 (verbatim shape)

Plan 04's Ghost-node shape (which is the cluster-side
companion to the now-landed `NodeSpecies::Publication` + `Contained`
schema):

```
node ghost
  species: Publication
  size: medium
  trust: medium
  placement: Contained {
    host: hyacinth
    substrate: NixosContainer
    resources: { cores: 2, ram_gb: 2 }
    network: { localAddress: 10.42.0.10, hostAddress: 10.42.0.1 }
    state: { persistent_paths: [ "/var/lib/ghost", "/var/lib/mysql" ] }
    trust: Medium
    user_namespace_policy: PrivateUsersPick
  }
  capabilities: {
    public_endpoint: Some(PublicEndpoint {
      domains: [
        PublicDomain::External(ExternalDomainName("blog.example.com")),
        PublicDomain::Criome(CriomeDomainName::for_subdomain("ghost", cluster)),
      ],
      tls_policy: AcmeLetsEncrypt,
      reverse_proxy_policy: HostTerminates,
    }),
  }
```

Today's landed `NodePlacement::Contained { host, user }` is a
simpler shape than Plan 04's `Contained { host, substrate,
resources, network, state, trust, user_namespace_policy }`.
Step 9 cleanup deferred the richer fields. When Ghost actually
lands, decide whether to extend `Contained` or carry the
substrate/resources/network/state metadata via separate
records.

The Ghost workload itself is **native NixOS service**, not OCI
(closed §18 below; reinforced by `~/primary/skills/nix-discipline.md`
§"Services are NixOS modules, not OCI workloads").

`Publication` species is narrow on purpose — covers
publication/membership web surfaces (Ghost, future
replacements). It does **not** imply `cache`, `builder`,
`container_host`, or `router` behavior. If a non-Ghost
candidate later has significantly different shape (a wiki, a
static-site generator), that's the moment to evaluate
splitting `Publication` into finer variants — not in the first
cut.

## Side notes

- **No-hashes preference.** The user wants commit hashes,
  store paths, and content-addressed identifiers kept out of
  chat output. Reference by branch + commit subject when
  needed. Inline commit hashes in this report serve as
  pickup pointers, not identity.
- **Wired connection on this workstation is load-bearing.**
  Don't bring it down without permission — other agents on
  it need it. The `nmcli +ipv4.routes "<dst>/32 <wifi-gw>"`
  + `nmcli device reapply <iface>` recipe is the
  polkit-allowed per-destination route override that lets
  you test wifi-only paths without disturbing wired.
- **systemd-run for prometheus activations.** When activating
  on prometheus over an SSH path that depends on Wi-Fi (e.g.
  hostapd will restart),
  `systemd-run --collect --unit=<name> --no-block

  the activation from the SSH session so it survives the
  disconnect. Then poll `systemctl status <name>` after
  reconnect.

### Operational note from former report 01 (whisrs widget)

When deploying CriomOS-home plugin changes to Quickshell
widgets (the dictation/Whisrs widget specifically), a
`home-manager switch` alone does not pick up the plugin —
Quickshell scans the plugins directory only at startup. The
restart procedure: kill the running quickshell PID and
relaunch `noctalia-shell` detached. Niri's
`spawn-at-startup` only fires at session start; there is no
reload IPC for plugin scan. Symptom of a stale quickshell
after a plugin deploy is *not* "widget missing" — it is
"widget renders but never updates," because the older
`BarWidgetRegistry` may have a stale entry while the new
plugin's socket plumbing is invisible. Treat a "permanently
neutral" widget on a CriomOS-home deploy day as a
stale-quickshell smell first, before suspecting the widget
source or the daemon.

This belongs as an inline note next to `startWhisrs` in
`CriomOS-home/modules/home/profiles/min/dictation.nix` —
filed as bd as a small follow-up.
